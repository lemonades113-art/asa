│         │                 │                 │                           │

**节点职责速查表**

| 节点 | 职责 | 关键判断 |
|-----|------|---------|
| Supervisor | 编排决策、验证数据、路由分发 | has_data_tag / is_data_empty / has_partial_result |
| Coder | 生成Python代码调用API | 生成带[DATA]:标签的代码 |
| Tools | 执行代码、返回结果 | 执行成功/失败/超时 |
| ErrorHandler | 错误分类、触发自愈 | ErrorClassifier + 4级自愈 |
| Reviewer | 校验结果、生成报告 | 结构化校验 + 融合 |
| ProfileUpdater | 更新用户偏好 | 基于对话内容 |

## 二、错误分类体系（ErrorClassifier）

**代码位置：** error_classifier.py 第23-67行

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         错误分类器（5类错误）                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ① auth_error (认证错误)                                                │
│     ├─ 模式：["401", "403", "auth", "token", "授权", "认证失败"]        │
│     ├─ 可重试：❌                                                       │
│     ├─ 策略：graceful_degradation (优雅降级)                            │
│     ├─ Token成本：低                                                    │
│     └─ 示例：Tushare Token过期                                         │
│                                                                         │
│  ② rate_limit (限流错误)                                               │
│     ├─ 模式：["429", "limit", "频率", "过于频繁"]                       │
│     ├─ 可重试：✅                                                       │
│     ├─ 策略：exponential_backoff (指数退避)                            │
│     ├─ Token成本：中                                                    │
│     └─ 示例：API调用超过频率限制                                        │
│                                                                         │
│  ③ network_error (网络错误)                                            │
│     ├─ 模式：["timeout", "connection", "超时", "无法连接"]             │
│     ├─ 可重试：✅                                                       │
│     ├─ 策略：exponential_backoff (指数退避)                            │
│     ├─ Token成本：中                                                    │
│     └─ 示例：网络超时、连接失败                                          │
│                                                                         │
│  ④ data_vacuum (数据真空)                                               │
│     ├─ 模式：["empty", "无数据", "null", "停牌", "退市"]               │
│     ├─ 可重试：❌                                                       │
│     ├─ 策略：graceful_degradation (优雅降级)                            │
│     ├─ Token成本：极低                                                  │
│     └─ 示例：停牌股票、财报未发布                                      │
│                                                                         │
│  ⑤ code_error (代码错误)                                               │
│     ├─ 模式：["KeyError", "TypeError", "ValueError", "IndexError"]    │
│     ├─ 可重试：✅                                                       │
│     ├─ 策略：immediate_retry (立即重试)                                │
│     ├─ Token成本：低                                                    │
│     └─ 示例：字段名写错、类型不匹配                                     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优先级顺序：** auth_error > rate_limit > network_error > data_vacuum > code_error

┌─────────────────────────────────────────────────────────────────────────┐
│                       Supervisor 状态判断逻辑                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. has_data_tag (格式验证)                                             │
│     判断："[DATA]:" in last_message_str                                  │
│     含义：Coder是否返回了结构化数据                                      │
│     失败→返回Coder，要求使用[DATA]:标签                                 │
│                                                                         │
│  2. is_data_empty (数据有效性)                                          │
│     判断：any([                                                       │
│         "[DATA]: {}" in last_message_str,                               │
│         "[DATA]: []" in last_message_str,                               │
│         "[DATA]: null" in last_message_str,                              │
│         "[DATA]: {'error'" in last_message_str,                          │
│     ])                                                                  │
│     含义：数据是否为空或包含错误                                          │
│     失败→触发数据真空处理                                                │
│                                                                         │
│  3. has_partial_result (熔断降级)                                       │
│     判断："[PARTIAL RESULT]" in last_message_str                        │
│     含义：是否是熔断器触发后的降级结果                                    │
│     失败→跳转到Reviewer生成降级报告                                     │
│                                                                         │
│  4. has_diagnostics (诊断信息)                                          │
│     判断："[DIAGNOSTICS]:" in last_message_str                          │
│     含义：是否包含性能诊断信息                                            │
│     用于Reviewer的报告优化                                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 六、完整的状态流转表

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         完整状态流转表                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌────────────────┬────────────────────────┬─────────────────────────┐ │
│  │     状态       │       触发条件         │       目标动作          │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ start          │ 用户发起查询           │ Supervisor编排          │ │
│  │                │                        │ → decompose_task()      │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ executing       │ 进入Coder              │ 生成代码 → Tools执行    │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ success        │ Tools返回[DATA]:       │ → Reviewer生成报告     │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ execution_error│ Tools返回[ERROR]:     │ → ErrorHandler分类     │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ level_1_retry  │ code_error + retry<3  │ Coder立即重试           │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ level_2_switch │ retry>=3 + 策略未耗尽  │ BacktrackingRouter切换 │ │
│  ├────────────────┼────────────────────────┼─────────────────────────┤ │
│  │ level_3_fallback│ 策略耗尽/不可重试   │ [PARTIAL RESULT]        │ │

## 八、满分口述总结

### 完整流程一句话总结

> "用户发起查询后，Supervisor进行编排，Coder生成代码并由Tools执行。如果执行成功，数据流向Reviewer生成报告；如果执行失败，ErrorHandler首先用ErrorClassifier将错误分为5类（auth/rate_limit/network/data_vacuum/code_error），然后根据错误类型触发4级自愈：Level 1立即重试（针对code_error）、Level 2策略切换（BacktrackingRouter尝试备选API）、Level 3优雅降级（返回[PARTIAL RESULT]）、Level 4硬性拒绝（不再浪费资源）。整个过程通过failure_history和策略上限防止循环自愈，recovery_level硬性停止确保系统不会无限运行。"

### 面试官可能的追问及应对

| 追问 | 应对 |
|-----|------|
| "为什么5类错误不是4类或6类？" | 基于Tushare API实际错误归纳，代码注释有说明 |
| "Level 1和Level 2有什么区别？" | Level 1是同策略重试，Level 2是换策略重试 |
| "怎么防止RAG纠错让模型越跑越偏？" | Level 1禁用原始RAG结果，只给纠错提示 |
| "auth_error为什么不自动更换Token？" | 系统边界控制，不操作凭证 |
| "100个用户同时报错会OOM吗？" | 线程隔离+超时熔断，守护线程自动清理 |
