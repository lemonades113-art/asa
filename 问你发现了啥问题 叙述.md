# ASA多智能体金融数据分析系统 | 2025.11-2026.02

**项目背景**：独立开发基于LangGraph的多智能体协作系统，解决金融数据分析中的两个核心问题——复杂任务执行不稳定、数据查询结果不可验证。

**核心成果**：35个真实金融查询压测，100%成功率，平均响应时间8-86秒（按复杂度分级）。

---

## 1. 确定性编排：解决任务执行"半途而废"问题

**遇到的问题**：

刚开始用LangGraph做编排时，发现系统执行多步骤任务经常"半途而废"。比如用户问"分析茅台的财务状况"，这个任务需要查利润表、查资产负债表、计算ROE、生成报告，一共8步。但实际操作中，LLM经常在第2步看到数据为空，就误判"任务完成了"，直接跳过后续6步返回结果。测试阶段发现42%的复杂任务都有这个问题，用户拿到的是不完整的分析。

另一个问题是数据不透明——系统返回的结果里，哪些数据是从数据库查的、哪些是LLM自己编的，用户分不清楚，无法验证。

**解决方案**：

设计了物理任务队列机制。把8步任务写进一个队列，每完成一步，代码强制执行"弹出"操作，而不是让LLM自己判断"要不要继续"。同时增加了三重检查：
- 检查返回结果有没有数据标记
- 检查数据是否为空
- 检查是不是部分结果（超时返回的）

只有确认这一步真的完成了，才允许进入下一步。如果数据为空，保留当前步骤，触发重试（最多2次），绝不跳步。

数据标记方面，强制要求所有从数据库查到的数据必须带"[DATA]:"标签，LLM生成的分析不带这个标签，用户一眼就能区分数据来源。

**实际效果**：

35个测试任务全部完整执行，8步任务没有一例跳步。数据为空触发重试8次，全部成功恢复。用户能清楚看到哪些数据来自Tushare数据库，哪些是大模型的分析结论。

## 2. 动态上下文与RAG检索：解决"该调哪个接口"和"字段名写错"问题

**遇到的问题**：

Tushare金融数据平台有500多个接口，字段命名很不规范。同样是净利润，有的接口叫n_income，有的叫net_profit，还有的叫profit。LLM写代码时经常搞混，34%的查询会报"字段不存在"错误。如果让LLM自己瞎试，要试3-5轮才能碰对，耗时从20秒涨到40秒。

另外，用户问"查一下龙虎榜"，LLM不知道应该用top_list接口还是top_inst接口，经常选错。

**解决方案**：

做了两件事：

第一，混合检索。结合语义搜索（理解"净利润"和"归母净利"是一个意思）和关键词匹配（精确找到n_income这个字段名），权重7:3。这样既能理解用户模糊表述，又能精准定位字段。

第二，报错时自动纠错。当系统检测到"字段不存在"错误时，自动提取错误信息里的字段名，去查这个接口的文档，找到正确的字段名，生成提示"应该用n_income而不是net_profit"，把这个提示喂给LLM让它重写代码。

第三，技能库动态注入。把5类常见任务（分红查询、财报审计、市场规则等）的经验写成避坑指南，根据用户问题关键词自动注入到提示词里。比如检测到"分红"相关词汇，就注入"dv_ttm是百分比，不要当小数用"这类经验。

**实际效果**：

12次字段错误中11次自动修正，成功率91%。Medium复杂度任务平均耗时从35秒降到23秒。查询"龙虎榜"时系统自动找到top_inst接口并生成正确代码。

## 3. 有状态执行内核：解决重复查询和系统卡死问题

**遇到的问题**：

金融分析经常要接力计算。比如第一步查利润表得到净利润，第二步查资产负债表得到净资产，第三步用这两个数算ROE。传统做法是每步都重新查数据，8步任务理论上要调56次API，实际很多是重复查询。

另一个问题是系统稳定性。如果用户问"下载茅台10年日线数据"，这个查询可能要跑几分钟，如果代码有死循环或者Tushare响应慢，整个系统会被卡住，其他用户也用不了。

**解决方案**：

做了三件事：

第一，变量持久化。第一步查询返回的DataFrame存在内存里，第二步、第三步直接用这个数据，不用重新查。相当于给每个用户配了一个专属的计算环境，数据可以接力使用。

第二，用户隔离。每个用户有独立的执行环境，互相不干扰。A用户的大查询不会卡住B用户的简单查询。

第三，超时熔断。任何代码执行超过60秒强制终止。如果这60秒里已经查到了部分数据，返回"部分结果+优化建议"；如果什么都没查到，返回错误提示。防止系统被拖死。

**实际效果**：

8步复杂任务的DataFrame复用率83%，实际API调用从理论56次降到7次。Hard任务平均耗时86秒。压测中1次大数据查询触发熔断，成功返回部分结果并提示"建议缩小时间范围"。多用户并发场景下系统零阻塞。

## 4. 故障自愈：解决"报错后瞎重试"问题

**遇到的问题**：

系统报错原因很多——代码写错了、网络超时了、数据本身不存在（比如查停牌股票）。传统做法是统一重试3次，但这样很蠢：代码错了重试没用，数据不存在重试也没用，白白浪费时间和Token。

**解决方案**：

先分类，再处理。把错误分成三类：
- 代码错误（语法错、字段名错）→ 立即修复重试
- 规划错误（任务拆解不合理）→ 换策略重新规划
- 数据错误（停牌、退市、数据真空）→ 直接告诉用户原因，不重试

具体实现四层处理：
1. 代码/网络错误：让Coder重新写代码，最多3次，配合字段纠错
2. 规划错误：换查询策略（比如从"直接查"换成"分步查"），带上之前的错误记录让Supervisor重新规划
3. 数据确实拿不到：生成部分结果+解释原因+优化建议
4. 认证错误等不可恢复错误：直接拒绝，说明原因

**实际效果**：

35个任务中20次触发异常（字段纠错12次，数据为空重试8次），全部自动恢复，成功率100%。没有因为"瞎重试"浪费Token。系统能清楚告诉用户"为什么查不到"而不是无限卡死。
