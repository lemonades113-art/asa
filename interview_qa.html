难题一 字段幻觉 — LLM 把"赚钱能力"映射到不存在的字段名 Schema Linking & RAG

痛点

Tushare 有 340+ 个接口，4000+ 个字段，每个字段名都是 roe_ttm/n_income_attr_p 这样非直觉的英文缩写。LLM 用通用语义理解很容易生成根本不存在的字段名，导致 KeyError。标准接口级 RAG 把整篇 API 文档当一个 chunk，语义粒度太粗，精确字段命中率只有 ~62%。

机制

① 字段级分块：把 340 篇文档拆解为每个字段一个 chunk，格式："API名称 字段名 类型 描述"，共 4032 个语义单元（日志证据：[检索器] 接口级=340, 字段级=3692）

② 混合检索：FAISS 向量分（α=0.7）+ BM25 词频分（β=0.3），_expand_query() 用 FINANCIAL_SYNONYMS 扩展查询，把"股息率"扩展为 dv_ttm dv_ratio，提升 BM25 的 TF 命中

③ 三路召回兜底：正则规则路 + LLM 结构化意图路（SchemaIntent: stocks/metrics/time_range）+ RAG 路，三路并行，结果合并注入 Coder prompt

量化结果

RAG Hit@3: 93.1%（原接口级 ~62%，提升 +31pp）
MRR@5: 87.1%
Context Precision@3: 83.8%
解析失败率: ~1%（原 ~5%）

难题二 代码执行失败 — 出错了没人修，或反复用相同策略无效循环 Execution & Fault Attribution

痛点

LLM 生成的代码在真实 API 调用时频繁出错：字段名拼错（KeyError）、API 参数格式错误、数据为空（停牌/财报未发布）、网络超时。单纯重试不够：① 同样的错误会重试 N 次毫无意义；② 不同类型的错误需要完全不同的修复策略；③ 50 行 Python traceback 直接塞给 LLM 浪费大量 Token。

机制

① 错误分类：classify_error_simple() 按优先级分类：field_error > auth_error > network_error > code_error

② 四级策略链：L1 压缩重试 → L2 字段别名替换（_FIELD_ALIAS_TABLE O(1) 查表）→ L3 传播图定位根因并制定针对策略 → L4 放弃+写入学习日志

③ Traceback 压缩：compress_error_traceback(max_lines=8)，保留前2行（异常类型）+后6行（调用链），50行→8行节省84%的上下文

④ 特殊处理：data_vacuum 类型（停牌/财报未发布）直接标记不可重试，避免无效循环

量化结果

10个真实故障Case，9个零干预自愈（90%）

代表性案例：
· dv_ttm 字段查询 → 用户写 dv_ratio → L2 自动修正
· 茅台净利润 CAGR=-7.01%（3年真实计算）
· 工行 vs 建行总资产：52.8万亿 vs 45.4万亿

难题三 上下文爆炸 — 多轮金融对话消息膨胀，LLM 遗忘原始查询目标 Planning & Token Budget

痛点

金融分析是多步骤任务，每轮 Supervisor→Coder→Reviewer 会产生大量工具调用消息、Python 执行结果、错误回溯。20轮之后消息列表超过40条，① LLM 注意力被稀释；② Token 成本线性增长；③ 最严重的是：LLM 忘记了用户最初问什么，出现"目标漂移"。

机制

① LLM 语义压缩（主动）：消息 > 20条（_COMPACT_THRESHOLD=20）触发 compact_messages()，用 fast_model 把历史对话摘要为 ≤500字 SystemMessage，显式保留：原始意图 + [DATA]结果 + 错误修复记录 + 剩余步骤

② 确定性截断（兜底）：trim_messages_for_context(max_keep=15) 强制锁定 first HumanMessage（原始查询），永久不丢

③ 全局熔断：total_step_count > 25 强制 FINISH，防止 RecursionLimit

④ 错误压缩：compress_error_traceback(max_lines=8) 防单次错误消耗整个窗口

确定性保证

原始查询永不丢失：
first_human = next(m for m in messages if isinstance(m, HumanMessage))
if first_human not in recent_msgs: trimmed.append(first_human)

这一行代码确保即使对话第 50 轮，LLM 仍然知道用户最初问什么。

Token 上限：最多 15 条 messages 进入 LLM，≤25步全局熔断

难题四 数据流与指令流混合 — 中间财务数据随步骤膨胀导致 LLM 注意力衰减 Execution Kernel

痛点

金融分析需要多步骤：第1步查日线数据，第2步查财务指标，第3步做计算，第4步生成报告。如果把每步的 DataFrame 结果都放进 messages，LLM 上下文会被数百行数据填满，后续推理质量严重下降，而且每步都要重新传输整个数据集。

机制（StatefulPythonKernel）

StatefulPythonKernel 维护一个持久化的 globals Python 字典，所有执行步骤共享同一个命名空间：

步骤1：df = pro.income(ts_code='600519.SH') → 存入 globals['df']
步骤2：margin = df['n_income'] / df['revenue'] → 直接从 globals 读 df

LLM 上下文只流转代码指令和推理逻辑，数据存在 Python 内存里，跨步骤直接复用，不需要重新下载和传输。

工程价值

避免数据重复下载，效率提升10倍

API 缓存包装器注入 daily_basic, income 等高频接口：
[APICache] 初始化完成，最大缓存数: 500（实际运行日志）

LLM 上下文瘦身：不再塞入 DataFrame 原始数据，只传计算结果和推理步骤
