### Q1: RAG 如何解决幻觉问题？

**结论**：RAG 通过"检索锚定 + 上下文约束"把 LLM 的自由生成空间压缩到一个可核查的文档子集内。

**逻辑**：
幻觉的本质是：模型参数里的隐式知识和当前问题不对齐，导致它"编"出看上去合理但不可验证的内容。RAG 的反制逻辑有三层：
1. **信息源替换**：把"模型记忆中的知识"替换成"当前检索到的文档"，模型回答的是文档里写的内容，不是参数里压缩的联想。
2. **Prompt 约束**：在 system prompt 里强制："只根据以下上下文回答，若无依据请说不知道"——这是软约束。
3. **结果验证**：我们项目里有**三重验证机制**（`validate_coder_result`），验证输出里必须含 `[DATA]:` / `[RESULT]:` 标记、数据非空、无 `[ERROR]` 状态——这是硬约束，LLM 遵循 Prompt 有概率失效，验证函数是代码层兜底。

**反思**：RAG 不能完全消除幻觉，只能压缩概率。如果检索结果本身是错的（召回不准、文档噪音大），模型仍可能在错误基础上自信生成。所以召回质量是天花板，这就是为什么我们做混合检索而不只用单路向量召回。

### Q2: 稠密检索 vs 稀疏检索的区别？

**结论**：稠密捕语义泛化，稀疏捕精确字符匹配，二者互补。

| 维度 | 稀疏（BM25） | 稠密（FAISS+向量）|
|---|---|---|
| 信号来源 | TF-IDF 词频倒排 | Embedding 余弦相似度 |
| 优势 | 精确词命中、无需训练、可解释 | 语义泛化（"净利润"≈"归母净利润"）|
| 劣势 | 词汇表外词(OOV)全失效 | 需要 GPU/大模型，近邻近似有误差 |
| 项目实现 | `BM25Okapi` + `jieba.cut_for_search` | `faiss.IndexFlatIP` + `BAAI/bge-small-zh-v1.5` |

我们项目里 `vector_weight=0.7`，即 70% 向量 + 30% BM25。金融场景有大量精确 API 字段名（如 `n_income_attr_p`），纯语义可能漂移，BM25 做精确命中补充。

### Q5: RRF（倒数排名融合）展开讲一下？

**结论**：RRF 是一种无需调权重的列表融合算法，核心思想是"排名越靠前，贡献越大，且用倒数惩罚尾部"。

**公式**：`RRF_score(d) = Σ 1/(k + rank_i(d))`，k 通常取 60。

**直觉**：假设 BM25 给文档 A 排第 1，向量检索给 A 排第 5。RRF 的分数 = 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318。 另一篇文档 B，BM25 第 3，向量第 3：1/63 + 1/63 = 0.0317。A 略高——它在 BM25 里有压倒性优势。

**我们项目的实际做法**：没有用标准 RRF，而是用**加权线性融合**：`0.7*vector_normalized + 0.3*bm25_normalized`。区别在于：
- RRF：只依赖排名，不用分数，对分数量纲不敏感，更鲁棒
- 我们的加权融合：先对两路分数各自 min-max 归一化到 [0,1]，然后加权求和——依赖分数分布，但在金融 API 检索场景里更直接可调

如果面试官深追 RRF，可以说："标准 RRF 在多路召回排名融合时更健壮，因为它不依赖绝对分数；我们当前用的是归一化加权融合，金融场景里 70/30 的比例是通过多组实验调出来的，如果将来接入 cross-encoder rerank，可以切换到 RRF 做 pre-ranking 融合层。"

### Q2：Agent 的"规划-执行-反思"闭环如何实现？

### Q（RAG 的回答方式区别）：RAG 怎么做的，只是 query-doc 结果划分？

**结论**：RAG 不只是 query-doc 相似度划分，而是一个完整的"检索-注入-生成-验证"闭环。

从信息利用角度讲，传统检索是把文档作为最终答案返回；RAG 是把检索结果作为生成器的**上下文约束**，模型在这个约束空间内重新"理解+整合+生成"。

### Q1：RAG 如何解决幻觉问题？

面试官问这个，真正想听的是你对"为什么模型会幻觉"的理解，以及你的系统做了什么来对抗它。

**场景触发**：我们系统要处理 340 多个 Tushare 金融接口的调用，每个接口有十几个参数，参数名全是英文缩写比如 `n_income_attr_p`。如果直接让大模型凭记忆生成代码，它会以极高的自信心写出根本不存在的参数名——这就是幻觉。

**核心矛盾**：大模型的幻觉根源是"生成空间无约束"。模型在海量文本上预训练，它知道"要查净利润大概用什么格式"，但不知道"这个系统里净利润的精确字段名是什么"。这两件事是有本质区别的——前者是泛化知识，后者是领域事实。

**逻辑闭环**：

第一步，用户的查询进来——比如"查茅台的净利润"——系统先走查询扩展，把"净利润"这个自然语言词汇映射到一张同义词字典里，扩展出 `n_income`、`n_income_attr_p`、`归母净利润` 等精确字段名。这一步的输入是自然语言，输出是包含领域术语的扩展查询串。

第二步，用这个扩展后的查询串，同时发给两个检索引擎——一个是 FAISS 向量检索，捕捉语义相似性；一个是 BM25 精确匹配，捕捉字符串命中。两路各自返回候选文档，然后按 70% 向量、30% BM25 的比例加权融合，排序后取前 5 个最相关的接口文档片段。这一步的输入是查询串，输出是最相关的 Tushare 接口说明——告诉 Coder "你要查的数据，应该用这个接口、这几个参数"。

第三步，这 5 条接口说明被注入到 Coder 节点的提示词里。现在 Coder 不是凭空生成代码，而是在"检索到的文档"这个约束空间里生成——就像开卷考试，答案不能超出参考材料的范围。这一步的输入是接口文档 + 用户需求，输出是有文档依据的可执行代码。

第四步，代码执行完之后，还有三重验证做最后把关：确认输出里有 `[DATA]:` 这个数据标记、数据不为空、没有报错标记。三道都过才算成功，少一道就打回 Coder 重新生成。这是硬约束，不依赖模型是否"遵循"提示词。

**价值判定**：如果不这么做，会发生什么？没有 RAG，模型生成的代码会用错字段名，执行直接报 `KeyError`；没有三重验证，模型可能返回一个空 DataFrame 却说"数据获取成功"——这在金融场景里是致命的，用户看到的是一个看起来正常、实际上是空数据的回答。这两件事合起来，就是 Tool Output Grounding 没有闭合的代价。

### Q2：稠密检索 vs 稀疏检索的区别？

**场景触发**：我们在做第一版检索时只用了向量检索，发现一个问题——用户输入"查 `dv_ttm` 字段"，向量模型不认识这个缩写，把它理解成跟"股息"相关的语义，但召回的文档却不是 `daily_basic` 这个正确接口。

**核心矛盾**：向量检索是语义层面的近邻，它擅长"净利润"≈"归母净利润"这种语义等价，但对于精确的 API 字段名、股票代码这类**字符串不可替代**的词汇，向量模型往往失效——因为这些词不在它的语义空间里。BM25 则相反，它纯靠词频和逆文档频率做评分，精确字符匹配无敌，但"净利润"和"net profit"之间的语义跨越它完全看不见。

**逻辑闭环**：

第一步，同一个查询串同时送给两个引擎。BM25 的输入是 jieba 分词后的词片段列表，它拿这些词去倒排索引里查每个文档的命中频率，再做归一化打分；向量引擎的输入是整句话，BGE 模型把它压缩成一个 512 维的浮点向量，然后在 FAISS 索引里搜最近邻，用内积（等价于余弦相似度）打分。

第二步，两路各自返回打分，都做 min-max 归一化到 0-1 区间，然后按 7:3 加权合并——向量分乘 0.7，BM25 分乘 0.3，加在一起就是最终的混合得分。这个权重不是拍脑袋定的，是因为金融场景里语义泛化的需求更高（用户说话不会用精确字段名），所以向量权重更重。

第三步，按混合得分降序排列，取前 5 条，同一个 API 接口只保留分数最高的一条，防止结果被同一个文档占满。输出是最终送给 Coder 的 5 段接口说明。

**价值判定**：如果只用向量，精确字段名查询会漂移；如果只用 BM25，同义词查询会漏检。两者互补才能在 340 个接口里做到高召回率。这是 Retrieval 中"语义覆盖"和"精确命中"两个目标之间天然的张力，混合检索是当前最实用的折中解法。

### Q3：整个 RAG 流程，用户输入进去就直接召回吗？

**核心矛盾**：用户的自然语言表达和索引里的文档表达之间有一个"语言鸿沟"。用户说"茅台赚了多少钱"，文档里写的是 `n_income_attr_p`——如果直接拿用户原话去检索，什么都找不到。

**逻辑闭环**：

第一步，用户原始输入进来，先经过查询理解层。如果是复杂查询比如"分析茅台近3年营收增速和利润率"，查询重写器会把它拆成两个独立的子问题——一个是营收增速查询，一个是利润率查询。这一步的输出是 1 到 N 个聚焦的子查询，每个子查询只问一件事。

第二步，每个子查询进入同义词扩展——把用户的中文词汇扩展成包含 API 字段名的复合查询串。这一步解决的是"用户不知道精确字段名"的问题，输出是富含领域术语的扩展查询。

第三步，扩展后的查询进入混合检索，向量 + BM25 并行检索，加权融合，去重取 top-5。

第四步，召回结果注入 Coder 提示词，Coder 生成代码，执行，三重验证。

**价值判定**：如果省掉第一步查询理解，复杂问题的子任务会相互干扰，Coder 不知道先做什么；如果省掉第二步扩展，精确 API 字段名永远召回不到。查询预处理是让检索"看懂用户意图"的前提，缺少它 RAG 系统的召回上限就被锁死了。

### Q（RRF 展开）：RRF 倒数排名融合的本质是什么？

**场景触发**：当你有两路检索结果，向量打 0.93 分，BM25 打 12.7 分，这两个数字没有任何可比性——量纲完全不同。直接相加等于垃圾。

**核心矛盾**：不同检索系统的分数分布不同，无法直接比较绝对分数。加权融合需要归一化，但 min-max 归一化在分布极端不均匀时会把尾部的差异压平，导致真正相关的文档分数和无关文档分数非常接近。

**RRF 的解法逻辑**：RRF 的核心思想是——与其比较分数，不如只比较排名。公式是：每个文档的最终得分 = 所有检索路的排名倒数之和，每个排名倒数是 `1 / (k + 排名)`，k 通常取 60，作用是"惩罚尾部、平滑头部"。

举个具体例子：文档 A 在向量检索里排第 1，在 BM25 里排第 5，RRF 得分 = `1/(60+1) + 1/(60+5)` ≈ 0.0318。文档 B 在两路都排第 3，得分 = `1/63 + 1/63` ≈ 0.0317。文档 A 略高——它有一路里的压倒性优势，这在直觉上是合理的。

**我们项目的实际选择**：我们用的是归一化加权融合而非标准 RRF。区别在于 RRF 对分数分布不敏感、更鲁棒；我们的方案依赖分数分布，但 70/30 这个权重是针对金融 API 场景调出来的，在这个特定分布下效果稳定。如果接入第三路检索或者更换 embedding 模型，切换到 RRF 会更安全。

**价值判定**：融合策略的选择本质上是在"鲁棒性"和"可调性"之间做权衡。RRF 不用调参，跨系统迁移不会崩；加权融合可以针对业务特征精调，但每次换场景都要重新实验。

### Q：设计一个生产级 AI Agent 系统架构，说明数据流转

**场景触发**：我们要处理的是金融分析类查询，这类查询天然是多步骤的——用户说"分析茅台近三年的盈利能力"，这背后至少包含"拿历史财报数据"、"计算 ROE/毛利率"、"生成分析结论"三个独立动作，而且每一步的失败都会影响下一步。

**核心矛盾**：如果让一个 LLM 一次性处理所有这些，它会在哪一步悄悄失败而不报错，给你一个看起来完整但数据是伪造的结论。单 Agent 模式下，你没有办法在任务执行的中间做质量检查。

**逻辑闭环**：

第一步，用户的查询进来，第一个接触点是 Supervisor 节点。它做三件事：先把消息历史做压缩（超过 20 条旧消息，用轻量模型生成一段摘要替换掉，保留最近 8 条原始消息），然后用快速打分函数评估这个查询的复杂度——涉及几只股票、几个指标、有没有时间跨度——打出 0 到 10 分，分数决定后续允许执行多少步。最后把查询分解成一个有序的步骤列表，比如"第一步查财报，第二步计算指标，第三步生成结论"。这一步的输入是用户自然语言，输出是一个结构化的任务计划。

第二步，Supervisor 把第一个步骤交给 Coder 节点。Coder 接到步骤描述之后，先去召回相关的 Tushare 接口文档（这就是 RAG 层），然后根据文档生成 Python 代码，调用沙箱执行。代码执行完后返回结果，但不是直接往后走——这里有三重验证：输出里必须有数据标记、数据不为空、没有报错标记。三道都过，Supervisor 才把任务推进到下一步；有一道没过，打回 Coder 重新生成，最多重试两次。这一步的输入是一个步骤描述，输出是有数据标记的执行结果。

第三步，所有步骤执行完成之后，Reviewer 节点汇总所有数据，如果来自多个 Coder 调用，先用结果融合模块去重和合并，然后生成面向用户的分析报告。Reviewer 审核完成后，链路不是直接结束——它的下游是 ProfileUpdater，这个节点基于完整的对话历史更新用户画像，比如记录"这个用户习惯查 A 股财务指标"，把成功的查询策略写入长期记忆，供下次同类查询复用。

**价值判定**：如果不做任务分解、不做步骤验证，系统在面对多步骤查询时的失败是静默的——模型不会告诉你第二步失败了，它只会把第一步的部分结果包装成一个完整答案输出来。这在金融场景是不可接受的，错误的财务数据比没有数据危害更大。状态机控制的本质是把"模型的隐式推理"变成"系统的显式验证"。

### Q：Agent 的"规划-执行-反思"闭环如何实现？

**场景触发**：Agent 系统有一个经典问题叫做 Reasoning Loop——模型反复用同一个错误策略重试，每次都失败，但系统没有任何机制判断"这条路走不通了，要换一种思路"。我们在早期测试里就遇到了这个：Coder 查某个港股数据，Tushare 根本不支持，但它会一直尝试不同的接口参数，死循环直到超时。

**核心矛盾**：LLM 的局部最优问题。它在当前上下文里看到"上次失败了，这次换个参数试试"，但它看不到全局——这个数据源根本没有这条数据，换多少个参数都无效。系统需要一个跳出局部的机制。

**逻辑闭环**：

第一步，规划阶段。当 Supervisor 收到一个新查询时，先启动查询重写器——把一个复杂问题分解成若干个聚焦的子问题，每个子问题对应一个可执行步骤。步骤生成之后做有效性校验：如果步骤描述里没有"查询"、"获取"、"计算"这类动作词，判定为无效计划，回退到用子问题直接作为步骤的保底策略。这一步保证进入执行阶段的计划是可行的。

第二步，执行阶段。每个步骤按序交给 Coder 执行，执行结果用三重验证做质量门控。关键机制在这里：系统维护了一个全局步骤计数器，超过 25 步就强制终止，这是防止无限循环的硬限制。同时还维护了一个错误类型字段——当前失败是代码写错了、还是网络超时、还是这个数据根本不存在——不同的错误类型走不同的重试策略：代码错误立即重试，网络超时指数退避，数据不存在最多一次然后降级。

第三步，反思阶段。当同一个错误类型连续出现两次以上，系统触发自我纠正机制。触发条件是：执行状态为"失败"且重试次数达到阈值。触发后，系统构造一个反思提示，把"用户原始查询、错误类型、重试次数"打包送给强模型，请它分析"为什么失败、下一步应该怎么调整策略"。反思结果作为一条系统消息注入对话流，影响 Coder 的下一次代码生成。同时，回溯路由器把这次失败标记到策略列表，自动切换到备选策略——比如从查日线数据切换到查周线数据。

**价值判定**：如果没有反思机制，失败就是终态——系统要么超时退出，要么输出一个错误提示。有了反思，失败变成了"有信息的中间状态"——系统知道哪条路走不通，并且主动切换方向。这是从"被动重试"到"主动调整"的本质区别。

### Q：多轮对话中，记忆如何分级管理？缓存策略怎么设计？

**场景触发**：我们系统面对两个场景：一是单次长对话，用户连续追问七八个问题，消息列表膨胀，Token 费用和延迟都爆炸；二是用户隔天再来问一个相关问题，系统完全不记得上次讨论了什么，又重新走一遍完整流程。这是两个不同维度的记忆问题：一个是"单次对话内的上下文管理"，一个是"跨会话的知识沉淀"。

**核心矛盾**：内存和持久化之间的张力。如果所有历史全部带进上下文，Token 爆炸；如果全部丢掉，系统每次都从零开始。需要一个分层的解法，让"最近的"保持活跃、"重要的"长期留存、"无用的"自动淘汰。

**逻辑闭环**：

第一步，短期记忆层处理单次会话内的消息膨胀。这一层的容量是硬限制——最多保存 10 条，超时 30 分钟自动过期，容量满了就用先进先出的策略淘汰最老的消息。但光截断不够，因为截断会丢掉中间重要的推理过程。所以在截断之前，先做一次摘要压缩：当消息总数超过 20 条，把最老的那批消息送给轻量模型生成一段摘要，用这段摘要替换掉原始消息，保留最近 8 条原文。最终效果是：Supervisor 每次看到的上下文是 `[摘要] + [最近8条]`，既知道对话历史，又不超过 Token 限制。

第二步，长期记忆层处理跨会话的知识沉淀。这一层存的不是对话原文，而是"成功策略"——比如"查茅台财报用 `income()` 接口、参数格式是 `ts_code='600519.SH'`、时间范围用 `period`"这样的结构化知识。每次 Reviewer 完成审核之后，系统自动触发画像更新，把这次成功的查询模式写入长期记忆，持久化到 JSON 文件。每条记忆有一个权重公式：重要性基础分 × 时间衰退因子 × 访问频率加成 × 成功次数加成。访问越频繁、越近期、成功次数越多的记忆，权重越高，越容易在下次检索时被召回。

第三步，多用户隔离层处理并发场景。每个用户会话有一个独立的 ID，每个 ID 对应一个独立的短期记忆实例，彼此完全隔离。但长期记忆是共享的——所有用户的成功策略都会沉淀到同一个知识库里，相当于"个人上下文隔离，全局经验共享"。

**价值判定**：如果没有这个分层机制，会发生什么？单层无限追加消息，第 15 轮对话时 Token 费用是第 1 轮的十几倍；简单截断则丢掉关键推理上下文，Coder 不知道前几步已经查了什么，会重复查询。分层记忆解决的是"有限上下文窗口内的无限对话"这个根本矛盾——这不是工程优化问题，是系统能不能 scale 的前提。

### Q：向量数据库选型对比，淘天场景怎么选？

**场景触发**：我们项目最初用 Chroma 做向量存储，它嵌入式、开箱即用，开发阶段没问题。但升级到 Python 3.14 之后，Chroma 依赖的底层 C++ 库 hnswlib 编译失败，整个 RAG 链路直接挂掉。这是一个典型的"开发阶段隐藏、升级时暴露"的技术债。

**核心矛盾**：不同向量数据库在"部署复杂度"、"并发能力"、"生态兼容性"三个维度上存在根本性的取舍。没有一个选项在三个维度上同时最优，选型的本质是确认你的瓶颈在哪个维度。

**逻辑闭环**：

第一步，我们把 Chroma 换成了 FAISS。选择的理由是：FAISS 是纯 Python 包，没有复杂的 C++ 编译依赖，Python 3.14 完全兼容；它是内存内索引，检索延迟极低；对于我们 2000 多个 chunk 的规模，用最简单的 `IndexFlatIP`（内积暴力搜索）精度比近似索引更高，速度也完全够用。第一次构建索引需要 3 到 5 分钟做向量化，之后全量缓存到磁盘，下次启动直接加载，命中缓存的冷启动时间小于 3 秒。这一步的输出是：一个在内存里常驻的精确向量索引，支持毫秒级检索。

第二步，明确这个选型的边界。FAISS 是单机库，没有内置的持久化服务、没有多租户隔离、没有水平扩展能力。这在我们原型阶段完全够用，但如果要上生产，就需要换 Milvus。

第三步，说清楚淘天场景的选型逻辑。淘天场景有三个核心诉求：商品库是亿级向量，规模远超单机 FAISS 的上限；大促期间并发峰值是平时十倍，需要水平扩展；新品上架需要实时插入向量，不能重建整个索引。Milvus 原生支持分布式部署、动态插入、多租户隔离，正好对应这三个诉求。Pinecone 是托管服务，运维省心，但数据出境有合规风险，阿里体系内不现实。所以淘天选 Milvus，自部署在阿里云。

**价值判定**：向量数据库选型不是技术偏好问题，是"系统扩展边界"的选择。用 Chroma 的代价是生态兼容性脆弱；用 FAISS 的代价是放弃水平扩展；用 Milvus 的代价是引入运维复杂度。我们选 FAISS 是因为在原型阶段，开发速度比扩展能力更重要，并且我们清楚这个选型的天花板在哪里。

### Q：Function Calling 的可靠性如何保障？模型一直调用同一个错误工具怎么办？

**场景触发**：在我们的系统里，Coder 节点有一个工具叫"执行 Python 代码"，有一个工具叫"搜索 Tushare 文档"。我们测试时发现：有时候 Coder 在应该先查文档的情况下直接去执行代码，代码里用了它凭记忆"猜"出来的接口名，执行失败；然后它还会继续调用执行代码工具去"修复"，在同一个错误上反复打转，这就是 Tool Hallucination 问题。

**核心矛盾**：LLM 做工具选择是一个分类决策，它依赖的是工具描述和当前上下文的匹配程度。当工具描述模糊、或者模型在上下文里没有足够信息做判断时，它会走向"惯性路径"——调用它最熟悉的那个工具，哪怕这个工具不适合当前步骤。

**逻辑闭环**：

第一步，从工具描述层面做预防。工具描述不是给人看的说明书，而是给模型做分类的特征输入。一个好的描述必须包含四个要素：什么情况下调用、输入参数的合法值域和格式示例、输出的数据结构、明确禁止的使用场景。比如"执行 Python 代码"这个工具，描述里要写清楚"仅在已知接口名和参数格式的情况下调用，如果不确定接口名，必须先调用文档搜索工具"。这是软约束，把分类边界告诉模型。

第二步，从系统层面做硬限制。我们维护了两个计数器：一个是工具调用总次数，超过阈值系统强制介入；一个是全局步数计数器，超过 25 步强制终止整个链路。这两个计数器不依赖模型是否"意识到"自己在循环，是代码层面的兜底。

第三步，从历史数据层面做替代推荐。我们有一个工具使用图，记录每个工具的历史成功率和平均延迟。当某个工具连续失败时，系统查询这个图，从可用工具列表里找出历史成功率最高的替代工具，把推荐信息注入 Coder 的下一次提示词里——"上次调用失败，建议改用 XXX 工具"。这把"历史失败经验"变成了"下次调用的先验知识"。

**价值判定**：如果只靠工具描述做约束，在模型能力不稳定的情况下可靠性上限大约是 80%。加了调用计数熔断和替代推荐之后，即使模型做出错误选择，系统也能在固定步数内跳出死循环并切换策略。这是把"模型行为不确定性"转化成"系统行为确定性"的关键。

### Q：Chunk 大小怎么确定？不同业务场景策略差异？

**场景触发**：我们早期的 RAG 版本，每个 Tushare 接口文档是一整篇作为一个 chunk，340 个接口就是 340 个 chunk。用户问"净利润对应哪个字段名"，召回的是整篇接口文档，里面有几十个字段，模型从中找答案，经常找错或者找到同名字段的错误版本。

**核心矛盾**：Chunk 粒度和查询意图的粒度不匹配。文档是接口级的，但用户的查询经常是字段级的——他要的不是"这个接口怎么用"，而是"这个接口里的某一行输出参数叫什么"。用接口级 chunk 回答字段级查询，等于让用户在一本书里自己找答案。

**逻辑闭环**：

第一步，识别查询意图的粒度。金融 API 文档的查询天然分两类：一类是"我该用什么接口查某类数据"，这是接口级意图；一类是"某个接口里净利润字段叫什么"，这是字段级意图。两类意图需要不同粒度的 chunk 才能精确命中。

第二步，建立两级 chunk 体系。对每篇接口文档，系统生成两种 chunk：第一种是接口级 chunk，保留接口标题加上完整描述和参数列表，用来回答接口选择类问题；第二种是字段级 chunk，解析文档里的"输出参数"表格，把每一行字段名、字段类型、字段描述拼成一条独立的 chunk，格式是"接口名 + 字段名 + 字段类型 + 描述"。340 篇文档拆分后，接口级保留 340 条，字段级新增约 1700 条，总量来到 2000 多条 chunk。

第三步，检索时做粒度感知的去重。召回结果里，同一个接口的字段级 chunk 可能命中很多条，但展示时合并——同一个接口只展示最高分的那条，防止 top-5 结果被同一个接口的不同字段占满，挤掉其他接口的信息。这一步保证输出的 5 条结果来自 5 个不同的接口，覆盖更广。

**价值判定**：从接口级到字段级的粒度升级，本质上解决了"召回精度和召回范围"之间的矛盾。接口级 chunk 召回范围广但精度低；字段级 chunk 精度高但如果没有去重会让结果过于集中。两级体系 + 去重策略，让字段级查询的 Hit@3 能提升 20% 以上，同时接口选择类查询不受影响。

---

### Q：检索结果相关性不高，怎么优化召回？

**场景触发**：这个问题的答案取决于相关性低的原因。在我们系统里，遇到过两种情况：一种是用户说"查分红数据"，但索引里存的是 `dv_ttm`、`dv_ratio` 这些缩写，向量模型不认识这些词，召回结果乱；另一种是用户查询太复杂，把多个意图混在一句话里，任何一个 chunk 都只能部分匹配。两种原因对应两种不同的优化路径。

**核心矛盾**：召回不准的根源可以在三个位置之一——查询端（用户表达和索引语言不匹配）、索引端（chunk 粒度和查询意图粒度不匹配）、融合端（单路检索有盲区）。修哪里取决于症状在哪里出现。

**逻辑闭环**：

第一步，从查询端修。用同义词字典做查询扩展——用户说"分红"，扩展成包含 `dv_ttm`、`dv_ratio`、`股息率`、`dividend` 的复合查询串，让 BM25 能精确命中这些 API 字段名。同时用查询重写器把复杂多意图查询拆成独立子查询，每个子查询聚焦一件事，独立召回后合并结果。这一步的输出是：从用户模糊自然语言到精确领域术语的桥接。

第二步，从索引端修。调整 chunk 粒度——从文档级降到字段级，让细粒度查询直接命中。同时在向量入库时做 L2 归一化，把每条 chunk 的向量长度统一到单位球面上，这样 FAISS 的内积搜索等价于余弦相似度，消除向量模长对相似度计算的干扰。这一步的输出是：更精细的索引结构，减少"召回到了正确主题但错误字段"的情况。

第三步，从融合端修。向量和 BM25 两路互补——向量解决同义词和语义漂移，BM25 解决精确字符匹配。加权融合参数 70/30 是经过实验选出的，金融场景语义泛化需求更高所以向量权重更大。如果还不够，下一步可以接入 Cross-Encoder Reranker，把召回的 20 个候选重新用 query-doc 拼接过精排模型打分，精度更高，但每次检索延迟会增加 100 到 200 毫秒。我们当前没有加，因为 2000 个 chunk 的规模下，召回精度已经满足需求，引入 Reranker 的延迟代价不合算。

**价值判定**：召回优化没有银弹，每一层优化都有代价——查询扩展增加构建复杂度，字段级 chunk 增加索引体积，Reranker 增加检索延迟。我们的选择是"在当前规模下找到精度和延迟的最优平衡点"，并且清楚地知道当规模扩大时，每个阶段该引入什么新的优化手段。这是系统设计能力的体现，不是把所有技术堆在一起。

### Q：Agent 推理链路串行工具多，响应延迟高，如何优化？

**场景触发**：我们系统的一次完整请求，要经过 Supervisor 路由、RAG 检索、Coder 代码生成、工具执行、Reviewer 审核，最少也要走五个节点，每个节点都要调一次大模型。早期测试里，一个多步骤的财报分析查询，端到端延迟可以达到几分钟，用户体验完全不可接受。

**核心矛盾**：Agent 系统的延迟来源是叠加的——每个节点的 LLM 调用延迟、工具执行延迟、上下文 Token 数随对话轮次增长带来的推理变慢，三者叠加。如果每个节点都用同等规格的模型，成本和延迟都是线性放大的。

**逻辑闭环**：

第一步，做模型分层，让不同节点用不同规格的模型。核心判断是：哪些节点的质量对最终结果起决定性作用，哪些节点只是辅助。我们把 Coder 代码生成和 Supervisor 路由决策定为高质量节点，用 qwen-plus 这类强模型，超时容忍是 120 秒；把 ErrorHandler、ProfileUpdater、消息摘要压缩定为辅助节点，用 qwen-turbo 这类快模型，超时容忍是 60 秒。这个分层让大约 60% 的节点调用切换到了低成本快速模型，整体延迟和成本都有 10% 到 20% 的改善。

第二步，控制 Token 总量，防止上下文膨胀拖慢每轮推理。消息历史超过 20 条时，触发摘要压缩——把旧消息浓缩成一段摘要，保留最近 8 条原文。同时工具输出超过设定的最大字符数时截断，加上截断提示告知模型数据已被截断。这两个操作保证每次 LLM 调用的输入 Token 数维持在可控范围内，不随对话轮次线性增长。

第三步，用复杂度感知的步骤预算，避免简单查询走重型流程。在任务分解之前，先给查询打一个复杂度分数——涉及几只股票、几个指标、有没有时间跨度、需不需要可视化，每项加分，总分 0 到 10。分数低的简单查询，步骤上限设为 3，Supervisor 不会让它走 8 步；分数高的复杂查询，步骤上限才扩展到 10。这避免了"用多步骤流程回答一个单步骤问题"的浪费。

**价值判定**：延迟优化的本质是识别链路里的"价值密度"。强模型放在价值密度高的节点，快模型放在价值密度低的节点，Token 控制保证每次调用的输入规模不失控，步骤预算避免流程浪费。这三件事加起来，才能在不牺牲质量的前提下把延迟压下来。如果只做其中一件，效果是局部的；三件同时做，才能从系统层面解决问题。

### Q：高并发场景下，如何设计 Agent 服务的弹性伸缩策略？

**场景触发**：我们项目是单机原型，但在设计上预留了并发隔离的基础接口。面试官问这道题，真正想听的是你对"Agent 服务的哪些部分是有状态的、哪些是无状态的"有清晰的认知，然后基于这个认知来谈扩缩容。

**核心矛盾**：Agent 服务的有状态性是水平扩展的障碍。每次对话的上下文、任务计划、重试计数，这些状态如果存在 Worker 进程本地，那这个请求就只能被同一个 Worker 处理，无法把流量分散到多台机器上。

**逻辑闭环**：

第一步，识别系统里哪些是有状态的。在我们的实现里，有状态的部分有三个：对话消息历史、任务计划（当前执行到第几步）、短期记忆实例。这三个如果放在进程内存里，就把 Worker 变成了有状态服务，无法水平扩展。

第二步，把有状态部分外置。对话消息历史和任务计划通过 LangGraph 的 `MemorySaver` 做持久化——在我们原型里是内存存储，生产上替换成 Redis 即可，Worker 本身变成无状态。短期记忆已经用 `session_id` 做了隔离，每个会话有独立实例，外置到 Redis 后不同 Worker 处理同一个会话的不同轮次也能拿到正确的上下文。FAISS 索引在内存中只读，所有 Worker 共享同一份，不存在写冲突。

第三步，设计扩缩容触发机制。我们的 `ToolCallMonitor` 已经在实时记录每个工具的调用延迟和失败率，这个数据天然可以作为扩缩容的信号源。当工具平均延迟超过 5 秒，说明工具服务有压力；当失败率超过 10%，说明系统过载。基于这两个指标配置告警，触发 Worker 水平扩容。对于 LLM API 的速率限制，我们代码里已有 429 检测加指数退避，生产上升级为全局令牌桶，按用户或租户隔离配额，防止单用户的突发请求消耗掉整个 API 配额。

**价值判定**：弹性伸缩能不能做，取决于架构设计时有没有把有状态和无状态分清楚。如果一开始就把状态混在 Worker 里，后来想扩容要推倒重来；如果一开始就把状态外置，扩容就是加机器这一件事。我们设计时已经做了这个分离，让后续的水平扩展有了技术可行性。

### Q：Agent 系统的核心监控指标有哪些？怎么设计告警策略？

**场景触发**：Agent 系统和传统服务的区别在于，传统服务失败了会立刻报错；Agent 系统失败可以是静默的——它执行了，没有报错，但结果是错的、数据是空的、或者走了一条低效的路径。这意味着传统的"成功率"指标不够用，你需要能感知"模型行为质量"的指标。

**核心矛盾**：Agent 系统的质量问题不在接口层，在推理层。接口返回 200 不代表答案正确，工具调用成功不代表数据完整。监控体系需要深入到推理链路内部，而不只是看外部接口的 HTTP 状态码。

**逻辑闭环**：

第一步，建立工具层监控。我们的工具调用监控器记录每个工具的调用总次数、成功次数、失败次数、累计延迟。从这里派生出三个核心指标：失败率、平均延迟、是否有工具从未被调用。失败率超过 10% 触发告警，因为一个工具持续失败会影响整个链路；平均延迟超过 5 秒触发告警，说明工具服务有性能问题；有必要工具从未被调用，触发警告，因为这可能意味着某条推理路径从未被激活，影响模型能力的覆盖。

第二步，建立链路层监控。全局步数计数器是最基础的链路健康指标——正常查询 5 到 8 步内应该完成，超过 15 步说明系统在低效重试，超过 25 步系统强制终止。错误类型分布是另一个维度：代码错误、网络超时、数据不存在，这三类的比例变化能反映出系统健康状态的变化——比如网络超时突然增加，说明外部 API 有问题；代码错误突然增加，说明 Coder 节点的提示词或者 RAG 召回质量下降了。

第三步，建立质量层监控。通过评测框架定期跑 Pass@K 指标——用一批有标准答案的测试用例，测试 K 次内有多少次成功。这个指标每天自动运行，结果写入日志，一旦 Pass@1 低于设定阈值触发告警。同时收集 Trajectory 轨迹数据，记录每次请求从 Supervisor 到 FINISH 的完整节点路径，用来事后分析哪类查询最容易失败，在哪个节点失败的概率最高。

**价值判定**：这三层监控的逻辑是从外到内的覆盖。工具层监控发现"工具坏了"，链路层监控发现"流程卡住了"，质量层监控发现"答案错了"。少了任何一层，都有盲区——只有工具层，你不知道流程是否在低效循环；只有质量层，你发现问题的时候滞后太多，用户已经受影响了。

### Q：如何追踪 Agent 单次请求完整链路？跨多个服务的 Trace 怎么设计？

**场景触发**：Agent 系统排查问题比传统服务难得多。传统服务报错，看堆栈就知道在哪一行；Agent 系统失败，可能是 Supervisor 路由错了，可能是 Coder 生成的代码有 bug，可能是 Reviewer 审核太严格打回了，可能是工具调用超时——失败的原因分布在多个节点，没有追踪能力的话根本不知道从哪里开始排查。

**核心矛盾**：Agent 系统的一次请求不是一条线，而是一个图——它在多个节点之间跳转，每次跳转都可能改变方向。传统的请求追踪是记录一条调用链，Agent 系统需要记录的是一个决策树。

**逻辑闭环**：

第一步，每次请求启动时，创建一个 Trajectory 对象，分配唯一 ID，记录用户原始查询和开始时间。这个 ID 贯穿整个请求生命周期，所有后续操作都挂在这个 ID 下面。

第二步，每个节点执行时，调用 `record_step` 记录一个 Step——动作类型（比如"Coder 代码生成"、"工具调用"、"Reviewer 审核"）、输入内容的摘要、输出内容的摘要、是否成功、如果失败是什么错误信息。这些 Step 串联起来，就是这次请求在整个状态机里的完整行走路径。

第三步，同时运行的还有一个故障传播分析器。它在运行时实时维护一个有向图，每当一个节点失败，它检查上一个节点的输出是否有隐患：如果上游成功、下游失败，判定为掩蔽传播——上游输出看起来正常但埋了隐患；如果上下游都失败，判定为直接传播——上游的错误直接导致了下游失败。最终生成一个根因分析结果，指出"根因在哪个节点、错误类型是什么、传播路径是什么、传播深度是几层"。

请求结束时，整个 Trajectory 序列化为 JSON 文件存储到磁盘，同时传入到评测框架做成功率统计。这些轨迹数据还有一个副产品——用来构建 DPO 微调数据集，成功轨迹作为 chosen，失败轨迹作为 rejected，用来持续优化模型的决策质量。

**价值判定**：如果没有这个追踪系统，排查一次 Agent 失败需要翻日志、看每个节点的输出、手动推断失败原因，可能要花半小时。有了 Trajectory 追踪加故障传播分析，10 秒内能定位到根因节点，知道是代码错误、数据缺失还是工具超时，直接针对性修复。这是 Agent 系统可维护性的基础，没有它，系统越复杂越难运维。

### Q：意图识别不准确，工程上有哪些方法优化？

**场景触发**：我们 Supervisor 节点做路由决策，输入是用户查询，输出是"下一步去 Coder 还是直接结束"。早期测试里发现：当 LLM 返回的路由决策 JSON 格式不对、或者 API 限流导致调用失败时，整个路由会崩掉，请求直接报错退出，根本到不了执行阶段。这是意图识别链路可靠性问题，不是准确率问题。

**核心矛盾**：LLM 做路由是一个概率性决策，它有一定概率输出错误格式或者走向错误分支。如果系统完全依赖 LLM 的路由输出，一旦 LLM 出问题，整条链路就断了。需要一个确定性的兜底路径，让系统在 LLM 路由失效时也能有合理的降级行为。

**逻辑闭环**：

第一步，主路径用 LLM 做语义路由。Supervisor 把路由决策的输出格式严格约束为 Pydantic 模型——只允许"去 Coder"、"去 Reviewer"、"结束"三个值，解析失败直接触发异常而不是产生歧义结果。这是软约束层，给 LLM 一个明确的输出规范。

第二步，主路径失败时，回退到关键词确定性路由。这个回退路由完全不依赖 LLM，只看两个字段：上一个节点是谁、当前执行状态是成功还是失败。比如"上一个节点是 Coder、状态是成功"，无论如何都去 Reviewer；"上一个节点是 Reviewer、状态是失败且失败次数超过 2 次"，直接结束。这是硬约束层，确定性规则，不会因为 LLM 输出异常而崩掉。

第三步，对于路由精度的持续优化，用复杂度感知来辅助意图分类。用户的查询在进入路由之前先打复杂度分，分数影响后续允许的步骤数和模型选择。这相当于在路由之前先做了一层查询理解，给 LLM 路由提供了更多的上下文线索，降低路由误判的概率。同时路由配置外化到 JSON 文件，规则调整不需要重新部署服务，直接修改配置热更新。

**价值判定**：意图识别的优化不只是提升准确率，更重要的是提升鲁棒性。准确率 90% 意味着 10% 的请求走错路，如果没有兜底路由，这 10% 的请求会崩溃；有了确定性回退，这 10% 的请求虽然走了次优路径，但系统不会崩。从"10% 的请求崩溃"到"10% 的请求走次优路径"，这是用户体验的本质差异。

### Q：RAG 系统如何评测？有哪些评测维度？如何提升相关度和回答效果？

**场景触发**：我们系统上线之前，没有办法靠人工测试来验证质量——340 个接口、几百种查询组合，手动测一遍需要几天时间。需要一套自动化评测体系，能在每次改动后快速告诉我们"系统变好了还是变坏了"。

**核心矛盾**：Agent 系统的质量评测有两个难点：第一，对于金融查询，"正确答案"本身就不好定义——同一只股票的利润，用不同接口查、用不同时间参数查，结果可能合法地不同；第二，规则匹配无法处理语义等价——"净利润 100 亿"和"归母净利润约 1000 亿元"字面上不同，但后者可能只是单位不一致，实质上是正确的。

**逻辑闭环**：

第一步，建立多维度评测体系。每次请求的结果从四个维度打分：第一个维度是执行成功率，代码能不能无异常跑完；第二个维度是数据完整性，输出里有没有数据标记且数据不为空；第三个维度是语义正确性，通过 LLM-as-Judge 来判断——把用户问题、系统回答、参考答案三者送给一个强模型，让它判断回答是否在语义上正确，可以处理单位换算、同义表达等规则无法覆盖的情况；第四个维度是延迟，记录每次请求的端到端耗时。

第二步，用 Pass@K 做主指标。Pass@K 的含义是：对同一个问题，系统最多尝试 K 次，K 次内有一次成功就算通过，通过率就是 Pass@K。这个指标比单次成功率更宽容，更适合有重试机制的 Agent 系统。我们用 K=1、3、5 三个值，Pass@1 反映首次成功率，Pass@3 反映重试机制的贡献。

第三步，如果要提升相关度，从召回端优化——增加同义词词典覆盖度、把 chunk 粒度细化到字段级、调整向量和 BM25 的融合权重。如果要提升回答效果，从生成端优化——改进 Reviewer 的审核 Prompt、在 Coder 的提示词里注入更多成功案例、用 LLM-as-Judge 的评判结果反向筛选高质量训练样本做微调。这两个优化方向是独立的，相关度低通常是召回问题，回答效果差通常是生成问题，混在一起优化效率很低。

**价值判定**：评测体系的核心价值是把"感觉系统变好了"变成"数据证明系统变好了"。没有评测，每次改动都是在黑暗里摸索——你不知道某次 Prompt 调整是真的有帮助还是在测试集上的偶然。有了 Pass@K 加 LLM-as-Judge 的评测，每次迭代都有一个数字作为锚点，可以做对比实验，可以量化每个优化手段的贡献。这是从"工程直觉"走向"工程科学"的前提。

### 2. 项目主要存在哪些Bad Case？怎么解决？

**场景/输入**：在GRPO训练初期，输入同样的数独题目，模型输出会出现三种典型错误模式。

**核心机制**：

**价值判定**：这样做解决了Reward Hacking导致的"表面合规、实质错误"问题，最终使得线索保真率达到99%，格式合规率达到100%。

## 第二部分：SFT核心流程（重新组织版）

### 3. SFT的数据到底是怎么"喂"给模型的？Loss是怎么算的？

**场景/输入**：在SFT阶段，原始数据是JSON格式，每个样本包含`question`（题目描述+数独矩阵）和`answer`（包含CoT的完整回答）。

**核心机制**：
- **数据协议**：`answer`字段的格式是`<think>\n推理过程\n</think>\n\n<answer>81位数字</answer>`，推理过程包含按坐标定位的详细步骤。
- **Chat Template拼接**：我使用Qwen的Chat Template将系统指令、用户问题、助手回答拼接成完整序列。系统指令定义了输出格式要求，用户问题包含数独矩阵，助手回答是训练目标。
- **Loss计算**：采用Causal LM Loss，对助手回答部分的每个token计算交叉熵。具体来说，Input IDs是整个拼接后的序列，Labels与Input IDs相同，但系统指令和用户问题的位置被Mask为-100（IGNORE_TOKEN_ID），只有助手回答部分参与Loss计算。公式是`Loss = -Σ log P(token_i | token_<i)`，求和范围仅限于助手回答的token。

**价值判定**：这样做解决了"模型不知道应该学什么"的问题，通过Mask机制明确告诉模型"只需要学习如何回答，不需要学习问题本身"，最终使得SFT后的格式完备率达到100%。

### 4. CoT是加在SFT的输出里训练，还是加在输入里？

**场景/输入**：在构造训练数据时，需要决定推理过程（CoT）的位置。

**核心机制**：CoT是加在**输出（Output）**里训练的，不是输入。具体来说：
- **输入（Input）**：系统指令 + 用户问题（数独矩阵）+ 助手角色标记。
- **输出（Output）**：`<think>`标签 + 推理过程 + `</think>`标签 + `<answer>`标签 + 81位答案 + `</answer>`标签。

模型在训练时，需要根据输入预测整个输出序列，包括推理过程和最终答案。推理过程是通过对比题目和答案自动合成的，每个空格对应一步"坐标定位→逻辑分析→填入数字"的推理链。

**价值判定**：这样做解决了"模型只会给答案不会解释"的问题，通过强制模型生成中间推理步骤，诱导模型学习数独的约束规则，最终使得GRPO阶段的逻辑奖励有稳定的提升基础。

**场景/输入**：在SFT阶段，如果推理过程太长，会占用大量token导致答案被截断；如果太短，可能逻辑不完整。

- **max_seq_length设定**：设为4096，为推理过程预留足够空间。如果某条数据超过这个长度，会被截断。

**价值判定**：这样做解决了"推理过长导致答案丢失"的截断问题，最终使得GRPO阶段的平均生成长度稳定在2000-3000token之间，既保证逻辑完整又避免话痨。

## 第三部分：GRPO算法详解（重新组织版）

### 6. GRPO运行时的具体数据流向？Group里的优势函数是怎么算的？

**场景/输入**：在GRPO训练时，输入是一个Prompt（系统指令+数独题目），需要输出一组回答并计算梯度更新。

**核心机制**：
- **数据流向**：
  1. 从训练集采样一个Batch的Prompt；
  2. 对每个Prompt，当前策略模型生成G=12个回答（Group Sampling）；
  3. 7个奖励函数分别给这12个回答打分，加权求和得到最终奖励；
  4. 计算这12个奖励的均值μ和标准差σ；
  5. 对每个回答计算相对优势：A_i = (r_i - μ) / σ；
  6. 用PPO-Clip目标函数计算Loss，约束策略更新幅度；
  7. 反向传播更新LoRA参数。

- **优势函数计算**：GRPO去掉PPO的Critic模型，用组内统计替代价值估计。公式是`A_i = (r_i - μ) / (σ + ε)`，其中ε=1e-4防止除零。这样做的好处是：不需要训练Value模型，节省显存；组内相对比较天然适合稀疏奖励场景。

**价值判定**：这样做解决了PPO中Value模型训练不稳定、估计偏差大的问题，最终使得GRPO训练在1000步内收敛，KL散度稳定在0.5以下。

### 7. 如果一个Batch里的G个样本奖励全是0，梯度还会更新吗？怎么处理？

**场景/输入**：在GRPO训练初期，模型输出格式混乱，7个奖励函数可能全部给0分。

**核心机制**：
- **梯度更新分析**：如果G个样本奖励全是0，均值μ=0，标准差σ=0，此时优势函数A_i = 0/0无定义。实际实现中，σ会加上一个极小值ε=1e-4，所以A_i = 0。PPO-Clip目标函数中，优势为0意味着Loss为0，**梯度不会更新**。
- **我的处理方案**：
  1. **格式奖励保底**：`soft_format_reward_func`只要输出包含`<answer>`就给0.1分，包含`</answer>`再给0.2分，确保即使答案全错也有非零奖励。
  2. **密集奖励设计**：`exact_answer_reward_func`中，81位答案如果部分匹配，按匹配率的平方给分，而不是只有全对才给分。
  3. **早停监控**：如果连续50步平均奖励低于0.5，自动触发检查，确认是否是奖励稀疏导致的训练停滞。

**价值判定**：这样做解决了"奖励稀疏导致梯度消失"的训练冷启动问题，最终使得GRPO训练从第10步开始就有非零奖励，第100步左右出现第一个满分样本。

### 8. LoRA秩r设为16，Alpha设为16，这个比例对训练稳定性有什么影响？

**场景/输入**：在SFT和GRPO阶段，我都使用了LoRA进行参数高效微调，需要确定秩r和缩放系数Alpha。

**核心机制**：
- **参数含义**：r=16表示低秩矩阵的秩，Alpha=16表示缩放系数。实际缩放比例是Alpha/r = 1.0。
- **数学原理**：LoRA的前向传播是`h = W·x + (Alpha/r)·B·A·x`。Alpha/r控制新引入的低秩更新对原始权重的"干扰程度"。
- **比例影响**：
  - 如果Alpha/r > 1（如Alpha=32, r=16），低秩更新的权重被放大，模型学习更快但可能不稳定，容易过拟合或偏离原始分布太远。
  - 如果Alpha/r < 1（如Alpha=8, r=16），低秩更新被压缩，模型学习保守，收敛慢但更稳定。
  - 我设Alpha/r = 1.0是标准做法，在"学习速度"和"稳定性"之间取平衡。

**价值判定**：这样做解决了"全量微调显存不足"和"低秩更新力度不够"的矛盾，最终使得在单卡A100上能训练3B模型，SFT阶段显存占用约18GB，GRPO阶段约35GB（含Group Sampling开销）。

### 9. SFT和GRPO的数据格式分别是什么？CoT的位置有何不同？

**SFT数据格式**：

- **Input（模型看到的）**：`<|im_start|>system\n你是一个数独助手...<|im_end|>\n<|im_start|>user\n{question}<|im_end|>\n<|im_start|>assistant\n`

- **Loss计算范围**：从`<think>`开始到`<|im_end|>`结束的所有token，前面的系统指令和用户问题被Mask。

**GRPO数据格式**：
```json
{
  "prompt": [
    {"role": "system", "content": "你是一个数独专家..."},
    {"role": "user", "content": "请填满以下数独：\n800000000...\n直接在<answer>标签中输出81位纯数字答案。"}
  ],
  "target_solution": "完整81位答案",
  "quiz_str": "原始题目81位"
}
```

**GRPO的Input-Output**：
- **Input**：Prompt列表，经过Chat Template拼接后变成字符串。
- **Output**：模型生成的回答，包含（可能正确也可能错误的）推理过程和答案。
- **与SFT的关键区别**：GRPO没有标准答案的CoT，只有目标答案。模型需要自己探索推理路径，奖励函数根据输出质量打分。

**CoT位置总结**：
- SFT：CoT在**输出**中，是训练目标的一部分，模型必须学习生成。
- GRPO：CoT也在**输出**中，但不是给定的，需要模型自己生成，奖励函数评估其质量。

## 第一部分：Attention机制与Transformer基础

### 1. Attention的本质是什么？从数学角度解释一下

**场景/输入**：在处理序列数据时，输入是查询向量、键向量、值向量三个矩阵，形状分别是（序列长度 × 向量维度）。

**核心机制/计算流**：Attention的本质是**加权求和**，权重由查询和键的相似度决定。具体计算分三步：第一步，查询矩阵乘以键矩阵的转置，得到相似度分数，形状是（查询长度 × 键长度）；第二步，除以根号下键向量的维度，防止数值过大；第三步，经过softmax归一化得到权重矩阵；第四步，权重矩阵乘以值矩阵，得到输出。数学上，这相当于对值矩阵做加权平均，权重反映了"当前查询应该关注哪些位置的值"。

**故障/反作弊逻辑**：如果不除以根号下维度，当维度较大时（如64、128），点积的数值范围会变大，导致softmax进入饱和区，梯度变得极小，模型无法学习。除以根号下维度后，方差被缩放回1，保持梯度流动性。

**价值/量化判定**：这样做解决了"长序列注意力权重趋于极端"的问题，最终使得在维度为64时，softmax的输入值稳定在合理范围，梯度不会消失。

---

### 2. 为什么注意力机制要除以根号dk？为什么是除这个数？

**场景/输入**：当键向量的维度是64时，查询和键的点积结果可能是一个很大的数。

**核心机制/计算流**：假设查询和键的每个元素都是独立随机变量，均值为0，方差为1。那么点积的方差等于维度数（因为每个元素的方差是1，一共有维度数个元素相加）。为了让点积的方差回到1，需要除以根号下维度。这是一个概率论推导的结果，不是经验值。

**故障/反作弊逻辑**：如果不除以根号下维度，softmax的输入会呈现"极端化"——少数位置接近1，大多数位置接近0，注意力变成"硬选择"而非"软加权"，模型失去捕捉多重相关性的能力。

**价值/量化判定**：这样做使得注意力权重分布更加平滑，最终使得模型能够同时关注多个相关位置，而不是只关注一个。

---

### 3. 如果不除以根号dk，有没有替代方式？

**场景/输入**：在某些变体架构中，可能会考虑其他数值稳定化方案。

**核心机制/计算流**：替代方式包括：第一，使用可学习的温度系数，让模型自己学习缩放比例；第二，使用层归一化对点积结果进行归一化；第三，使用稀疏注意力，只计算部分位置的相似度。但在标准Transformer中，除以根号下维度是最简洁、最有效的方案，不需要引入额外参数。

**故障/反作弊逻辑**：可学习的温度系数可能收敛到极端值，导致注意力失效；层归一化会增加计算开销；稀疏注意力会丢失全局信息。

**价值/量化判定**：除以根号下维度是"无参、稳定、高效"的最优解，最终使得标准Transformer成为主流架构。

---

### 4. Q、K、V矩阵是如何得到的？

**场景/输入**：输入是一个词嵌入矩阵，形状是（批次大小 × 序列长度 × 模型维度）。

**核心机制/计算流**：通过三个独立的线性变换（全连接层）得到。具体来说，输入分别乘以三个权重矩阵，得到查询、键、值。这三个权重矩阵是模型需要学习的参数。在多头注意力中，每个头有自己独立的三个权重矩阵。

**故障/反作弊逻辑**：如果三个矩阵共享参数，模型会失去灵活性，无法分别学习"查询什么"、"记忆什么"、"输出什么"的不同功能。

**价值/量化判定**：分离三个矩阵使得注意力机制具有表达能力，最终使得模型能够学习复杂的对齐模式。

### 5. 多头注意力（MHA）解释一下

**场景/输入**：输入是词嵌入，需要捕捉不同子空间的依赖关系。

**核心机制/计算流**：将模型维度分成多个头（如8个或16个），每个头独立计算注意力。具体来说，查询、键、值分别被切分成多份，每份进入一个头做独立的注意力计算，最后将所有头的输出拼接起来，再经过一个线性变换。数学上，这相当于在多个低维子空间并行计算注意力。

**故障/反作弊逻辑**：如果只用单头，模型只能捕捉一种类型的依赖关系（如局部语法），无法同时捕捉长距离语义和局部语法。

**价值/量化判定**：这样做解决了"单一注意力模式表达能力不足"的问题，最终使得模型能够同时学习局部和全局、语法和语义的多种依赖关系。

### 6. 多头注意力存在什么问题？如何解决计算开销过大的问题？

**场景/输入**：当序列长度很长时（如4096），注意力矩阵的形状是（4096 × 4096），计算和存储开销都是平方级增长。

**核心机制/计算流**：主要问题是**计算复杂度高**，标准注意力的复杂度是O(序列长度的平方 × 模型维度)。解决方案包括：第一，稀疏注意力，只计算局部窗口或特定模式的注意力；第二，线性注意力，用核函数近似softmax，将复杂度降到线性；第三，Flash Attention，通过IO感知的分块计算，减少显存读写，实际速度提升2-4倍。

**故障/反作弊逻辑**：稀疏注意力可能丢失远距离依赖；线性注意力近似可能损失精度。

**价值/量化判定**：Flash Attention在保持精度的同时大幅提升速度，最终使得长序列训练成为可能。

### 7. 手撕带attention mask的MHA

**场景/输入**：输入是批次数据，包含查询、键、值三个张量，以及一个掩码张量表示哪些位置需要被屏蔽。

**核心机制/计算流**：第一步，通过线性变换得到查询、键、值；第二步，计算点积相似度，形状是（批次 × 头数 × 查询长度 × 键长度）；第三步，**关键步骤**：将掩码位置（通常是上三角）填充为负无穷；第四步，softmax归一化，被掩码的位置变为0；第五步，乘以值矩阵，得到输出。

**故障/反作弊逻辑**：如果不加掩码，Decoder在生成第t个token时会"偷看"第t+1个及之后的token，破坏自回归性质。

**价值/量化判定**：掩码机制保证了生成的因果性，最终使得Decoder能够正确地进行自回归生成。

---

## 第二部分：位置编码

### 8. 比较常用的位置编码方式？RoPE是什么，解决了什么问题？

**场景/输入**：Transformer本身对位置不敏感，需要注入位置信息。

**核心机制/计算流**：
- **绝对位置编码（Sinusoidal）**：为每个位置生成固定的正弦/余弦向量，直接与词嵌入相加。
- **可学习位置编码**：将位置编码作为可训练参数。
- **旋转位置编码（RoPE）**：将相对位置信息编码到查询和键的旋转矩阵中。具体来说，将每对维度看作一个复数，乘以旋转因子，旋转角度与位置索引成正比。

**故障/反作弊逻辑**：绝对位置编码无法处理比训练时更长的序列；可学习位置编码需要为每个位置存储参数，无法外推。

**价值/量化判定**：RoPE通过旋转操作天然支持相对位置，最终使得模型能够外推到比训练时更长的序列，且不需要额外参数。

---

### 9. 位置编码应该怎么加？训练上下文变长之后，应该改哪个参数？

**场景/输入**：模型训练时最大长度是2048，现在需要处理4096的序列。

**核心机制/计算流**：RoPE的位置编码是在计算注意力时动态应用的，通过旋转矩阵将位置信息注入查询和键。当上下文变长时，需要调整两个参数：第一，`max_position_embeddings`改为4096；第二，如果长度差距很大，需要开启`rope_scaling`（位置插值或NTK-aware扩展），将旋转角度进行缩放。

**故障/反作弊逻辑**：如果不调整位置编码参数，模型在超过训练长度的位置会看到"陌生"的旋转角度，导致注意力失效。

**价值/量化判定**：正确调整位置编码参数使得模型能够外推，最终使得长文档处理成为可能。

---

## 第三部分：归一化

### 10. LayerNorm和BatchNorm的区别？为什么用RMS Norm而不用Layer Norm？

**场景/输入**：在深度网络中，需要稳定每层的数值分布。

**核心机制/计算流**：
- **BatchNorm**：对一个批次内的所有样本的同一维度做归一化，减均值除标准差。依赖批次统计，对批次大小敏感。
- **LayerNorm**：对单个样本的所有维度做归一化，不依赖批次，适合序列模型。
- **RMS Norm**：LayerNorm的简化版，只保留缩放（除均方根），去掉中心化（减均值）。

**故障/反作弊逻辑**：在LLM的深层网络中，LayerNorm的均值可能趋近于0，导致梯度消失；RMS Norm只关注二阶矩，更稳定。

**价值/量化判定**：RMS Norm计算更快、更稳定，最终使得Qwen、LLaMA等现代LLM都采用这一方案。

---

### 11. Transformer中的LayerNorm是放在残差前还是后？两种顺序有何不同？

**场景/输入**：在Transformer块中，归一化层可以放在子层之前或之后。

**核心机制/计算流**：
- **Post-LN（残差后）**：原始Transformer论文使用，子层输出先加残差，再归一化。
- **Pre-LN（残差前）**：现代LLM使用，先对输入做归一化，再进入子层，最后加残差。

**故障/反作弊逻辑**：Post-LN在深层网络中容易出现梯度消失，训练不稳定；Pre-LN将归一化放在残差路径之外，梯度流动更顺畅。

**价值/量化判定**：Pre-LN使得训练更稳定、收敛更快，最终使得现代LLM能够训练到上百层。

---

## 第四部分：PPO、DPO、GRPO、DAPO演进

### 12. 讲一下PPO、DPO、GRPO、DAPO的改进脉络

**场景/输入**：在LLM对齐阶段，需要让模型输出符合人类偏好。

**核心机制/计算流**：

**PPO（Proximal Policy Optimization）**：
- 需要4个模型：策略模型、参考模型、奖励模型、价值模型。
- 价值模型估计状态价值，计算优势函数。
- 使用Clip机制约束策略更新幅度。
- 问题：价值模型训练困难，显存开销大，容易训崩。

**DPO（Direct Preference Optimization）**：
- 只需要2个模型：策略模型、参考模型。
- 去掉显式的奖励模型和价值模型，直接用偏好对（好回答vs坏回答）优化。
- 核心公式将奖励函数隐式表达为策略与参考模型的对数概率比。
- 问题：依赖静态偏好数据，无法在线探索。

**GRPO（Group Relative Policy Optimization）**：
- 只需要2个模型：策略模型、参考模型。
- 去掉价值模型，用组内相对优势替代。
- 对同一Prompt采样G个回答，用组内均值和标准差标准化奖励。
- 优势函数：A_i = (r_i - μ) / σ。
- 适合规则奖励场景（如数独），不需要训练奖励模型。

**DAPO（Direct Alignment from Preferences Optimization）**：
- DeepSeek提出的新范式，进一步简化流程。
- 直接用偏好数据对齐，不需要显式的奖励模型或组采样。

**故障/反作弊逻辑**：PPO的价值模型估计不准会导致优势函数偏差；DPO的静态偏好无法适应新场景；GRPO的组采样增加了计算开销。

**价值/量化判定**：GRPO在规则奖励场景下最优，最终使得我的数独项目能够在双卡A100上完成训练，而不需要4卡。

---

### 13. PPO和GRPO有什么区别？Group代表了什么？

**场景/输入**：在强化学习对齐阶段，需要计算优势函数指导策略更新。

**核心机制/计算流**：
- **PPO**：用价值模型估计状态价值V(s)，优势函数A(s,a) = Q(s,a) - V(s)。需要单独训练价值模型。
- **GRPO**：对同一Prompt采样G=12个回答，计算这12个奖励的均值μ和标准差σ，优势函数A_i = (r_i - μ) / σ。

**Group的含义**：Group是"同一Prompt下的多个采样输出"。它想替换的是PPO中价值模型提供的"基线估计"——用组内均值替代价值函数的期望回报估计。

**故障/反作弊逻辑**：如果G太小（如2或4），组内方差大，优势估计不准；如果G太大（如32），显存开销大。

**价值/量化判定**：GRPO去掉价值模型，节省显存约30%，最终使得在相同硬件下可以训练更大的模型或更长的序列。

---

### 14. PPO、DPO、GRPO分别是on-policy还是off-policy？

**场景/输入**：在强化学习中，on-policy和off-policy决定了数据能否重复使用。

**核心机制/计算流**：
- **PPO**：on-policy。使用当前策略采样的数据计算优势，更新后数据就丢弃，不能重复使用。
- **DPO**：off-policy。使用静态偏好对数据，可以重复训练，数据利用率高。
- **GRPO**：on-policy。使用当前策略组采样的数据计算相对优势，更新后丢弃。

**故障/反作弊逻辑**：on-policy方法训练稳定但样本效率低；off-policy方法样本效率高但可能不稳定。

**价值/量化判定**：GRPO保持on-policy的稳定性，同时通过组采样提高样本效率，最终使得在数独这种稀疏奖励场景下仍能收敛。

---

### 15. PPO的clip函数解决什么问题？不clip会怎样？

**场景/输入**：在策略更新时，新旧策略的概率比可能变化很大。

**核心机制/计算流**：PPO的clip函数约束概率比在[1-ε, 1+ε]范围内，ε通常设为0.2。如果概率比超出这个范围，梯度会被截断。公式是`min(概率比 × 优势, clip(概率比, 1-ε, 1+ε) × 优势)`。

**故障/反作弊逻辑**：如果不clip，策略可能在一次更新中发生剧烈变化，导致模型崩溃（collapse）。比如，某个动作的优势估计为正，策略会无限增大该动作的概率，失去探索能力。

**价值/量化判定**：clip机制保证了策略更新的平滑性，最终使得PPO和GRPO训练稳定，KL散度不会飙升。

---

### 16. PPO的loss包含哪些？如何考虑loss weight？

**场景/输入**：PPO需要同时优化多个目标。

**核心机制/计算流**：PPO的loss包含三部分：
1. **策略loss**：带clip的重要性采样目标，权重1.0。
2. **价值loss**：价值模型的均方误差，权重通常0.5。
3. **熵奖励**：鼓励探索，防止策略过早收敛，权重通常0.01。

**故障/反作弊逻辑**：如果价值loss权重太大，价值模型会主导训练，策略无法改进；如果熵奖励太小，模型会过早收敛到局部最优。

**价值/量化判定**：合理的loss weight平衡了策略改进、价值估计和探索，最终使得训练既稳定又高效。

---

### 17. 介绍一下GAE，为什么λ经常设置成0.95？

**场景/输入**：在计算优势函数时，需要权衡偏差和方差。

**核心机制/计算流**：GAE（Generalized Advantage Estimation）是一种优势估计方法，通过参数λ平衡TD误差的多步累积。λ=0时，只使用单步TD，偏差大、方差小；λ=1时，使用完整回报，偏差小、方差大。λ=0.95是常用值，意味着主要使用多步回报，但略有衰减。

**故障/反作弊逻辑**：如果λ太小，优势估计偏差大，策略更新方向不准；如果λ太大，方差大，训练不稳定。

**价值/量化判定**：λ=0.95在偏差和方差之间取得平衡，最终使得优势估计既准确又稳定。

---

## 第五部分：GRPO深度解析

### 18. GRPO的reward、advantage计算，每个token是怎么分配的？

**场景/输入**：在GRPO训练中，一个回答包含多个token，需要确定奖励如何分配到每个token。

**核心机制/计算流**：
- **Reward计算**：奖励是**句子级别**的，对整个回答打一个总分。我的7个奖励函数分别评估格式、逻辑、正确性等维度，加权求和得到最终奖励。
- **Advantage计算**：优势也是**句子级别**的，A_i = (r_i - μ) / σ，同一组内的G个回答共享相同的均值和标准差。
- **Token级别分配**：在PPO-Clip目标函数中，**每个token的loss都乘以相同的优势值**。也就是说，如果某个回答的奖励高，该回答的所有token都会获得正向梯度；如果奖励低，所有token都获得负向梯度。

**故障/反作弊逻辑**：这种"信用分配"是粗糙的——回答中可能有正确的部分也有错误的部分，但所有token共享同一个优势值。这是策略梯度方法的固有缺陷。

**价值/量化判定**：虽然信用分配不精细，但GRPO通过组内相对比较降低了方差，最终使得训练能够收敛。

---

### 19. GRPO里面clip的作用？去掉了会怎么样？

**场景/输入**：在GRPO策略更新时，需要约束新旧策略的差异。

**核心机制/计算流**：GRPO继承了PPO的clip机制，约束概率比在[1-ε, 1+ε]范围内。公式与PPO相同，取`min(概率比 × 优势, clip(概率比, 1-ε, 1+ε) × 优势)`。

**故障/反作弊逻辑**：如果去掉clip，当某个回答的奖励远高于组内均值时，策略会急剧增加该回答的生成概率，可能导致：第一，模式崩溃，只生成这一种回答；第二，KL散度飙升，模型偏离参考模型太远；第三，训练不稳定，loss震荡。

**价值/量化判定**：clip机制是GRPO训练的"安全阀"，最终使得KL散度稳定在0.5以下，模型不会崩溃。

---

### 20. 如果用更好的traj替换rollout出来的数据，训练会发生什么？

**场景/输入**：假设有人工标注的高质量回答，想直接替换模型自己采样的数据。

**核心机制/计算流**：GRPO是on-policy方法，理论要求数据必须由当前策略生成。如果用外部数据替换：
- 概率比（当前策略概率 / 旧策略概率）会严重偏离1，clip机制会频繁触发。
- 优势估计基于组内统计，外部数据破坏了组内一致性。
- KL散度约束可能失效，因为参考模型与当前策略的差异无法准确估计。

**故障/反作弊逻辑**：训练可能会发散，或者模型"学会"生成外部数据的模式，但无法泛化到新题目。

**价值/量化判定**：GRPO不适合混合外部数据，最终使得数据构造必须依赖规则奖励而非人工标注。

---

### 21. KL估计器K1、K2、K3是什么？正向KL和反向KL在大模型RL里有什么不同？

**场景/输入**：在约束策略与参考模型的差异时，需要估计KL散度。

**核心机制/计算流**：
- **K1（Forward KL）**：E_ref[log(ref/policy)]，期望在参考模型分布下计算。强迫策略覆盖参考模型的所有高概率区域。
- **K2（Reverse KL）**：E_policy[log(policy/ref)]，期望在当前策略分布下计算。允许策略忽略参考模型的低概率区域。
- **K3（JS散度）**：对称版本，较少使用。

**正向KL vs 反向KL**：
- 正向KL是"包容性"的，策略不能遗漏参考模型的任何高概率行为，可能导致过度分散。
- 反向KL是"选择性"的，策略可以专注于高奖励区域，忽略参考模型的其他行为，更保守。

**故障/反作弊逻辑**：大模型对齐几乎都用反向KL，因为：第一，我们希望模型在保持通用能力的同时，在特定任务上表现更好；第二，正向KL会强迫模型覆盖所有可能回答，导致训练不稳定。

**价值/量化判定**：GRPO使用反向KL约束，最终使得模型在数独任务上专业化，同时保持通用语言能力。

---

### 22. GRPO为什么不需要单独训练奖励模型RM？它的奖励从哪来？

**场景/输入**：在传统RLHF中，需要训练一个奖励模型来打分。

**核心机制/计算流**：GRPO的奖励来自**规则函数**而非神经网络。在我的项目中，7个奖励函数都是基于规则的：
- 格式奖励：检查标签是否闭合。
- 逻辑奖励：检查行/列/宫格的唯一值。
- 正确性奖励：检查答案是否与标准答案匹配。

这些奖励函数不需要训练，是确定性的代码逻辑。

**故障/反作弊逻辑**：规则奖励的缺点是设计困难，需要领域知识；优点是不需要收集偏好数据，不会过拟合。

**价值/量化判定**：规则奖励使得GRPO适合有明确评估标准的任务（如数学、代码、数独），最终使得项目能够在没有人工标注的情况下完成训练。

---

### 23. GRPO不稳定的原因，有什么可以改进的地方？

**场景/输入**：在GRPO训练中，可能遇到奖励震荡、KL散度飙升、模型崩溃等问题。

**核心机制/计算流**：不稳定的原因包括：
1. **组内方差大**：如果G=12个回答的奖励差异很小，优势估计不准，梯度噪声大。
2. **奖励稀疏**：早期训练时大部分回答得0分，梯度信号弱。
3. **KL约束不足**：beta=0.001可能不够强，模型偏离参考模型太远。

**改进方案**：
1. **增大组大小**：从12增加到16或24，降低优势估计方差。
2. **密集奖励设计**：如我的`exact_answer_reward_func`，部分匹配也给部分分。
3. **动态KL系数**：根据KL散度动态调整beta，KL大时增大惩罚。
4. **早停机制**：KL超过2.0连续20步自动停止。

**价值/量化判定**：这些改进使得GRPO训练稳定收敛，最终使得Pass@1从11%提升到73%。

---

## 第六部分：LoRA与微调

### 24. LoRA的原理是什么？如何初始化？

**场景/输入**：在微调大模型时，显存有限，无法更新全部参数。

**核心机制/计算流**：LoRA的核心假设是"微调发生在低维子空间"。具体来说：
- 原始权重矩阵W保持不变。
- 引入两个低秩矩阵A（输入维度 × 秩）和B（秩 × 输出维度），秩r远小于输入/输出维度。
- 前向传播时，输出 = W·输入 + (Alpha/r)·B·A·输入。
- A用随机高斯初始化，B用零初始化。这样初始时B·A=0，LoRA不产生任何影响，训练从参考模型开始。

**故障/反作弊逻辑**：如果B也用随机初始化，初始输出会偏离参考模型太远，训练不稳定。

**价值/量化判定**：LoRA将可训练参数量从全量的100%降到约1%，最终使得在单卡A100上能够微调3B模型。

---

### 25. LoRA的超参数有什么？效果不理想时如何改进？

**场景/输入**：LoRA训练效果不如预期，需要调优。

**核心机制/计算流**：主要超参数：
- **秩r**：控制低秩矩阵的维度，通常8、16、32。
- **Alpha**：缩放系数，通常等于r或2r。
- **目标模块**：哪些层应用LoRA，通常包括查询、键、值、输出投影。

**改进方案**：
- 效果差，过拟合：增大r，增加表达能力；减小Alpha/r比例，降低LoRA影响。
- 效果差，欠拟合：检查目标模块是否包含所有注意力层；增大学习率。
- 不稳定：开启dropout（如0.05），增加正则化。

### 27. 在推理阶段，为了消除LoRA带来的额外延迟，你会进行权重Merge吗？

## 模块一：多步推理轨迹受控与任务编排



**痛点判定（先说如果不这么做会发生什么）：**



大模型在做长链财务分析时，天然有一个倾向——它会在中间某个步骤遇到空数据、或者接口限流的时候，开始"自我脑补"，要么凭空生成一个看起来合理但完全错误的答案，要么就直接宣布"任务完成"其实根本没算完。这就是所谓的推理漂移和逻辑早停。



**[输入是什么]**



用户输入一句话：比如"分析茅台和五粮液过去三年的毛利率差异"。这句话被系统解析后，变成一个需要多步完成的任务序列——先拉茅台的营收和成本数据，再拉五粮液的，然后分别计算毛利率，最后做对比分析，共计四到五个独立步骤。



**[核心逻辑处理了什么]**

**[输出了什么]**



每个步骤的推进都有明确的门控验证，整条推理链路变成了可审计、可追溯的状态机，而不是一个不透明的黑盒子。最终效果是中等难度任务的Pass@1（也就是一次就给出正确答案的概率）达到了87.5%。

## 模块二：符号语义对齐与混合RAG优化



**痛点判定：**



在金融领域，大模型写代码时有一个典型错误——它知道你要查"股息率"，但它会在代码里写成`dividend_yield`，而真实接口的字段名叫`dv_ttm`。这两个词语义上完全一致，但符号上不等价，代码直接报错。这就是"字段幻觉"。如果不解决这个问题，系统会频繁因为字段名写错而触发异常，用户体验极差。



**[输入是什么]**



用户说"查一下茅台的股息率"，系统需要去API文档库里找到"股息率"对应的真实字段名，然后把它交给代码生成模块去用。



**[核心逻辑处理了什么]**



系统放弃了传统的"按接口文档整篇检索"的方式，而是把340篇接口文档拆解成了4032个最小语义单元——每一个单元只描述一个字段，比如"dv_ttm：近十二个月股息率，单位为百分比"。检索粒度从接口级下沉到了字段级。在查询端，系统还维护了一张金融同义词表，当用户说"股息率"的时候，系统会自动把这个词扩展成它的所有等价表达，再去匹配。检索采用混合策略：语义向量检索负责理解意图，BM25关键词检索负责精确命中字段名，两路结果融合排序。对于确实很冷门的查询，还有正则规则兜底，覆盖低频场景。



**[输出了什么]**



检索命中的不再是一篇文档，而是精确到字段级别的语义单元，直接告诉代码生成模块"这个意图对应的字段名是`dv_ttm`"。最终RAG Hit@3（前三个检索结果里有正确答案的概率）达到93.1%，代码解析失败率从5%降到了1%。

## 模块三：指令流/数据流分离内核与故障归因自愈



**痛点判定：**



多步任务有一个天然的上下文膨胀问题——每执行一步，如果把中间计算结果都塞进大模型的对话历史里，到第五步、第六步的时候，上下文已经几千个token了，大模型开始"记不住前面说了什么"，推理质量直线下降。另一个问题是，遇到报错怎么办？如果每次报错都无脑重试，碰上数据本来就不存在的情况（比如公司停牌），系统会陷入死循环。



**[输入是什么]**



一个多步任务执行到中间，第二步产出了一个数据帧，第三步需要用这个数据帧做进一步计算，同时第三步的代码报了一个错：`KeyError: 'revenue'`。



**[核心逻辑处理了什么]**



首先，数据不进大模型的上下文。第二步产出的数据帧被持久化在Python的命名空间（可以理解为一个内存变量池）里，第三步直接通过变量名去取，大模型的对话历史里只流转推理指令，不携带数据。这就是指令流和数据流的物理分离。对于报错，系统有一套根因分类机制：如果是字段名写错（字段幻觉），系统会压缩报错堆栈取关键的8行，加上正确的字段名提示，重新发给大模型修正代码；如果是数据本身不存在（比如公司停牌导致的数据真空），系统会直接把这个步骤标记为"不可重试"，跳过无效循环，在报告里如实说明数据缺失原因。



**[输出了什么]**



上下文长度保持在一个可控范围内，不随步骤数量线性膨胀。故障能被分类处理，而不是一刀切地无限重试。实测10个真实故障场景中，9个实现了零人工干预的自动修复，最终产出了茅台CAGR增长趋势解读、平安转型信号识别等有分析深度的报告。

系统放弃了传统的"按接口文档整篇检索"的方式，而是把340篇接口文档拆解成了4032个最小语义单元——每一个单元只描述一个字段，比如"dv_ttm：近十二个月股息率，单位为百分比"。检索粒度从接口级下沉到了字段级。在查询端，系统还维护了一张金融同义词表，当用户说"股息率"的时候，系统会自动把这个词扩展成它的所有等价表达，再去匹配。检索采用混合策略：语义向量检索负责理解意图，BM25关键词检索负责精确命中字段名，两路结果融合排序。对于确实很冷门的查询，还有正则规则兜底，覆盖低频场景。



**[输出了什么]**



检索命中的不再是一篇文档，而是精确到字段级别的语义单元，直接告诉代码生成模块"这个意图对应的字段名是`dv_ttm`"。最终RAG Hit@3（前三个检索结果里有正确答案的概率）达到93.1%，代码解析失败率从5%降到了1%。

## 模块三：指令流/数据流分离内核与故障归因自愈



**痛点判定：**



多步任务有一个天然的上下文膨胀问题——每执行一步，如果把中间计算结果都塞进大模型的对话历史里，到第五步、第六步的时候，上下文已经几千个token了，大模型开始"记不住前面说了什么"，推理质量直线下降。另一个问题是，遇到报错怎么办？如果每次报错都无脑重试，碰上数据本来就不存在的情况（比如公司停牌），系统会陷入死循环。



**[输入是什么]**



一个多步任务执行到中间，第二步产出了一个数据帧，第三步需要用这个数据帧做进一步计算，同时第三步的代码报了一个错：`KeyError: 'revenue'`。



**[核心逻辑处理了什么]**



首先，数据不进大模型的上下文。第二步产出的数据帧被持久化在Python的命名空间（可以理解为一个内存变量池）里，第三步直接通过变量名去取，大模型的对话历史里只流转推理指令，不携带数据。这就是指令流和数据流的物理分离。对于报错，系统有一套根因分类机制：如果是字段名写错（字段幻觉），系统会压缩报错堆栈取关键的8行，加上正确的字段名提示，重新发给大模型修正代码；如果是数据本身不存在（比如公司停牌导致的数据真空），系统会直接把这个步骤标记为"不可重试"，跳过无效循环，在报告里如实说明数据缺失原因。



**[输出了什么]**



上下文长度保持在一个可控范围内，不随步骤数量线性膨胀。故障能被分类处理，而不是一刀切地无限重试。实测10个真实故障场景中，9个实现了零人工干预的自动修复，最终产出了茅台CAGR增长趋势解读、平安转型信号识别等有分析深度的报告。

## 模块四：错题本驱动的在线策略对齐与记忆进化



**痛点判定：**



大模型有一个让人抓狂的特点：它会在同一类问题上反复犯一样的错。比如港股股票代码后面要加`.HK`，它忘了一次，你纠正了，下次它还是忘。如果每次都要靠人工干预来修正，这个系统就不能叫"智能"系统了。



**[输入是什么]**



用户问了一个跟上次失败案例高度相似的问题，比如同样是查港股的分红数据。



**[核心逻辑处理了什么]**



系统在每次任务执行结束后，会自动归纳这次轨迹是成功还是失败，失败的原因是什么，成功的关键步骤是什么，并把这条经验存入"错题本"。当新的请求进来时，系统用关键词匹配判断这个请求跟哪些历史经验相关，然后把相关经验（上限1500个字符，确保不撑大上下文）动态注入到系统提示词里，相当于在推理之前就把"上次踩过的坑"告诉大模型。用户的个人偏好（比如他喜欢用表格格式、关注净利润而不是营收）由一个独立节点在任务完成后提取并持久化，下次自动适配，不需要用户重复说明。



**[输出了什么]**



系统具备了跨会话的在线纠偏能力，同类错误的重复率显著下降。每一条分析结论都带有`[DATA]:`协议标注的数据来源锚点，可以逐条追溯到具体的API调用和字段值，报告的可信度是可验证的，而不是大模型编出来的。
