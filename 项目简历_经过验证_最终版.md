# 智能金融数据分析多Agent系统（ASA v2.0）- 项目简历

## 项目概述

设计并实现了基于LangGraph的**Supervisor多Agent架构**，集成动态技能加载、自适应错误处理、金融逻辑审查的金融数据分析系统。面向股息率、分红、送转等复杂金融指标的语义理解与数据查询。项目核心代码7800+行，已通过实际压测验证架构可行性和瓶颈定位。

## 核心技术方案

### 1. 多Agent编排框架（LangGraph Supervisor Pattern）

**文件**：`multi_agent.py` 第317-336行

**架构设计**：

- **Supervisor节点**：中心化路由器，基于`MultiAgentState`全局上下文分配任务
- **Coder节点**：代码生成与API调用，集成动态技能注入
- **Reviewer节点**：金融逻辑审查，识别数据异常
- **ErrorHandler节点**：4级Backtracking策略（优先级降级）
- **ProfileUpdater节点**：用户画像动态更新

**关键实现**（代码验证）：

```python
class MultiAgentState(TypedDict):
    """Multi-Agent状态容器"""
    messages: Annotated[List[BaseMessage], operator.add]  # 消息累积
    next: str  # 下一步节点(Supervisor/Coder/Reviewer/FINISH)
    retry_count: int  # 重试计数
    remaining_steps: list  # 剩余任务队列
    ...
```

**技术亮点**：使用`operator.add`实现消息列表的原子操作累积，避免消息重复或遗漏。

### 7. Thread-aware Kernel隔离

**文件**：`lib.py` 第481-548行（KernelManager类）

**机制**：

```python
class KernelManager:
    """为每个thread_id提供隔离的执行环境"""
    def __init__(self):
        self._kernels: Dict[str, Union[StatefulPythonKernel, OpenSandboxKernel]] = {}
        self._use_sandbox = os.getenv("USE_ASA_SANDBOX", "false").lower() == "true"
    
    def get_kernel(self, thread_id: str):
        if thread_id not in self._kernels:
            if self._use_sandbox and OPENSANDBOX_AVAILABLE:
                self._kernels[thread_id] = OpenSandboxKernel()
            else:
                self._kernels[thread_id] = StatefulPythonKernel()
        return self._kernels[thread_id]
```

**效果**：

- **隔离性**：每个thread_id拥有独立的Python命名空间（`self.globals`字典）
- **持久性**：同一用户多次请求共享同一Kernel，支持多轮对话中的变量接力
- **扩展性**：支持切换为OpenSandbox沙箱内核

## 已知问题与改进方向

### 问题1：递归循环控制不足 ⚠️

**现象**：数据真空时，Coder→ErrorHandler→Supervisor循环至超限

**根本原因**：
- ErrorHandler无状态序列审计
- 缺乏"连续N次无数据→自动熔断"的逻辑

**改进方案**（已识别，待实现）：

```python
class ErrorHandler:
    def check_loop_pattern(self, last_k_steps):
        # 检查最近5步是否形成"Coder→ErrorHandler"循环
        if all(step.node in ["Coder", "ErrorHandler"] for step in last_k_steps[-5:]):
            return self.fallback_strategy()  # 触发L4降级
        return self.retry()
```

**预期效果**：将系统稳定性从"0%成功率"向"部分降级可用"推进

### 问题2：Tushare API数据真空

**现象**：某些股票分红数据为空（可能因上市时间短、无分红历史、或API同步延迟）

**当前处理**：重试无效，最终超限失败

**需要**：多数据源备选、部分结果降级返回

### 问题3：OpenSandbox集成预留 ⚠️

**现象**：`lib.py`第32-38行import OpenSandbox但失败时fallback

**影响**：代码执行仍使用本地StatefulPythonKernel，无沙箱隔离

**改进**：若需沙箱隔离，需完成OpenSandbox SDK集成与权限配置

## 面试实证答题（基于代码验证）

### Q1：我看压测日志里出现了Recursion limit报错，怎么回事？

**答**：这是一个非常真实的工程问题。系统在处理"数据真空"场景时（如查询尚未披露的财报），会陷入Coder→ErrorHandler→Supervisor→Coder的循环。日志清晰地显示：

- **Step 1-3**：Supervisor正确理解意图，Coder成功加载了dividend_expert技能 ✅
- **Step 4-5**：代码逻辑正确，但Tushare API返回空数据 ❌
- **Step 6-25**：ErrorHandler逐级尝试降级（调整参数、切换接口、尝试备选源），但都失败了
- **第25步超限**：系统无法判断"何时放弃"，因此触发递归限制

这证明我不仅理解LLM和多Agent架构，更重要的是**能精准定位工程瓶颈并提出解决方案**——引入状态序列审计，在"连续3次Coder→ErrorHandler"时强制熔断。

### Q2：你的load_skill逻辑流转是怎样的？

**答**：这是一个完整的三步流程：

1. **Supervisor识别意图**（multi_agent.py第320行的next字段）：根据用户query的关键词（如"分红""股息率"），Supervisor判断"需要调用Coder"并在RouteResponse的reason中建议"加载分红专家技能"

2. **Coder调用load_skill工具**（第450+行）：Coder接收建议后，主动调用`load_skill("dividend_expert")`工具函数

3. **Tools返回技能指导**（skills.json第2-8行）：该函数物理读取skills.json，返回包含"单位别乘100""状态过滤规则"等硬约束的技能包。这些规则以ToolMessage的形式注入Coder的上下文

**效果**：Coder在写代码的前一秒，Context里已经包含了最纯净、最精准的业务规则，大幅提升了API调用的准确性。

### Q3：为什么不用LangChain默认的Memory隔离？

**答**：LangChain的默认Memory（如ConversationBufferMemory）只解决了**聊天历史的管理**，但忽视了**物理计算环境的隔离**。

在金融数据分析场景中，多个用户可能并发查询同一只股票，如果共享同一个Python执行环境，会出现变量污染：用户A的`df`可能被用户B覆盖。

我在`lib.py`第481-548行实现了`KernelManager`：

```python
class KernelManager:
    def __init__(self):
        self._kernels: Dict[str, Union[StatefulPythonKernel, OpenSandboxKernel]] = {}
    
    def get_kernel(self, thread_id: str):
        if thread_id not in self._kernels:
            self._kernels[thread_id] = StatefulPythonKernel()  # 为每个thread_id创建独立内核
        return self._kernels[thread_id]
```

**关键机制**：每个thread_id映射一个独立的Python字典命名空间（StatefulPythonKernel的`self.globals`）。这样既实现了**多用户并发下的变量防污染**（thread隔离），又保留了**同一用户多轮对话间的变量接力**（session持久化）。
