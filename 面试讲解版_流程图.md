# ASA多智能体系统 - 面试讲解版（含流程图）

## 核心洞察
这个系统解决的问题：**LLM容易在金融数据查询中出现三类错误**
1. **代码错误**（KeyError、TypeError）→ 需要RAG辅助纠错
2. **数据真空**（停牌、退市）→ 需要优雅降级
3. **无限重试**（死循环）→ 需要4级自愈机制

## 第二部分：混合检索的均衡性设计

### 流程图：向量检索 vs BM25 的权衡

```
┌────────────────────────────────────────────────────────────────┐
│ 用户问：dv_ttm的含义                                            │
└────────────────────────────────────────────────────────────────┘
                         ↓
    ┌────────────────────┴────────────────────┐
    │                                         │
    ▼                                         ▼
【向量检索】                          【BM25精确匹配】
BGE-M3 Embedding                    分词+倒排索引
                                     
查询转向量：                          查询分词：
"dv_ttm含义"                         ["dv", "ttm", "含", "义"]
  → 向量 v = [0.2, 0.5, ...]        
                                     与语料库对比：
计算余弦相似度：                      找出包含"dv_ttm"的文档
                                     
得分范围：0-1                         得分范围：0-1000+
(Cosine相似度)                       (BM25原始分数)
                                     
Top-20文档：                         Top-20文档：
[                                   [
  ("doc1", 0.89),  ← 高              ("doc1", 456),  ← 大
  ("doc2", 0.76),                    ("doc2", 234),
  ("doc3", 0.65),                    ("doc3", 123),
  ...                                ...
]                                   ]

    │                                 │
    └────────────────────┬────────────┘
                         ▼
        【混合评分：Min-Max归一化】
        
        Step 1：向量分归一化
        向量最高分：0.89
        doc1: 0.89/0.89 = 1.0
        doc2: 0.76/0.89 = 0.85
        
        Step 2：BM25分归一化
        BM25最高分：456，最低分：123
        范围：456 - 123 = 333
        doc1: (456 - 123) / 333 = 1.0
        doc2: (234 - 123) / 333 = 0.33
        
        Step 3：加权求和（向量权0.7，BM25权0.3）
        doc1_混合分 = 1.0 * 0.7 + 1.0 * 0.3 = 1.0 ✓
        doc2_混合分 = 0.85 * 0.7 + 0.33 * 0.3 = 0.595 + 0.099 = 0.694
        
        Step 4：排序取Top-5
        最终返回：
        [doc1, doc2, doc3, doc4, doc5]
        
```

### 关键问题预判

**面试官可能问：为什么要Min-Max归一化？**

```
问题：BM25分值太大（100-1000），向量分太小（0-1）
      怎么比较？
      
答案：Min-Max归一化
      把两个分值都缩放到0-1区间
      
      向量分：原本0.89，归一化后仍是0.89（已是0-1）
      BM25分：原本456，先减去min(123)=333，再除以范围(333)
              最终(456-123)/333 = 1.0
      
好处：
      1. 两者数值可比
      2. 防止大分值掩盖小分值
      3. 权重有实际意义（0.7 vs 0.3）
```

## 第三部分：高级应用 - 带上下文的精准检索

### 流程图：ErrorHandler的精准RAG策略

```
┌──────────────────────────────────────────────────────────────────┐
│ 【错误场景】Coder查询失败多次                                      │
│ 第1次：df['wrong_field1'] → KeyError                             │
│ 第2次：df['wrong_field2'] → KeyError                             │
│ 现在第3次还要再试？                                               │
└──────────────────────────────────────────────────────────────────┘
                         ↓
┌──────────────────────────────────────────────────────────────────┐
│ 【ErrorHandler Level 1】                                         │
│                                                                  │
│ 前2次都失败了，但我还是尝试RAG纠错                               │
│ search("daily 接口 字段 wrong_field2", top=2)                   │
│ → 搜索空间很大（几百篇文档），可能搜不准                         │
└──────────────────────────────────────────────────────────────────┘
                         ↓
          retry_count >= 3 且 retry_count 递增
                         ↓
┌──────────────────────────────────────────────────────────────────┐
│ 【ErrorHandler Level 2】策略切换                                  │
│                                                                  │
│ BacktrackingRouter.mark_failure() 记录失败                       │
│ → 认知：用daily接口一直失败，可能这个API不对                     │
│                                                                  │
│ BacktrackingRouter.get_next_strategy()                           │
│ → 返回备选策略：试试daily_basic接口？或扩大日期范围？            │
│                                                                  │
│ 带上下文的精准检索：                                             │
│ 旧搜索：search("字段名")                                         │
│        → 几百篇文档，容易漂移                                    │
│                                                                  │
│ 新搜索：search("daily_basic 接口 close", top=2)                 │
│        → 搜索空间从几百压缩到2-3篇                               │
│        → BM25精确匹配"daily_basic"，即便权重不变也能锁定答案    │
└──────────────────────────────────────────────────────────────────┘
                         ↓
        Coder收到新的API建议 + 上下文
                         ↓
┌──────────────────────────────────────────────────────────────────┐
│ 【第3次重试】用新API                                              │
│ df = pro.daily_basic(...)['close']  ← 换API                     │
│ → 成功！返回数据                                                 │
└──────────────────────────────────────────────────────────────────┘
```

### 为什么"带上下文"很关键

```
场景：查询"ROE"字段

❌ 坏方案：
   search("ROE", top=5)
   → 返回所有提到"ROE"的文档
   → 可能包括：
     - fina_indicator的ROE
     - fina_mainbusiness的ROE
     - 论文中的ROE概念讨论
     - 总共10篇，容易混淆

✓ 好方案：
   search("fina_indicator 接口 ROE", top=2)  ← 带API名称
   → BM25精确匹配"fina_indicator" + "ROE"
   → 直接返回2篇相关文档
   → 上下文清晰，Coder不会再搞错
```

## 第四部分：4级自愈机制 - 防止无限重试

### 流程图：逐级加重惩罚

```
┌─────────────────────────────────────────────────────────────────┐
│ 错误发生：Coder执行异常                                           │
└─────────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────────┐
│ 【Level 1】Immediate Retry（立即重试）                           │
│ ✓ 条件：recovery_level == 1 且 retries < 3                      │
│ ✓ 动作：提供RAG纠错提示                                          │
│ ✓ 成本：token成本最低（只提示，不重建）                         │
│                                                                │
│ 流程：                                                          │
│ retry_count++ (从0→1→2→3)                                      │
│ 若 retry_count >= 3 且还是错：                                  │
│   recovery_level++ (1→2)                                       │
│   进入Level 2                                                  │
└─────────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────────┐
│ 【Level 2】Strategy Switch（策略切换）                           │
│ ✓ 条件：recovery_level == 2                                     │
│ ✓ 动作：BacktrackingRouter切换API或参数                         │
│ ✓ 成本：token成本中等（换个思路重新尝试）                        │
│                                                                │
│ BacktrackingRouter有内置策略库：                                │
│ [                                                              │
│   {"name": "daily接口", "params": {...}},     ← 第1个策略      │
│   {"name": "daily_basic接口", "params": {...}}, ← 第2个策略    │
│   {"name": "扩大日期范围", "params": {...}},    ← 第3个策略    │
│   ...                                          ← 最多5个策略   │
│ ]                                                              │
│                                                                │
│ 流程：                                                          │
│ BacktrackingRouter.mark_failure() 记录第1个策略失败              │
│ BacktrackingRouter.get_next_strategy() 返回第2个策略             │
│ 若 所有策略都试完还是错：                                       │
│   recovery_level++ (2→3)                                       │
│   进入Level 3                                                  │
└─────────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────────┐
│ 【Level 3】Graceful Degradation（优雅降级）                     │
│ ✓ 条件：recovery_level == 3                                     │
│ ✓ 动作：停止重试，返回部分结果                                   │
│ ✓ 成本：0（不再调用Coder，转到Reviewer生成报告）                │
│                                                                │
│ 降级方案：                                                      │
│ 1. 返回缓存数据：有历史查询结果吗？返回上次的数据                │
│ 2. 返回部分结果：能查到一半吗？返回那一半                        │
│ 3. 返回诊断信息：告诉用户"无法查询，可能停牌"                    │
│                                                                │
│ 流程：                                                          │
│ degraded_msg = f"""[PARTIAL RESULT]                            │
│ 原始查询：{user_query}                                          │
│ 降级方案：{fallback_strategy['hint']}                          │
│ 最后错误：{error_msg}                                           │
│ """                                                             │
│ return {"next": "Reviewer", "execution_status": "success"}     │
│        ↑ 注意：即便有错，也标记为"success"，因为降级是有意的    │
└─────────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────────┐
│ 【Level 4】Reject（拒绝处理 + 诊断报告）                         │
│ ✓ 条件：recovery_level >= 4                                     │
│ ✓ 动作：停止所有重试，返回诊断报告给用户                         │
│ ✓ 成本：0（完全停止）                                           │
│                                                                │
│ 诊断报告内容：                                                   │
│ [REJECT]: 已重试 9 次，无法解决                                 │
│ - 尝试策略 1：daily接口 → KeyError: 'xxx'                      │
│ - 尝试策略 2：daily_basic接口 → KeyError: 'yyy'               │
│ - 尝试降级：缓存为空，无可用数据                                │
│ - 建议：请检查输入参数或联系系统管理员                          │
│                                                                │
│ recovery_history 示例：                                        │
│ [                                                              │
│   {"level": 1, "error": "KeyError: net_profit", ...},         │
│   {"level": 1, "error": "KeyError: n_income", ...},           │
│   {"level": 1, "error": "KeyError: profit", ...},             │
│   {"level": 2, "strategy": "daily_basic", ...},               │
│   {"level": 2, "strategy": "expand_date_range", ...},         │
│   {"level": 3, "fallback": "use_cache_or_reject", ...},       │
│   {"level": 4, "action": "REJECT", ...}                       │
│ ]                                                              │
│                                                                │
│ 这份报告会被推送到监控后台，供人工分析：                        │
│ - 系统遇到了什么底层问题？                                     │
│ - 是Prompt设计的问题？还是API文档的问题？                       │
│ - 下次如何改进？                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 关键数字

```
Level 1 重试上限：3次
Level 2 策略上限：5个备选策略
Level 3 降级：1次
Level 4 拒绝：立即停止

总耗时成本：
- Level 1：3 × 50ms = 150ms（token成本：纠错提示）
- Level 2：5 × 100ms = 500ms（token成本：新的API提示）
- Level 3：1 × 50ms = 50ms（token成本：降级方案说明）
- Level 4：0ms（不再调用LLM）

最坏情况：150 + 500 + 50 = 700ms，然后明确拒绝，不会无限循环
```

## 第五部分：全局检索器的并发安全设计

### 流程图：延迟初始化 + 内存复用

```
┌──────────────────────────────────────┐
│ 系统启动                              │
│ global_retriever = None              │
└──────────────────────────────────────┘
                 ↓
        【没有任何初始化成本】
        系统启动快速，不卡顿
                 ↓
    ┌────────────┴────────────┬─────────────┐
    │            │            │             │
    ▼            ▼            ▼             ▼
【线程1】    【线程2】     【线程3】    【线程4】
查询1        查询2         查询3         查询4


【第一次search被调用】（比如线程1）
  ↓
  if global_retriever is None:
      initialize_retriever(use_gpu=False)
      ↓
      Step 1：加载Tushare CSV（几秒）
      Step 2：初始化BM25倒排索引（几秒）
      Step 3：加载ChromaDB向量库（如果已存在）
             或 创建新向量索引（需要GPU，30-60秒）
      ↓
      global_retriever = <retriever对象>

  return retriever.search(query)


【之后的search调用】（线程2、3、4）
  ↓
  global_retriever 已存在
  ↓
  直接使用内存中的 retriever 对象
  ↓
  检索延迟：100ms（GPU）或 300ms（CPU）
  
  
【ChromaDB持久化的作用】
  ↓
  第一次创建索引：
    Coder → "查茅台PE"
    → BGE-M3计算embedding → 存入 /chroma_db/
    → 耗时30秒
    
  之后启动：
    加载 /chroma_db/ 目录
    → 直接读取预计算的embedding
    → 无需重新计算
    → 耗时几秒

【文档更新（生产设计）】
  ↓
  旧的实现：全量重建（很浪费）
  
  新的实现：基于Hash的增量更新
  ┌─────────────────────────────────────┐
  │ 每个文档都有Hash值（存在metadata中）│
  │ 文档内容：{"title": "daily接口", ...}│
  │ 计算Hash：md5(文档内容) = "abc123"  │
  │                                     │
  │ 下次更新检查：                       │
  │ - 新文档Hash != 旧文档Hash            │
  │   → 只重新embedding这一篇           │
  │ - 新文档Hash == 旧文档Hash           │
  │   → 跳过，复用旧embedding          │
  │                                     │
  │ ChromaDB.upsert()：                 │
  │ - 如果文档id存在：更新              │
  │ - 如果文档id不存在：新增            │
  │ - 自动处理向量重建                  │
  └─────────────────────────────────────┘
```

### 高并发下的一致性保证

```
场景：10个线程同时调用search

【线程1】 【线程2】 【线程3】 ... 【线程10】
  │        │        │              │
  └────────┴────────┴──────────────┘
          all call search()
                │
        ┌───────┴────────┐
        │                │
        ▼                ▼
   【第一次】         【之后9次】
initialize_retriever  retriever已存在
(锁定资源)            (直接用内存引用)
     │                      │
     └──────────┬───────────┘
                ▼
        all use same retriever
        
关键：Python全局变量保证了single instance
     即便多线程访问，也是同一个对象


【为什么GPU很重要】
没GPU（CPU检索）：
  embedding计算：1秒/篇
  10000篇文档：10000秒 = 2.7小时
  高并发下，response时间：300ms

有GPU（GPU检索）：
  embedding计算：100ms/篇
  10000篇文档：1000秒 = 17分钟
  高并发下，response时间：100ms

use_gpu=True 的收益：
  - 初始化时间：减少30倍
  - 并发响应：快3倍
```

## 面试讲解指南

### 讲解顺序（10分钟）

```
1分钟：系统整体设计
    "这个系统核心解决3个问题：代码错误、数据真空、无限重试"
    
2分钟：Coder反思能力（流程图 Part 1）
    "当KeyError时，ErrorHandler不是直接告诉Coder答案，
     而是用RAG搜索文档，然后让Coder自己推导。
     这样Coder会真正理解，而不是机械替换"
    
2分钟：混合检索的均衡设计（流程图 Part 2）
    "为什么既要向量又要BM25？一个擅长语义，一个擅长精确。
     通过Min-Max归一化，让两者权重合理分配"
    
2分钟：4级自愈机制（流程图 Part 4）
    "Level 1重试、Level 2策略切换、Level 3降级、Level 4拒绝。
     每级失败都自动升级，recovery_level递增确保不死循环"
    
1分钟：高并发设计（流程图 Part 5）
    "延迟初始化 + 内存复用，全局只有一个retriever对象。
     ChromaDB持久化避免每次都重建索引"
    
2分钟：答面试官的"但是"
    - "但是如果正则失效呢？" → Level 2策略切换
    - "但是如果Level 2还失败呢？" → Level 3降级
    - "但是如果4级都失败呢？" → Level 4拒绝，推送诊断报告
```

### 面试官可能的追问

```
Q1: "为什么不直接告诉Coder答案？"
A: "我想让Coder理解原理，而不是机械替换。
   通过RAG文档 + 错误反馈，Coder会推导出正确答案。
   这样下次遇到类似问题，Coder能自己解决"
   
Q2: "如果RAG搜错了怎么办？"
A: "Level 2会介入，BacktrackingRouter切换API或参数。
   或者进入Level 3降级，返回缓存或部分结果"
   
Q3: "但这样成本不会很高吗？"
A: "Level 1成本最低（只提示），Level 2成本中等（切换策略），
   Level 3成本为零（不再调用LLM）。
   而且大多数查询能在Level 1就成功，不会进到Level 2"
   
Q4: "为什么要Min-Max归一化？"
A: "BM25分值范围是0-1000+，向量分范围是0-1。
   直接加权会让BM25掩盖向量信号。
   归一化后两者都是0-1，权重才有意义"
   
Q5: "高并发下会有race condition吗？"
A: "不会。Python全局变量在单进程内是thread-safe的。
   initialize_retriever被调用第一次时才初始化，
   之后所有线程复用同一个retriever对象。
   ChromaDB本身支持并发读"
```
