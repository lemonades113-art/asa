# API文档RAG：从粒度对齐到领域适配

---

## 零、我的系统是什么？

**面试官问**："你的RAG是匹配什么？是字段名还是字段描述？"

---

"我的RAG系统是一个**金融API文档检索器**，核心目标是让LLM生成正确的API调用代码。

先回答你的问题：**匹配的是两者**。

我的Chunk长这样：

```
利润表接口 字段: n_income (float) - 净利润
每日指标 字段: pe_ttm (float) - 滚动市盈率
```

每个Chunk同时包含：
- **字段名**（英文）：`n_income`、`pe_ttm` —— Tushare API实际使用的名字
- **字段描述**（中文）：`净利润`、`滚动市盈率` —— 人工可读的描述

**一个Chunk = 一个向量**。我把340篇API文档拆成4032个字段级Chunk，每个Chunk独立embed。

用户问"净利润"，我的Query扩展会把它变成：

```
"净利润 n_income net_profit 归母净利润 扣非净利润"
```

然后去匹配Chunk。BM25精确匹配命中`n_income`和`净利润`，FAISS语义匹配也命中。

**这就是双向对齐**：用户输入中文，扩展出英文，Chunk里中英文都有，两边都能匹配上。

## 一、和前沿工作的对比

### 1. OpenAPI Chunking论文（2024.11）

**他们研究的问题**：OpenAPI文档怎么做Chunking？endpoint级别还是path级别？

**他们的发现**：
- **Endpoint-level chunking**：把整个endpoint作为一个chunk，信息完整但粒度粗
- **Path-level chunking**：按path拆分，粒度更细
- **结论**：不同任务需要不同粒度，没有最优解

**我的做法**：
- 我做的是**Field-level chunking**——比endpoint更细一级
- 因为金融API的字段是"原子操作单元"：用户问"净利润"，我要返回的是`n_income`这个字段名，不是整个接口文档

**为什么更细？因为金融场景的需求粒度就是字段级**。用户问"查茅台的市盈率"，这是一个字段级的问题。如果我返回整个`daily_basic`接口的文档（500字），用户还要从里面找市盈率字段——这是粒度错配。

---

### 2. RepoAgent（EMNLP 2024 Demo）

**他们解决的问题**：怎么自动生成代码仓库的文档？

**他们的核心设计**：
- **AST解析**：用抽象语法树分析代码结构，提取函数、类、模块
- **对象级文档**：为每个代码对象生成独立文档
- **双向调用关系**：识别函数之间的调用关系，丰富文档上下文

**对比我的做法**：

| 维度 | RepoAgent | 我的系统 |
|------|-----------|----------|
| 数据源 | 代码仓库（Python文件） | API文档（CSV） |
| 解析方式 | AST语法树 | 正则+表格解析 |
| Chunk粒度 | 对象级（函数/类） | 字段级 |
| 目标 | 生成文档 | 检索字段名 |

**可以借鉴的地方**：
- RepoAgent用AST解析代码结构——这是"结构化理解"
- 我可以类比：我用表格解析API文档结构，提取字段名和描述的关系
- **本质相同：都是把非结构化文本变成结构化知识**

---

### 3. CodeRAG（EMNLP 2025）

**他们解决的问题**：Repository-level代码生成时，怎么检索"支撑代码"？

**他们的核心设计**：

```
Requirement Graph ←→ DS-Code Graph
      ↓                    ↓
  需求节点            代码节点
      ↓                    ↓
   子需求              被调用的函数
   相似需求            语义相似的代码
```

**双图映射**：
- Requirement Graph：建模需求之间的父子关系、相似关系
- DS-Code Graph：建模代码之间的依赖关系、语义关系
- **映射**：把需求节点映射到代码节点，找到"应该被参考的代码"

**对比我的做法**：

| 维度 | CodeRAG | 我的系统 |
|------|---------|----------|
| 图建模 | 需求图+代码图，双图映射 | 无图，扁平检索 |
| 检索目标 | 支撑代码（被调用函数） | 字段名+描述 |
| 召回策略 | 图遍历+外部搜索 | BM25+FAISS混合 |

**CodeRAG更复杂，但我更垂直**：
- CodeRAG解决的是通用代码生成——要考虑函数调用、模块依赖
- 我解决的是API调用——核心就是字段名对齐，不需要复杂的图结构

**可以借鉴的地方**：
- CodeRAG的"需求→代码"映射思想，我可以类比成"用户意图→字段名"映射
- 我的`FINANCIAL_SYNONYMS`字典就是一种简化的映射：中文概念→英文字段名

### 4. API Documentation RAG论文（2025.03）

**他们研究的问题**：LLM用冷门API时，RAG能帮多少？

**他们的核心发现**：

1. **RAG有效**：冷门API的正确使用率提升83%~220%
2. **BM25好用**：对于代码补全任务，BM25比向量检索更可靠
3. **示例代码最重要**：API文档里，示例代码对生成质量贡献最大
4. **LLM能容忍噪声**：文档里有错误信息，LLM也能从上下文推断

**这验证了我的设计**：
- 我用BM25+FAISS混合检索，BM25权重0.3——论文说BM25好用，我做对了
- 我的Chunk包含字段描述——相当于"微型示例"，告诉LLM这个字段是干什么的

**但我缺少的**：
- 论文说示例代码最重要——我的Chunk里没有代码示例
- 后续改进：可以在Chunk里加入"使用示例"，比如`n_income`字段的典型用法

## 二、我的设计的本质

**面试官问**："你为什么要做Field-level Chunking？"

---

"这个问题我从两个角度回答。

**第一，问题的本质是什么？**

用户问"查茅台的净利润"，这是一个**字段级的问题**。

用户想知道的是：Tushare里"净利润"对应的字段名是什么？是`n_income`？`net_profit`？还是别的？

如果我用传统的文档级Chunk，返回整个`income`接口文档（500字），用户（或LLM）要从里面找到净利润字段——这是**粒度错配**。

正确的做法是：**让检索粒度和问题粒度对齐**。字段级问题 → 字段级Chunk。

这和**ColBERT的late interaction**是同一个逻辑。ColBERT保留token级表示，我保留字段级表示。底层原理一致：**检索的最小单元要和问题的最小单元对齐**。

**第二，为什么不用更复杂的结构？**

CodeRAG用了双图映射——需求图+代码图。RepoAgent用了AST解析——语法树遍历。

我没用这些，因为**金融API文档是表格结构，不需要复杂的解析**。

API文档天然就有"字段名+类型+描述"的表格，我只需要把表格每一行拆出来变成Chunk。这不是偷懒，是**适配数据结构**——如果数据源是代码仓库，我会用AST；如果数据源是API文档，我就用表格解析。

**这体现了一个设计原则：方案复杂度要匹配问题复杂度**。金融API的字段检索是一个相对简单的问题，用相对简单的方案就够了。"

```

    ↓

    ↓

    ↓

```

## 五、可改进方向

1. **加入示例代码**：API Documentation RAG论文说示例代码最重要——可以在Chunk里加入典型用法

2. **图建模**：如果要考虑接口之间的依赖关系（比如`income`接口需要先用`daily`接口获取ts_code），可以引入CodeRAG的双图映射思想

3. **AST解析**：如果数据源变成代码仓库而不是API文档，可以引入RepoAgent的AST解析

4. **动态权重**：当前BM25权重固定0.3，可以根据查询类型动态调整——精确查询用更高BM25权重，语义查询用更高向量权重
