项目一：GRPO符号推理——框架选型

面试官问：你为什么选GRPO而不是PPO或DPO？这三个方法的核心区别是什么？

我从三个维度来回答：机制差异、适用场景、我的选择依据。

第一，机制层面的核心差异

PPO

完整链条：先训练Reward Model → 再用RM给LLM输出打分 → 用policy gradient更新LLM。关键点：需要Critic模型来估计value function，计算advantage：

$$A_t = \sum_{l=0}^{\infty} \gamma^l r_{t+l} - V(s_t)$$

右边第一项是实际累积奖励，第二项是Critic估计的期望奖励。两者的差就是advantage——这个动作比期望好多少。Critic降低了variance，但代价是要额外训练和维护一个模型。

DPO

把RL问题直接转化成偏好学习问题。核心观察：如果有偏好数据$(x, y_w, y_l)$，可以直接从偏好数据中学policy，不需要显式训练RM。目标函数：

$$\mathcal{L}_{DPO} = -\mathbb{E} \left[ \log \sigma \left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)} \right) \right]$$

不需要在线采样、不需要RM、不需要Critic——但代价是依赖高质量的偏好对数据，且无法处理"两个答案都不对"的情况。

GRPO（我的选择）

DeepSeek在R1论文里提出，核心创新是用group内的均值替代Critic。对于一个prompt，采样G个response，计算每个的reward，用group mean做baseline：

$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_j\}_{j=1}^G)}{\text{std}(\{r_j\}_{j=1}^G)}$$

不需要训练Critic，baseline是从采样中直接算出来的。

第二，适用场景的差异

方法适用场景不适用场景
PPO需要精细控制、长期规划的任务（对话、游戏）显存受限、Critic难训练的场景
DPO有高质量偏好对数据、offline学习答案可验证但不易比较优劣的任务
GRPO答案可验证、显存受限需要长期credit assignment的任务

我的数独场景特点：① 答案可验证——填完逐格检查对错，reward信号明确；② 推理链相对短——不需要算几十步之后的收益；③ 资源受限——3B模型、单卡A100，显存吃紧。这三个特点让GRPO成为最优选。

第三，我的实际选择过程

为什么没选DPO：数独场景下，同一题目的两个解，可能一个错5格、一个错20格——都是错的，但"错得更少"和"错得更多"之间，DPO会强制模型学习"错得更少的是preferred"，这个信号有噪声。GRPO的设计天然适合：一个prompt生成G=4个response，按正确率排序，group内的相对排序比绝对偏好更可靠。

为什么没选PPO：PPO需要同时加载policy模型、reference模型、Critic模型、Reward模型——3B规模下四个模型光是参数就占48GB，单卡根本放不下。GRPO只需要policy和reference，RM用规则实现，省掉两个模型。

学术关联

DeepSeek-R1论文（Incentivizing Reasoning Capability in LLMs via Reinforcement Learning）里，他们用GRPO训练数学推理，在AIME 2024上达到和PPO相当的效果，显存开销低约30%。我的实现基本follow他们的设置：G=4，group内归一化，KL约束系数beta=0.001。

总结：选GRPO不是因为它"更先进"，而是因为它和我的任务特性最匹配——答案可验证、资源受限、推理链短。每个方法的选型都应该基于任务特性，而不是哪个更fancy。

GRPO去掉Critic的代价 · Reward Hacking设计

GRPO去掉Critic的代价

追问：GRPO去掉Critic的代价是什么？方差问题怎么解决？

代价是方差更大，这直接影响训练稳定性和收敛速度。

理解Critic的作用

Critic的核心作用是降低policy gradient的方差。在REINFORCE（无Critic）里，advantage直接用return：$\hat{A}_t = \sum_{l=0}^{T} \gamma^l r_{t+l}$，方差极大——同一状态同一动作，因为后续采样路径不同，return可能差几倍。

PPO引入Critic后：$A_t = \sum_{l=0}^{\infty} \gamma^l r_{t+l} - V(s_t)$，$V(s_t)$是学出来的baseline，它估计"从状态s出发期望能拿多少分"。因为V是在很多样本上训练的，比单次return更稳定——方差降低了。

GRPO的替代方案和代价

GRPO没有Critic，用group mean当baseline：$\hat{A}_i = \frac{r_i - \text{mean}}{\text{std}}$

两个问题：① 采样噪声：如果G太小（G=2），group mean可能和真实期望差很远；② 方差传导：如果G个response的reward都很接近，std接近0，advantage会爆炸（实现里加了epsilon=1e-8防止除零）。

我的解决方案

方案内容效果
增大G值设置G=4（DeepSeek论文默认值）。G=2时训练不稳定（KL波动大），G=4后稳定下来。代价是每次需要采样更多response。KL偏离度波动从0.1-0.8降到0.3-0.4
KL散度约束beta=0.001的KL约束，限制策略偏离reference太远：$L = \mathbb{E}[A_t] - \beta \cdot \text{KL}(\pi_\theta \| \pi_{ref})$相当于给高方差梯度加了"安全带"
梯度裁剪PPO clip机制：$\text{clip}(r_t, 1-\epsilon, 1+\epsilon) \cdot A_t$，$\epsilon=0.2$防止单次更新步子迈太大

Reward Hacking与奖励设计

面试官问：你说"线索保真奖励"，具体怎么设计的？这是为了解决什么问题？

问题发现的过程

训练初期，发现一个奇怪现象：模型的reward曲线在涨，但正确率没涨。仔细看生成结果，发现模型在篡改原题。

数独题目有已填好的数字（clues）和空白格。正常推理是填空白格，但模型发现：把原题的clue改成一个错误的数字，再在错误基础上"推理"，就能骗过格式检查拿到partial reward。这是典型的environment tampering——模型不是在解决问题，而是在改问题。

方案设计：硬约束 + 重惩罚

def clue_preservation_reward_func(generated, original, clue_mask):
    changed = (generated != original) & (clue_mask != 0)
    if changed.any():
        return -2.0  # 任何篡改直接重罚
    return 0.0

为什么是-2.0而不是-1.0或-3.0？

先算一笔期望收益账：

  • 作弊后可能拿到format reward：约+0.5 / 拿到partial logic reward：约+0.3 → 期望收益约+0.8
  • 惩罚-1.0：作弊期望收益 = +0.8 - 1.0 = -0.2，但方差大，模型可能"运气好就不被抓"继续作弊
  • 惩罚-2.0：作弊期望收益 = +0.8 - 2.0 = -1.2，明显为负，模型会探索合法路径
  • 惩罚-3.0：期望更负，但梯度信号太强，训练不稳定，实验中观察到loss震荡，策略崩坏

最终选了-2.0，是"刚好能压住作弊，又不至于崩训练"的阈值。惩罚是硬约束——只要检测到篡改就给-2.0，不考虑"改了多少个"，堵死整条路径。

效果

指标加线索保真前加线索保真后（-2.0）
线索保真度负值（模型在改题）0.72
作弊行为占比约30%<5%
Pass@1约50%73%

学术关联：Reward Tampering: Pathways to Manipulating Reward 这篇论文系统讨论了各种Reward Hacking形式。我的案例属于"environment tampering"——模型修改环境（题目）来获取奖励。另一个相关研究是Constitutional AI，用规则约束模型行为。我的硬约束设计本质上是类似思路：把"不可篡改原题"作为宪法条款，违规直接惩罚。

细粒度信用分配 · KL约束调参

细粒度信用分配

面试官问：细粒度信用分配怎么做的？27个逻辑单元是什么？

稀疏奖励问题

数独是"全对才得分"的任务——9×9=81格，错一个就是0分。模型随机生成一个解，81格大概能蒙对9格（每行1格），但这9格分散在不同行、列、宫，没有任何一个单元全对，最终reward=0。问题：模型其实做了"部分正确"的事情，但没收到任何正反馈，RL无法知道"哪些格子填对了"。

拆解思路

数独有天然结构：9行 + 9列 + 9个宫格 = 27个逻辑单元。每个单元可以独立验证唯一性——即使全盘没解对，某一行填对了，这个单元就应该拿到partial reward。

具体实现

def row_col_block_logic_reward_func(board):
    reward = 0.0
    for i in range(9):  # 检查9行
        row = board[i, :]
        non_zero = row[row != 0]
        if len(non_zero) > 0:
            if len(set(non_zero)) == len(non_zero):  # 无重复
                reward += 1.0 / 27
    # 同理检查9列、9宫格
    ...
    return reward  # 27个单元全对 = 1.0

为什么这个设计有效

方案模型填对10格模型填对50格模型全对
全局奖励001.0
27单元奖励约0.15约0.601.0

在全局奖励下，填对10格和填对0格reward都是0——模型分不出来。在27单元奖励下，填对10格能拿到约0.15的reward，模型能收到正反馈。梯度信号更dense。

注：27个单元的奖励是累加的，不是取最大。数独约束全局耦合——一个数字填错影响所在行、列、宫格三个单元。模型如果想优化局部奖励，必须保证这个数字在三个维度上都不冲突，这其实就逼近了全局正确。

量化效果

指标加细粒度奖励前加细粒度奖励后
平均每步reward0.170.81
Pass@1约50%73%

学术关联：Process Reward Model（Let's Verify Step by Step，Lightman et al. 2023）——把最终答案的奖励拆解到中间推理步骤。我的27单元reward是简化版PRM：不需要训练额外verifier模型，直接用规则校验。

训练稳定性控制——KL约束与参数调优

面试官问：你说beta=0.001的KL散度约束，这个值怎么选的？大了小了会怎样？

理解KL约束的作用

KL散度约束防止policy偏离太远：$$\text{KL}(\pi_\theta \| \pi_{\text{ref}}) = \sum_a \pi_\theta(a|s) \log \frac{\pi_\theta(a|s)}{\pi_{\text{ref}}(a|s)}$$

在目标函数里：$L = \mathbb{E}[A_t \cdot r_t(\theta)] - \beta \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}})$。$\beta$是两者的平衡系数。

beta值对训练的影响

beta值KL偏离度观察到的现象结论
0.010.1-0.2reward几乎不涨，模型原地踏步太大，不探索
0.0010.3-0.4reward稳步上升，收敛平滑合适
0.00010.6+第50步开始崩，格式错乱太小，跑飞了

监控和干预机制

approx_kl = (log_probs - old_log_probs).mean()  # 估计KL散度
# 自适应调节beta
if KL > target_kl * 1.5:  beta *= 2
if KL < target_kl * 0.5:  beta /= 2

当approx_kl超过0.5时，手动降学习率或增大beta，防止策略崩坏。

量化效果

指标无KL约束beta=0.01beta=0.001（最终）
训练稳定性崩坏稳定但无提升稳定且有提升
Pass@1训练崩了约55%73%
收敛步数N/A>200~100

学术关联：PPO原论文（Proximal Policy Optimization Algorithms，Schulman 2017）里详细讨论了KL约束的作用。Schulman建议用adaptive KL penalty，让KL自动调节到目标值。DeepSeek-R1的设置也类似，他们用的KL系数在1e-4到1e-3量级。

效果评估·基线对比 · 项目二架构设计

效果评估与基线对比

面试官问：73%正确率是怎么评估的？基线是什么？怎么算的？

测试数据集构建

  • 数独生成器生成的题目：400道
  • 在线数独网站的题目：100道
  • 总计500道题目，确保：不在训练集里出现、难度均匀（简单/中等/困难各约1/3）、每道题有唯一解

评估指标：Pass@K

来自代码生成领域，定义：$\text{Pass@}K = \mathbb{P}(\text{至少1个正确} | K\text{次采样})$

实际计算用无偏估计：$\text{Pass@}K \approx 1 - \frac{\binom{n-c}{K}}{\binom{n}{K}}$（n=采样次数，c=正确次数）

我的设置：每题采样K=4个答案。Pass@1：只看第一个答案的正确率。Pass@4：4个答案里有一个对就行。

基线对比

方法Pass@1Pass@4备注
随机生成<1%<5%几乎不可能蒙对
纯SFT（无RL）约45%约65%格式正确但逻辑弱
SFT + DPO约58%约75%偏好学习有噪声
SFT + PPO约70%约82%需要更多显存
SFT + GRPO（我的方案）73%85%效果和PPO相当，显存更低

为什么不是SOTA：73%不是业界最好结果（GPT-4在数独上90%+），但模型只有3B参数、训练资源有限（单卡A100约10小时）。更重要的是定性分析：训练后模型产生了明显的推理行为变化——从"瞎猜"变成"有逻辑的尝试"，会检查行列宫已有数字，排除后再填。这种"排除法推理"行为的涌现，说明模型真的学到了数独规则，而不是背答案。

后续改进方向：增大G值（从4到8）降低方差；引入curriculum learning；结合MCTS提升推理质量。

项目二：金融Agent系统——架构设计

面试官问：你说"状态机驱动架构"，具体是什么状态机？为什么要这么设计？

问题背景

金融数据分析任务需要多步推理。传统Agent架构（ReAct范式）让LLM自由输出thought+action，存在两个核心问题：

  • 漂移：模型执行到Step 2时突然跳到不相关任务（茅台→五粮液→泸州老窖）
  • 早停：遇到数据为空，模型不知道怎么处理，直接返回"数据不可用"

状态机设计

┌─────────┐  [规划完成]   ┌─────────┐  [执行完成]   ┌─────────┐  [验证通过]  ┌────────┐
│  PLAN   │─────────────→│ EXECUTE │─────────────→│ VERIFY  │────────────→│ OUTPUT │
└─────────┘              └─────────┘              └─────────┘             └────────┘
     ↑                                                  │
     └──────────────────[VERIFY失败→重新规划]───────────┘
     连续失败≥3次 → 终止，返回降级答案

核心设计：[DATA]:协议验证作为门控条件

def verify_output(output: str) -> bool:
    if "[DATA]:" not in output: return False  # 必须含[DATA]:标记
    if is_empty(extract_data(output)): return False  # 数据不能为空
    return True

权限隔离设计（分离规划权和执行权）

def supervisor_node(state):  # 只规划，不执行
    plan = model.invoke("规划下一步: " + state.context)
    return {"next_action": plan.action, "params": plan.params}

def coder_node(state):        # 只执行，不规划
    result = execute_tool(state.next_action, state.params)
    return {"output": result}

def reviewer_node(state):     # 只验证，不修改
    if verify_output(state.output): return {"status": "success"}
    else: return {"status": "retry"}

第一性原理：把推理过程从"自由文本生成"变成"受约束的状态转移"。每个状态只做一件事，输出格式固定，下游不需要parse——从架构层面消除不确定性。

量化效果

指标传统自由输出状态机驱动
Pass@1（中等难度）约65%87.5%
推理漂移率约15%<3%
无效重试率约20%<5%

项目二：RAG优化——字段级Chunking

面试官问："340篇文档重构为4032个字段级chunk"，重构逻辑是什么？为什么能解决字段幻觉？

字段幻觉的根源

用户问"获取贵州茅台的净利润"，模型要生成代码调用Tushare API。问题是：用户说"净利润"，Tushare的API里字段名是n_income。如果模型不认识这个映射，可能臆造net_profit或profit，代码就报错了。

传统RAG用接口级chunk（每个API文档一篇，500-2000字）：

  • 字段名n_income被稀释在几百字的文档里
  • 向量检索时，query "净利润"和整篇文档的相似度不高
  • BM25也没用，因为文档里根本没有"净利润"这四个字

字段级chunk重构

Tushare API文档是Markdown格式，输出参数表是这样的：

| 名称 | 类型 | 描述 |
|------|------|------|
| ts_code | str | 股票代码 |
| n_income | float | 净利润 |

混合检索融合

# lib.py:1223
for idx in all_ids:
    v = vector_scores.get(idx, 0.0) * vector_weight   # FAISS 0.7权重
    b = bm25_norm.get(idx, 0.0) * (1 - vector_weight) # BM25  0.3权重
    hybrid_scores[idx] = v + b

向量检索召回语义相关的文档，BM25召回精确匹配字段名的文档。为什么混合？因为金融字段有两类查询模式："帮我查茅台今天的市盈率"属于语义查询，向量好；"给我pe_ttm字段的说明"属于精确关键词查询，BM25好。

效果对比

接口级（改前）48%62%71%

78%→93.1%的提升主要来自同义词扩展。有些query用"ROE"搜，字段名是"roe"，大小写不匹配；扩展后追加了"roe"关键词，BM25就能匹配上了。

Hit@3=93.1%怎么算的？Ground truth怎么来的？

Ground truth构建：准备100个真实用户query（从历史日志采样），人工标注每个query对应的正确字段名。例如：query="获取茅台ROE" → 正确字段="roe"

Hit@K计算：对每个query，用RAG检索top-K结果，如果top-K里包含正确字段算命中。Hit@3 = 命中query数 / 总query数

局限性：Ground truth只有100条，样本量偏小；评估是离线的，线上真实效果可能有差距；没有对比其他Embedding模型（如BGE-large vs BGE-small）

学术关联

ColBERT是一种late interaction的检索方法，在token级别计算相似度。我的字段级chunking可以看作是一种更粗糙的粒度控制——不是token级别，而是字段级别。

RAGAS是RAG评估框架，提出用faithfulness、context precision等指标评估。我的Hit@K是simpler版本，专门针对字段召回场景。

错题本驱动的在线策略对齐

面试官问：错题本驱动的在线策略对齐，具体怎么实现的？为什么不重新训练？

问题背景

典型case：港股代码格式。用户问"获取腾讯控股的股价"，正确格式是00700.HK，但模型经常输出0700.HK（少一个0）或TCEHY（美股代码）。这种错误不是随机的，是模型对港股格式规范不理解导致的——每次遇到港股都会错。

传统解决方案是重新微调，但代价太大：成本高（GPU时间+数据收集）、周期长（几天到几周）、不灵活（更新需重新训练）。

推理时动态注入

def get_memory_context(query: str, session_id: str) -> str:
    lessons = memory_store.search(query=query, filters={"session_id": session_id, "type": "lesson"})
    if lessons: return f"【历史教训】\n{lessons[0].content}\n"
    return ""

def supervisor_node(state):
    memory_context = get_memory_context(state.query, state.session_id)
    prompt = f"{memory_context}\n用户问题: {state.query}\n请规划下一步..."
    plan = model.invoke(prompt)

在线注入 vs 重新训练

维度重新训练在线注入
成本高（GPU时间+数据收集）低（推理时额外检索）
更新周期天到周分钟级更新
灵活性低（更新需重新训练）高（动态增删）
效果更彻底，改变权重针对性强，但覆盖有限

局限：在线注入只能处理"规则型"错误，无法改变模型的核心能力。但对于格式规范、特殊case，这是更高效的方案。
