# 多智能体系统 - 项目能力测试Prompt集

> **目的**：通过6个递进式测试，验证系统的数据获取、算法透明、错误恢复、消息管理、上下文继承等核心能力。

---

## 📌 **测试1：基础数据查询 + 元数据输出** ⭐

**验证能力**：
- Coder能否生成正确的API调用
- Tools能否成功执行代码
- 是否输出完整的[DATE][SOURCE][DATA]标签
- 是否自然语言无Markdown

**Prompt：**
```
查询贵州茅台（600519.SH）最新一个交易日的收盘价和成交量。

要求：
1. 获取最新交易日的行情数据
2. 明确输出：日期、收盘价、成交量
3. 必须说明数据来源和采集时间
4. 给出简要分析：这个成交量水平是否正常？
```

**预期验证点**：
```
✅ 代码执行成功（Coder写出pro.daily()）
✅ 数据输出格式完整（[DATE] [SOURCE] [DATA]都在）
✅ [DATA]包含JSON结构（ts_code, trade_date, close, vol）
✅ Reviewer输出纯自然语言，无Markdown
✅ 报告长度400-800字
```

**失败信号**：
```
❌ 输出包含Markdown符号（###、**、-列表）
❌ 没有[SOURCE]标记
❌ [DATA]输出为空或{}
❌ 报告内容与[DATA]不对应
```

## 📌 **测试2：多步骤任务 + 动态分解** ⭐⭐

**验证能力**：
- Supervisor能否正确分解复杂任务
- remaining_steps是否被正确执行
- Coder是否依次完成多个步骤
- 任务计划能否指导代码生成

**Prompt：**
```
对中国平安（000001.SZ）进行综合分析，要求：

1. 获取最新的股息率数据（使用daily_basic表的dv_ttm字段）
2. 获取最近30天的股价走势（最高价、最低价、收盘价）
3. 计算近30天的平均股价
4. 判断：当前股价相对于30日均价是溢价还是折价？
5. 基于以上数据，给出投资建议
```

**预期验证点**：
```
✅ Supervisor输出task_plan，包含5个步骤
✅ Coder第1次调用获取股息率
✅ Coder第2次调用获取30天行情
✅ Coder计算平均价格和溢折价幅度
✅ Reviewer输出综合分析建议
✅ 元数据引用完整（[META]定义了30日均价的计算）
```

**失败信号**：
```
❌ 只完成了前1-2个步骤，中途停止
❌ 步骤顺序错乱（先算均价，再获取数据）
❌ Reviewer报告与actual [DATA]不一致
❌ 没有体现"溢价"或"折价"的判断
```

## 📌 **测试3：错误自救 + 数据质检** ⭐⭐

**验证能力**：
- ErrorHandler能否检测代码错误
- Coder能否根据错误信息自动修复
- retry_count机制是否工作
- 数据为空时的自救策略

**Prompt：**
```
查询某个小盘股的PE值（代码：002001.SZ），并分析其估值水平。

要求：
1. 获取该股的PE（市盈率）数据
2. 获取同行业的平均PE
3. 对比：该股PE是否高于或低于行业平均？
4. 分析原因（可能与上市时间短、业绩差异有关）

注意：如果某个API返回空数据，请自动换备选API或修改参数重试。
```

**预期验证点**：
```
✅ 第1次查询可能失败（PE为空）
✅ ErrorHandler检测到[ERROR]标记
✅ retry_count增加到1
✅ Coder自救：换用daily_basic或其他API
✅ 第2次查询成功，获得数据
✅ Supervisor质检通过（[DATA]非空）
✅ Reviewer报告中包含"备选API"的说明
```

**失败信号**：
```
❌ 代码出错后直接返回Error，没有重试
❌ retry_count没有增加
❌ 没有尝试备选API或修改参数
❌ Reviewer报告[DATA]为空仍输出分析（说明质检失效）
```

## 📌 **测试4：缓存机制 + 话题继承** ⭐⭐

**验证能力**：
- ToolCache能否命中缓存
- 跨轮变量是否被保留（StatefulPythonKernel）
- 用户话题继承时是否复用数据
- metadata字段标注缓存来源

**Prompt（连续两轮）：**

**第1轮：**
```
查询茅台（600519.SH）的最新股价和成交量。
```

**等待系统完成，然后立即进行第2轮：**

**第2轮：**
```
把茅台的股价数据画成近30天的走势图。
```

**预期验证点**：
```
✅ 第1轮：Coder执行查询，Tools保存缓存
✅ 日志出现 [ToolCache] 保存缓存：run_script
✅ 第2轮：state["messages"]保留了第1轮的[DATA]
✅ Coder识别出"把它画成图"中的"它"指的是茅台
✅ 第2轮的代码直接用df_maotai（如果有保留）或重新定义
✅ 日志出现 [DEBUG Reviewer] 指代补全：它 → 【茅台】
```

**失败信号**：
```
❌ 第2轮重新调用API而不是复用缓存
❌ 没有[ToolCache] 命中缓存的日志
❌ Coder不知道"它"指茅台，提示需要明确
❌ 画的图表来源于新查询的数据而非第1轮数据
```

## 📌 **测试5：数据质检 + 降级方案** ⭐⭐⭐

**验证能力**：
- Supervisor的三层质检机制（[DATA]标记、非空、[SUMMARY]）
- Coder遇到数据为空时的自救
- 动态重规划（retry_count≥3）
- 降级方案的有效性

**Prompt：**
```
分析沪深300指数中股息率最高的5支股票。

要求：
1. 获取沪深300成分股列表
2. 对每支股票获取最新的股息率（dv_ttm）
3. 排序找出股息率最高的5支
4. 输出这5支股票的：代码、名称、股息率、当前价格
5. 给出投资建议：是否值得投资？为什么？

注意：某些小盘股或新股可能没有dv_ttm数据，请自动转用备选指标（如分红收益率）。
```

**预期验证点**：
```
✅ Supervisor分解任务：
   - 步骤1：获取沪深300名单
   - 步骤2：批量查询股息率
   - 步骤3：排序筛选
   - 步骤4：输出和分析

✅ 数据质检第1阶段：检查[DATA]标记
✅ 数据质检第2阶段：检查数据非空
✅ 数据质检第3阶段：检查[SUMMARY]摘要

✅ 如果某个股票数据为空：
   - ErrorHandler检测
   - Coder尝试自救（换API或指标）
   - retry_count递增
   
✅ 如果重试3次仍失败：
   - Supervisor触发重规划
   - 生成新的任务分解
   - 采用新策略（如只分析前50支而不是全部）

✅ Reviewer报告包含：
   - 明确说明哪些股票用了备选指标
   - [META]定义了股息率和分红收益率的区别
```

**失败信号**：
```
❌ 某个股票返回空数据，Coder没有自救
❌ retry_count达到3次仍继续重试（应该重规划）
❌ 没有显式的重规划日志：[Supervisor] 🔄 重规划
❌ Reviewer报告声称"分析完成"但[DATA]里有大量空值
```

## 📌 **测试6：完整流程 + 跨轮学习** ⭐⭐⭐⭐

**验证能力**：
- 完整的Supervisor→Coder→Tools→ErrorHandler→Reviewer→ProfileUpdater流程
- 用户画像的动态更新（ProfileUpdater）
- 多轮对话中系统逐渐了解用户偏好
- 消息清洗和上下文补全的综合效果

**Prompt（模拟3轮对话）：**

**第1轮：**
```
我是一个稳健型投资者，主要关注银行和保险股。
请分析中国平安（000001.SZ）是否值得长期持有？
```

**等待报告完成，然后：**

**第2轮：**
```
现在分析一下中国银行（601988.SH）的分红情况。
相比平安，银行股的股息率是否更稳定？
```

**等待报告完成，然后：**

**第3轮：**
```
我想要对比这两只股票。哪一个更适合我这样的稳健投资者？
```

**预期验证点**：
```
✅ 第1轮：
   - Supervisor分解任务
   - Coder获取平安的财务数据、股息率、ROE等
   - ErrorHandler处理任何错误
   - Reviewer生成投资分析报告
   - ProfileUpdater更新：
     * interested_sectors: ["金融", "保险"]
     * risk_preference: "低"
     * investment_style: "稳健"
   
✅ 第2轮：
   - state["user_profile"]已包含第1轮的更新
   - Supervisor可以看到用户喜欢金融股
   - Coder获取银行股数据
   - Reviewer在分析时可能说：
     "与您第一轮查询的平安相比，银行股的股息率..."
   - ProfileUpdater根据用户询问"分红情况"，增强了
     "preferred_analysis_depth": "深度"
   
✅ 第3轮：
   - Coder获取两支股票的对比数据（复用缓存）
   - Reviewer输出对比分析，引用[META]定义对比指标
   - 结论明确针对"稳健投资者"的建议
   - ProfileUpdater最终确认用户画像已学习
```

**验证点：消息管理**
```
✅ 第1轮后：state["messages"]包含
   HumanMessage(用户输入) 
   → AIMessage(Supervisor决策)
   → AIMessage(Coder回复)
   → ToolMessage(代码结果)
   → AIMessage(Reviewer报告)

✅ 进入第2轮时：
   - messages不清空，追加新的HumanMessage
   - trim_messages()可能删除最早的部分历史
   - 但保留首个HumanMessage和最近N条

✅ Reviewer清洗消息时：
   - 保留用户原始问题（第3轮会用到"对比"）
   - 保留最后的[DATA]（两支股票的数据）
   - 删除中间的ToolMessage和ErrorHandler记录
   
✅ 第3轮Reviewer输出时：
   - 包含对第1、2轮的引用（但不显式提及）
   - 自然地说出"相比第一轮的平安和第二轮的银行..."
```

**失败信号**：
```
❌ 第2轮Reviewer不知道平安的信息（缓存丢失）
❌ ProfileUpdater没有更新user_profile
❌ 第3轮对比报告无法关联前两轮的数据
❌ 消息列表膨胀过大（应该被修剪）
❌ 没有显式的"Reviewer清洗消息"日志
```

## 💡 **执行建议**

1. **从测试1开始**，逐步增加复杂度
2. **记录日志**，对照预期验证点检查
3. **关键日志标记**：
   - `[Supervisor]` - 主决策
   - `[Coder]` - 代码生成
   - `[ToolCache]` - 缓存命中/保存
   - `[ErrorHandler]` - 错误分类和重试
   - `[DEBUG Reviewer]` - 消息清洗
   - `[ProfileUpdater]` - 画像更新
4. **观察报告格式**：应该是纯自然语言，无Markdown
5. **检查元数据**：Reviewer是否引用了[META]定义
