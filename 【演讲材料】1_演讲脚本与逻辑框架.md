## 第一部分：项目定位（开场30秒）

### 讲解框架

**开场白**：
> "大家好。我们今天要讲的项目解决了一个真实的金融行业问题：初级分析师需要花大量时间做重复工作——查数据、清数据、写报告。我们的方案是用AI替代这个过程。"

**核心定位**：
- **问题**：人工投研流程低效（1个分析报告需要2-3小时）
- **解决方案**：虚拟投研团队（5个AI角色的自动化系统）
- **结果**：同样的分析流程自动化完成（10分钟内）

## 第二部分：系统架构（讲解1.5分钟）

### 用一个类比讲清楚

**类比**：传统做法 vs AI做法

```
【传统方式】
分析师 → 打开Tushare → 手动查数据 → 手动计算指标 → 手动写报告 → 输出结果
（需要专业人员，耗时2-3小时）

【AI系统方式】
用户问题 → 自动分解 → 自动查数据 → 自动计算 → 自动写报告 → 输出结果
（秒级完成，0人力成本）
```

### 系统的5个角色（这是核心，要讲清楚）

| 角色 | 比喻 | 职责 | 能做什么 |
|-----|------|------|--------|
| **Supervisor** | 🧠 大脑/经理 | 理解用户要求，分解任务，决定谁去做什么 | 听懂"帮我分析中国平安"，分解成5个子任务 |
| **Coder** | 💻 手脚/执行员 | 写Python代码，调用数据API，进行数据处理 | 从Tushare API获取实时数据，计算PE、PB等指标 |
| **Reviewer** | 👁️ 眼睛/质检员 | 检查数据是否正确，检查是否有异常，写分析报告 | 识别"股息率>10%是异常"，写成专业报告 |
| **ErrorHandler** | 🛡️ 免疫系统 | 当出错时自动修复问题 | 代码执行失败→自动改进代码→重新执行 |
| **ProfileUpdater** | 💾 记忆库 | 记住用户的偏好和历史查询 | 下次用户问类似问题时，记得上次的设置 |

### 架构流程图（用这个讲执行流程）

```
用户提问："帮我分析中国平安（000001.SZ）的价值"
    ↓
[Supervisor] 分解任务
    ├─ 任务1: 获取基本面数据（PE、PB、股息率）
    ├─ 任务2: 获取财务数据（营收、净利）
    ├─ 任务3: 计算估值指标
    └─ 任务4: 撰写分析报告
    ↓
[Coder] 执行代码获取数据
    ├─ 调用Tushare API获取实时行情
    ├─ 调用财务数据接口获取财报数据
    └─ 返回结构化数据（有[DATE]、[SOURCE]、[DATA]标签）
    ↓
[ErrorHandler] 检查是否出错
    ├─ 如果代码出错？→ 自动修复代码 → 重新执行
    ├─ 如果数据缺失？→ 换个API试试
    └─ 如果网络超时？→ 等一会儿再试
    ↓
[Reviewer] 质检和写报告
    ├─ 检查数据是否合理（PE<0? 股息率>50%?都是异常）
    ├─ 识别核心结论
    └─ 写成400-800字的专业分析报告
    ↓
[ProfileUpdater] 记住这次的结果
    ├─ 记住用户关心"中国平安"
    ├─ 记住用户关心"估值指标"
    └─ 下次用户问类似问题时，自动应用这些偏好
    ↓
输出结果给用户（数据+分析报告）
```

## 第三部分：核心能力展示（讲解1分钟）

### 能力1：自动数据获取和处理

**说这个**：
> "系统可以自动从Tushare（国内专业金融数据库）获取实时数据。不是爬网页，而是通过官方API直连。数据包括：股票基本面（PE、PB、股息率）、财务数据（营收、净利、现金流）、行情数据（价格、成交量）等。"

**演示内容**：
- 可以获取100支股票的数据（而不是手动查1支）
- 数据自动清洗（处理缺失值、单位转换等）
- 返回结构化结果（表格格式，可直接用）

### 能力2：自动异常检测

**说这个**：
> "系统可以自动识别数据中的异常值。比如股息率超过50%（正常是2-5%），PE小于0（不可能），这些系统都能自动检测并告警。"

**演示内容**：
- 对比正常数据：中国平安 股息率4.5%（正常）
- vs 异常数据：某ST股票 股息率65%（异常，标记红警告）

### 能力3：自动撰写分析报告

**说这个**：
> "获取到数据和指标后，系统自动生成专业的分析报告。这不是简单的表格拼接，而是理解数据逻辑，用自然语言写出结论。"

**演示内容**：
```
【示例报告】
基于最新数据，中国平安当前PE为9.8倍，低于行业平均10.2倍。
股息率4.5%，超过最近3年平均3.2%，说明公司盈利向好。
建议：当前估值处于底部区间，适合长期布局。
风险提示：金融监管变化可能影响估值。
```

### 能力4：自动故障修复

**说这个**：
> "当执行过程中出现错误，系统不会直接崩溃或返回错误。而是自动诊断问题类型，采取对应的修复策略。"

**演示内容**：
- 代码错误？→ 自动改进代码逻辑
- API超时？→ 等一会儿后重试
- 数据格式不对？→ 自动转换格式

## 第四部分：技术创新点（讲解1分钟）

### 创新1：消息链完整性验证

**概念**：
> "一个好的AI系统需要能够'解释自己的思考过程'。我们采用了消息链完整性验证机制。"

**具体是什么**：
- 系统处理过程中的每一步都有记录
- 用户可以追溯：问题→分解→执行→结果 的全过程
- 每条数据都标注了来源（[SOURCE]）、时间（[DATE]）、格式说明（[META]）

**为什么重要**：
- 透明性：用户信任系统
- 可追溯性：出问题时能快速定位原因
- 审计性：金融应用需要能被审计

### 创新2：多角色分工而不是单体Agent

**概念**：
> "传统AI只有一个Agent处理所有任务。我们不同，分成5个专业角色。"

**优势**：
- **降低出错率**：每个角色只专注自己领域，出错率低
- **提高透明性**：Reviewer角色充当质检员，拦截坏数据
- **便于调试**：出问题时能快速定位是哪个环节
- **成本更优**：不是所有步骤都用"最强模型"，简单任务用轻量级模型

### 创新3：四层错误自愈机制

**概念**：
> "系统不但会出错，而且能自动修复大部分错误。"

**具体分类**：
1. **代码错误**（占比最高）→ 自动改进代码逻辑，重试最多3次
2. **网络错误**（API超时）→ 指数退避重试（等1秒→2秒→4秒）
3. **数据错误**（格式不对）→ 自动格式转换
4. **业务错误**（逻辑不对）→ 重新分解任务，换个策略

**效果**：
- 44个真实任务的最终成功率：86%以上
- 通过自愈机制，很多本来会失败的任务都被救活了

## 第五部分：真实案例演示（讲解2分钟）

### 案例1：一个分析师日常任务

**用户问题**：
> "帮我筛选当前估值最低的10支金融股，并给出估值分析报告"

**系统执行**：
1. Supervisor：分解成"筛选金融股" → "查估值数据" → "排序" → "写报告"
2. Coder：获取200支金融股的PE、PB数据
3. Reviewer：检查数据合理性，识别出3条异常数据（标记为风险）
4. 自动生成：
   ```
   【估值最低的10支金融股排名】
   
   1. 招商银行（600036）PE: 6.8, PB: 0.65
   2. 工商银行（601398）PE: 5.2, PB: 0.48
   ...
   
   分析结论：当前金融股整体估值处于历史低位，
   特别是大行和股份制银行具有配置价值。
   ```

**用时**：10分钟（人工需要2-3小时）

### 案例2：发现数据异常

**背景**：某公司的股息率突然从2%跳到85%

**系统行为**：
1. Reviewer自动识别"异常"
2. 自动查证数据来源
3. 自动写告警：
   ```
   ⚠️ 风险提示：某公司股息率显示85%，显著高于历史平均2%。
   可能原因：(1)数据接口异常 (2)公司发生重大变化
   建议：查证官方公告后再参考此数据
   ```

**效果**：防止用户基于错误数据做决策

## 第七部分：常见问题应答

### Q1: 系统的数据准确吗？

**答**：
> "系统直连Tushare官方API，数据来源可信。但AI写的报告是基于数据的逻辑分析，需要用户审核。我们的角色是初级分析师，不是投资顾问。"

### Q2: 为什么需要5个Agent，而不是1个超级Agent？

**答**：
> "原因有三：(1)降低出错率——每个Agent只做自己擅长的事 (2)提高透明性——Reviewer充当质检员 (3)降低成本——不是所有任务都需要最强模型"

### Q3: 系统能处理实时交易吗？

**答**：
> "不行。我们做的是日级别的分析报告，不是高频交易。实时交易需要毫秒级反应，不是这个系统的设计目标。"

### Q4: 用户学习（ProfileUpdater）的实现程度？

**答**：
> "这部分还在优化中。基础框架已实现，能记住用户的查询历史和偏好。但深度学习（如"学会推荐用户喜欢的股票"）还需要更多数据积累。"

## 重要提示

### 演讲时要强调

✅ 系统做了什么（功能）
✅ 系统能带来什么价值（收益）
✅ 系统相比人工的优势（对比）
✅ 系统当前的限制（透明度）

### 演讲时要避免

❌ 代码细节（如"我用了StatefulPythonKernel来...）
❌ 模型选择细节（如"为什么用qwen-plus而不是gpt-4"）
❌ 技术实现细节（如"怎样通过消息链去去重"）
❌ 太多数字和指标（只说核心的）

### 演讲时的应急应答

**如果评委问：代码怎么写的？**
> "这部分由我负责开发。演讲重点是系统能做什么，代码实现细节可以课后详聊。"

**如果评委问：PE怎么算的？**
> "这是基础金融知识，PE=股价/每股收益。系统从Tushare API直接获取，不需要自己算。"

**如果评委问：准确率多少？**
> "系统的数据准确率（基于API）接近100%。但分析报告是AI生成的逻辑推导，需要人工审核，不能直接作为投资建议。"
