## 开场定调：我想先把问题本身讲清楚

面试官您好，我先不急着讲技术栈，我想先说一件事：这个系统从立项到最终压测，让我印象最深的不是做了什么，而是发现了什么不能做、以及为什么不能做。

过去两三年 Agent 领域有一个越来越清晰的共识，2026 年的几篇论文把它说得很明确了——PALADIN、Self-Healing Router、谷歌的多 Agent 目标漂移研究、ACL 2025 的 DRAG、还有今年的 FGR-ColBERT。这些工作切入点各不相同，有的在研究工具调用失败，有的在研究检索噪声，有的在研究多智能体目标漂移，但我把它们放在一起看，发现它们其实都在回答同一个更底层的问题：为什么一个在单步表现优秀的大模型，在真实的多步工具调用场景里会系统性地失控？

这个问题的答案，我现在能用一句话说清楚：因为大模型是概率系统，而多步执行是一个概率连乘的过程，连乘的结果是指数级衰减，这不是某个模型做得不够好，是数学上的必然。

正是基于这个判断，我这个项目的整个设计逻辑就变了。我没有去追求用更强的模型、更精心的 Prompt 来提高单步成功率，而是把整个系统重新设计成：尽一切可能缩小概率模型参与决策的比例，把能变成确定性代码的部分全部变成代码，把不得不用模型的部分严格隔离在明确边界内，并且为所有可预期的失败场景预先准备好确定性的恢复路径。

这是整个设计的核心哲学，接下来我讲的每一个具体机制，都能回溯到这个判断上。

## 第一性原理：概率为什么会崩塌

我先把数学摆出来，因为这是理解后面所有设计决策的基础。

一个用户问"分析贵州茅台的财务健康状况，给出投资建议"，这在我们的系统里会被分解成六个步骤：查利润表、查资产负债表、计算 ROE、计算资产负债率、做财务健康分析、生成投资建议。假设每一步大模型生成的代码正确率是 95%——这已经是个相当乐观的估计了——那六步串行执行下来，整体成功率是 0.95 的 6 次方，也就是 73.5%。四分之一的概率这个任务会在某个步骤出错。而且这还不是线性退化，是指数退化——步骤越多，崩塌越快。

这个数字背后有三种具体的崩塌形式，每一种在我的压测数据里都有真实的案例。

第一种，我叫它字段幻觉性崩塌。大模型的训练语料里不包含 Tushare API 的私有字段名——它知道"净利润"是什么，但它不知道在 Tushare 的利润表接口里净利润字段叫 `n_income` 而不是 `net_profit`。它会非常自信地写出 `df.iloc[0]['net_profit']`，然后代码运行时直接 KeyError，整个步骤崩掉。我在 35 个压测任务里发现，所有 Hard 难度任务里字段名写错的概率接近 88%，也就是说几乎每一次复杂的财务分析，第一步就会出错。这不是偶发问题，是系统性的必然。

第二种，我叫它数据真空性崩塌。大模型生成的代码在语法上完全正确，接口调用也成功了，但返回的是一个空 DataFrame。比如查贵州茅台 2024 年第四季度的资产负债表——季报可能还没发布，接口成功执行但返回空数据。这时候模型面对一个空 DataFrame，如果没有硬性门控，它会把"查询成功但无数据"解读为"任务已完成"，然后直接跳到生成结论阶段，用语言包装一个看起来合理但完全没有数据支撑的分析。这是金融场景里最危险的失败形式，因为它是无声的——用户拿到一个格式完整、措辞专业的分析报告，但里面的数字是模型编出来的。

第三种，我叫它目标漂移性崩塌。这是谷歌那篇多 Agent 论文（Problem Drift，2026）里研究的问题。他们发现多轮辩论场景里，超过一定轮次之后，Agent 会开始"忘记原始问题"，在一个已经偏离的方向上越讨论越远。他们定义了一个 FOCUS 指标来量化这种漂移，发现 35% 的案例出现了"缺乏进展"。在我的系统里，这个问题的表现形式更隐蔽——Supervisor 在多步执行中，偶尔会出现规划漂移，比如用户问"计算茅台的 ROE"，规划到第三步的时候，生成了一个"分析茅台的护城河"这样的步骤，完全偏离了原始目标。

这三种崩塌，我的系统用三套完全不同的工程机制来对抗，接下来逐一讲。

## 机制一：用状态机把概率决策关进笼子

先解释清楚"关进笼子"这个比喻是什么意思。

传统的 ReAct 框架——LangChain 里最常见的那种模式——是让大模型在每一步工具调用之后，自己判断接下来该做什么、要不要再调工具、应该调哪个工具。这种设计的问题是：每一次"该做什么"的判断都是概率性的，模型可能判断正确，也可能漂移。Self-Healing Router 这篇论文（2026 年 3 月发表）把这个问题量化了——他们测了 ReAct 在 19 个场景下的 LLM 调用次数，平均要 123 次，而且每次工具失败之后 ReAct 都要重新调用 LLM 来决策下一步，这意味着控制流完全依赖 LLM 的概率判断，稳定性极低。他们的解法是把控制流从 LLM 推理改成图路由——工具失败时把对应的边权重设成无穷大，然后用 Dijkstra 算法重新计算最短路径，不需要调用 LLM。结果 LLM 调用从 123 次降到了 9 次。

这个思路给了我很大启发。我的状态机设计的核心思想跟这篇论文是一致的：**把能够确定性化的控制流决策，从 LLM 推理里剥离出来，变成代码逻辑。**

具体实现上，整个系统是一个 LangGraph 构建的有向状态图，节点包括 Supervisor、Coder、Reviewer、ErrorHandler、ProfileUpdater。关键在于节点之间的路由逻辑是怎么写的——Supervisor 节点的输出会触发 P1 断言检查，检查逻辑在 multi_agent.py 的 729 到 795 行。P1 断言只做一件事：检查 Coder 返回的消息里是否包含 `[DATA]:` 前缀的有效数据标记。这个检查是纯代码判断，不经过 LLM，是二进制的：有 `[DATA]:` 且不为空，步骤队列 `pop(0)` 推进下一步；没有，要么触发 retry_count 机制，要么转 ErrorHandler。整个推进逻辑不依赖 LLM 的"判断"，是完全确定性的状态转移。

`[DATA]:` 标记本身也不是靠提示词要求 LLM 自觉加的——在工具执行的后处理函数里，代码特征匹配到执行结果后会自动添加这个标记，是代码层的强制行为。这个细节很重要：如果依赖 LLM 主动加标记，它偶尔会忘，就给了目标漂移留了后门。

另一个关键机制是职责隔离。Supervisor 是规划节点，它持有步骤队列 `remaining_steps`，决定做什么；Coder 是执行节点，它写代码调接口，没有任何权限修改队列；Reviewer 是审核节点，只做输出质量评估。这三个职责在代码层强制隔离，任何节点都不能越权。这直接堵死了模型在执行阶段偷偷修改计划的可能性，也防止了规划阶段提前执行导致的状态污染。

全局步数计数器是最后一道保险，超过 25 步强制终止整个链路。这不是"提示词里告诉模型你最多做 25 步"，是代码里的硬限制，直接 raise 一个异常。

结果如何？在 35 个真实压测任务里，Medium 难度任务的 E2E Pass@1 达到 87.5%，Hard 难度任务里的目标漂移事件降到了可控范围。

## 机制二：把 RAG 的粒度对准问题的粒度

这是整个项目里我认为最值得深聊的一个机制，因为它涉及一个在很多人做 RAG 的时候会忽略的基础性问题：**检索单元的粒度，要和问题的粒度完全匹配，否则召回是假召回。**

让我从问题出发来解释。用户说"查一下茅台的股息率"，系统需要知道：在 Tushare API 里，股息率对应的字段叫 `dv_ttm`，所属接口是 `daily_basic`，参数格式是 `ts_code` 加交易日期。这是一个极度精确的、符号级别的对齐需求——不是"语义上差不多"，是字节层面的精确匹配。

传统做法是把每篇 Tushare 接口文档作为一个检索单元——340 篇文档就是 340 个 chunk，每个 chunk 几百到几千个字。问题在于，当你用"股息率"这个词去检索，最相关的那篇文档被正确地召回了，但这篇文档里列了几十个字段，模型需要在几十个字段里自己找到 `dv_ttm`。这个"从文档里找字段"的步骤，模型完成得并不可靠——它会以为看到了 `dv_ratio`（另一个看起来相关的字段）就觉得自己找到了答案。这就是字段幻觉的技术根源：检索粒度在文档级，但问题的答案在字段级，粒度错配。

我的做法是把 340 篇接口文档解析成字段级的最小检索单元。具体来说，每一行参数表格拆成一个独立的 chunk，格式是接口名加字段名加字段类型加字段描述——比如"利润表接口 字段: n_income (float) - 净利润"。这样 340 篇文档变成了 4032 个字段级 chunk，存到 FAISS 索引里。当用户问"净利润字段是什么"，检索直接返回"利润表接口 字段: n_income (float) - 净利润"，这个 chunk 本身就是答案，Coder 节点读到这个 chunk，写代码的时候直接知道该用 `n_income`，不需要再从一篇长文档里猜。

FGR-ColBERT 这篇论文（2026 年 4 月）从一个不同的角度验证了同样的道理。他们研究的是 ColBERT 的 late interaction——ColBERT 的核心思想是保留 token 级别的表示向量，让 query 和 document 在 token 粒度上做交互，而不是把整个文档压成一个向量。FGR-ColBERT 更进一步，把大模型蒸馏出的细粒度相关性信号直接集成到检索函数里。他们发现 110M 参数的小模型通过这个机制，在检索质量上能超越 27B 参数的大模型。这个结论背后的逻辑我认为完全适用于我的场景：**检索的精度不取决于模型参数量，取决于检索单元的粒度是否和问题的信息需求精确对齐。** 我做的字段级 chunking 是这个思想的一个工程化实现，不是 token 级，而是字段级，但解决的是同样的粒度错配问题。

粒度问题解决之后，还有第二个子问题：查询侧的语义鸿沟。用户说"股息率"，索引里存的是 `dv_ttm` 这个英文缩写，如果直接拿"股息率"去做关键词检索，BM25 什么都匹配不到。解法是在查询端做同义词扩展：系统维护了一张 `FINANCIAL_SYNONYMS` 字典（lib.py 的 933 到 972 行），把中文金融术语映射到对应的 API 字段名，"股息率"扩展为包含 `dv_ttm`、`dv_ratio` 的复合查询串。这样向量检索负责语义相似性的召回，BM25 关键词检索负责精确字段名的命中，两路结果按 0.7:0.3 的权重加权融合——这个权重是离线用真实查询评估调出来的，金融语义泛化需求更高所以向量权重更大。

最终结果：RAG Hit@3 从改造前的大约 62% 提升到了 93.1%，代码字段解析失败率从 5% 降到了 1%。这个数字背后的意义是：Coder 节点不再需要猜字段名，每次生成代码的时候，检索结果直接给了它正确答案。

DRAG 这篇论文（ACL 2025）研究了一个叫"幻觉叠加幻觉"的现象——当 RAG 检索结果本身就含有噪声，生成模型不仅会产生幻觉，还会把检索来的噪声和自己的幻觉叠加，产生更严重的错误。他们的解法是在检索阶段引入多 Agent 辩论，支持方和反对方对同一个检索结果做正反论证，用对抗性辩论来过滤噪声。我的多路召回融合是一个类似思想的简化实现：规则抽取、LLM 意图识别、RAG 三路分别做检索，取交集或做加权融合。单路召回有噪声，三路交叉验证能大幅降低噪声。不是学术上的对抗辩论，是工程上的多源验证。

## 机制三：让错误分类，而不是让错误重试

这是第三个核心机制，也是和大多数 Agent 系统区别最明显的地方。

先说为什么无脑重试是最差的策略。Tushare API 会产生四类本质不同的错误：字段名写错触发 KeyError、网络超时、Token 权限不够、数据真空（接口正常返回但数据不存在）。这四类错误的正确处理方式是完全不同的。字段名写错，重试有意义，因为下次可以用正确的字段名；网络超时，指数退避重试有意义；Token 权限不够，重试没有任何意义，是配置问题；数据真空，最关键，比如查一家已经停牌的公司的交易数据，接口永远返回空 DataFrame，无限重试只会把系统拖死。

如果不做错误分类，直接无脑重试，遇到数据真空场景就是死循环，直到超时。我在设计 ErrorHandler 节点的时候，花了相当多时间在错误分类上。

分类逻辑在 error_classifier.py 里，核心是一个 `ERROR_PATTERNS` 字典（108 到 139 行），对每一种错误类型定义了识别规则和对应的处理策略。识别是多维度的：异常类型（KeyError、TimeoutError、PermissionError 等）、报错消息里的关键词（比如"KeyError"加引号包着的字段名说明是字段幻觉）、API 返回的状态码。分类完了之后走完全不同的处理路径。

字段幻觉的处理路径是这样的（代码在 error_handlers.py 的 682 到 720 行）：先用正则从报错堆栈里提取出错误字段名，比如从 `KeyError: 'net_profit'` 里提取出 `net_profit`；然后压缩错误上下文，只取关键的 8 行；接着用提取出的字段名和相关 API 名组合成新的 RAG 查询，在字段级索引里检索正确字段名；最后把"字段名 `net_profit` 不存在，正确字段名是 `n_income`"这条信息打包成结构化反馈注入 Coder 的下一次生成。整个流程不调用 LLM 做判断，是纯规则加 RAG 查询。PALADIN 那篇论文（2025 年 9 月）做的是类似的事情，但规模大得多——他们在五万条失败轨迹上微调模型，训练了一个能够检索相似失败案例并生成恢复策略的 Agent，把恢复率从 32.76% 提升到了 89.68%。他们的系统泛化能力更强，能处理训练中没见过的失败模式；我的系统是有限集合上的规则覆盖，牺牲了泛化，换取了确定性和速度——字段别名表是 O(1) 查表，不需要 LLM 推理。在金融 API 场景里，失败模式是有限集合，这个取舍是合理的。

数据真空的处理路径是另一条完全不同的路：系统识别到空 DataFrame 但没有任何异常抛出的组合信号，立即把这个状态标记为"不可重试"，在最终报告里注明"查询时间段内数据不存在"，跳出循环，不触发 retry_count。这条路的代码判断是：接口调用成功（没有异常），但 DataFrame 为空，且没有"无权限"类的错误消息——三个条件同时成立才判定为数据真空，精确度很重要，避免把网络超时误判成数据真空。

还有一类特殊情况是我额外处理的：大数据查询触发 60 秒熔断。系统里的代码执行内核是一个有状态的 Python 内核，执行超过 60 秒的代码会被强制终止。60 秒这个参数是从压测数据里定的——Hard 难度任务的 P95 完成时间是 62 秒，Medium 难度的平均是 23 秒，30 秒会误杀大量正常任务，90 秒用户等待体验太差，60 秒是那个平衡点。熔断触发之后，系统不是直接返回错误，而是检查 `stdout_capture` 里有没有部分输出——如果已经有超过 100 字节的部分数据（说明任务执行到了一定进度），就返回 `[PARTIAL RESULT]` 标记加已获取的部分结果，Supervisor 识别到这个标记之后不继续执行剩余步骤，直接路由到 Reviewer 生成降级回复，告诉用户任务执行了多少进度、推荐缩小查询范围。这不是简单的超时失败，是带信息量的降级。

## 机制四：有状态执行内核解耦指令流和数据流

这个机制解决的问题更隐蔽，很多人做多步 Agent 的时候不会意识到它，但它是系统能不能 scale 到真实复杂任务的关键。

多步任务里，每一步都会产出中间数据。比如查利润表拿到了茅台的净利润，这个数据在下一步计算 ROE 的时候需要用到。最简单的处理方式是把这个数据文本化，塞进对话历史里，下一步的 Coder 从历史消息里读。问题是每往下走一步，对话历史就多一段数据，到第五步第六步的时候，历史里已经有几千个 token 的财务数据，LLM 的上下文被大量原始数据填满，推理质量直线下降，这就是"上下文膨胀"。

系统的解法是把数据流和指令流物理分离。中间数据不进对话历史，而是持久化在 StatefulPythonKernel 的 `globals` 字典里——这是一个跨步骤共享的 Python 命名空间，变量在内核里是持久的。第二步生成的 `df_income`（利润表数据帧）存在 globals 里，第三步的代码直接用变量名 `df_income` 来引用它，不需要把数据文本化后传给 LLM。对话历史里只有推理指令——"用 df_income 计算 ROE，公式是净利润除以净资产"——而不是实际的数据内容。上下文长度因此不随步骤数量线性增长，10 步任务和 3 步任务消耗的 token 差不多。

这个设计的副产品是一个重要的安全机制：AST 静态分析。在执行代码之前，系统对代码做静态分析，检查是否包含 `os.system`、`subprocess`、`__import__` 等高危调用。这不依赖沙箱，是在代码字符串级别做的规则拦截，误杀率低，性能开销可以忽略。

KernelManager 在多用户并发场景里负责隔离：每个会话有独立的内核实例和独立的 Lock。用户间并行，用户内串行——同一个用户的两次查询会抢同一个 Lock，确保步骤执行的顺序性；不同用户的查询互不干扰，globals 完全隔离。

## 整体架构的一句话总结

用 10 秒钟说清楚这个系统的本质：我用三层确定性约束对抗概率模型的天然不稳定性。第一层，状态机架构隔离概率决策，把控制流变成确定性路由；第二层，字段级 RAG 对齐检索精度和代码生成的信息需求，消灭字段幻觉；第三层，错误分类加有状态执行内核，让每种失败都有精确的恢复路径。

系统不是最复杂的，也不是 SOTA，但在金融 API 调用这个具体场景里，它把可预期的失败模式逐一覆盖了。35 个真实压测任务全部完成、87 次 Tushare API 调用 100% 成功、10 个真实故障场景里 9 个零人工干预自愈——这些数字背后是对金融场景特有失控模式的逐一工程化应对，不是靠更强的模型，是靠更克制的架构设计。

## 常见追问的深度回答

### 追问一：你的 RAG 跟普通 RAG 有什么本质区别？

普通 RAG 是文本对文本的检索，问题和文档在同一个语义空间里，向量相似度能直接衡量相关性。我的场景是自然语言对 API Schema 的检索，这是跨模态符号对齐问题——用户的自然语言意图，需要映射到严格的编程接口符号。这两件事的性质完全不同。

举一个具体例子：用户说"查股票分红情况"，你用普通文档级 RAG，检索到的是一篇叫"分红相关接口介绍"的文档，里面列了十几个字段。模型从这篇文档里读完之后，写出了 `pro.dividend(ts_code='600519.SH', dv_ratio=dv_ratio)` ——语法上看起来合理，但 `dividend` 接口里其实没有 `dv_ratio` 这个参数，正确的字段叫 `per_share`，代码运行直接 KeyError。

用字段级 RAG，检索直接命中"股利分红接口 字段: per_share (float) - 每股股利（税前）"，Coder 读到这个 chunk，直接知道应该写 `df['per_share']`，这一步出错的概率接近零。

这就是本质区别：普通 RAG 给 LLM 提供参考材料，LLM 自己判断，判断过程有概率出错；字段级 RAG 直接给 LLM 提供答案，LLM 不需要判断，出错的空间被消除。

### 追问二：你的自愈和 PALADIN 比，差在哪里？

我自己分析是泛化能力上的差距。PALADIN 的系统能处理训练时没见过的失败模式，因为它有一个专门训练过的恢复模型，在五万条失败轨迹上学过怎么从各种失败状态中恢复。我的系统是规则覆盖，如果出现一种我的 `ERROR_PATTERNS` 字典里没有预定义的新型错误，系统只能走通用重试逻辑，不能做针对性修复。

但我对这个权衡是接受的，原因有两个。第一，金融 API 调用场景的失败模式是相对有限的集合：字段名错误、网络超时、权限问题、数据真空，这几类覆盖了 95% 以上的真实错误。用规则把这些高频场景全部覆盖，比训练一个泛化的恢复模型性价比更高。第二，规则的确定性是一个优势而不是劣势——PALADIN 的 LLM 恢复策略也可能判断错，我的规则在预定义的场景里是 100% 正确的。PALADIN 的论文说恢复率是 89.68%，我在覆盖到的场景里恢复率是 100%——代价是我只覆盖了有限的场景。

### 追问三：你讲的"概率崩塌"，有没有更准确的数据支撑？

我从真实压测数据里拿数字来说。35 个任务里，Hard 难度任务（6 步以上）的首次尝试成功率，在加入所有自愈机制之前，我做过对比测试，大约是 42%——跟 0.95 的 6 次方（77%）比差距更大，因为真实场景里单步成功率不到 95%，Tushare 字段幻觉的出错概率在 Hard 任务里接近 88%，说明单步在字段层面的成功率只有 12%，只有在语义层面是高的。

加入字段级 RAG 之后，字段幻觉率从 88% 降到了不到 10%；加入状态机门控之后，目标漂移问题消失；加入错误分类自愈之后，整体 E2E Pass@1 在 Hard 任务上达到了 85% 以上。这个数字的提升路径是清晰的：不是靠单步成功率的提升，是靠系统架构对失败的系统性覆盖。

### 追问四：为什么要用 LangGraph，用普通循环不行吗？

用普通循环完全可以实现状态转移的逻辑，但 LangGraph 给了两个工程上有价值的东西。第一是节点间状态的类型安全：AgentState 是一个 TypedDict，节点的输入输出类型在代码层强制，不会出现某个节点返回了一个多出一个字段的字典导致下游崩溃。第二是路由逻辑的可读性：LangGraph 里每条路由边都是显式声明的，整个执行流程是一个可视化的图，节点之间的依赖关系一目了然，调试和追踪容易。用普通循环，状态转移逻辑散落在 if/else 里，复杂度一上来就很难维护。

但我要补充一点：LangGraph 也带来了一个额外成本，就是调试时的状态追踪比较复杂，尤其是并发多用户场景里，每个 thread_id 对应一个独立的状态图实例，出了问题要定位到具体是哪个节点的哪次执行，需要完善的日志体系。这个成本是真实存在的，不是可以忽略的。

## 关于这个系统在更大图景里的位置

我自己对这个项目的定位是很清醒的：它不是一个追求 SOTA 的研究系统，它是一个在特定领域、用有限工程手段、把 Agent 的可靠性提升到可以实际落地的程度的工程系统。

2025 年到 2026 年，Agent 领域真正值得关注的趋势不是某个新的 Benchmark 刷榜，而是整个社区开始认真对待"Agent 能不能在真实生产环境里稳定运行"这个问题。PALADIN 研究失败恢复，Self-Healing Router 研究控制流确定性，Problem Drift 研究目标保持，DRAG 研究检索噪声，FGR-ColBERT 研究检索粒度——这些工作从不同角度在构建同一件事：让 Agent 从实验室走向工业系统。

我做的这个系统，正好落在这个趋势里一个比较扎实的位置上：不够聪明，但够稳定；不够通用，但在金融场景里够可靠；不是最先进，但是可解释、可追踪、可复现。

这是我认为现阶段工业级 Agent 应该有的样子。
