### 4.4 熔断器机制详解

#### 4.4.1 熔断器代码实现（lib.py 第391-470行）

```python
class StatefulPythonKernel:
    def execute(self, code: str, max_output_length: int = 8000, timeout: int = 60) -> str:
        """
        执行Python代码，带60秒超时熔断器
        
        Args:
            code: 待执行的Python代码
            timeout: 超时时间（秒），默认60秒
        
        Returns:
            执行结果字符串：
            - 正常完成: "[DATA]: {...}" 或 "[ERROR]: ..."
            - 超时有输出: "[PARTIAL RESULT] ..."
            - 超时无输出: "[ERROR] 代码执行超过60秒..."
        """
        result_container = {"output": None, "error": None, "done": False}
        stdout_capture = io.StringIO()
        
        def target():
            """守护线程中执行代码"""
            try:
                with contextlib.redirect_stdout(stdout_capture):
                    exec(code, self.globals)
                result_container["output"] = stdout_capture.getvalue()
                result_container["done"] = True
            except Exception as e:
                result_container["error"] = str(e)
                result_container["done"] = True
        
        # 启动守护线程
        exec_thread = threading.Thread(target=target)
        exec_thread.daemon = True  # 关键：主线程退出时自动终止
        exec_thread.start()
        
        # 等待timeout秒
        exec_thread.join(timeout=timeout)
        
        # ⏰ 检查是否超时
        if not result_container["done"]:
            # 超时处理：智能降级
            partial_output = stdout_capture.getvalue()
            
            if partial_output and len(partial_output) > 100:
                # 有部分输出 → 返回PARTIAL RESULT
                return f"[PARTIAL RESULT] 数据查询耗时较长（>{timeout}秒），已获取部分结果：\n{partial_output[-500:]}"
            else:
                # 无有效输出 → 返回ERROR
                return f"[ERROR] 代码执行超过{timeout}秒，已强制终止。无有效输出。"
        
        # 正常完成
        if result_container["error"]:
            return f"[ERROR] {result_container['error']}"
        return result_container["output"]
```

### 4.6 为什么是60秒而不是30秒

**设计权衡（来自实际压测）**:

| 超时时间 | 优势 | 劣势 | 实测效果 |
|---------|------|------|---------|
| 30秒 | 响应快，用户体验好 | **误杀率高**：Medium任务平均23秒，Hard任务平均86秒，30秒会导致大量误杀 | ❌ 35个任务中8个被误杀 |
| 60秒 | **容错性强**：Medium任务全部通过，仅超大查询触发熔断 | 超时等待时间较长 | ✅ 35个任务中34个正常完成 |
| 90秒 | 几乎不误杀 | **体验差**：用户等待过长，系统反馈不及时 | ✅ 35个任务全部完成，但3个任务>90秒 |

**最终选择60秒的原因**:
1. **覆盖99%的正常查询**：压测数据显示，Hard任务平均86秒，但P95分位数为62秒
2. **避免系统卡死**：超大查询（如全市场扫描）会导致系统长时间阻塞
3. **保留降级能力**：60秒足够产生partial_output，便于Reviewer生成有价值的降级回复

## 第五部分：关键技术决策与权衡

### 5.1 为什么选择threading而不是multiprocessing

**决策背景**: Python代码执行需要进程/线程隔离，防止用户之间相互干扰。

| 方案 | 优势 | 劣势 | 是否采用 |
|------|------|------|---------|
| **multiprocessing** | - 真正的CPU并行<br>- 完全内存隔离<br>- GIL无影响 | - **进程创建开销大**（Windows上约1-2秒）<br>- 跨进程通信复杂<br>- 内存占用高（每个进程独立） | ❌ 未采用 |
| **threading** | - 创建开销小（<10ms）<br>- 共享内存，通信方便<br>- 适合I/O密集任务 | - GIL限制<br>- 内存共享需要Lock管理 | ✅ 已采用 |
| **asyncio** | - 单线程高并发<br>- 资源占用最小 | - **无法支持同步代码**<br>- Tushare/LLM SDK均为同步调用<br>- 改造成本极高 | ❌ 未采用 |

**最终选择threading的原因**:

1. **I/O密集型场景**:
   - 系统90%时间在等待：LLM调用（2-15秒）、Tushare API（1-8秒）
   - CPU计算极少：仅有数据清洗、指标计算
   - threading在I/O等待时自动释放GIL，性能损失<5%

2. **用户级并行需求**:
   - **用户之间并行**：多个用户同时访问，每个用户独立线程
   - **用户内串行**：单个用户的请求必须串行（依赖globals状态）
   - threading.Lock完美实现：
     ```python
     # lib.py 第560-645行
     class KernelManager:
         def __init__(self):
             self._kernels: Dict[str, StatefulPythonKernel] = {}
             self._locks: Dict[str, threading.Lock] = {}
         
         def get_kernel(self, thread_id: str):
             if thread_id not in self._kernels:
                 self._kernels[thread_id] = StatefulPythonKernel()
                 self._locks[thread_id] = threading.Lock()  # 每用户独立Lock
             return self._kernels[thread_id], self._locks[thread_id]
     ```

3. **启动速度**:
   - threading创建耗时：5-10ms
   - multiprocessing创建耗时：1000-2000ms（Windows）
   - **实测差异**：Easy任务8秒 vs 9.5秒（加进程开销）

4. **代码兼容性**:
   - Tushare SDK：同步调用，不支持async
   - LangChain LLM：同步调用，异步改造需重构3668行代码
   - threading零改造，直接使用

### 5.3 为什么保留最近10轮消息而不是15轮

```python
# multi_agent.py 第3200-3300行
def trim_messages(messages: List[BaseMessage], max_messages: int = 10) -> List[BaseMessage]:
    """
    保留最近N轮对话，防止上下文爆炸
    
    保留策略:
    1. 第1条消息：用户原始查询（必须保留）
    2. 最近N轮：Coder代码、Supervisor决策、Reviewer审核
    3. 变量保留：globals字典持久化，不受trim影响
    """
    if len(messages) <= max_messages:
        return messages

    return [messages[0]] + messages[-(max_messages-1):]
```

### 5.4 为什么max_retries=2而不是3

**retry_count重试机制配置**:
```python
# multi_agent.py 第743-757行
max_retries = 2  # 最多重试2次（共3次执行机会）

if is_data_empty:
    retry_count = state.get("retry_count", 0)
    if retry_count >= max_retries:
```

```
    ↓ 返回: [DATA]: {}（数据为空）
    ↓ retry_count=0 → 1

    ↓ 返回: [DATA]: {}（Q3也未发布）
    ↓ retry_count=1 >= max_retries(1)

```

**max_retries=2（推荐）**:

```
    ↓ 返回: [DATA]: {}
    ↓ retry_count=0 → 1
T+8000ms: Step 1重试1，改为2024Q3数据
    ↓ 返回: [DATA]: {}
    ↓ retry_count=1 → 2
T+16000ms: Step 1重试2，改为2024Q2数据
    ↓ 返回: [DATA]: {'revenue': 123...}  ✅ 成功
    ↓ pop(0) 继续下一步
```

```
    ↓ Q2已返回数据，但继续尝试Q1（浪费8秒）
    ↓ 总耗时：32秒（比max_retries=2多16秒）

成功率：98%（仅提升1%）
用户体验：⚠️ 等待时间过长
```

**为什么max_retries=2最优**:
1. **覆盖常见时间滞后**：季报发布延迟1-2季度，2次重试足够
2. **避免无效循环**：停牌股、退市股永远无数据，3次尝试后应降级
3. **用户体验平衡**：86秒可接受，112秒开始焦虑
4. **成本可控**：每次重试消耗1次Tushare调用+1次LLM调用
