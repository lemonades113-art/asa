## 二、5类错误分类详解

### 2.1 错误分类器定义

**代码位置：** `error_classifier.py` 第23-67行

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         5类错误分类体系                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ① auth_error (认证错误) - 401/403                                      │
│     ├─ 触发模式：["401", "403", "auth", "token", "授权", "认证失败"]   │
│     ├─ 是否可重试：❌                                                    │
│     ├─ 策略：graceful_degradation                                        │
│     ├─ 原因：Token过期、API Key无效                                      │
│     └─ 示例：Tushare Token未续费                                        │
│                                                                         │
│  ② rate_limit (限流错误) - 429                                          │
│     ├─ 触发模式：["429", "limit", "频率", "过于频繁"]                   │
│     ├─ 是否可重试：✅                                                    │
│     ├─ 策略：exponential_backoff (1→2→4→8→16秒)                         │
│     ├─ 原因：超过API调用频率限制                                         │
│     └─ 示例：1分钟内调用超过500次                                         │
│                                                                         │
│  ③ network_error (网络错误)                                             │
│     ├─ 触发模式：["timeout", "connection", "超时", "无法连接"]          │
│     ├─ 是否可重试：✅                                                    │
│     ├─ 策略：exponential_backoff                                         │
│     ├─ 原因：网络波动、服务器无响应                                       │
│     └─ 示例：Tushare服务器响应超时                                        │
│                                                                         │
│  ④ data_vacuum (数据真空)                                               │
│     ├─ 触发模式：["empty", "无数据", "null", "停牌", "退市"]           │
│     ├─ 是否可重试：❌                                                    │
│     ├─ 策略：graceful_degradation                                        │
│     ├─ 原因：股票停牌、财报未发布、数据真空期                           │
│     └─ 示例：查询停牌股票的实时价格                                       │
│                                                                         │
│  ⑤ code_error (代码错误)                                               │
│     ├─ 触发模式：["KeyError", "TypeError", "ValueError", "IndexError"] │
│     ├─ 是否可重试：✅                                                    │
│     ├─ 策略：immediate_retry                                             │
│     ├─ 原因：字段名写错、类型不匹配、索引越界                            │
│     └─ 示例：df['pe'] 但字段名是'pe_ttm'                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 三、BacktrackingRouter 策略路由器

### 3.1 核心概念

**代码位置：** `multi_agent.py` 第155-287行

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     BacktrackingRouter 策略路由器                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【核心职责】                                                            │
│  ├─ 管理失败历史（failure_history）                                      │
│  ├─ 切换备选策略（STRATEGIES）                                          │
│  ├─ 避免重复错误（记忆学习）                                            │
│  └─ 提供RCA建议（根因分析）                                            │
│                                                                         │
│  【failure_history 结构】                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  {                                                          │   │   │
│  │    "strategy": "daily接口",      ← 失败的策略名            │   │   │
│  │    "error": "KeyError: 'pe'",    ← 错误信息（截断200字）   │   │   │
│  │    "error_type": "code_error",   ← 错误类型                 │   │   │
│  │    "agent": "Coder",             ← 失败节点                 │   │   │
│  │    "timestamp": 1700000000.0     ← 时间戳                   │   │   │
│  │  },                                                         │   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  【STRATEGIES 策略库】                                                  │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  序号  │     策略名           │           说明                   │   │
│  ├───────┼─────────────────────┼─────────────────────────────────┤   │
│  │   0   │  daily接口          │  默认使用 pro.daily()          │   │
│  │   1   │  daily_basic接口    │  使用 pro.daily_basic()        │   │
│  │   2   │  income接口          │  使用 pro.income()             │   │
│  │   3   │  fina_indicator接口  │  使用 pro.fina_indicator()     │   │
│  │   4   │  备用策略            │  兜底策略                      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 策略切换流程

```
场景：用户问"查平安银行的PE"

【第1次尝试】Strategy 0: daily接口
  │
  ├─ 代码：df = pro.daily(ts_code='000001.SZ', fields='pe')
  ├─ 执行：失败！
  ├─ 错误：KeyError: 'pe' (daily接口没有pe字段)
  └─ 处理：mark_failure() → failure_history + ["daily接口 失败"]

【第2次尝试】Strategy 1: daily_basic接口 (自动切换)
  │
  ├─ BacktrackingRouter 检测到 daily接口失败
  ├─ 切换到备选策略：daily_basic
  ├─ 代码：df = pro.daily_basic(ts_code='000001.SZ', fields='pe')
  ├─ 执行：成功！
  └─ 返回：[DATA]: {"ts_code": "000001.SZ", "pe": 8.5}

【成功！✅】
```

```

第1次失败后：
[
  {

已重试 3 次，错误类型 network_error

## 四、完整错误处理流程示例

### 4.1 场景：查询停牌股票

```
用户问题："查金鸿顺（603158）的最新价格"
实际状态：该股票已停牌

执行流程：
═══════════════════════════════════════════════════════════════════════

Step 1: Coder 生成代码
        │
        ├─ df = pro.daily(ts_code='603158.SZ')
        └─ print(f"[DATA]: {df.to_dict()}")

Step 2: Tools 执行代码
        │
        ├─ 调用 pro.daily()
        ├─ 返回：[] (空DataFrame)
        └─ 包装为：[DATA]: []

Step 3: Supervisor 检查
        │
        ├─ has_data_tag = True ✓
        ├─ is_data_empty = True ❌
        └─ 跳转到 ErrorHandler

Step 4: ErrorHandler 分类
        │
        ├─ 错误信息："[]" 或 空数据
        ├─ ErrorClassifier 匹配：
        │    ├─ patterns: ["empty", "无数据", "停牌"]
        │    └─ 匹配到 data_vacuum
        ├─ is_retryable = False ❌ (停牌了，再试100次也一样)
        └─ 策略：graceful_degradation

Step 5: Level 3 降级
        │
        ├─ 生成 [PARTIAL RESULT]
        ├─ execution_status = "success" (降级成功)
        └─ 跳转到 Reviewer

Step 6: Reviewer 生成报告
        │
        ├─ 检测 [PARTIAL RESULT]
        ├─ 生成降级回复：
        │   "抱歉，金鸿顺（603158.SZ）目前处于停牌状态，
        │    暂无实时行情数据。建议：
        │    1. 查看公司公告了解停牌原因
        │    2. 查看最近一次复牌日期
        │    3. 如需分析基本面，可查询财报数据"

═══════════════════════════════════════════════════════════════════════
```

```

┌─────────────────────────────────────────────────────────────────────────┐
│                    Tushare接口 vs 常见错误对应表                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │

│  ├─ 成功返回：open, close, high, low, vol, pct_chg                    │
│  ├─ 常见错误：KeyError (字段拼写错误)                                   │
│  └─ 备选接口：daily_basic (包含PE/PB)                                  │
│                                                                         │
│  【daily_basic() - 每日指标】                                            │
│  ├─ 成功返回：ts_code, trade_date, close, dv_ttm, pe, pb              │
│  ├─ 常见错误：Empty DataFrame (股票停牌)                               │
│  └─ 降级策略：data_vacuum → 返回"停牌提示"                              │
│                                                                         │
│  【income() - 利润表】                                                  │
│  ├─ 成功返回：n_income, n_income_attr_p, total_revenue                 │
│  ├─ 常见错误：report_type 类型错误                                      │
│  ├─ 修复方案：pd.to_numeric(df['report_type'], errors='coerce')       │
│  └─ 备选接口：fina_indicator (已计算好的指标)                          │
│                                                                         │
│  【dividend() - 分红数据】                                              │
│  ├─ 成功返回：div_proc, dv_before_tax, dividend, cash_div              │
│  ├─ 常见错误：KeyError (字段不存在)                                     │

## 七、面试满分口述总结

### 7.1 一句话总结

> "这套系统的错误处理分为三步：先分类（5类错误）、再自愈（4级流转）、最后停止（硬性边界）。每个环节都有明确的条件和边界，不会让系统无限循环。"

### 7.2 流程图速记

```
用户查询
    │
    ▼
Coder生成代码 → Tools执行
    │
    ├─ 成功 → Reviewer报告
    │
    ▼
ErrorHandler分类
    │
    ├─ auth_error → Level 3降级 (不重试)
    ├─ rate_limit → 指数退避重试
    ├─ network_error → 指数退避重试
    ├─ data_vacuum → Level 3降级 (不重试)
    └─ code_error → Level 1重试 (RAG纠错)
              │
              ├─ 成功 → Reviewer报告
              │
              ├─ 失败(3次) → Level 2切换策略
              │
              ├─ 成功 → Reviewer报告
              │
              └─ 失败(5策略) → Level 4拒绝
                          │
                          ▼
                   [REJECT] 报告
```
