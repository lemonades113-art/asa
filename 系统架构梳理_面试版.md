# 多智能体金融投研系统 - 完整架构梳理

## 一、项目核心定位

**项目名称**：基于多智能体协作的金融投研自动化系统 (FinAgent v2.0)  
**核心问题**：GPT等通用大模型在金融领域的**"不可能三角"**
- ❌ **时效性缺失**：无法获取实时数据
- ❌ **计算能力弱**：无法进行复杂量化运算
- ❌ **逻辑不可控**：无法处理多步骤任务链

**解决思路**：不用单体Agent，而是构建**5个专业智能体的虚拟投研团队**

---

## 二、系统架构概览

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────┐
│                    用户输入                          │
│           (查询/分析/数据需求)                      │
└────────────────────┬────────────────────────────────┘
                     │
                     ↓
         ┌───────────────────────┐
         │    Supervisor(大脑)    │  ← 路由决策中枢
         │  任务分解 + 动态路由    │
         └───────────┬───────────┘
                     │
        ┌────────────┼────────────┐
        ↓            ↓            ↓
    ┌─────┐   ┌──────┐   ┌──────────┐
    │Coder│   │Review│   │ErrorHand │
    │执行 │   │ 分析 │   │  自愈    │
    └──┬──┘   └──┬───┘   └────┬─────┘
       │         │            │
       └─────────┼────────────┘
               (反馈)
                 ↓
    ┌─────────────────────────┐
    │  ProfileUpdater(记忆)     │
    │  用户画像 + 个性化服务     │
    └─────────────┬────────────┘
                  │
                  ↓
          ┌──────────────┐
          │  输出报告     │
          │(含数据透传)   │
          └──────────────┘
```

### 2.2 5个核心智能体

| 角色 | 职能 | 使用模型 | 关键输出 |
|:---|:---|:---|:---|
| **🧠 Supervisor** | 任务分解、路由决策、重规划 | smart(GPT-4o) | {next, reason} |
| **💻 Coder** | Python代码生成、API调用、数据处理 | smart+工具 | [DATA]、[META]、[IMAGE] |
| **👁️ Reviewer** | 报告撰写、数据质检、异常拦截 | smart | 专业分析报告 |
| **🛡️ ErrorHandler** | 错误诊断、自动修复、指数退避 | fast(GPT-4o-mini) | 修复策略、恢复建议 |
| **👤 ProfileUpdater** | 用户画像管理、偏好学习、个性化 | fast | 更新后的用户档案 |

**模型分层原理**：
- **smart**（强逻辑）：Supervisor/Coder/Reviewer需要复杂推理 → 用GPT-4o
- **fast**（轻量级）：ErrorHandler/ProfileUpdater是辅助操作 → 用GPT-4o-mini（省40% Token）

## 三、核心工作流程

### 3.1 标准执行流程（以"查询中国平安股息率"为例）

```
Step 1: 用户输入
   Input: "请查询中国平安的股息率，并评估投资价值"
   
Step 2: Supervisor 任务分解
   Action: decompose_task()
   Output: {
      "steps": [
         "获取中国平安(000001.SZ)的最新股息率数据",
         "分析股息率是否在合理范围内(0%-8%)",
         "与行业平均值对比",
         "评估投资价值并给出建议"
      ]
   }
   
Step 3: Supervisor 路由决策
   Logic: "首个步骤是获取数据" → 派给 Coder
   Decision: next = "Coder"
   
Step 4: Coder 代码执行
   Action: run_script("""
      import tushare as ts
      pro = ts.pro_api()
      
      # 获取股息率
      df_basic = pro.daily_basic(
         ts_code='000001.SZ',
         fields='ts_code,trade_date,dv_ttm,close',
         limit=1
      )
      
      print(f"[DATE]: 2025-11-30")
      print(f"[SOURCE]: Tushare Pro API")
      print(f"[META]: 股息率定义 = 最新TTM股息率(%)")
      print(f"[DATA]: {json.dumps(data)}")
   """)
   Output: [DATA]: {"stock": "中国平安", "dividend_yield": 5.15}
   
Step 5: Supervisor 数据质检
   Check: [DATA]标记是否存在 ✓
   Check: 数据是否为空 ✓
   Check: 数据范围是否合理(0%-8%) ✓
   Decision: 质检通过 → 派给 Reviewer
   
Step 6: Reviewer 报告撰写
   Input: Coder的执行结果 + 用户查询
   Action: 撰写专业分析报告
   Output: """
   【中国平安股息率分析】
   
   最新股息率：5.15%
   数据来源：Tushare Pro API
   数据日期：2025-11-30
   
   分析意见：
   - 股息率5.15%处于中等水平（A股平均3-4%）
   - 符合金融类企业的特点
   - 风险提示：过去分红不代表未来
   """
   
Step 7: ProfileUpdater 画像更新
   Extract: "用户感兴趣的领域" = 金融投研
   Extract: "用户分析深度" = 中等（要求对比、评估）
   Update: user_profile = {
      "domain": "金融投研",
      "depth": "medium",
      "interested_stocks": ["000001.SZ"],
      "analysis_preference": "定量+定性"
   }
   
Step 8: 返回用户
   Output: Reviewer的专业报告 + ProfileUpdater的个性化记录
```

### 3.2 错误处理与自愈流程（以API字段名不匹配为例）

```
场景：查询海康威视财务数据时，API返回的列名与预期不同

Step 1: Coder 第一次执行失败
   Code: df = pro.income(fields='accounts_receivable')
   Error: KeyError: 'accounts_receivable' ❌
   Coder状态: execution_status = "error"
   
Step 2: ErrorHandler 错误诊断
   Classify: 错误类型 = code_error （代码层面）
   Analyze: 根本原因 = 字段名不存在于API返回值
   Suggest: 修复策略 = "尝试多个可能的列名"
   Output: {
      "error_type": "code_error",
      "recovery_action": "modify_field_mapping",
      "suggestion": "检查可用列: ar, accounts_receivable, note_receivable"
   }
   
Step 3: Coder 自动修复代码
   Old Code: df['accounts_receivable']  ❌
   New Code: 
      ar_col = None
      for col in ['ar', 'accounts_receivable', 'note_receivable']:
         if col in df.columns:
            ar_col = col
            break
      if ar_col:
         ar_value = df[ar_col].iloc[0]
   Modified: execution_status = "pending" (重新尝试)
   
Step 4: Coder 第二次执行成功
   Result: 获取应收账款数据 ✓
   Status: execution_status = "success"
   Output: [DATA]: {"应收账款占比": 1.975}
   
Step 5: Supervisor 继续流程
   Check: 数据质检通过 ✓
   Next: 派给 Reviewer 撰写报告
```

### 3.3 重规划流程（当连续失败时）

```
场景：Coder 连续3次API失败（例如网络波动、API配额耗尽）

Step 1: 标记失败
   retry_count = 3 （超过阈值）
   error_type = "network_error"
   execution_status = "error"
   
Step 2: Supervisor 触发重规划
   Condition: retry_count >= 3 AND error_type = "network_error"
   Action: 调用 replan_task()
   
   Original Plan:
   [
      "获取最近5年日线数据",
      "计算波动率",
      "计算最大回撤"
   ]
   
Step 3: 重新分解任务
   New Plan:
   [
      "获取最近1个月日线数据（减少数据量）",
      "基于现有数据计算波动率",
      "生成备选报告（不依赖最大回撤计算）"
   ]
   
Step 4: 重新执行
   Coder 采用新策略，避免之前的失败路径
   remaining_steps 重置，retry_count 清零
```

## 四、关键设计要点

### 4.1 数据透传协议（100% 可追溯）

每个Coder执行必须按顺序输出这些标签：

```python
print(f"[DATE]: 2025-11-30")              # 数据时间
print(f"[SOURCE]: Tushare Pro API")       # 数据来源
print(f"[TIME_RANGE]: 2025-11-01 至 2025-11-30")  # 数据范围

print("[META]: 股息率定义 = TTM股息率(%)")        # 计算逻辑透传
print("[META]: 波动率定义 = 日收益率标准差 × √252")
print("[META]: 评分方法 = Min-Max归一化")

print(f"[DATA]: {json.dumps(data)}")      # 结构化数据输出
print(f"[IMAGE]: ./output/chart.png")     # 图表路径(可选)
print(f"[WARNING]: 数据异常提示")         # 异常警告(可选)
print(f"[DIAGNOSTICS]: 诊断信息")        # 执行诊断(可选)
```

**价值**：Reviewer 可以直接引用这些标签来撰写报告，确保"数据→计算→结论"完全可验证

### 4.2 质检体系（三层递进）

```
┌──────────────────────┐
│  P0: 数据存在性检查   │  [DATA]标记是否存在？数据是否为空？
└──────┬───────────────┘
       │ ✓ 通过
       ↓
┌──────────────────────┐
│  P1: 数值范围检查     │  股息率在0%-8%范围内？
└──────┬───────────────┘   无异常的极值？
       │ ✓ 通过
       ↓
┌──────────────────────┐
│  P2: 逻辑一致性检查   │  收入>支出？现金流符合预期？
└──────────────────────┘   行业对标合理？
```

**实例**：
- 数值555%股息率 → P1失败 → 输出警告，建议核实
- 应收账款超过收入100倍 → P2失败 → 标记异常，需要解释

### 4.3 模块化工具系统

Coder可以调用的工具：

| 工具 | 用途 | 权限 |
|:---|:---|:---|
| `run_script()` | 执行Python代码 | ✅ 必须调用 |
| `search_tushare_docs_local()` | 查询Tushare API文档 | ✅ 可选（遇到API问题时） |
| `get_current_datetime()` | 获取当前时间 | ✅ 可选（记录数据时间） |

**严格限制**：Coder 每条消息**必须包含至少一个tool call**（通常是run_script），否则视为失败

## 五、状态管理机制

### 5.1 MultiAgentState 结构

```python
class MultiAgentState(TypedDict):
    # 核心通信字段
    messages: List[BaseMessage]              # 对话历史
    next: str                                # 下一个执行者 ("Coder"/"Reviewer"/...)
    
    # 任务追踪字段
    task_plan: dict                          # 分解计划 {"steps": [...]}
    remaining_steps: list                    # 剩余步骤
    
    # 执行状态字段
    execution_status: str                    # "pending"/"success"/"error"
    retry_count: int                         # 重试计数
    error_type: str                          # "code_error"/"network_error"/...
    
    # 上下文字段
    user_profile: dict                       # 用户画像
    last_sender: str                         # 上一个发送者
    last_execution_data: dict                # 上次执行数据（用于验证）
    
    # 安全防护字段
    tool_call_count: int                     # 防止无限循环
    reviewer_fail_count: int                 # 防止Reviewer死循环
    message_window_size: int                 # 消息修剪窗口
```

**关键字段说明**：
- `last_sender` + `execution_status` 组成"二维决策空间"，驱动状态转移
- `retry_count` 分类管理不同错误的重试策略
- `remaining_steps` 实现"任务惯性"，避免虚假FINISH

### 5.2 状态转移矩阵

```
当前状态: last_sender=User, execution_status=pending
   ↓
Supervisor决策: 用户提交新任务 → next="Coder"
   ↓
Coder执行成功: execution_status="success"
   ↓
Supervisor数据质检: 检查[DATA]标记 + 数值范围
   ├─ ✓ 通过 → next="Reviewer" (还有剩余步骤) 或 next="FINISH"
   └─ ❌ 失败 → 生成诊断反馈，next="Coder" (重试)
   ↓
Reviewer撰写报告: 生成markdown分析
   ↓
ProfileUpdater更新画像: 提取用户偏好
   ↓
next="FINISH" → 任务完成，返回用户

错误分支:
Coder执行失败: execution_status="error"
   ↓
ErrorHandler诊断: classify_error() → 生成修复建议
   ↓
retry_count < 3 → next="Coder" (自动重试)
retry_count >= 3 → Supervisor触发重规划
```

## 六、关键创新点

### 6.1 创新点1：四级错误自愈

**问题**：传统Agent遇到API错误就停止，用户必须手动干预

**解决**：ErrorHandler自动诊断并修复
```
code_error (代码错误) → 字段映射修复 + 重新执行
network_error (网络错误) → 指数退避重试 + 限流处理
auth_error (授权错误) → Token刷新 + 重新认证
unknown_error (未知错误) → 降级方案 + 备选API
```

**实际效果**：95% 的错误能在3次重试内自动修复，无需人工干预

### 6.2 创新点2：数据完全透传

**问题**：LLM输出"推荐格力电器"但无法说明理由，用户无法验证

**解决**：强制输出 [META] + [CODE] + [DATA] + [REPORT] 四层结构
```
[META]  → 定义计算公式（例：综合评分 = 70%×股息率 + 30%×波动率）
[CODE]  → 代码执行过程（通过run_script输出）
[DATA]  → 结构化结果（JSON格式）
[REPORT] → Reviewer引用上述三项撰写分析
```

**价值**：用户能完整追溯"数据→计算→结论"，实现100%可验证

### 6.3 创新点3：智能路由与重规划

**问题**：单体Agent不知道什么时候该"放弃"某个策略

**解决**：Supervisor根据"task_plan"和"remaining_steps"动态调度
```
剩余步骤 > 0 → 继续执行后续步骤（任务惯性）
剩余步骤 = 0 → 判断是否派给Reviewer还是FINISH
连续3次失败 → 触发重规划，采用新策略
```

**效果**：避免无限循环，同时最大化任务完成率

## 九、面试中如何讲解这个项目

### 讲解框架（10分钟版）

**第1分钟**：定位问题
> "通用LLM在金融领域有三个问题：无法获取实时数据、计算能力弱、逻辑不可控。我们的解决思路是不用单体Agent，而是构建5个专业智能体的虚拟投研团队。"

**第2-3分钟**：架构介绍
> "系统由5个模块组成：Supervisor是'大脑'负责任务分解和路由，Coder是'手脚'执行代码和查询数据，Reviewer是'眼睛'进行质检和报告撰写，ErrorHandler是'免疫系统'自动修复故障，ProfileUpdater是'记忆'维护用户画像。"

**第4-6分钟**：核心流程
> 演示一个完整的请求流程：用户提问→Supervisor分解→Coder执行→质检→Reviewer报告→返回

**第7-9分钟**：三大创新
1. "四级错误自愈"：自动诊断代码错误、网络错误等，95%的问题3次重试内解决
2. "数据完全透传"：强制输出 [META]+[CODE]+[DATA]，用户能完全追溯计算过程
3. "智能重规划"：当策略失败时自动调整，避免无限循环

**第10分钟**：成效指标
> "系统的响应时间平均8秒（vs人工85分钟），修复成功率95%，Token成本比全量GPT-4低40%。"

### 常见面试题预案

**Q1: 为什么要5个智能体？一个不行吗？**
> A: 职能分离有三个优势：(1)降低单体复杂度，每个Agent有明确目标；(2)支持动态替换，比如可以把Coder替换为C++专家；(3)模型分层，Supervisor/Coder/Reviewer用smart模型，ErrorHandler/ProfileUpdater用fast模型，节省成本。

**Q2: Supervisor是怎么路由的？**
> A: 不是简单的关键字匹配，而是二维决策空间：(1)last_sender = 上一个发送者是谁；(2)execution_status = 上一步是成功还是失败。基于这两个维度，Supervisor用LLM生成结构化JSON {next, reason}，然后用降级路由保证兼容性。

**Q3: 怎么保证不会无限循环？**
> A: 四层防护：(1)retry_count达到上限就停止；(2)tool_call_count防止工具无限调用；(3)reviewer_fail_count防止Reviewer死循环；(4)最后都有FINISH端点作为保底。

**Q4: 怎么实现"数据完全透传"的？**
> A: 强制所有Coder的输出都包含 [DATE][SOURCE][META][DATA][IMAGE] 五个标签，Reviewer看到这些标签后直接引用，不需要解析。这样做的好处是整个计算链条完全可验证。

**Q5: ErrorHandler怎么修复代码错误？**
> A: 不是"猜"，而是"学习"。比如遇到KeyError，我们知道是字段名不对，就会生成一个fallback代码：尝试多个可能的列名。如果还失败，就降级方案。这个过程自动发生，用户无感知。

## 十、项目亮点总结

| 亮点 | 实现方式 | 业界对标 |
|:---|:---|:---|
| **鲁棒性** | 四级错误分类+自动修复 | 传统Agent无修复能力 |
| **可验证性** | 元数据+代码+数据三层透传 | RAG系统的黑盒输出 |
| **智能性** | Reviewer质检+逻辑验证 | 通用LLM无常识库 |
| **效率** | 8秒 vs 人工85分钟 | 单Agent无法并行 |
| **成本** | 模型分层40%节省 | 全GPT-4过昂贵 |
| **可定制** | 配置化路由+工具框架 | 代码写死不灵活 |

---

**总结**：这个项目的核心价值是通过多智能体协作、完整的错误自愈机制、和数据完全透传，解决了通用LLM在金融领域的三大问题，是从"聊天机器人"升级到"专业投研助手"的典型案例。
