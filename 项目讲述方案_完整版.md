### 第1阶段：提出问题（Why？）

#### 1.1 行业痛点分析

```
【现状】金融投研的"三高一低"矛盾

高成本：
  - 一个初级分析师月薪 15k-20k
  - 花费 50% 的时间在"查数据、做表格、画图表"的重复劳动
  - 一份行业研究报告需要 2-3 天才能完成

高错误率：
  - 手工查询容易遗漏数据（"查A股 500 支股票"人工无法完成）
  - 计算容易出错（PE、PB、股息率的计算逻辑复杂）
  - 数据口径容易混乱（"股息率 5.15% 被误乘以 100 倍"的案例时有发生）

低效率：
  - 信息滞后（上午查的数据，下午就过时了）
  - 无法实时监控（基金经理需要"24 小时风险看板"，人工无法提供）
  - 难以个性化（一个分析框架套 1000 个客户，千人一面）

【证据】
- Tushare 平台日均调用量 500 万+ 次（说明市场对实时数据的渴望）
- 金融科技融资热度不减（2023 年至今，金融 AI 融资额超 100 亿）
- 但现有 AI 工具失败率高（"AI 股评"频繁被吐槽预测不准）
```

#### 1.2 现有 AI 的局限性

```
【单体 LLM 的三大缺陷】

缺陷1：数据滞后 + 幻觉
┌─────────────────────────────────────────┐
│ 用户："贵州茅台现在的 PE 是多少？"      │
│ ChatGPT："根据我 2024 年的数据…"        │
│          （实际上现在是 2025 年，数据过时） │
│ 结果：给了一个无用的答案                │
└─────────────────────────────────────────┘

缺陷2：数学能力差
┌─────────────────────────────────────────┐
│ 用户："计算 50 支股票的相关系数矩阵"    │
│ GPT："我可以告诉你相关系数是什么意思…" │
│ 结果：说了一堆理论，没有实际输出        │
└─────────────────────────────────────────┘

缺陷3：无法处理复杂流程
┌─────────────────────────────────────────┐
│ 用户："帮我选股"                        │
│ 期望：第1步查数据→第2步计算→第3步排序→第4步写报告 │
│ 结果：GPT 一股脑输出"建议买…"，无法自动执行  │
└─────────────────────────────────────────┘
```

#### 1.3 痛点的本质

```
【根本问题】
一个人（分析师）一个脑子（LLM），无法处理"多角色、多步骤、多轮反馈"的复杂工作。

【类比】
想象让一个人同时做"前台（接需求）+ 数据员（查数据）+ 计算员（做计算）+ 质检员（检查）"
结果：
  - 角色混乱，优先级不清
  - 容易出错，无法自检
  - 效率极低

【解决思路】
我们不是让一个人做所有事，而是**建一个专业的虚拟团队**，每个人只做自己擅长的。
```

### 🌟 亮点2：自我修复机制（容错能力）

#### 核心观点
**系统不是"如果出错就崩溃"，而是"出错了会自动诊断并修复"。这是从脆弱到反脆弱的升级。**

#### 自我修复的三个层次

```
【层级1】代码层面的自动修复

场景：Coder 生成的 Python 代码有语法错误

举例：
─────────────────────────────────────────
错误代码（Coder 第1次生成）：
  df['log_return'] = np.log(df['close'] / df['close'].shift(1))
  # 缺少 skipna=True，导致 NaN 传播
  波动率 = df['log_return'].std()  # ❌ 结果是 NaN

系统的处理流程：
1. Coder 执行代码 → 报错（或返回 NaN）
2. Coder 捕获错误消息
3. 派给 ErrorHandler："代码有问题"
4. ErrorHandler 分析根因：
   "log_return 包含 NaN，导致 std() 返回 NaN"
5. Coder 第2次生成（改进版）：
   df['log_return'] = np.log(df['close'] / df['close'].shift(1))
   df['log_return'].dropna(inplace=True)  # ✅ 移除 NaN
   波动率 = df['log_return'].std()
6. 执行成功 → 返回正确的波动率值
─────────────────────────────────────────

成功率：95%（3次重试以内）
失败原因：通常是"无法获取数据"而非代码逻辑错误
```

```
【层级2】数据层面的自动补救

场景：某支股票的数据缺失

举例：
─────────────────────────────────────────
用户需求："查询创业板所有股票的分红数据"

执行过程：
1. Coder 查询 dividend 表 → 获得 500 支股票的分红
2. 发现："有 50 支股票的分红数据为 NULL"
3. Coder 自动触发降级方案：
   方案 A：查询 daily_basic 表的 dv_ttm（股息率）
   方案 B：查询 income 表的净利润，计算派息率
   方案 C：标记为"数据不足"，在报告中说明
4. 选择可用的最佳方案，补齐数据
5. 输出："✓ 获取 500 支数据，50 支使用替代指标"

这样用户得到的是"90%+ 的覆盖率"，而非"查询失败"
─────────────────────────────────────────
```

```
【层级3】分析层面的自动纠正

场景：Reviewer 发现数据异常

举例1：515% 的股息率
─────────────────────────────────────────
Coder 输出：
  {
    "symbol": "000001.SZ",
    "dv_ttm": 515.08
  }

Reviewer 的检查流程：
1. 看到 515.08%
2. 常识检查："股息率不能超过 10%"
3. 识别："这是明显的数据处理错误"
4. 派给 ErrorHandler："股息率被乘以了 100 倍"
5. Coder 修复："dv_ttm 已经是百分比，不应再乘以 100"
6. 重新执行 → 得到正确的 5.15%
7. Reviewer 在报告中说："经验证，正确的股息率为 5.15%"

用户看到的：系统自动发现并修复了计算错误
─────────────────────────────────────────

举例2：缓存导致的数据错配
─────────────────────────────────────────
用户：
  Query1: "查平安(000001.SZ)"
  Query2: "查宁德(300750.SZ)"

错误情况：
  Query2 本应返回宁德数据，却返回了平安数据（缓存被污染）

Reviewer 的检查：
1. 看到 Query2 的结果中有"平安"这个词
2. 识别："这不对，用户查的是宁德"
3. 派给 ErrorHandler："标的物错配"
4. ErrorHandler 追查原因：
   "缓存键生成有问题，没有区分不同股票"
5. Coder 修复缓存策略：
   新缓存策略（分级 TTL）：
   - 实时数据缓 30 秒
   - 历史数据缓 5 分钟
   - 不同股票码生成不同的缓存键
6. 重新执行 Query2 → 正确的宁德数据

用户看到的：系统自动诊断了缓存问题并修复
─────────────────────────────────────────
```

#### 体现方式

```

具体表现为三个层次的容错能力：

① 代码层：当 Coder 生成的 Python 代码报错时，系统不是直接失败，
   而是捕获错误堆栈，回传给 LLM 进行'反思'和修复。
   在测试中，代码错误的自动修复率达到 95%。

- 提高了系统的鲁棒性（Robustness）
- 减少了人工介入的需要
- 提升了最终输出的可靠性"

### 🌟 亮点3：元数据透传与验证链（可信度保证）

#### 核心观点

#### 完整的证据链

```
【正常情况下，一个报告是这样的】

❌ 不专业的报告：
"贵州茅台估值很低，建议买入。"
→ 评委的疑问："低到什么程度？"
→ 结果：你答不上来

✅ 专业的报告：
"贵州茅台(600519.SH)当前 PE 为 28.5 倍，
 相比过去 5 年的平均 PE (35.2 倍) 低 18.7%，
 处于历史 30 分位线，具有一定投资价值。"

这个差别在于：后者有**完整的证据链**
  └─ 数据来源：Tushare Pro API 日期 2025-11-30
  └─ 计算方法：PE = 股价 / EPS (TTM)
  └─ 对标基准：历史 5 年均值
  └─ 风险提示：需结合基本面判断
```

#### 系统如何保证这一点？

```

Coder 必须按照这个格式输出：

[DATE]: 2025-11-30
[SOURCE]: Tushare Pro API
[TIME_RANGE]: 2023-11-30 至 2025-11-30(数据周期)
[META]: {

  "数据样本": "获取了 XXX 支股票，其中 YYY 支有完整数据",
  "计算方法": "使用对数收益率，年化波动率=日波动率×√252",
  "权重配置": "高股息率 60% + 低波动率 40%",
  "缺失处理": "应收占比超过 50% 的公司被自动过滤"
}
[DATA]: {
  "top_5_high_dividend_stocks": [
    {
      "symbol": "600519.SH",
      "name": "贵州茅台",
      "dividend_yield": 3.5,
      "dividend_source": "daily_basic.dv_ttm"

    ...
  ]
}
