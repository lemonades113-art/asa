# 从第一性原理看Agent：概率崩塌与确定性约束

---

## 零、本质问题

面试官问："你的Agent系统解决了什么问题？"

我先说一个观察：

**Agent领域2025-2026年最重要的发现，不是Agent能做什么，而是Agent在哪里失败。**

PALADIN（2025.09）发现：现有Agent训练只用成功轨迹，从不暴露失败场景——但真实世界里工具失败是常态。

Self-Healing Router（2026.03）发现：ReAct每次工具失败都调用LLM重规划，19个场景要123次LLM调用——成本不可控。

Problem Drift论文（2026）发现：多Agent辩论超过一定轮数，会"忘记原始问题"——35%的案例出现"缺乏进展"，越讨论越偏。

DRAG（ACL 2025）发现：RAG引入的外部知识可能带噪声——"幻觉叠加幻觉"，错误的检索让生成更错。

FGR-ColBERT（2026.04）发现：文档级检索不够精确——需要token级的细粒度相关性信号。

**这些论文从不同角度指向同一个本质问题：概率模型在长链条执行中会"崩塌"。**

我做的金融Agent系统，是在这个大图景里找一个具体的落地点。

---

## 一、问题定义：概率崩塌的三种形态

### 形态一：错误累积

**[输入]**：用户问"分析茅台和五粮液的毛利率差异"，一个5步推理链。

**[核心问题]**：每一步都有小概率错误，错误会传播。

ATTS论文有一个公式：

```
P(整体成功) = P(步骤1) × P(步骤2) × ... × P(步骤N)
```

如果单步准确率是95%，5步链条的准确率是：
```
0.95^5 = 77%
```

近四分之一的任务会失败。这不是"某一步做错了"，是**概率的必然崩塌**。

**[我的输出]**：状态机架构，把不确定性隔离在PLAN节点，EXECUTE节点是确定性代码。

---

### 形态二：目标漂移

**[输入]**：用户问"查茅台净利润"，系统开始多轮辩论/规划。

**[核心问题]**：轮次越多，越可能"忘记原始问题"。

Problem Drift论文定义了FOCUS指标：

```
FOCUS_r = M(ŷ^(r), y) - M(ŷ^(r-1), y)
```

如果FOCUS连续为负，说明每轮讨论都在远离正确答案。论文发现35%的案例出现"缺乏进展"——agents在原地打转，越讨论越偏。

**[我的输出]**：状态机的PLAN→EXECUTE→VERIFY循环，每一步都有明确的目标约束。VERIFY节点检查"是否包含[DATA]:标记"，不是软性评估，是硬性门控。

---

### 形态三：检索噪声

**[输入]**：用户说"净利润"，RAG检索返回一篇讲"利润分析"的文章。

**[核心问题]**：检索结果语义相关但字段名不对。

DRAG论文称之为"Hallucination on Hallucination"——错误的检索让生成更错。你检索到一篇讲利润的文章，但里面没有`n_income`这个字段名，LLM就开始编。

**[我的输出]**：Field-level Chunking + 三路Schema召回。每个字段独立成chunk，检索直接命中字段名。三路投票融合，避免单路幻觉。

---

## 二、解决方案：三层确定性约束

### 约束一：结构约束——状态机架构

**[输入]**：用户问"查茅台市盈率"

**[核心逻辑]**：

```
用户Query → PLAN节点（LLM推理，输出结构化JSON）
          → EXECUTE节点（确定性代码执行，无LLM）
          → VERIFY节点（硬性门控，检查[DATA]:标记）
          → OUTPUT或回到PLAN重试
```

**Self-Healing Router论文**的启发：

他们把"控制流决策"建模成图上的路由，用Dijkstra最短路径算法做确定性路由。工具失败时，边权重变成无穷大，路径自动重算——**不需要调用LLM**。

我的状态机是类似的思想：**把概率决策限制在特定节点，其他节点用确定性逻辑**。

PLAN节点有LLM（概率），EXECUTE节点没有LLM（确定），VERIFY节点是代码判断（确定）。

**[输出]**：不确定性被"隔离"，不会扩散到整个系统。

**关键代码**（lib.py multi_agent架构）：

```python
# 状态转移：确定性路由
if state["execution_status"] == "success":
    return "output"
elif state["retry_count"] < 3:
    return "plan"  # 重试
else:
    return "reject"  # 止损
```

---

### 约束二：粒度约束——Field-level Chunking

**[输入]**：用户说"净利润"

**[核心逻辑]**：

传统RAG：文档级Chunk → 返回整个API文档 → LLM从文档里找字段名

我的RAG：字段级Chunk → 返回`n_income`字段定义 → LLM直接看到字段名

**FGR-ColBERT论文**的启发：

他们发现文档级检索不够精确，需要token级的细粒度相关性信号。ColBERT的late interaction保留token级表示，FGR-ColBERT更进一步——把LLM蒸馏的细粒度信号直接集成到检索函数。

我的Field-level Chunking是类似的粗糙近似：不是token级，而是字段级。但底层逻辑一致——**检索的最小单元要和问题的最小单元对齐**。

**代码实现**（lib.py:1158）：

```python
# Chunk格式：API名 + 字段名 + 类型 + 描述
field_chunk = f"{api_title} 字段: {field_name} ({field_type}) - {field_desc}"

# 例子：
# "利润表接口 字段: n_income (float) - 净利润"
```

**[输出]**：Hit@3从62%提升到93.1%。检索直接命中字段名，不需要LLM从文档里"猜"。

---

### 约束三：经验约束——错误分类与教训注入

**[输入]**：执行报错 "KeyError: 'pe_ratio'"

**[核心逻辑]**：

```
错误发生 → ErrorClassifier分类（字段错误）
        → 别名表查表（pe_ratio → pe_ttm）
        → 生成纠错反馈
        → 注入下一轮PLAN的System Prompt
```

**PALADIN论文**的启发：

他们训练了一个"恢复增强"的Agent——在5万条失败轨迹上训练，推理时检索最相似的失败案例，执行对应的恢复动作。恢复率从32.76%提升到89.68%。

我做的是一个简化版：不需要训练，用规则+检索实现。核心思想相同——**从失败中提取可复用的策略**。

**代码实现**（error_handlers.py:682-720）：

```python
# 别名表：高频易错字段的映射
_FIELD_ALIAS_TABLE = {
    "pe_ratio": "pe_ttm",
    "市盈率": "pe_ttm",
    "净利润": "n_income",
    # ...
}
```

**[输出]**：字段错误不需要LLM推理，O(1)查表纠错。PALADIN用LLM判断恢复策略，我用规则——更快，更确定。

## 三、与前沿论文的对话

### 对话PALADIN：训练 vs 规则

**PALADIN的做法**：
- 在5万条失败轨迹上微调
- 推理时检索相似失败案例
- LLM生成恢复策略
- 恢复率：32.76% → 89.68%

**我的做法**：
- 规则分类+别名表
- O(1)查表纠错
- 不需要训练

**对话点**：

PALADIN的泛化能力更强——能处理训练中没见过的失败模式。但我的场景更垂直——金融API的字段错误是有限集合，用规则覆盖就够了。

**权衡**：我牺牲了泛化，换取了速度和确定性。PALADIN需要LLM推理恢复策略，我是查表。

---

### 对话Self-Healing Router：LLM调用 vs 图路由

**Self-Healing Router的做法**：
- 工具失败 → 边权重变无穷大 → Dijkstra重算路径
- 19个场景，ReAct要123次LLM调用，他们只要9次
- LLM调用减少93%

**我的做法**：
- 错误分类 → 部分错误直接纠错（不调LLM）
- 部分错误走指数退避（不调LLM）
- 只有不可恢复错误才终止

**对话点**：

Self-Healing Router的洞见是：**大多数控制流决策应该是路由，而不是推理**。

路由是确定性的——图上两点之间的最短路径是唯一的。推理是概率性的——LLM可能选A也可能选B。

我的状态机走了类似的路：**EXECUTE节点是路由（确定性），只有PLAN节点是推理（概率性）**。

---

### 对话Problem Drift：目标保持

**Problem Drift论文的发现**：
- 多Agent辩论会"忘记原始问题"
- 35%的案例出现"缺乏进展"
- FOCUS指标量化漂移程度

**我的做法**：
- 状态机的每个状态都有明确目标
- VERIFY节点检查输出是否满足条件
- 最大重试次数防止无限循环

**对话点**：

Problem Drift论文提出了DRIFTJudge来检测漂移。我的VERIFY节点是类似的思想——**在执行过程中持续检查"是否还在正确轨道上"**。

但我更激进：VERIFY是硬性门控，不通过就回到PLAN重试。不是"软性评估"，是"二进制判断"。

---

### 对话DRAG：检索噪声

**DRAG的做法**：
- 多Agent辩论在检索和生成两个阶段
- 支持方vs反对方的对抗性辩论
- 解决"Hallucination on Hallucination"

**我的做法**：
- 三路Schema召回（规则+LLM意图+RAG）
- 投票融合，过滤单路幻觉
- Field-level Chunking避免粒度错配

**对话点**：

DRAG用辩论来清洗检索结果——多个Agent对同一个检索结果做正反论证。我用的是更简单的方法：**三路召回+投票**。

本质相同：**不信任单一来源，用多源验证清洗噪声**。

---

### 对话FGR-ColBERT：检索粒度

**FGR-ColBERT的做法**：
- ColBERT的late interaction保留token级表示
- 用LLM蒸馏的细粒度信号增强检索
- 110M参数的模型超越27B的Gemma 2

**我的做法**：
- Field-level Chunking
- 每个字段独立成chunk
- 不是token级，是字段级

**对话点**：

FGR-ColBERT证明了：**检索粒度越细，定位越精确**。

我的实现是粗糙近似——没有token级的late interaction，但有字段级的独立chunk。底层逻辑一致：**让检索单元匹配问题单元**。

## 六、核心洞察

1. **概率崩塌是Agent的本质问题**——不是"能不能做"，是"能做多稳定"

2. **确定性约束是解法**——不是消除概率性（做不到），是隔离和约束

3. **方案复杂度要匹配问题复杂度**——金融场景是有限失败集合，用规则够了

4. **与前沿对话的本质**——不是"引用论文"，是"回答同一个本质问题"
