## 一、问题与方案

### 核心问题

| 问题 | 表现 |
|:---|:---|
| **时效性** | 难以获取实时数据和快速响应 |
| **计算能力** | 无法自动执行复杂量化分析 |
| **可靠性** | API故障时无法自我修复 |
| **可信度** | 黑盒输出，难以追溯数据来源 |

### 解决思路

采用**多智能体协作**而非单体Agent，构建虚拟投研团队
```

## 三、核心创新

### 创新1：四级错误自愈

**案例**：海康威视财务数据抓取

| 阶段 | 内容 |
|:---|:---|
| **第1次执行** | `KeyError: 'accounts_receivable'` - 字段不存在 |
| **错误诊断** | 属于`code_error`，触发字段名模糊匹配策略 |
| **自动修复** | 生成新代码：`ar_col in ['ar', 'accounts_receivable', 'note_receivable']` |
| **第2次执行** | ✓ 成功获取数据，净现比=0.944，应收占比=1.975 |

**时序**：出错(2s) → 诊断(1s) → 生成修复(2s) → 重试成功(3s) = **约8秒自动修复，无需人工干预**

| 错误类型 | 修复策略 |
|:---|:---|
| code_error | 代码重写 |
| network_error | 指数退避重试 |
| auth_error | Token刷新 |
| unknown_error | 降级方案 |

**数据支撑**：基于50次真实API调用测试，修复成功率95%

---

### 创新2：数据透传与可追溯

| 传统方式 | 本系统 |
|:---|:---|
| 输入 → [黑盒] → 输出 | 输入 → [META定义算法] → [CODE执行] → [DATA结构化] → [REPORT引用] → 输出 |
| 用户无法验证 | 每个结论都可以追溯到源代码和原始数据 |

**例**：推荐格力电器，综合评分8.64
- [META]: 70%×股息率 + 30%×波动率
- [CODE]: yield_score=normalize(4.96), vol_score=normalize(0.15), final=8.64
- [DATA]: {"stock":"格力电器", "div_yield":4.96%, "source":"Tushare Pro"}
- [REPORT]: 引用上述计算过程完整论证

---

### 创新3：异常数据拦截

**案例**：平安银行股息率异常

| 阶段 | 处理方式 |
|:---|:---|
| **数据获取** | Coder输出：515.08% |
| **Reviewer质检** | 触发常识规则：股息率>10% → 标记异常 |
| **报告输出** | ⚠️ 数据异常警告，建议核实后再使用 |

**意义**：发现错误比隐瞒错误更重要，保护用户是第一位

**质检维度**：
- 数值范围（股息率0%-8%, 净现比0.5-2.0等）
- 历史变化（与上期/行业平均对比）
- 逻辑自洽（收入>支出, 现金流符合预期等）

### 创新点总结

| 维度 | 提升 |
|:---|:---|
| **容错能力** | 3次自动重试，95%成功修复 |
| **可追溯性** | 完整数据链路透传，100%可验证 |
| **质检能力** | 拦截明显的异常数据和逻辑错误 |
| **成本优化** | 模型分层（Smart/Fast），降低Token消耗 |

## 五、实现细节

### 数据流转示例

**案例**：高股息股票筛选

执行流程：
1. [META] 定义算法：综合评分 = 70% × 股息率 + 30% × 波动率
2. [CODE] Python代码执行：yield_score=normalize(4.96), vol_score=normalize(0.15), final=8.64
3. [DATA] 结构化输出：{"stock": "格力电器", "score": 8.64, "div_yield": 4.96%, "source": "Tushare"}
4. [REPORT] 生成报告：基于上述数据和计算逻辑，推荐格力电器

**优势**：用户可以完整追溯从数据→计算→结论的全流程，100%可验证

## 六、局限与改进

### 当前局限

| 问题 | 影响 | 改进方向 |
|:---|:---|:---|
| **数据源** | 依赖单一Tushare API | 多源备份（Wind/EastMoney等） |
| **会话记忆** | 仅支持单次对话 | 向量数据库长期记忆 |
| **时效性** | 数据可能有1-2天延迟 | 接入实时行情源 |
| **可视化** | 报告仅为Markdown文本 | 前端在线图表渲染 |

### 改进计划

- **短期（3个月）**：前端UI优化，支持图表展示
- **中期（6个月）**：向量数据库集成，跨会话记忆
- **长期（1年）**：多数据源互备，实时数据接入

### 常见问题

| 问题 | 回答 |
|:---|:---|
| **为什么不用RAG？** | RAG做检索，我们做计算。金融分析需要动态编程，不是查库 |
| **如何控制成本？** | 模型分层：复杂任务用GPT-4, 简单任务用mini, 整体降低40% |
| **数据安全怎样保证？** | 本地存储用户画像，API不共享token，无敏感数据上传 |
| **架构能否迁移？** | 可以，替换Coder工具库和Reviewer知识库，迁移成本约30% |
