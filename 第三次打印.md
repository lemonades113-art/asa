## 第一层：多步推理轨迹受控与任务编排

### 【痛点：不是技术难，是工程上的命门】

你想象这样一个场景：用户说"分析茅台和五粮液的毛利率差异"，LLM 需要：①拿到两个股票的利润表，②分别算毛利率，③做差值比较，④给出解读。这是四步，任何一步的 `DataFrame` 返回空值、或者 Coder 只做了前两步就在第三步输出了"分析完成"——整条链就早停了，而且从外部看结果"好像没问题"。

**如果不解决这个问题**，系统的 Pass@1 会大概率卡在 40% 以下，因为 LLM 的 Prompt 遵循是概率性的，不能作为硬约束。

---

### 【核心方案：三个机制联动】

**机制一：`[DATA]:` 协议是一个门控条件，不是一个标签**

`[DATA]:` 这个字符串在后处理中的具体用法是：在 `validate_coder_result()` 函数里，用 `"[DATA]:" in result` 做布尔判定，这个判定的结果，作为 `execution_status` 字段写入 `AgentState` 这个 `TypedDict`。

具体来说：

```python
# multi_agent.py - validate_coder_result()
has_data_tag = "[DATA]:" in result
if not has_data_tag and not has_result_tag and not has_partial_tag:
    return False, "缺少输出标记 [DATA]:", 1
```

这个函数返回 `(False, reason, 1)` 的时候，`Supervisor` 节点就不会弹出步骤队列里的当前步骤，而是继续把 `"pending"` 写回 `execution_status`，强制 Coder 重跑这一步。

**这才是三重验证的意义：不是验证格式，而是用格式状态控制步骤推进。**

判断链条是：
```
[DATA]:存在 → execution_status="success" 
→ remaining_steps.pop(0) → 弹出已完成步骤
→ 进入下一步

[DATA]:不存在 → execution_status="pending"  
→ remaining_steps 不变 → 重发给 Coder
```

---

**机制二：`RouteResponse` 是结构化输出约束路由，不是 Prompt 约束**

`Supervisor` 节点的决策结果必须符合：

```python
class RouteResponse(BaseModel):
    next: Literal["Coder", "Reviewer", "ErrorHandler", "FINISH"]
```

这是用 `with_structured_output(RouteResponse)` 在 Logits 层约束 Token 选择。LLM 在生成 `next` 字段的 Token 时，FSM 会过滤掉所有不属于这四个值的 Token ID——不是事后校验，是生成期就不可能产生非法路由。

**为什么不用正则事后解析？** 因为真实测试日志里出现过这个：

```
# e2e_test_output.txt 第 138 行
[StructuredOutput] with_structured_output 失败 (fallback 到正则):
Error code: 400 - {'message': 'This response_format type is unavailable now'}
```

API 限制导致 `with_structured_output` 失效，这时候降级到正则 JSON 解析。所以设计了双层：结构化输出是主路径，正则解析是兜底。这两层加在一起，格式对齐率才能达到 100%。

---

**机制三：任务计划 `task_plan` 是物理步骤队列，不是 Prompt 里的思维链**

```python
# AgentState TypedDict - multi_agent.py
task_plan: dict       # {"steps": [...], "current_step_index": 0}
remaining_steps: list # 剩余未完成步骤的物理队列
```

`remaining_steps` 这个 `list` 存在 `AgentState` 里，LangGraph 每次节点执行都把它透传下去。`Supervisor` 节点弹出第一个步骤、发给 Coder，Coder 执行后带着 `[DATA]:` 返回，验证通过后才弹出。这是**消息之外的状态存储**，和 LLM 的推理上下文完全隔离。

**效果**：这就是"从架构层隔离非法路径"的实际含义——规划节点 `Supervisor` 无执行权（它不调用 `run_script`），执行节点 `Coder` 无路由权（它不能直接跳到 `Reviewer`），路由必须由 `Supervisor` 的 `RouteResponse.next` 决定。

**结果**：中等难度任务 Pass@1 达到 87.5%（7/8）。

## 第二层：符号语义对齐与混合 RAG 优化

### 【痛点：字段幻觉是金融 Text-to-Code 的命门】

什么叫字段幻觉？LLM 说"我要查市盈率"，然后生成 `df['pe_ratio']`，但 Tushare `daily_basic` 接口里这个字段叫 `pe_ttm`。这两个词语义上 100% 相同，但符号不等价，`KeyError: 'pe_ratio'` 就这么崩了。

更糟糕的是，如果你用接口级索引做 RAG，检索"市盈率"会返回整篇 `daily_basic` 文档，文档里有 40 多个字段，LLM 要在这 40 多个字段里自己判断哪个是"市盈率"——这是在给 LLM 制造歧义。

---

### 【核心方案：B2 字段级 Chunking + 混合检索】

**第一步：将 340 篇 API 文档拆成 ~4032 个字段级语义单元**

代码在 `lib.py` 的 `_prepare_data()` 里：

```python
# 每个字段行独立成一个 chunk
field_chunk = f"{api_title} {field_name} {field_type} {field_desc}"
# 例如："daily_basic接口 pe_ttm float 市盈率（TTM）"
```

这样当 Coder 搜索"市盈率对应的字段名"时，检索候选集里直接有 `"daily_basic pe_ttm float 市盈率（TTM）"` 这个完整的字段级单元。Hit@3 就是在这个粒度下衡量的，不是接口级。

从日志可以看到：
```
[检索器] 缓存命中！接口级=340, 字段级=3692, 总计=4032
```

---

**第二步：BM25 + FAISS 混合检索，融合评分**

```python
# 融合公式
hybrid_score[idx] = vector_score[idx] * 0.7 + bm25_score[idx] * 0.3
```

FAISS 用的是 `IndexFlatIP`（内积相似度），向量用 `BGE-Small-zh-v1.5` 编码，embedding 维度 512，归一化后做内积相当于余弦相似度。BM25 用的是 `BM25Okapi`，基于 `jieba.cut_for_search` 分词。

权重 0.7/0.3 的来源：向量检索擅长语义相近（"股息率" → `dv_ttm`），BM25 擅长精确匹配（"pe_ttm"这种字母串，向量模型反而容易偏移）。

---

**第三步：查询侧同义词扩展 `_expand_query()`**

```python
# query: "查询茅台的净利润"
# 扩展后: "查询茅台的净利润 n_income net_profit"
```

"净利润" → BM25 TF 词频加权到 `n_income`、`net_profit` → 精确命中字段级 chunk。这就是"增强 BM25 精确匹配权重"的具体工程实现，不是说在 Prompt 里写"请使用标准字段名"。

**结果**：RAG Hit@3 = 93.1%，解析失败率从 5% 降到 1%。

## 第三层：指令流/数据流分离内核与故障归因自愈

### 这一层面试官最感兴趣，我用真实崩溃日志来讲】

---

### 子问题1：为什么要"数据流分离"？

考虑这个场景：Coder 第一步查了茅台 2024 年的完整利润表，得到 30 行 × 20 列的 DataFrame。如果这个数据直接作为消息放进 `messages` 列表，下一步 LLM 的 context 就多了 600 个数值。5 步下来，context 里全是数字，推理空间全被数据占满，LLM 开始在数据里找规律而不是推理步骤。

**解决方案：`StatefulPythonKernel.execute(code, self.globals)` 共享命名空间**

```python
# lib.py - StatefulPythonKernel
exec(code, self.globals)  # 所有 exec 共享同一个 self.globals 字典
```

第一步执行 `df = pro.income(ts_code='600519.SH')`，`df` 这个变量就保存在 `self.globals` 里。第二步执行 `gross_margin = df['n_income'] / df['revenue']` 时，直接从 `self.globals` 里取 `df`，是 O(1) 的 dict 查找，没有任何序列化/反序列化，也没有把 df 的内容传回 LLM。

LLM 的 context 里只有两行代码指令，不是 600 个数值。这就是"数据流与指令流分离"的具体实现：**变量在 Python 命名空间里流转，指令在 LangGraph messages 里流转，两个流永远不合并**。

---

### 子问题2：9/10 故障自愈，具体是哪 9 个、靠什么机制？

try:

`[核心逻辑]`：走 `error_handlers.py` 的四级策略路由：

**Level 1 - `ImmediateRetryHandler`**：
1. `classify_error_simple()` 检测到 `"keyerror"` 关键词 → `error_type = "field_error"`
2. `compress_error_traceback(max_lines=8)` 将 50 行压缩为 8 行（保留前 2 行错误类型 + 后 6 行定位信息），防止 context 被 traceback 撑满
3. `extract_field_error()` 用正则 `r"KeyError:\s*['\"](\w+)['\"]"` 提取到 `"dv_ratio"`
4. `_FIELD_ALIAS_TABLE.get("dv_ratio")` → O(1) 查到... 如果不在表里，走 `difflib.get_close_matches("dv_ratio", list(_FIELD_ALIAS_TABLE.values()), cutoff=0.6)` 模糊匹配到 `"dv_ttm"`
5. 生成反馈：`"[Alias Table 快速路由] 'dv_ratio' 建议替换为 'dv_ttm'"`，写入 messages，`next = "Supervisor"`，`recovery_level` 从 1 → 2

`[输出]`：Coder 下一次生成 `df['dv_ttm']`，Tushare 正常返回，`[DATA]:` 出现，验证通过，该步骤完成。

这就是一次完整的 Level 1 自愈。**关键在于 `_FIELD_ALIAS_TABLE` 是 O(1) 查表**，不需要 LLM 调用，修复速度在毫秒级。

---

**故障四：认证失败 → 直接 Level 4，不浪费重试次数**

```python
# error_handlers.py - RejectHandler
def can_handle(self, ctx: ErrorContext) -> bool:
    return ctx.recovery_level >= 4 or ctx.error_type == "auth_error"
```

如果 Tushare token 过期，`classify_error_simple()` 检测到 `"401"` 或 `"authentication"` → `error_type = "auth_error"`。`ErrorHandlerRouter.route()` 遍历策略列表，`ImmediateRetryHandler.can_handle()` 返回 False（因为 `error_type` 不在 `["code_error", "field_error", "network_error"]`），`SmartFallbackHandler`、`RCAEnhancedHandler` 也返回 False，直到 `RejectHandler` 命中。

**为什么跳过 Level 1-3？** 因为 auth 错误重试 3 次也没用，浪费 3 次 API 调用。直接 Level 4 拒绝，写入 `.learnings/ERRORS.md`，告诉用户检查 token，这才是工程上的正确决策。

---

**故障五：RCA 传播链分析 → Level 3 的根因定位**

当 Coder 成功生成代码、但 Reviewer 验证失败时，`FaultPropagationAnalyzer._detect_propagation()` 检测到：
- 上一个 Agent（Coder）success=True
- 当前 Agent（Reviewer）success=False

判定为 `PropagationType.MASKED`，confidence=0.7。

```python
# rca_module.py
AGENT_SUSPICION_WEIGHTS = {
    "Supervisor": 0.3,
    "Coder": 0.5,  # 最高嫌疑
    "Reviewer": 0.15,
    "ErrorHandler": 0.05
}
```

`analyze_root_cause()` 构建入度图，入度为 0 的节点是根因候选。再按可疑度权重排序，最终指向 Coder 生成的代码有潜在 Bug，而不是 Reviewer 自己的问题。这个诊断信息注入到下一次的 System Prompt，Coder 会在知道"上次是我的代码有问题"的前提下重新生成。

## 第四层：错题本驱动的在线策略对齐与记忆进化

### 【痛点：LLM 永远记不住上次犯的错】

比如，港股股票代码格式是 `00700.HK`，LLM 有概率生成 `0700.HK`（少一个 0）。这个错第一次犯完，下次同类问题进来，LLM 完全不知道上次的教训——因为每次对话都是全新的无状态 context。

---

### 【核心方案：`LearningRecorder` + 关键词注入】

**写入端**：每次 `RejectHandler` 触发（Level 4 放弃），调用：

```python

record_error_to_learnings(
    query=state.get("original_query", ""),
    error_message=ctx.error_message[:300],
    error_type=ctx.error_type,
    fix_hint=ctx.fallback_strategy.get("hint", "")
)
```

写入 `.learnings/ERRORS.md`，格式如：
```

[code_error] 2026-02-04 | 查询港股腾讯股价

建议: 港股代码需要5位数字前缀，如 '00700.HK'
```

每次 Reviewer 通过时，写入 `.learnings/LEARNINGS.md`：
```

成功案例: 查询茅台2024年股息率
方案: dv_ttm字段，单位已是百分比，无需除以100
```

**读取端**：下一次 Supervisor 构建 System Prompt 时：

```python

learnings_context = get_learnings_context(user_query_for_learnings, max_chars=1500)
if learnings_context:
    system_prompt += f"\n\n{learnings_context}"
```

`get_learnings_context()` 的具体实现：从 ERRORS.md 和 LEARNINGS.md 各取最近 10 条，提取 2 字以上的关键词，做 `sum(1 for kw in keywords if kw in entry)` 的关键词命中评分，按分数排序，截断到 1500 字符注入。

从测试日志可以看到：
```

```

这 235 字符是系统从历史错误里提取的，包含具体的字段修正建议，在这次推理开始前就已经注入了。这就是"在线纠偏"——不是改模型权重，而是在推理侧把历史教训硬编进 System Prompt。

**效果**：分析报告中数值 100% 具备 `[DATA]:` 来源锚点，可逐条追溯。

## 问题一：字段错误处理的真实流程——是 ErrorHandler 还是 Supervisor 在做 RAG？

你的疑问是对的，我上次有一处说法混淆了。让我把两条路清楚地分开。

### 真实链路：ErrorHandler 触发字段纠错，Supervisor 触发数据为空重试，这是两个完全独立的路径。

**路径 A：`KeyError` 字段名报错——ErrorHandler 接管**

**【输入】** Coder 写了 `df['dividend_yield']`，Python 执行后抛 `KeyError: 'dividend_yield'`，作为 ToolMessage 返回。

**【核心逻辑——全在 ErrorHandler 内部完成，分四步】**

**第一步，错误检测。** `error_handler_node` 第 2730-2733 行检查 ToolMessage 是否包含 "Error" 或 "Traceback"，命中，进入错误处理分支。

**第二步，错误压缩。** 第 2738 行调用 `error_handlers.compress_error_traceback`，把可能几十行的 Traceback 压缩为 8 行，只保留第一行（错误类型）和最后几行（出错位置），防止后续对话上下文窗口被撑爆。

**第三步，字段纠错。** 这是关键。第 2751 行调用 `extract_field_error(last_msg.content)`，用正则 `r"KeyError:\s*['\"](\w+)['\"]"` 提取出错误字段名 `dividend_yield`。然后第 2758-2765 行扫描最近 5 条消息，用 `re.search(r'pro\.(\w+)', msg.content)` 找到本次调用的是 `pro.daily_basic` 接口，保存为 `interface_hint`。

**这里不做 RAG 检索。** 注意看 `error_handlers.py` 第 217 行——ErrorHandler 在 `ImmediateRetryHandler.handle()` 里走的是先查 `_FIELD_ALIAS_TABLE`（一个硬编码字典，第 682-720 行），如果命中就直接给出建议字段名；如果没命中，就用 `difflib.get_close_matches` 做字符串模糊匹配（第 220-222 行）。它找的是已知别名，而不是重新做 RAG。

**第四步，路由回 Supervisor。** ImmediateRetryHandler 的 `handle()` 方法（第 247-258 行）返回 `{"next": "Supervisor", "last_sender": "ErrorHandler", "retry_count": ctx.retry_count + 1, ...}`，同时把字段纠错反馈放进 messages 里。

**路径 B：`[DATA]: {}` 数据为空——Supervisor 接管，且 Supervisor 在这里做 RAG**

这是我上次解释的第二个场景。代码执行没有报错，但 DataFrame 是空的，Coder 打印了 `[DATA]: {}`。三重验证在 Supervisor 里拦截到"内容为空"，Supervisor 自己在 Python 层调用 `search(user_query, top=3)`，把文档塞进给 Coder 的重试提示里（`multi_agent.py` 第 1500-1544 行）。

**这两条路不会同时走。`KeyError` 进 ErrorHandler→Supervisor 路径；`[DATA]: {}` 进 Supervisor 内部的重试路径。**

**【输出】** ErrorHandler 处理字段错误后，输出类似：

```
[Schema 字段纠错] 检测到字段 'dividend_yield' 不存在
  可能原因：字段名拼写错误 / 该接口不包含此字段
[Alias Table 快速路由] 'dividend_yield' 建议替换为 'cash_div'
[Error Preview] KeyError: 'dividend_yield' ...
```

这段消息被打包成 SystemMessage 发给 Supervisor，Supervisor 再把它转发给 Coder 做下一轮尝试。

## 问题二：第二次 RAG 检索为什么能找到正确字段？

这是一个非常尖锐的问题，真实答案是：**字段纠错机制的第一道防线根本不靠 RAG，而是靠别名表（`_FIELD_ALIAS_TABLE`）和模糊匹配。RAG 是作为 Coder 下次生成代码时的背景知识，不是字段纠错本身的手段。**

让我把这个拆清楚：

**为什么第一次 RAG 检索可能给出不准确的字段名？**

`multi_path_schema_recall` 在 Coder 进入时，三路召回返回的是一段"参考文档文本"，格式是：
```
# daily_basic → dv_ttm (float) - 股息率(税前,TTM)
```

这段文本被拼进 System Prompt，但 LLM 生成代码时有一定概率不遵守——可能因为 Prompt 太长被稀释，也可能模型"记忆"了训练集里 `dividend_yield` 这个更常见的字段名。**所以 RAG 第一次并不是"召回错了"，而是"召回了正确文档，但 LLM 没采纳"。**

**第二次为什么准了？**

看 ErrorHandler 的处理逻辑（`error_handlers.py` 第 208-234 行）：

```python
if ctx.error_type == "field_error":
    # 1. 先查 _FIELD_ALIAS_TABLE（O(1)，无 LLM）
    suggested = _FIELD_ALIAS_TABLE.get(wrong_field)
    # 2. 没命中就模糊匹配
    if not suggested:
        suggested = suggest_field_correction(wrong_field, list(_FIELD_ALIAS_TABLE.values()), cutoff=0.6)
    # 3. 生成反馈
    correction = generate_field_correction_feedback(wrong_field, suggested, ...)
    feedback_parts.append(f"[Schema 字段纠错] {correction}")
    if suggested:
        feedback_parts.append(f"[Alias Table 快速路由] '{wrong_field}' 建议替换为 '{suggested}'")
```

第二次 Coder 拿到的 System Prompt 里多了一段非常直接的硬指令：**"[Alias Table 快速路由] 'dividend_yield' 建议替换为 '...'"**。不是"参考文档"这种软提示，而是明确的字段替换指令。LLM 在有了这个明确指令的情况下，采纳率接近 100%。

但是这里有一个你需要知道的细节——`_FIELD_ALIAS_TABLE`（第 682-720 行）里并没有 `dividend_yield` 这个条目。所以真实情况是：`_FIELD_ALIAS_TABLE.get('dividend_yield')` 返回 `None`，然后走 `difflib.get_close_matches`。如果 `dividend_yield` 和别名表里的 `cash_div`、`div_procf` 相似度不到 0.6，那么 `suggested` 也是 `None`，最终 `generate_field_correction_feedback` 生成的只有：

```
【字段纠错】检测到字段 'dividend_yield' 不存在
【建议】请使用 search 工具查询 "daily_basic 接口字段列表" 获取正确的字段名。
常见字段：ts_code, trade_date, open, high, low, close, vol, amount
```

Coder 第二次收到这个反馈后，会主动调用 `search_tushare_docs_local` 工具，用 "daily_basic 接口字段列表" 作为查询词去做 RAG。这次 RAG 是 Coder 自己主动发起的工具调用，查询词比第一次更精准（直接问某个接口的字段列表），检索到 `dv_ttm` 的字段级 chunk 概率更高。这就是第二次能对的根本原因——不是 RAG 本身变准了，而是查询词被精化了，从"茅台股息率"变成了"daily_basic 接口字段列表"。

---

## 问题三：字段不存在错误的根本原因——RAG 文档本身是正确的，是 LLM 没有严格遵守

这里有三个真实原因，要分清楚：

**原因 A：LLM 在生成代码时"忽视"了 RAG 召回的正确字段名。**

三路召回已经返回了 `dv_ttm`，但 RAG 的结果是作为 System Prompt 里的"参考文本"注入的。Prompt 内容越长，LLM 对不同段落的注意力权重越分散。如果 Prompt 里同时有任务描述、用户历史、技能注入、Schema 召回提示，那么 Schema 召回那段可能被稀释——LLM 生成代码时没有把它当成强制约束，而是"参考"，最终还是按训练集中更熟悉的写法（比如 `dividend_yield`）写了代码。这是 Prompt 软约束 vs 代码硬约束的本质矛盾。

**原因 B：RAG 文档本身有时效问题。**

`error_handlers.py` 第 374 行的注释说得很直白："不是删除 RAG，而是作为 RAG 的补充——当 RAG 检索不到或字段报错时，实时查询官方 Schema。" `TushareSchemaMonitor`（第 474 行）也是专门为此设计的——监控 Tushare API 的字段变动，自动更新别名表。系统本身承认了 RAG 文档可能过期这个问题，`dv_ttm_is_percentage` 知识库（`multi_agent.py` 第 855 行）里也有明确说明："dv_ttm 可能是小数(如 5.2)或百分比(如 5.2%)"，说明即使字段名对了，字段含义也可能和预期不一样。

**原因 C：字段级 chunking 覆盖有盲区。**

`lib.py` 第 1084-1118 行的字段级 chunking 逻辑有一个过滤条件：`field_type in ('str', 'float', 'int', 'double', 'bigint', 'date')`。如果某个字段在文档里没有明确标注字段类型，或者标注格式不规范（比如写了 `varchar` 而不是 `str`），这个字段就不会被切分成独立的字段级 chunk，只保留在接口级 chunk 里。接口级 chunk 的信息粒度更粗，RAG 检索时返回的是整个接口文档，LLM 需要从一大段文字里自己找字段名，出错概率上升。

---

## 问题四：三路召回的必要性——不仅仅是找字段名

### 三路召回的主要目的是什么？

`multi_path_schema_recall` 的返回值（`lib.py` 第 1473-1495 行）包含了四类信息：

1. **股票代码**（ts_code）：`600519.SH` 这种格式需要精确匹配，规则路径用正则提取，这不是字段名
2. **指标字段名**：`dv_ttm` 这种 Tushare 字段
3. **时间范围**："近4个季度"、"2020年以来"，这是 LLM 意图路径才能理解的语义
4. **分析维度**：单只股票 / 对比 / 行业，影响 Coder 决定调用哪个接口

所以三路召回提供的是一套完整的**任务参数包**，不是单纯找字段名。

### 规则路径的独特价值

**【输入】** 用户原始问题字符串。

**【核心逻辑】** `_rule_based_schema_hints`（`lib.py` 第 1382 行）：正则提取 `\d{6}\.(SH|SZ|BJ)` 格式的股票代码；关键词匹配 `metric_alias_map`，把"股息率"直接映射到 `dv_ttm`。整个函数不调用任何外部服务，纯 Python 字典 + 正则，延迟 < 1ms。

**【输出】** `{"stocks": ["600519.SH"], "metrics": ["dv_ttm"], "raw_keywords": ["股息率"]}`

**价值：** 零延迟、零幻觉、100% 确定性。对于系统预定义的 ~8 个核心指标（ROE、PE、净利润、股息率等），这条路永远是准的，不会因为 LLM 的随机性或 RAG 的向量距离抖动而给出错误答案。

### LLM 意图路径的独特价值（以及幻觉风险如何被控制）

**【输入】** 用户原始问题，使用 `with_structured_output(SchemaIntent)` 调用大模型。

**【核心逻辑】** `extract_schema_intent`（`lib.py` 第 1422 行）让大模型把问题解析为一个严格约束的 Pydantic 对象：

```python
class SchemaIntent(BaseModel):
    stocks: List[str]          # 目标标的
    metrics: List[str]         # 关心的指标
    time_range: Optional[str]  # 时间范围
    dimension: Optional[Literal["single_stock", "comparison", "industry", "index"]]
```

**【输出】** 比如 `SchemaIntent(stocks=["600519.SH"], metrics=["股息率"], time_range="最近一期", dimension="single_stock")`

**幻觉风险如何被控制：** `with_structured_output` 工作在 Logits 层，通过 FSM（有限状态机）约束 token 选择，强制输出符合 Pydantic schema 的 JSON。`dimension` 字段只能是四个 Literal 之一，不能乱填。`lib.py` 第 1442 行还有一个 `except Exception` 兜底：LLM 路径失败时静默降级，不中断主流程——"`SchemaIntent] 结构化意图抽取失败，已降级为规则+RAG`"。

**LLM 路径的不可替代价值——时间语义理解：** 规则路径只能提取明确出现的股票代码和指标词，但 "近四个季度营收趋势" 里的 "近四个季度" 是什么日期范围？"2020年以来" 的起始日期？这些时间语义，正则表达式写不出来，只有 LLM 能理解。Coder 生成代码时如果不知道时间范围，就不知道 `start_date` 和 `end_date` 参数填什么，要么猜，要么需要再问用户。LLM 意图路径解决了这个具体问题。

### RAG 路径的独特价值

**【输入】** 经过 `_expand_query` 同义词扩展后的查询，例如"茅台股息率 dv_ttm dv_ratio dividend_yield"。

**【核心逻辑】** FAISS 向量检索（权重 0.7）+ BM25（权重 0.3）混合打分，从 ~4000 个字段级 chunk 里召回 top-5。

**【输出】** 字段级文档片段，例如：`# daily_basic → dv_ttm (float) - 股息率(税前,TTM)`

**RAG 路径的不可替代价值——长尾接口覆盖：** 规则路径的 `metric_alias_map` 只覆盖了 8 个核心指标。但 Tushare 有 300+ 个接口，每个接口有几十个字段。当用户问"季报里的商誉减值"、"港股的换手率"这类长尾问题时，规则路径和别名表都不命中，只有 RAG 能从文档里检索到对应的字段和接口组合。**三路召回的互补性在于：规则路径覆盖高频 8 个核心，LLM 路径覆盖时间语义，RAG 路径覆盖 300+ 接口的长尾字段。**

## 面试现场复述（口语化段落版，因果严密）

我在 ASA 里设计字段纠错的时候，首先要想清楚一件事：字段报错这个问题，本质上是 LLM 生成的字段名和 Tushare 实际字段名之间的对齐问题。这个对齐问题有两个来源：一是 LLM 从训练集里记了一个"看起来合理但 Tushare 不支持的"字段名，比如 `dividend_yield`；二是 Tushare 自己偶尔改字段名，导致 RAG 文档里记载的字段和 API 实际返回的不一致。

针对这两个来源，我设计了三道防线。第一道是在 Coder 进入之前，就用三路 Schema 召回给它打好预防针——规则路径零延迟锁定核心字段，LLM 意图路径解析时间语义，RAG 路径覆盖长尾接口。这三路结果拼成一段提示注入 System Prompt，本质是把"字段文档"前移到代码生成之前，减少 LLM 靠记忆猜字段名的概率。

第三道防线是 Supervisor 里的数据真空识别——当代码没报错但数据是空的，这是另一条路，和字段纠错完全分开，不要把两条路混在一起讲。

**典型的工具调用顺序（茅台股息率查询）：**
```

第二轮：LLM 收到 ToolMessage → 看到了dv_ttm字段 → tool_call: run_script("...代码...")
         → ToolMessage 返回执行结果
第三轮：LLM 收到执行结果 + 第2004行 final_answer_instruction 指令 → 直接输出 [DATA]: {...}
```

## 问题二：RAG 检索的触发条件、判断标准、相似度量化算法

### 触发时机有两处，要区分清楚

**触发 A：Coder 节点进入时的 Schema 多路召回（被动、系统触发）**

`coder_node` 第 1963-1970 行，只要 `ENABLE_SCHEMA_RECALL=true`（默认开启）且有用户查询，每次 Coder 进入都会调用 `multi_path_schema_recall(user_query)`，里面的 RAG 路径调用 `search(query, topk=5)`。这是无条件触发的，不看任何状态。

**触发 B：Coder 自己在执行过程中主动调用 `search_tushare_docs_local` 工具（主动、LLM 决策）**

这是 LLM 自主决定的工具调用。什么时候触发取决于 LLM——通常是它判断"我还不知道用哪个接口"或者"我收到了字段错误的纠错提示，要去查文档"时触发。

### 相似度计算的具体算法（对照 lib.py 第 1177-1255 行）

**【输入】** 用户查询字符串，比如"茅台600519股息率"。

**第 0 步：Query Expansion（lib.py 第 1159-1175 行）**

先把查询扩展。扫描 `FINANCIAL_SYNONYMS` 字典，发现"股息率"在 key 里，于是把对应 value 全部追加：

```
原始 query：  "茅台600519股息率"
扩展后 query：  "茅台600519股息率 dv_ttm dv_ratio dividend_yield"
```

这一步的意义后面问题七专门解释。

**第 1 步：FAISS 向量检索**

用 BGE-Small-zh-v1.5 对扩展后的 query 做 embedding，得到一个 512 维向量。L2 归一化后用 `_faiss_index.search(q_emb, k)` 做内积搜索（`IndexFlatIP`），返回 top-k×3=15 个候选的 `(score, idx)` 对。

分数归一化：
```python
max_score = float(scores[0][0])   # 最高分（第一个结果）
vector_scores[idx] = float(score) / max_score  # 归一化到 [0, 1]
```

所以每个候选的向量得分是 [0, 1] 之间的相对值，不是绝对余弦相似度。

**第 2 步：BM25 词频检索**

jieba 分词后计算每个 chunk 的 BM25 得分。BM25 的核心公式是：
```
score(q,d) = Σ IDF(t) × (TF(t,d) × (k1+1)) / (TF(t,d) + k1 × (1 - b + b × |d|/avgdl))
```
其中 k1、b 是 BM25Okapi 的默认参数。得分归一化到 [0, 1]（min-max 归一化，第 1214-1217 行）。

**第 3 步：融合打分**

```python
hybrid_score[idx] = vector_score.get(idx, 0.0) * 0.7 + bm25_score.get(idx, 0.0) * 0.3
```

两路取并集，向量没召回到的 idx，`vector_scores.get(idx, 0.0)` 返回 0，BM25 补分，反之亦然。

**第 4 步：去重 + 排序输出**

按 `hybrid_score` 降序排，同一个 API 标题（`seen_titles`）只保留最高分 chunk 避免重复。返回 top_k=5 条结果字符串。

**【输出】** 格式化字符串，包含匹配度得分，如：
```
# 每日指标 → dv_ttm (匹配度: 0.923)
字段: dv_ttm - 股息率(税前,TTM)

# 每日指标 → dv_ratio (匹配度: 0.871)
字段: dv_ratio - 股息率(现金,TTM)
```

## 问题五：Supervisor 如何注入记忆？相似度基于什么计算？

### 两层记忆，注入方式不同

**短期记忆（会话内）：** `multi_agent.py` 第 1258-1267 行。每次 Supervisor 被调用，先把当前用户查询 `add` 进 `session_short_term`，然后调 `get_memory_context(user_query_for_memory)`。短期记忆的上下文是直接取最近 N 条（`get_context_string`），**不做相似度计算，按时间顺序全返回**（`memory_system.py` 第 166-176 行）。

**长期记忆（跨会话）：** 第 1278-1284 行，优先使用 `memory_system.long_term.search_hybrid(query=user_query_for_memory, top_k=5, alpha=0.7)`。

长期记忆的相似度计算，从 `memory_system.py` 第 97-112 行的 `get_current_weight` 公式：

```python
decay_factor = math.exp(-decay_rate * days_since_last_access)
access_bonus = math.log(1 + access_count)
success_bonus = 1.0 + (success_count * 0.2)
weight = importance * decay_factor * (1 + access_bonus * 0.1) * success_bonus
```

检索时是**用户查询 vs 记忆内容**做语义匹配（`search_hybrid` 的 alpha=0.7，70% 向量 + 30% BM25），然后把匹配分 × `get_current_weight()` 得到综合分数，超过阈值 0.5 才被召回（`multi_agent.py` SmartFallback 第 2624 行）。

注入方式：检索结果以文本格式拼到 `system_prompt` 末尾（第 1272 行 `system_prompt += f"\n\n{memory_context}"`），Supervisor 自己不分析这些记忆，直接透传给下游 LLM 决策用。

## 问题七：`dv_ttm` 和 `股息率` 如何产生高相似度？算法原理

这里有两层机制叠加，缺一不可。

### 第一层：Query Expansion 把它们显式绑定在同一个向量空间

`lib.py` 第 1159-1175 行的 `_expand_query`：

```python
FINANCIAL_SYNONYMS = {
    "股息率": ["dv_ttm", "dv_ratio", "dividend_yield"],
    ...
}
# 用户 query 里有"股息率" → 扩展词 dv_ttm dv_ratio dividend_yield 追加到 query 末尾
expanded = "茅台股息率 dv_ttm dv_ratio dividend_yield"
```

**BM25 层面**：扩展后的 query 包含了 `dv_ttm`，字段级 chunk `"每日指标 字段: dv_ttm (float) - 股息率(税前,TTM)"` 里也有 `dv_ttm`，BM25 的 TF 词频直接命中，得分大幅提升。

**FAISS 层面**：BGE-Small-zh-v1.5 是在中英文混合语料上预训练的语言模型。`"dv_ttm"` 这个词本身在金融领域训练数据里和"股息率"高度共现，所以它们在语义向量空间里天然相近。这不是巧合，是预训练模型学到的领域知识。

**如果没有 Query Expansion，只靠 FAISS：** BM25 那路完全无法把 `"股息率"` 和 `"dv_ttm"` 关联起来（两个词没有字符重叠），FAISS 那路有一定语义相似度但不稳定——embedding 模型未必每次都把 "茅台600519股息率" 和 "dv_ttm (float) - 股息率(税前,TTM)" 的余弦相似度排到第一。两者叠加后稳定性显著提升。

### 第二层：FAISS IndexFlatIP 做的是精确余弦相似度

`lib.py` 第 1133-1140 行：

```python
embs = embs / norms          # L2 归一化
self._faiss_index = _faiss.IndexFlatIP(dim)   # IndexFlatIP = 内积
self._faiss_index.add(embs)
```

向量先 L2 归一化，再用内积 (`IndexFlatIP`) 检索。数学上 **L2 归一化向量的内积 = 余弦相似度**。所以 ASA 用的是精确余弦相似度，不是近似搜索（不是 HNSW 或 IVF）。

`q_emb = q_emb / norm` 在查询时也做了归一化（第 1196-1197 行）。

查询向量 q 和文档向量 d 都是单位向量，`<q, d> = |q||d|cosθ = cosθ`，返回的 scores 就是余弦相似度值（范围 [-1, 1]，归一化后是 [0, 1]）。

## 问题八：粗排 vs 精排——当前实现是什么？Cross-Encoder 为何不用？

### 当前的"粗排"具体是什么

系统目前的排序机制全部是**双塔模型（Bi-Encoder）粗排**：

- FAISS 路径：BGE-Small-zh-v1.5 把 query 和每个 chunk **分别独立** encode 成向量，做内积。query 向量和 document 向量是两个独立过模型的结果，**不交互**。
- BM25 路径：完全基于词频统计，也没有任何跨 query-document 的注意力。

融合后 `hybrid_score = vector_score × 0.7 + bm25_score × 0.3`，排序，直接输出 top-5。

这就是"粗排"的本质——**query 和 document 的语义交互是隐式的，通过向量空间的距离来近似，而不是让模型直接对 (query, document) 对做判断**。

### Cross-Encoder 精排是什么，为什么没用

Cross-Encoder 的做法是：把 `(query, document)` 拼接成一个输入喂给 BERT/RoBERTa 这类 Transformer，让模型直接输出一个相关性分数。因为 query 和 document 在同一次前向传播里经过了 Self-Attention，每个 token 都能看到另一个序列的全部内容，所以相关性判断更精准。

**没用 Cross-Encoder 的原因，从三个维度来看：**

**延迟代价**：FAISS + BM25 粗排 top-5 的时间是毫秒级（向量内积是 O(n) 矩阵运算，FAISS 做了 SIMD 加速）。Cross-Encoder 精排需要对每个 (query, document) 对单独做一次 Transformer 前向传播，如果 top-5 就是 5 次，top-15 候选集就是 15 次。单次 BERT 推理约 10-50ms，精排 15 个候选集就是额外 150-750ms，对 Agent 实时响应场景不可接受。

**规模收益递减**：字段级 chunking 后约 4000 个 chunk，每条 chunk 平均只有 1-2 句话（格式是 `"API名 字段: field_name (type) - 描述"`）。文档极短时，Bi-Encoder 的语义向量已经能很好地区分相关性，Cross-Encoder 的精排收益不显著。

**领域覆盖已在前处理**：`_expand_query` 的同义词扩展 + 字段级 chunking 联合作用，已经把查询词和文档在词汇层面对齐了。BM25 的关键词命中在这个场景下本身就很准——当 query 里出现 `dv_ttm`，文档里也有 `dv_ttm`，BM25 是 100% 精准的，不需要再用 Cross-Encoder 来"理解语义"。

---

## 面试现场一段话总结（口语化版）

我们系统的 RAG 整体是"前处理硬对齐 + 混合双塔粗排"的架构，没有 Cross-Encoder 精排。具体来说，在构建索引时做了字段级切分，把每个 API 文档拆成一行一个 chunk，每个 chunk 就是一个字段的名字加描述，文档粒度很细。检索时先做同义词扩展，把"股息率"扩展成"股息率 dv_ttm dv_ratio"，再送 FAISS 做余弦相似度、BM25 做词频匹配，0.7:0.3 融合排序，返回 top-5。Cross-Encoder 没用的核心原因是：这 4000 条 chunk 每条很短，Bi-Encoder 的精度在这个场景下已经够用，而 Cross-Encoder 的延迟代价在实时 Agent 里是不可接受的。整个 RAG 管道没有任何一步调 LLM，从触发到返回结果全是确定性算法，延迟可控在 100ms 以内。

根据上面八个问题的代码分析，现在把它们重新组织成面试现场能直接开口说的口语化叙述版本，每个都按照"场景触发→核心矛盾→逻辑推导→价值判断"的结构来讲，不用子弹点，用流水段落。

## 面试复述版：八个机制的口语化表达

### 一、Coder 节点的工具调用机制

我当时在设计 Coder 节点时，发现了一个本质矛盾：Agent 要完成"查茅台股息率"这件事，至少需要两个完全不同的能力——一是查文档弄清楚用哪个接口，二是写代码调 API 拿到数据。如果把这两件事都交给一次 LLM 推理来完成，要么文档查不准，要么代码写不好，因为一次推理很难同时处理"检索"和"生成"两种模式。

我的解决方式是给 Coder 绑定三个工具：搜索 Tushare 文档的工具、执行 Python 代码的工具、获取当前时间的工具。绑定的方式是 `bind_tools`——LangChain 会把这三个工具的函数签名以 JSON Schema 的形式追加到每次 LLM 调用的消息里，模型读到这个 Schema 之后就知道自己有三把工具可以用，以及每把工具的参数格式是什么。

实际执行时，Coder 的调用是分轮进行的。第一轮，LLM 判断"我还不知道 daily_basic 接口的字段名"，于是 response 里带了一个 tool_calls 字段，内容是"调 search_tushare_docs_local，查询词是 daily_basic 股息率字段"；第二轮，search 的结果作为 ToolMessage 回来，LLM 看到了 dv_ttm 这个字段名，然后 response 里带的是另一个 tool_calls，内容是"调 run_script，代码是调 daily_basic 接口"；第三轮，执行结果回来，加上系统在 Coder 入口注入的"你已经拿到工具结果，不要再调工具，直接输出 [DATA]: 标记的答案"这条强制指令，LLM 才输出最终结果。

这个设计的价值在于把"查文档"和"写代码"分离成两次有状态的 LLM 调用，每次调用的任务更单一，准确率更高；而且工具白名单只有三个，不存在调错工具的可能，模型选工具的决策空间极小。

---

### 二、RAG 检索的触发条件和相似度算法

我在这个系统里遇到的核心问题是：用户说的"股息率"和 Tushare 文档里写的 `dv_ttm` 是完全不同的字符串，光靠字面匹配根本搜不到，但用户又不可能知道 API 字段名，他只会说自然语言。

触发的时机有两处。第一处是 Coder 每次进入时，系统自动用用户原始查询跑一次三路 Schema 召回，结果拼进 System Prompt 作为背景知识。第二处是 Coder 在执行过程中，LLM 主动发起工具调用 search_tushare_docs_local，这由 LLM 自己决定何时触发，通常是它判断自己不确定字段名的时候。

相似度计算分四步。第一步是查询扩展，遇到"股息率"这个词，就从同义词表里查出对应的 API 字段名 dv_ttm、dv_ratio、dividend_yield，全部追加到查询字符串末尾，这样后续计算时这些字段名本身就成了查询词的一部分。第二步是 FAISS 向量检索，用 BGE-Small-zh-v1.5 把扩展后的查询 encode 成 512 维向量，所有 chunk 的向量已经在建索引时算好了，查询时做内积搜索。这里用的是 IndexFlatIP，所有向量都做了 L2 归一化，所以内积等于余弦相似度，是精确搜索而不是近似搜索。第三步是 BM25 词频检索，jieba 分词后计算 TF-IDF 加权词频分数，经过同义词扩展后查询词里已经包含了 dv_ttm 这个字符串，文档里的字段级 chunk 也包含 dv_ttm，词频直接命中。第四步是 0.7 比 0.3 融合打分，取两路得分的加权和，再去重——同一个 API 标题只保留分数最高的那一条 chunk，最终返回来自五个不同 API 的 top-5 结果。

这个设计的价值是双层冗余：向量召回解决语义不匹配，BM25 解决字符串精确匹配，两路互补，任何一路失效另一路托底。

---

### 三、错误后的 RAG 重试——是谁触发、用什么查

这个问题有一个很容易混淆的地方，我当时就被自己的设计混淆过。字段 KeyError 之后的第二次 RAG，不是 ErrorHandler 发起的，也不是 Supervisor 发起的，而是 Coder 自己主动发起的工具调用。

事情的经过是这样的：Coder 第一次写了 `df['dividend_yield']`，代码执行报了 KeyError，结果作为 ToolMessage 回来。LangGraph 把这条 ToolMessage 路由给 ErrorHandler，ErrorHandler 做的事情是从报错信息里提取出错误字段名 dividend_yield，然后去查一张硬编码的别名表，找不到的话用字符串模糊匹配算编辑距离，最终生成一段纠错建议文本，里面会说"检测到字段 dividend_yield 不存在，请使用 search 工具查询 daily_basic 接口字段列表"。这段建议通过 Supervisor 转发给 Coder 的下一轮。

Coder 第二次进入时，LLM 读到了这段建议，于是主动发起了工具调用，查询词是"daily_basic 接口字段列表"——注意这个查询词和第一次的"茅台股息率"完全不同了。正因为查询词变了，BM25 对 daily_basic 这个词的词频得分极高，能精确召回 daily_basic 接口的所有字段级 chunk，LLM 在这些 chunk 里看到了 dv_ttm 这个字段名，第二次才写出了正确代码。

所以第二次 RAG 能得到不同结果的根本原因不是算法变了，而是输入变了——查询词从模糊的业务语言变成了精确的接口+字段列表描述，检索精度自然提升。

---

### 四、RAG 召回数量和选择逻辑

RAG 每次默认召回 5 条结果，Supervisor 在数据为空时直接调的是 top=3 的紧急版本。

选择逻辑在代码层面有一道去重机制：维护一个已出现的 API 标题集合，遍历按融合分数排序的候选列表，同一个 API 标题的第二个 chunk 直接跳过，这样最终 5 条结果来自 5 个不同的 API。之所以这么做，是因为同一个 API 可能有几十个字段级 chunk，如果不去重，top-5 很可能全是同一个接口的不同字段，失去了多接口覆盖的价值。

这 5 条结果全部以字符串形式拼在一起，要么注入 System Prompt，要么作为 ToolMessage 返回。代码层面没有任何"选择"步骤，是把全部 5 条都给 LLM 看，让 LLM 自己判断哪条最相关。这个设计的好处是系统不需要为 LLM 的判断负责，给足信息让 LLM 选，出错了也有 ErrorHandler 兜底。

---

### 五、Supervisor 的记忆注入机制

Supervisor 每次被调用时，会先把当前用户查询存一份进短期记忆，然后用这个查询去检索长期记忆，把检索结果拼进 System Prompt 末尾注入给下游 LLM。

短期记忆是会话内的，每个用户（thread_id）有独立的一份，最多保存 10 条，TTL 是 30 分钟，按时间顺序全部返回，不做相似度计算。它的作用是告诉 LLM "这个用户刚才问过什么"，防止同一次会话里的上下文丢失。

长期记忆是跨会话的，检索时用用户当前查询和历史记忆内容做语义匹配，70% 向量 + 30% BM25 混合打分，同时乘以记忆的衰退权重——权重公式是"重要性乘以时间衰退因子，再乘以对数访问次数，再乘以成功次数加成"，综合分数超过 0.5 才被召回。衰退机制保证了用的是最近、最常被证明有效的经验，而不是三个月前一次失败记录。

这两层记忆被设计成可选加载的，就是说就算记忆系统不可用，主流程也能正常跑，只是少了历史经验这个加速因子。

---

### 六、纠错建议的生成机制

纠错建议的生成者是 ErrorHandler 节点，具体是 error_handlers.py 里的 ImmediateRetryHandler 处理策略。整个过程不调用 LLM，纯规则加字典，延迟不超过 5 毫秒。

它的三级降级逻辑是这样的：第一级先去查一张叫 _FIELD_ALIAS_TABLE 的硬编码字典，这张表覆盖了大约 30 个高频易错的字段名映射，比如市盈率对应 pe_ttm，净利润对应 n_income，O(1) 时间就能查完。第二级是如果别名表查不到，就用 Python 标准库的 difflib 做字符串模糊匹配，计算编辑距离相似度，相似度超过 0.6 才给出建议——这一级解决的是字段名拼写错误的情况，比如 dv_ttm 被拼成了 dv_ttm_ 这种。第三级是两级都没找到，就生成一段降级提示："请使用 search 工具查询 {接口名} 接口字段列表"——把球踢给 Coder，让它自己主动发起 RAG 查文档。

这个设计不依赖 LLM 生成建议，原因是字段名纠错是一个有确定性答案的问题，不需要用 LLM 来猜，LLM 反而可能产生幻觉给出一个"听起来对但实际不存在"的字段名。规则加字典的确定性比 LLM 的概率性在这个场景下更可靠。

---

### 七、`dv_ttm` 和 `股息率` 的相似度来源

这个问题背后的核心矛盾是：用户语言和 API 文档语言天然存在语义鸿沟。用户说"股息率"，文档里写的是"dv_ttm"，这两个字符串没有任何字符重叠，BM25 无法关联它们；而 BGE 向量模型虽然在金融语料上有一定语义理解，但如果文档 chunk 只包含 dv_ttm 而不包含任何中文描述，向量相似度也不稳定。

解决这个矛盾的关键是两层机制叠加。第一层是字段级 chunking：文档被切成"每日指标 字段: dv_ttm (float) - 股息率(税前,TTM)"这种格式的小 chunk，注意这个 chunk 里同时包含了 dv_ttm 和"股息率"这两个词。所以只要有人搜"股息率"，这个 chunk 里的"股息率"就会被 BM25 命中，同时"dv_ttm"也在同一个 chunk 里。第二层是 Query Expansion：用户的查询在送进检索前，会先被同义词表扩展，"股息率"被追加了 dv_ttm、dv_ratio、dividend_yield 这些 API 字段名，BM25 对扩展后查询词里的 dv_ttm 和 chunk 里的 dv_ttm 做词频计算，直接命中，分数极高。

向量侧的相似度来自于 BGE 模型的预训练语料——在金融领域文本里，dv_ttm 和股息率高度共现，模型学到了它们在语义空间里邻近。两路叠加后，就算其中一路在某次查询里失效，另一路也能召回正确结果，稳定性大幅提升。

---

### 八、粗排与精排——为什么不用 Cross-Encoder

我在设计检索方案时，考虑过加 Cross-Encoder 做精排，最终没有用，原因不是不懂，而是权衡后认为不值得。

先说清楚两者的区别。当前用的 FAISS 加 BM25 都属于双塔模型——查询向量和文档向量是分别独立计算的，它们在向量空间里的距离代表相关性，但 query 和 document 的 token 之间没有直接的注意力交互。Cross-Encoder 是把 query 和 document 拼成一个输入喂给 BERT，所有 token 在同一个 Self-Attention 里互相看，模型直接输出一个相关性分数，理论上更准。

不用 Cross-Encoder 的三个原因：第一个是延迟代价，FAISS 搜索加 BM25 合并的总耗时在 100ms 以内，Cross-Encoder 对每个候选对要做一次 Transformer 前向传播，如果候选集有 15 个 chunk，就是 15 次推理，单次推理约 10 到 50 毫秒，加起来 150 到 750 毫秒，在实时 Agent 场景里不可接受。第二个是文档粒度，字段级 chunk 每条平均只有一两句话，文档极短时双塔模型的语义向量已经能很好地区分相关性，Cross-Encoder 的精排收益不显著——你没必要用重锤打蚊子。第三个是前处理已经对齐，同义词扩展加字段级 chunking 在词汇层面把查询词和文档强制对齐了，BM25 在这个场景下本身就是精准的——当查询词里有 dv_ttm，文档 chunk 里也有 dv_ttm，BM25 是 100% 命中的，不需要 Cross-Encoder 来"理解语义"做补充。

## RAG定量指标（Q1-Q3）

**[输入是什么]**

`run_real_eval.py` 第 69-190 行定义了 100 道真实评测用例，每道题包含 `query`（自然语言查询）、`relevant_keywords`（ground truth 关键词列表）、`category`（分类标签）。

**[核心逻辑处理了什么]**

第 319-429 行的 `run_rag_evaluation()` 函数跑的是五个指标，我逐个拆开：

**Hit@1**（第 355-356 行）：把 `search(query, topk=5)` 返回的第一段文本拿出来，转小写，然后看 relevant_keywords 里有没有哪个词出现在里面——有就是 1，没有就是 0。本质是：前一名结果到底有没有命中正确答案。

**Hit@3**（第 359-360 行）：把前三个文档片段拼成一段字符串，同样看关键词有没有出现。这是系统实际用的召回判断——因为 Coder 拿到的是 top-5 结果，只要前三里有一个对的就算有效召回。计算公式极简单：`hit3 = any(kw in top3_text for kw in rel_kws)`，布尔值。

**MRR@5**（第 362-367 行）：Mean Reciprocal Rank，遍历 top-5 的每个文档，找到第一个命中的文档排名 `rank`，MRR = `1.0 / rank`。如果第一名命中，MRR = 1.0；第二名命中，MRR = 0.5；五名之内都没命中，MRR = 0.0。最后对全部 100 道题取平均。这个指标衡量"正确结果平均出现在第几位"，比 Hit@3 更细腻。

**Context Precision**（第 369-374 行）：`precision = relevant_docs / min(3, len(doc_sections))`。分子是 top-3 里面有多少个文档包含相关关键词，分母是 3。衡量的是检索结果的精准度——你拿回来的三个有多少是有用的。

**Context Recall**（第 376-378 行）：`recall = covered / len(rel_kws)`。分子是 ground truth 关键词里有多少被 top-3 的文本覆盖到，分母是关键词总数。衡量的是重要信息有没有被漏掉。

**E2E Pass@1**（第 529 行）则是端到端指标：`success = has_data and kw_recall >= 0.5`，必须同时满足——响应里出现 `[DATA]:` 标记（说明真的执行了 API 调用），且期望关键词至少命中 50%。

**[输出了什么]**

实测结果：RAG Hit@3 = 93.1%，MRR@5 = 0.87，Context Precision、Context Recall 和 E2E Pass@1 = 87.5%，所有数字都写在 `evaluation_results/` 目录下的 JSON 文件里，每次跑完自动落盘。

## RAG节点调用分布（Q4-Q7）

**[输入是什么]**

系统共有五个节点：Supervisor、Coder、Reviewer、ErrorHandler、ProfileUpdater。

**[核心逻辑处理了什么]**

先说结论：调用 RAG 的节点有两个，Coder 是主动调用，Supervisor 是在特定错误条件下临时调用，Reviewer 和 ErrorHandler 都不直接调 RAG。

**Coder 节点**：`multi_agent.py` 第 708 行，`coder_model = smart_model.bind_tools(coder_tools)`，Coder 绑定了工具白名单，里面包含 `search_tushare_docs_local`。每次 Coder 节点进入，它的系统提示词里会先调用 `multi_path_schema_recall(user_query, topk=5)`（`lib.py` 第 1446 行）把三路召回结果注入进来，然后 LLM 在生成代码过程中还可以再主动调用 `search` 工具。这是"每次进入都触发"的主动召回。

**Supervisor 节点**：`multi_agent.py` 第 1500 行附近，当 `validate_coder_result` 发现 `is_data_empty=True` 时，Supervisor 会直接在 Python 层调用 `from lib import search; rag_results = search(user_query, top=3)`，然后把结果拼进下一次发给 Coder 的提示里。这不是工具调用，不过 LLM，是纯 Python 函数调用——相当于 Supervisor 在帮 Coder "临时补课"。

**ErrorHandler 节点**：它本身不调 RAG，但是它路由 Coder 重新进入，Coder 进入时就会再次触发三路召回——这是间接驱动 RAG 的机制。

关键区别在 `multi_agent.py` 第 708 行和 1590 行之间的对比：Coder 的 RAG 是 LLM 工具调用（LLM 决定何时调用）；Supervisor 的 RAG 是 Python 直接 `search()` 调用（代码层强制执行）。

正常流程里，Coder 每次进入都先做三路 Schema 召回；错误恢复流程里，Supervisor 补做一次 Python 级别的文档检索，作为 Coder 第二次进入前的额外 context 补充。

## 三路融合召回（Q8-Q10）

**[输入是什么]**

用户说一句"查茅台的股息率"，这句话里没有任何 API 字段名，只有中文语义描述。

**[核心逻辑处理了什么]**

`lib.py` 第 1446-1495 行的 `multi_path_schema_recall` 函数把查询扔给三条并行路径处理：

**路径一：规则路径**（`lib.py` 第 1382 行 `_rule_based_schema_hints`）

纯 Python，零 LLM，延迟 < 1ms。用正则 `r"(?<!\d)\d{6}\.(SH|SZ|BJ)"` 提取股票代码，然后查硬编码字典 `metric_alias_map`（第 1398 行），把"股息率"映射到 `dv_ttm`，"市盈率"映射到 `pe_ttm`，"净利润"映射到 `n_income`。这是速度最快、最稳定的路径，专门保证核心金融术语不漏。

**路径二：LLM 意图路径**（`lib.py` 第 1422 行 `extract_schema_intent`）

调用 `get_chat_model(model_type="smart")`，用 `with_structured_output(SchemaIntent, method="function_calling")` 强制 LLM 返回一个 JSON 对象，schema 里有四个字段：`stocks`（股票代码列表）、`metrics`（指标字段列表）、`time_range`（时间范围字符串，如"近4个季度"）、`dimension`（分析维度，枚举：single_stock/comparison/industry/index）。规则路径只能匹配预定义的关键词，但 LLM 路径能理解"最近一个财年"、"Q3 以来"这种规则字典覆盖不到的时间语义。失败时静默降级，不中断主流程。

**路径三：RAG 路径**（`lib.py` 第 1463 行调用 `search`）

真正的向量+关键词混合检索。查询先经过 `_expand_query`（第 1159-1175 行）——扫描 `FINANCIAL_SYNONYMS` 字典（第 933-972 行），把"股息率"扩展成"股息率 dv_ttm dv_ratio dividend_yield"追加到查询末尾，让 BM25 的词频权重也能命中字段名。然后：FAISS IndexFlatIP 做向量检索取 `top_k * 3 = 15` 个候选，BM25Okapi 做关键词检索取前 30 个候选，按 `hybrid_score = 0.7 * vector_score + 0.3 * bm25_score` 融合（第 1220-1225 行），排序后去重——同一个 API 只保留最高分 chunk。

**为什么三路都要？** 规则路径快但覆盖有限；LLM 路径理解语义但有延迟和幻觉风险；RAG 路径覆盖所有字段文档但召回不精确。任意一路失败，另外两路托底。三路融合输出一段中文提示（第 1474-1494 行），格式是"规则识别到的标的 / 规则识别到的关键字段 / LLM意图抽取的指标 / 文档召回片段"，拼接到 Coder 的 System Prompt 里。

**[输出了什么]**

Coder 在写代码之前就知道："你要查的股息率，字段名是 dv_ttm，接口是 daily_basic，参数格式是 ts_code + trade_date"，不需要凭记忆猜，不会写出 `dividend_yield` 这种不存在的字段名。

## DAG 架构实现（Q11-Q13）

**[输入是什么]**

DAG 在这个项目里指的是 LangGraph 构建的有向状态图（Directed Acyclic Graph，但实际上是带条件边的有向图，可以有环）。

**[核心逻辑处理了什么]**

用 LangGraph 的 `StateGraph(MultiAgentState)` 定义图结构：

- **节点**：Supervisor、Coder、Reviewer、ErrorHandler、ProfileUpdater 各自是一个 Python 函数
- **状态**：`MultiAgentState`（第 609-630 行），是一个 TypedDict，包含 `messages`（消息历史，用 `operator.add` 做追加合并）、`remaining_steps`（剩余任务步骤队列）、`retry_count`（重试计数）、`error_type`（错误类型）、`total_step_count`（全局步数，防无限循环）等字段
- **边**：Supervisor 节点用 `RouteResponse` Pydantic 模型（结构化输出）决定下一个节点是 Coder / Reviewer / ErrorHandler / FINISH，形成条件边

执行流的 Trace 举例（从代码实际跑出来的日志可以看到这个顺序）：
```
用户 → Supervisor（分解任务，生成 remaining_steps）
      → Coder（三路召回 + 工具调用 + run_script 执行）
      → Supervisor（三重验证）
        ├── 验证通过 → Reviewer（生成最终回答）→ FINISH
        └── 验证失败 → ErrorHandler（根因分类）→ Supervisor（携带纠错建议）→ Coder（第二次）
```

DAG 的关键不是"无环"，而是"节点职责分离 + 状态流转可审计"。每次状态转移都有明确的条件判断，不靠 LLM 隐式决定。`total_step_count` 全局计数器超过 25 就强制 FINISH，是代码层的死限，不依赖大模型"意识到自己在循环"。

## 工具权限控制（Q14-Q16）

**[输入是什么]**

哪些节点可以调用哪些工具，必须在架构层固定，不能让 LLM 自己决定。

**[核心逻辑处理了什么]**

`multi_agent.py` 第 681-708 行：

```python
# YAML 驱动加载，硬编码 fallback
coder_tools = [_TOOL_REGISTRY[n] for n in _allowed_names if n in _TOOL_REGISTRY]
if not coder_tools:
    coder_tools = _HARDCODED_CODER_TOOLS
coder_model = smart_model.bind_tools(coder_tools)   # Coder 绑定工具
reviewer_model = smart_model                         # Reviewer 无工具
```

工具白名单优先从 YAML 配置文件（`routing_config.json`）加载 `_allowed_names`，加载失败则 fallback 到代码里硬编码的 `_HARDCODED_CODER_TOOLS` 列表。`bind_tools` 是 LangChain 的机制——把工具定义注入到 LLM 的 function calling schema 里，LLM 只有在工具被 bind 进去的情况下才能调用它。Supervisor 用 `smart_model.invoke(messages)`，没有 `bind_tools`，所以从 LLM 层面就没有调用任何工具的可能性——它能调 search 是因为代码层直接 `from lib import search` Python 调用，不是工具调用。

Reviewer 同样没有绑定任何工具，它只做文本生成。ErrorHandler 也不绑定 API 工具——它只做错误分析和路由决策。这是架构层的职责强制，不是靠 Prompt 里说"你不要调工具"来约束的。

## 召回验证不依赖相似度分数（Q17-Q19）

**[为什么不用相似度？]**

这个问题非常核心。相似度分数是 RAG 内部评估检索质量的指标，但它衡量的是"文档和查询语义有多接近"，不是"代码有没有跑通、数据有没有真正查出来"。你完全可以检索出相似度 0.95 的文档、但 Coder 生成的代码还是用了错误字段名、API 调用返回空 DataFrame——在这种情况下，相似度分数高没有任何意义。

真正能说明"这步任务成功了"的信号，只有一个：代码执行后打出了 `[DATA]:` 标记，且内容非空。`[DATA]:` 是在 `lib.py` 第 756-803 行的 `_format_execution_result` 函数里，根据代码执行结果自动添加的——不是 LLM 写进去的，是后处理函数写进去的。如果有 Traceback，加 `[ERROR]:`；如果返回 DataFrame 或 JSON，加 `[DATA]:`；如果是计算结果，加 `[RESULT]:`。这个标记反映的是代码执行的实际结果状态。

**[为什么是 Supervisor 验证而不是 Reviewer？]**

Reviewer 是最终输出节点，它的职责是把数据转成用户友好的自然语言——它需要的前提条件是数据已经就绪、经过验证。让 Reviewer 来做数据验证，相当于让"写报告的人"同时兼职"质检员"，职责混乱。

Supervisor 是路由控制节点，它在 Coder 每次执行后都要决定"下一步走哪条边"，天然就是验证的最佳位置——通过验证才放行到 Reviewer，不通过就路由到 ErrorHandler 或重发 Coder。这是状态机控制的核心逻辑：门控在中间节点，不在终点节点。

## 错误处理流（Q20-Q23）

**[输入是什么]**

Coder 执行代码报了 KeyError。

**[核心逻辑处理了什么]**

实际的路由路径是：**Coder → Supervisor（三重验证失败）→ ErrorHandler → Supervisor（携带纠错建议）→ Coder（第二次）**，不是直接 Coder → Supervisor → ErrorHandler 的线性流。

详细流程：
1. Coder 执行工具 `run_script`，结果作为 ToolMessage 回到图
2. 图路由回 Supervisor，Supervisor 调用 `validate_coder_result`（第三重验证检测到 `[ERROR]:` 标记），判断失败
3. Supervisor 根据错误内容路由到 `ErrorHandler`
4. ErrorHandler 做根因分类（`error_classifier.py` 第 142-172 行），KeyError 命中 `"keyerror"` 关键词，`error_type = "code_error"` 或 `"field_error"`，策略是 `rag_field_correction`
5. ErrorHandler 生成字段纠错建议，路由回 Supervisor
6. Supervisor 根据 `field_not_found` 策略（`ERROR_RETRY_CONFIG` 第 523-528 行，`max_retries=1`）补做一次 RAG，把纠错建议 + 新文档一起发给 Coder
7. Coder 第二次进入，再次触发三路召回

**ErrorHandler 如何决定纠正动作？**

`error_classifier.py` 第 108-139 行定义了 `ERROR_PATTERNS`，每种错误类型有关键词列表 + `is_retryable` + `strategy` 三个属性。分类器逐个 pattern 遍历，第一个匹配到的 pattern 决定处理策略。`data_vacuum` 的 `is_retryable = False`，`should_reject_query` 返回 True，直接跳到 Reviewer 生成说明；`field_error` 的 `is_retryable = True`，进字段纠错路径。

**重跑同一个查询结果会一样吗？**

不会，因为第二次进 Coder 时，System Prompt 里多了两段内容：ErrorHandler 注入的字段纠错建议（"你用了 dividend_yield，这个字段不存在，正确字段是 dv_ttm"）+ Supervisor 补做的 RAG 文档（直接命中字段级 chunk "daily_basic → dv_ttm (float) - 股息率(税前,TTM)"）。这两段新的 context 会让 LLM 在第二次生成代码时走完全不同的路径。

**RAG 每次返回几条，用哪几条？**

`search(query, topk=5)` 默认返回 top-5，但内部 FAISS 搜 `top_k * 3 = 15` 个候选，BM25 取前 30 个，融合后取最高分的 top-5（且每个 API 只保留最高分 chunk）。Coder System Prompt 里的三路召回结果会被截断到 1500 字符（第 1469-1470 行），全部注入，不是只用第一条。

## 字段纠错机制（Q24-Q28）

**[输入是什么]**

执行器报 `KeyError: 'dividend_yield'`。

**[核心逻辑处理了什么]**

"dv_ttm"和"股息率"的映射，是通过两个并行机制完成的：

**机制一：FINANCIAL_SYNONYMS 词典**（`lib.py` 第 933-972 行）

```python
"股息率": ["dv_ttm", "dv_ratio", "dividend_yield"],
```

这是 Python 字典，写死在代码里，用于查询扩展——不用余弦相似度，不用向量，就是字符串匹配。当用户说"股息率"，`_expand_query` 函数会把这三个字段名追加到查询末尾，BM25 因此能精确命中含有 "dv_ttm" 的字段级 chunk。

**机制二：字段级 Chunk 的向量检索**

字段级 chunk 的格式是 `"{API名} 字段: {field_name} ({field_type}) - {field_desc}"`（`lib.py` 第 1109 行），比如 `"每日指标 字段: dv_ttm (float) - 股息率(税前,TTM)"`。这个 chunk 经过 BGE-Small-ZH-v1.5 嵌入成 512 维向量，存入 FAISS IndexFlatIP。用户查询扩展后 embed，做内积相似度检索，相似度最高的字段 chunk 就包含正确字段名。这一路是余弦相似度（IndexFlatIP + normalize = cosine）。

**纠错建议是谁生成的？**

不是 LLM 生成的，是 `error_handlers.py` 里的 `generate_field_correction_feedback(wrong_field, interface_hint)` 函数——根据错误字段名和接口名，拼一段模板提示："你用了 `{wrong_field}` 这个字段，在 `{interface_hint}` 接口中不存在，请通过文档检查正确字段名。"这个函数是纯 Python，返回格式固定的字符串，不需要 LLM。真正靠 RAG 检索的是"正确字段名是什么"这个信息——它来自第二次 RAG 召回返回的字段级 chunk。

## 记忆注入机制（Q29-Q36）

**[输入是什么]**

Supervisor 每次启动时，在决定下一步之前，先查询记忆系统。

**[核心逻辑处理了什么]**

记忆系统由三层构成（`memory_system.py`）：

**短期记忆 ShortTermMemory**（第 124-193 行）：`max_size=10, TTL=30min`，用 `OrderedDict` 实现 FIFO 淘汰，存储最近的对话轮次内容。`get_context_string()` 返回最近 5 条记忆拼成的提示文本，注入 Supervisor System Prompt。

**长期记忆 LongTermMemory**：优先用 ChromaDB（`chromadb.PersistentClient`，hnsw:space=cosine）做语义检索，同时双写 JSON 文件做 fallback。ChromaDB 不可用时从 JSON 恢复。`add()` 方法在存入前做内容哈希去重——`memory_id = ltm_{md5(content)[:12]}`，相同内容不重复存储，只更新访问时间。

**记忆衰退公式**（`memory_system.py` 第 97-112 行）：

```python
weight = importance * exp(-decay_rate * days_since_access) 
       * (1 + log(1 + access_count) * 0.1) 
       * (1 + success_count * 0.2)
```

其中 `decay_rate = 0.1`，30 天未访问的记忆权重降到 `exp(-0.1 * 30) ≈ 0.05`，约初始值的 5%。访问次数通过对数加成——访问越多衰退越慢，`log(1+access_count)` 确保加成不会无界增长。`success_count >= 3` 的高价值策略通过 `cleanup_decayed(min_weight=0.1)` 豁免被清理。

**用户画像**由 ProfileUpdater 节点在每次任务完成后后置提取，用 `fast_model`（qwen-turbo）从对话历史里抽取用户偏好（喜欢表格还是图表、关注净利润还是营收），持久化到 `user_profile` 字段，下次 Supervisor 启动时直接读取注入。

**成功策略记忆**：每次任务成功后，ProfileUpdater 提取"这次成功用了什么路径、哪个接口、哪些字段"，存入 LongTermMemory，category="successful_strategy"。下次遇到语义相似查询时，`get_memory_context(user_query)` 用向量检索（ChromaDB cosine）召回最相关的历史策略，压缩到 1500 字符注入 Supervisor。

**触发时机**：Supervisor 每次被调用，在生成任务计划之前，先调用 `get_memory_context()` 和 `get_learnings_context()`（从 `.learnings/ERRORS.md` 和 `.learnings/LEARNINGS.md` 读取），把历史经验前置注入，相当于在模型推理开始之前就把"上次的教训"告诉它。

## 评估体系（Q37-Q41）

**[输入是什么]**

两套独立评测：RAG 召回质量评测 + 端到端系统评测。

**[核心逻辑处理了什么]**

评测数据集（`run_real_eval.py`）：100 道 RAG 评测题（5 类：接口选择、字段名查询、字段纠正、参数理解、多表关联）+ 100 道 E2E 评测题（easy/medium/hard 三档，都是真实 Tushare API 调用）。

**RAG 评测算法**：上面 Q1-Q3 已经详细讲过，核心是关键词匹配而不是 LLM 打分——ground truth 是人工标注的必须出现的字段名字符串，客观、可复现、不依赖 judge 模型。

**E2E 评测算法**：Pass@1 判定标准是 `has_data AND kw_recall >= 0.5`，不是让人工评测。`has_data` 用字符串匹配 `"[data]:"` 来判断（第 524 行），不是 LLM 评分——因为 `[DATA]:` 标记是代码层强制添加的，存在即代表真实 API 调用成功了。`kw_recall` 看期望关键词的覆盖率，比如查"茅台股息率"，期望关键词是 `["dv_ttm", "股息率", "600519"]`，至少要命中一半。

**文档格式**：评测结果存 JSON（`evaluation_results/real_eval_{timestamp}.json`），结构是 `{"rag": {"summary": {...}, "per_case": [...]}, "e2e": {"summary": {...}, "per_case": [...]}}`。per_case 里每道题都有 `retrieved_snippet`（实际召回的文档片段，前 300 字符）和 `final_answer`（实际的系统响应，前 500 字符），可以逐题审查。

## 幻觉防御机制（Q42-Q43）

**[输入是什么]**

金融场景里最危险的幻觉不是大模型"编故事"，而是生成语义正确但 API 里不存在的字段名——`dividend_yield` 而不是 `dv_ttm`。

**[核心逻辑处理了什么]**

系统用了三道防线，每道防线解决不同维度的幻觉：

**第一道：三路 Schema 召回前置注入**。在 Coder 开始写代码之前，强制把正确的接口名、字段名注入到 System Prompt 里。LLM 在有"dv_ttm 是股息率字段名"这条 context 的情况下，生成 `dividend_yield` 的概率极低——不是靠"禁止幻觉"的 Prompt，而是靠"提供正确答案"的信息注入。这对应"源头拦截"。

**第二道：代码执行层的协议标记**。`_format_execution_result`（`lib.py` 第 756 行）在代码执行后处理阶段根据实际执行结果自动打标记——不是让 LLM 决定打什么标记，是根据有没有 Traceback、有没有结构化数据来确定性地打。LLM 没办法伪造 `[DATA]:`，因为这个标记在 LLM 控制层之外。

**第三道：三重验证门控**。Supervisor 的 `validate_coder_result` 检查三件事：有没有 `[DATA]:` 标记（验证1）、内容是否非空（验证2，检查 `{}` / `[]` / `null` 等空指示符）、有没有 `[ERROR]:` 标记（验证3）。三关任何一关没过，Coder 的输出不会传递给 Reviewer——用户永远不会看到一个"看起来像数据但其实是编的"回答。

这三道防线形成了闭环：没有正确字段名 → 召回注入提供正确字段名；字段名还是错了 → 代码执行报 KeyError 打 `[ERROR]:` 标记 → 三重验证拦截 → ErrorHandler 字段纠错 → 再次注入正确字段名。系统的 `[DATA]:` 协议遵守率接近 99% 不是靠"告诉模型要遵守"，而是靠代码层强制打标 + 验证层拦截的工程双保险。
