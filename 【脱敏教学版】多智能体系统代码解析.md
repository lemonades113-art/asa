## 🧠 模块一：Supervisor（大脑）的动态路由逻辑

### 1.1 核心概念

Supervisor 是整个系统的智能路由器，负责：
- **任务分解**：将用户的复杂需求拆解成子任务
- **动态路由**：根据执行状态和错误信息，智能决策下一步
- **质检把关**：确保消息质量，防止 Token 爆炸
- **降级策略**：在智能决策失败时，使用关键字降级路由

### 1.2 脱敏代码片段

```python
# ============ 模块一：Supervisor 的动态路由逻辑 ============

from typing import Literal
from dataclasses import dataclass

@dataclass
class MultiAgentState:
    """统一的状态管理类"""
    messages: list                      # 完整的对话历史
    next: str                          # 下一步目标节点
    retry_count: int                   # 当前重试次数
    user_profile: dict                 # 用户画像（历史偏好、风险承受力等）
    execution_status: str              # 执行状态 (pending/running/success/error)
    last_sender: str                   # 上一步的执行者身份
    error_classification: str          # 错误分类 (code_error/network_error/auth_error/unknown)


class Supervisor:
    """Supervisor 节点：智能路由决策器"""
    
    def decompose_task(self, user_request: str, user_profile: dict) -> dict:
        """
        任务分解：将用户需求拆解成子任务清单
        
        思路：
        1. 分析用户请求的关键词（如"获取股票数据"、"生成投研报告"）
        2. 基于历史用户画像（user_profile），选择合适的执行路径
        3. 构建子任务的依赖关系
        """
        # 关键字识别：从请求中提取意图
        keywords = self._extract_keywords(user_request)
        
        # 经验库查询：查找已知问题的预防方案
        known_issues = self._query_experience_base(keywords)
        
        # 子任务生成：构建执行计划
        subtasks = []
        for keyword in keywords:
            subtask = {
                "type": keyword,
                "dependencies": [],
                "fallback_strategy": known_issues.get(keyword, "default")
            }
            subtasks.append(subtask)
        
        return {
            "subtasks": subtasks,
            "execution_plan": self._build_dag(subtasks)
        }
    
    def supervisor_node(self, state: MultiAgentState) -> dict:
        """
        Supervisor 的核心决策函数：
        1. 消息修剪（防止 Token 爆炸）
        2. 执行状态检查（判断是否出错）
        3. 动态路由决策（下一步执行谁）
        4. 质检把关（确保消息质量）
        """
        
        # ◆ 第一步：消息修剪逻辑
        cleaned_messages = self._prune_messages(state.messages)
        # 保留最近N条消息，删除超出时间窗口的消息
        # 删除冗余的错误堆栈和重复信息
        
        # ◆ 第二步：执行状态检查
        if state.execution_status == "error":
            # 存在错误，进入错误处理流程
            state.next = "ErrorHandler"
            return {"next": state.next}
        
        if state.execution_status == "success" and state.last_sender == "Coder":
            # Coder 执行完成，转交给 Reviewer 质检
            state.next = "Reviewer"
            return {"next": state.next}
        
        # ◆ 第三步：智能决策路由
        if state.last_sender == "Reviewer":
            # Reviewer 反馈质检结果
            quality_score = self._parse_quality_score(state.messages[-1])
            
            if quality_score < 0.7:
                # 质量不达标，返回给 Coder 重新执行
                state.retry_count += 1
                if state.retry_count > 3:
                    # 重试次数超过阈值，执行降级策略
                    state.next = self._fallback_keyword_route(state)
                else:
                    state.next = "Coder"
                return {"next": state.next}
            else:
                # 质量达标，任务完成
                state.next = "END"
                return {"next": state.next}
        
        # ◆ 第四步：降级路由（智能决策失败时使用关键字）
        if state.retry_count > 5:
            state.next = self._fallback_keyword_route(state)
            return {"next": state.next}
        
        # 默认路由：优先分配给 Coder
        state.next = "Coder"
        return {"next": state.next}
    
    def _prune_messages(self, messages: list) -> list:
        """
        消息修剪策略：
        - 保留最近 20 条消息
        - 删除超出 48 小时的消息
        - 合并连续的相同角色消息
        - 删除冗余的错误堆栈
        """
        # 时间窗口过滤
        current_time = time.time()
        filtered = [m for m in messages if current_time - m["timestamp"] < 48*3600]
        
        # 消息数量限制
        if len(filtered) > 20:
            filtered = filtered[-20:]
        
        return filtered
    
    def _fallback_keyword_route(self, state: MultiAgentState) -> str:
        """
        降级路由：当智能决策失败时，基于关键字进行路由
        
        例如：
        - 请求包含 "数据" 关键字 → 路由给 Coder
        - 请求包含 "报告" 关键字 → 路由给 Reviewer
        - 请求包含 "错误" 关键字 → 路由给 ErrorHandler
        """
        last_message = state.messages[-1]
        
        keyword_routes = {
            "数据": "Coder",
            "报告": "Reviewer",
            "错误": "ErrorHandler",
            "缓存": "ToolCache",
        }
        
        for keyword, target in keyword_routes.items():
            if keyword in last_message.get("content", ""):
                return target
        
        # 都不匹配时，返回初始状态
        return "Coder"


# ============ Supervisor 的路由决策树 ============
"""
执行状态流程图：
                ┌─ Success ──┐
                │            │
      Pending ──┤            ├─→ Reviewer ──┐
                │            │              │
                └─ Error ────┴─→ ErrorHandler
                    │
                    └─→ 质检失败 ──→ Coder (重试)
                            │
                            └─→ 重试超限 ──→ Fallback Route
"""
```

---

**[打开代码片段]**

各位，这是 Supervisor 的核心逻辑。我把它比作"大脑"的原因是：

1. **任务分解**（decompose_task）：当用户说"帮我分析这只股票"时，Supervisor 不是直接去做，而是拆成若干子任务。比如"获取数据"、"计算指标"、"生成报告"，并且记录它们的依赖关系。

2. **动态路由**（supervisor_node）：这是最关键的。看这个代码：
   - 如果上一步是 Coder，执行成功了，就转给 Reviewer 去质检
   - 如果 Reviewer 反馈质量不达标（质量分 < 0.7），就退回给 Coder 重做
   - 如果重试超过 3 次还失败，就执行"降级策略"

3. **消息修剪**（_prune_messages）：这解决了一个关键问题——Token 爆炸。想象对话进行到第 100 轮，历史消息堆积成山，会耗尽大模型的上下文。所以我们：
   - 只保留最近 20 条消息
   - 删除 48 小时以前的消息
   - 合并重复内容

4. **降级策略**（_fallback_keyword_route）：如果智能决策失败（比如重试 5 次还没解决），系统会切换到"关键字路由"——这是一个**完全基于规则的、不依赖大模型的路由方式**。比如请求包含"数据"字眼就转给 Coder，包含"报告"就转给 Reviewer。这确保了系统的容错能力。

**核心设计哲学**：Supervisor 不像传统的消息队列系统一直重复一个动作。它是**自适应的**——能根据执行结果动态调整下一步动作，并且有降级方案。

## 🔧 模块二：Coder（执行者）的有状态执行与数据标记

### 2.1 核心概念

Coder 是系统的执行者，负责：
- **有状态执行**：跨多次调用保持变量状态（不像普通函数调用一样每次重置）
- **数据标记**：为每条输出数据标记时间、来源、质量等元数据
- **防御性编程**：对输入进行校验，对输出进行断言
- **缓存机制**：避免重复的 API 调用，降低成本和延迟

### 2.2 脱敏代码片段

```python
# ============ 模块二：Coder 的有状态执行与数据标记 ============

from datetime import datetime

class StatefulExecutor:
    """有状态的执行器：保持跨调用的变量状态"""
    
    def __init__(self):
        # 持久化的执行环境（跨多次调用保持状态）
        self.execution_context = {}
        self.variable_store = {}        # 存储已声明的变量
        self.metadata_registry = {}     # 存储数据的元数据
    
    def execute_code(self, code_snippet: str, variables: dict) -> dict:
        """
        执行一段代码片段，并保持状态
        
        关键特性：
        1. 变量持久化：def a = 1 的 a 不会被销毁
        2. 输出标记：自动为输出加上时间戳、来源等标签
        3. 错误捕获：执行错误时返回完整堆栈和建议
        """
        
        # ◆ 第一步：变量注入（从历史状态恢复）
        execution_env = self.variable_store.copy()
        execution_env.update(variables)
        
        # ◆ 第二步：代码执行
        try:
            # 执行代码片段（这里隐藏具体的执行引擎实现）
            exec(code_snippet, execution_env)
            
            # 更新状态存储
            self.variable_store = {
                k: v for k, v in execution_env.items() 
                if not k.startswith("__")
            }
            
            result = {
                "status": "success",
                "output": execution_env.get("output", None),
                "variables_changed": list(self.variable_store.keys())
            }
        except Exception as e:
            result = {
                "status": "error",
                "error_type": type(e).__name__,
                "error_message": str(e),
                "suggestion": self._suggest_fix(str(e))
            }
        
        # ◆ 第三步：数据标记（为每个输出加上元数据）
        result["metadata"] = self._tag_output(result, variables)
        
        return result
    
    def _tag_output(self, result: dict, source_variables: dict) -> dict:
        """
        数据标记：为输出加上可追溯的标签
        
        标签格式：[DATE][SOURCE][META][DATA]
        """
        now = datetime.now()
        
        metadata = {
            "[DATE]": now.isoformat(),
            "[SOURCE]": "Coder",
            "[META]": {
                "quality_check": self._check_data_quality(result.get("output")),
                "data_type": type(result.get("output")).__name__,
                "size_bytes": len(str(result.get("output")).encode()),
            },
            "[DATA]": result.get("output")
        }
        
        return metadata
    
    def _check_data_quality(self, data) -> dict:
        """
        数据质量检查：
        - 空值检测
        - 类型一致性
        - 范围合理性
        """
        checks = {
            "is_empty": data is None or len(str(data)) == 0,
            "type_valid": isinstance(data, (dict, list, str, int, float)),
            "completeness": 1.0,  # 完整性评分
        }
        return checks
    
    def _suggest_fix(self, error_message: str) -> str:
        """
        错误建议：根据错误类型给出修复建议
        
        例如：
        - KeyError: "缺少必需的字段 'xxx'，请检查输入数据结构"
        - TypeError: "类型不匹配，期望 int，收到 str"
        - NetworkError: "网络连接失败，尝试重新连接"
        """
        if "KeyError" in error_message:
            return "缺少必需的字段，请检查输入数据结构"
        elif "TypeError" in error_message:
            return "类型不匹配，检查变量类型转换"
        elif "NetworkError" in error_message:
            return "网络连接失败，30秒后重试"
        else:
            return "未知错误，请查阅错误日志"


class CoderNode:
    """Coder 节点：执行任务的工作者"""
    
    def __init__(self):
        self.executor = StatefulExecutor()
        self.tool_cache = ToolCache()
    
    def coder_node(self, state: MultiAgentState) -> dict:
        """
        Coder 的执行流程：
        1. 从 Supervisor 获取任务
        2. 使用大模型生成代码
        3. 执行代码并获取结果
        4. 返回标记后的输出给 Reviewer
        """
        
        # ◆ 第一步：从 state 中提取任务
        task_description = state.messages[-1]["content"]
        
        # ◆ 第二步：调用大模型生成代码
        # 这里会传递 CODER_SYSTEM_PROMPT（含有详细的编码规范）
        generated_code = self._generate_code_from_llm(
            task_description,
            system_prompt=CODER_SYSTEM_PROMPT
        )
        # prompt 中包含的规范：
        # - 变量命名规范（snake_case）
        # - API 调用最佳实践
        # - 错误处理规范
        # - 数据质量检查清单
        
        # ◆ 第三步：执行生成的代码
        execution_result = self.executor.execute_code(
            code_snippet=generated_code,
            variables=state.user_profile  # 注入用户历史信息
        )
        
        # ◆ 第四步：缓存和标记
        if execution_result["status"] == "success":
            # 将成功的结果缓存（避免重复调用）
            self.tool_cache.set(
                key=task_description,
                value=execution_result,
                ttl=3600  # 1小时有效期
            )
        
        # ◆ 第五步：构造返回消息
        output_message = {
            "role": "Coder",
            "content": self._format_output(execution_result),
            "metadata": execution_result.get("metadata"),
            "status": execution_result["status"]
        }
        
        state.messages.append(output_message)
        state.execution_status = execution_result["status"]
        state.last_sender = "Coder"
        
        return {"messages": state.messages}
    
    def _generate_code_from_llm(self, task: str, system_prompt: str) -> str:

        调用大模型生成代码
        
        这里的 system_prompt 包含了所有编码规范（已脱敏删除）
        """
        # 实现细节已隐藏
        # 核心流程：task + system_prompt → LLM → Python code
        pass
    
    def _format_output(self, result: dict) -> str:
        """将执行结果格式化为可读的消息"""
        if result["status"] == "success":
            return f"✓ 执行成功\n数据：{result['output']}"
        else:
            return f"✗ 执行失败\n错误：{result['error_message']}\n建议：{result['suggestion']}"


class ToolCache:
    """
    工具缓存：用于存储 API 调用结果，避免重复调用
    
    设计思路：
    - Key: 任务描述的 hash
    - Value: 执行结果
    - TTL: 时间到期自动删除
    """
    
    def __init__(self):
        self.cache = {}
        self.ttl_map = {}
    
    def set(self, key: str, value, ttl: int):
        """存储结果，设置生存时间"""
        self.cache[key] = value
        self.ttl_map[key] = time.time() + ttl
    
    def get(self, key: str):
        """获取结果，检查是否过期"""
        if key in self.cache:
            if time.time() < self.ttl_map[key]:
                return self.cache[key]
            else:
                del self.cache[key]  # 删除过期数据
        return None
```

**[打开代码片段]**

现在讲 Coder，它是系统的"干活者"。关键词是**有状态执行**和**数据标记**。

**有状态执行是什么意思？**

想象你在用 Python 的 REPL（交互式终端）：
```
>>> a = 1
>>> print(a)
1
>>> a = a + 1
>>> print(a)
2
```

这里 `a` 的值被"记住了"。每次我输入新命令时，前面的 `a` 值都还在。

但如果你用普通函数调用：
```python
def add_one(x):
    x = x + 1
    return x

result1 = add_one(1)  # 返回 2
result2 = add_one(result1)  # 返回 3
```

每次调用前，我必须显式传入变量。函数内部没有"记忆"。

Coder 采用的是**第一种方式**（有状态的）。这样做的好处是：
- Coder 可以在一次执行中定义变量，下一次调用时继续使用这些变量
- 就像在 Jupyter Notebook 中写代码，Cell 之间是连贯的

**数据标记是什么意思？**

每条数据输出都被打上标签：
```
[DATE] 2025-01-15T10:30:45  ← 什么时候生成的
[SOURCE] Coder               ← 谁生成的
[META] {quality_check, type} ← 质量和类型信息
[DATA] {...实际数据...}       ← 真正的内容
```

为什么要标记？因为在一个多智能体系统中，数据来源很复杂。可能来自缓存，可能来自 API，可能来自本地计算。标记让后续的 Reviewer 能追溯每条数据的来源和质量。

**缓存机制**是另一个亮点。假设用户问"给我 Apple 的股票数据"，第一次 Coder 调用 API 获取，花了 2 秒。如果 5 分钟内再有人问同样的问题，不用重新调用 API，直接从缓存返回，秒级响应。而且缓存有生存时间（TTL），过期自动删除，确保数据不会太陈旧。

## 🛡️ 模块三：ErrorHandler（免疫系统）的自我修复逻辑

### 3.1 核心概念

ErrorHandler 是系统的"免疫系统"，负责：
- **错误分类**：将不同类型的错误区分对待（编码错误 vs 网络错误 vs 认证错误）
- **四层自救**：逐级提升修复策略，从轻量级（重试）到重量级（回退）
- **智能建议**：基于错误类型生成修复建议
- **状态恢复**：在修复后恢复执行状态

### 3.2 脱敏代码片段

```python
# ============ 模块三：ErrorHandler 的自我修复逻辑 ============

class ErrorClassifier:
    """错误分类器：将错误映射到不同的处理策略"""
    
    def classify_error(self, error: Exception, context: dict) -> str:
        """
        错误分类：根据错误信息和上下文判断错误类型
        
        分类结果：
        1. code_error: 代码逻辑错误（KeyError, TypeError, 等）
        2. network_error: 网络连接失败（Timeout, ConnectionError）
        3. auth_error: 认证/授权失败（401, 403, InvalidToken）
        4. unknown_error: 未知错误
        """
        
        error_name = type(error).__name__
        error_msg = str(error).lower()
        
        # 规则一：检查错误类型名
        if error_name in ["KeyError", "TypeError", "ValueError", "IndexError"]:
            return "code_error"
        
        # 规则二：检查错误消息包含的关键词
        if any(keyword in error_msg for keyword in ["timeout", "connection", "network"]):
            return "network_error"
        
        if any(keyword in error_msg for keyword in ["401", "403", "unauthorized", "forbidden"]):
            return "auth_error"
        
        # 默认：未知错误
        return "unknown_error"


class ErrorHandler:
    """ErrorHandler 节点：四层自救机制"""
    
    def __init__(self):
        self.classifier = ErrorClassifier()
        self.repair_history = []  # 记录修复历史，防止重复修复
    
    def error_handler_node(self, state: MultiAgentState) -> dict:
        """
        ErrorHandler 的核心决策：四层自救逻辑
        
        执行流程：
        Layer 1 (code_error)      → 重新生成代码 + 提示词优化
        Layer 2 (network_error)   → 等待 + 重试
        Layer 3 (auth_error)      → 刷新令牌 + 重新认证
        Layer 4 (unknown_error)   → 降级 + 人工介入
        """
        
        # ◆ 第一步：解析错误
        last_message = state.messages[-1]
        error_info = last_message.get("error", {})
        
        # 分类错误
        error_type = self.classifier.classify_error(
            Exception(error_info.get("message")),
            context={"last_sender": state.last_sender}
        )
        state.error_classification = error_type
        
        # ◆ 第二步：根据错误类型进行分层处理
        
        if error_type == "code_error":
            return self._handle_code_error(state)
        
        elif error_type == "network_error":
            return self._handle_network_error(state)
        
        elif error_type == "auth_error":
            return self._handle_auth_error(state)
        
        else:  # unknown_error
            return self._handle_unknown_error(state)
    
    def _handle_code_error(self, state: MultiAgentState) -> dict:
        """
        Layer 1：代码错误的修复
        
        策略：
        1. 提取错误堆栈和上下文
        2. 生成修复提示词（告诉 LLM 哪里出错了）
        3. 让 Coder 重新生成代码
        """
        error_msg = state.messages[-1].get("error", {}).get("message")
        error_stacktrace = state.messages[-1].get("error", {}).get("stacktrace")
        
        # 检查是否已经修复过同样的错误（防止死循环）
        if self._is_duplicate_error(error_msg):
            # 已经重复修复，执行降级
            state.next = "Supervisor"
            return {"next": state.next}
        
        # 生成修复提示词
        repair_prompt = f"""
        之前的代码执行出错了：
        错误类型：{type(error_msg).__name__}
        错误信息：{error_msg}
        错误位置：{error_stacktrace[:200]}  # 只保留前200字
        
        请重新生成代码，避免上述错误。
        """
        
        # 构造新的 Supervisor 指令
        repair_message = {
            "role": "ErrorHandler",
            "content": repair_prompt,
            "instruction": "regenerate_code"
        }
        state.messages.append(repair_message)
        state.next = "Coder"  # 转回给 Coder 重新执行
        state.retry_count += 1
        
        self.repair_history.append({
            "error": error_msg,
            "timestamp": datetime.now().isoformat()
        })
        
        return {"next": state.next}
    
    def _handle_network_error(self, state: MultiAgentState) -> dict:
        """
        Layer 2：网络错误的修复
        
        策略：
        1. 首先等待一段时间（指数退避算法）
        2. 重试相同的请求
        3. 如果多次失败，切换到降级方案（如使用缓存）
        """
        retry_count = state.retry_count
        
        if retry_count < 3:
            # 前三次失败：进行等待和重试
            wait_time = 2 ** retry_count  # 2秒、4秒、8秒的指数退避
            
            retry_message = {
                "role": "ErrorHandler",
                "content": f"网络连接失败，等待 {wait_time} 秒后重试",
                "action": "wait_and_retry",
                "wait_seconds": wait_time
            }
            state.messages.append(retry_message)
            state.next = state.last_sender  # 返回给上一步重试
            state.retry_count += 1
        else:
            # 重试超过3次：尝试使用缓存数据
            cached_data = self._try_get_cached_result(state)
            if cached_data:
                # 缓存命中
                cached_message = {
                    "role": "ErrorHandler",
                    "content": "使用缓存数据作为降级方案",
                    "data": cached_data,
                    "action": "use_cache"
                }
                state.messages.append(cached_message)
                state.next = "Reviewer"
            else:
                # 缓存也没有，进行完全降级
                state.next = "Supervisor"
        
        return {"next": state.next}
    
    def _handle_auth_error(self, state: MultiAgentState) -> dict:
        """
        Layer 3：认证错误的修复
        
        策略：
        1. 刷新认证令牌
        2. 重新进行认证
        3. 如果仍然失败，向用户申请新的凭证
        """
        
        # 尝试刷新令牌
        token_refreshed = self._refresh_auth_token()
        
        if token_refreshed:
            # 令牌刷新成功，重新执行上一步
            auth_message = {
                "role": "ErrorHandler",
                "content": "认证令牌已刷新，重新执行任务",
                "action": "token_refreshed"
            }
            state.messages.append(auth_message)
            state.next = state.last_sender  # 重新执行上一步
        else:

            auth_message = {
                "role": "ErrorHandler",
                "content": "认证失败且无法自动恢复，需要用户重新授权",
                "action": "require_user_intervention"
            }
            state.messages.append(auth_message)
            state.next = "END"  # 任务中止，等待用户手动处理
        
        return {"next": state.next}
    
    def _handle_unknown_error(self, state: MultiAgentState) -> dict:
        """
        Layer 4：未知错误的修复（最终降级）
        
        策略：
        1. 记录详细的错误信息用于事后调查
        2. 降级到 Supervisor，让其重新规划
        3. 如果多次都是未知错误，通知人工审查
        """
        
        error_msg = state.messages[-1].get("error", {}).get("message")
        
        # 记录详细日志
        debug_log = {
            "timestamp": datetime.now().isoformat(),
            "error_message": error_msg,
            "last_sender": state.last_sender,
            "full_context": state.messages[-5:]  # 保留最近5条消息作为上下文
        }
        
        # 将日志写入外部存储（用于事后分析）

            "action": "escalate_to_supervisor",

        }
        state.messages.append(fallback_message)
        state.next = "Supervisor"  # 降级给 Supervisor 重新规划
```

**[打开代码片段]**

ErrorHandler 是我最喜欢的模块，因为它体现了系统设计的"韧性"。

**为什么要有 ErrorHandler？**

在生产环境中，错误**一定会发生**。网络会断，API 会超时，令牌会过期。不能假设一切顺利。ErrorHandler 就是为了应对这些不可避免的故障。

**四层自救机制是什么？**

按照错误的严重程度，我设计了四层：

**Layer 1：代码错误** → 轻量级修复
- 问题：代码有 bug（比如 KeyError、TypeError）
- 修复：提取错误堆栈，把错误信息和错误位置告诉 LLM，让它重新生成代码
- 防护：记录修复历史，如果同样的错误重复出现（陷入死循环），就跳过去

**Layer 2：网络错误** → 等待 + 重试
- 问题：网络超时、连接断开
- 修复：使用指数退避算法（2秒、4秒、8秒）等待，然后重试
- 防护：如果重试 3 次还是失败，尝试使用缓存数据（用旧数据也比没数据好）

**Layer 3：认证错误** → 刷新令牌
- 问题：API 令牌过期（401 / 403）
- 修复：自动刷新令牌，然后重新执行
- 防护：如果刷新失败，需要用户手动重新授权，系统等待

**Layer 4：未知错误** → 记录 + 降级
- 问题：莫名其妙的错误，无法分类
- 修复：生成唯一的追踪 ID，记录详细的错误上下文（用于事后调查），降级给 Supervisor 重新规划

**为什么这个设计好？**

1. **渐进式修复**：先尝试轻量级修复（重新生成代码），不成功再升级到重量级修复（缓存降级）
2. **防止死循环**：记录历史，如果修复失败了多次，就放弃，不要无限重试
3. **可观测性**：每个错误都有追踪 ID 和详细日志，便于调试

### 4.1 核心概念

Reviewer 是系统的"质检官"，负责：

- **报告生成**：将原始数据转化为可读的投研报告
- **上下文清洁**：删除冗余消息，保留关键决策信息
- **质量评分**：对输出进行打分，反馈给 Supervisor

```python
class DataValidator:
    """数据验证器：检查数据的质量指标"""
    
    def validate_data(self, data: dict) -> dict:
        """
        数据质检清单：
        1. 完整性检查：是否包含所有必需字段
        2. 准确性检查：数据类型和范围是否合理
        3. 一致性检查：不同字段之间是否存在矛盾
        4. 及时性检查：数据是否过期
        """
        
        checks = {}
        
        # 检查一：完整性
        required_fields = ["symbol", "price", "volume", "timestamp"]
        missing_fields = [f for f in required_fields if f not in data]
        checks["completeness"] = len(missing_fields) == 0
        
        # 检查二：准确性
        checks["accuracy"] = self._verify_data_types(data)
        
        # 检查三：一致性
        checks["consistency"] = self._verify_data_consistency(data)
        
        # 检查四：及时性
        data_age = self._calculate_data_age(data.get("timestamp"))
        checks["freshness"] = data_age < 24 * 3600  # 数据不超过24小时
        
        # 综合评分
        total_checks = len(checks)
        passed_checks = sum(1 for v in checks.values() if v)
        quality_score = passed_checks / total_checks
        
        return {
            "checks": checks,
            "quality_score": quality_score,
            "passed": quality_score >= 0.7  # 70% 通过率为达标
        }
    
    def _verify_data_types(self, data: dict) -> bool:
        """验证数据类型是否正确"""
        try:
            float(data.get("price"))
            int(data.get("volume"))
            return True
        except:
            return False
    
    def _verify_data_consistency(self, data: dict) -> bool:
        """检查字段之间的逻辑一致性"""
        # 例如：价格应该是正数，交易量应该是正数
        if data.get("price", 0) < 0 or data.get("volume", 0) < 0:
            return False
        return True
    
    def _calculate_data_age(self, timestamp: str) -> float:
        """计算数据的年龄（单位：秒）"""
        data_time = datetime.fromisoformat(timestamp)
        age = (datetime.now() - data_time).total_seconds()
        return age

        """
        Reviewer 的执行流程：
        1. 从 Coder 的输出中提取数据
        2. 进行数据质检
        3. 如果质检通过，生成报告；否则要求 Coder 重新执行
        4. 返回质量评分给 Supervisor

        coder_output = self._extract_coder_output(state)
        
        # ◆ 第二步：数据质检
        validation_result = self.validator.validate_data(coder_output)
        
        # ◆ 第三步：决策 - 通过还是拒绝
        if not validation_result["passed"]:
            # 质检失败，返回给 Coder
            rejection_message = {
                "role": "Reviewer",
                "content": self._generate_rejection_reason(validation_result),
                "quality_score": validation_result["quality_score"],
                "action": "require_revision"
            }
            state.messages.append(rejection_message)
            state.execution_status = "quality_failed"
            state.last_sender = "Reviewer"
```

最后一个模块，Reviewer。简单来说，它就是系统的"质检官"。

**数据质检的四个维度：**

1. **完整性**（Completeness）：数据是否包含所有必需字段？比如股票数据必须有 symbol、price、volume 和 timestamp，少一个都不行。

2. **准确性**（Accuracy）：数据类型对不对？价格应该是浮点数，交易量应该是整数。

3. **一致性**（Consistency）：字段之间有没有逻辑矛盾？比如价格或交易量出现负数，这就不合理。

4. **及时性**（Freshness）：数据有没有过期？24 小时以前的股票数据就不新鲜了。

所有检查都通过了，质量评分才能 >= 70%，这样才算达标。如果没达标，Reviewer 会生成"拒绝理由"，告诉 Coder 具体哪里出错了，Coder 然后重新执行。

**报告生成：**

- 数据摘要（关键的股价、涨跌幅等）
- 技术分析（趋势、支撑位、压力位等，这些是由 LLM 生成的）
- 投资建议（买/卖/持有）
- **强制风险提示**（这很重要）

**上下文清洁：**

多轮对话下来，消息会堆积成山。Reviewer 最后的工作是清理垃圾：
- 删除 ErrorHandler 的中间重试消息（只保留最终成功的结果）
- 删除 Supervisor 的中间决策（只保留关键决策）
- 合并重复的角色消息（如果 Coder 连续说两句话，就合并成一句）
- 限制消息总数（最多保留 20 条，删除中间的冗余部分）

这样最后的输出既完整（保留了必要信息），又精简（删除了垃圾信息），Token 使用更高效。

```
    │         │        └──→ [返回给 Coder] (retry_count++)
    │         │
    │         ▼ (if retry > 3)
    │     [降级：使用缓存]
    │
    └─→ Error ──→ ErrorHandler (四层自救)
                  │
                  ├─→ Layer 1: Regenerate Code ──→ Coder
                  ├─→ Layer 2: Wait & Retry ──→ Coder
                  ├─→ Layer 3: Refresh Token ──→ Coder
                  └─→ Layer 4: Escalate ──→ Supervisor
    
    Final Output (Report)
```

4. **ErrorHandler**（10 分钟）：重点讲"四层自救"机制

### 可能的提问和回答

**Q: 为什么要有 Supervisor？直接让 Coder 执行不行吗？**

A: Supervisor 的作用是"分治"。一个复杂的请求（比如"帮我分析 3 只股票的对比"）需要拆成 3 个子任务。而且，Supervisor 能根据执行结果动态调整，比如 Coder 失败了，Supervisor 可以重新规划。这比一条直线的执行流更灵活。

**Q: 有状态执行和普通函数调用的区别是什么？**

A: 普通函数调用像工厂的单个流水线，每次开始前都要把所有原料重新进来。有状态执行像 Jupyter Notebook 的多个 Cell，前一个 Cell 定义的变量，后一个 Cell 可以直接使用。这样效率更高，代码也更简洁。

**Q: 为什么要标记数据？直接返回数据不行吗？**

A: 标记数据是为了"可追溯性"。想象 Reviewer 收到一份数据，它需要知道：这份数据是从哪里来的（Coder 还是缓存）？什么时候生成的（新鲜不新鲜）？质量如何？这些信息都很重要。标记就像给数据贴标签。

**Q: 如果网络错误一直发生怎么办？**

A: 有两个降级方案。第一，指数退避重试（等待 2、4、8 秒）。第二，尝试使用缓存数据。如果缓存也没有，就向用户返回"暂时无法获取最新数据"，并给出最后一次成功的结果。总比直接报错好。

**Q: ErrorHandler 的四层设计有什么好处？**

A: 不同的错误需要不同的修复策略。代码错误需要重新生成代码，网络错误需要等待重试，认证错误需要刷新令牌，未知错误需要人工介入。如果都用同一套策略，会很低效，而且容易出错。

**Q: 如何防止 Coder 陷入无限循环？**

A: Supervisor 记录了 retry_count，每次重试都 +1。如果超过 3 次还失败，就停止重试，执行降级策略。同时，ErrorHandler 也会记录修复历史，如果发现同样的错误重复出现，就放弃修复，升级到下一层。

**Q: Reviewer 为什么要删除消息？**

A: 多轮对话下来，消息会堆积。如果不清理，最后的消息列表可能有几百条，这会浪费大模型的 Token 预算。清理的规则是：删除冗余的中间步骤，只保留关键的决策点和最终结果。

## 📝 总结：三个核心设计哲学

### 1. **分治思想**（Divide and Conquer）
- Supervisor 负责决策，Coder 负责执行，ErrorHandler 负责修复，Reviewer 负责把关
- 每个角色职责清晰，不会互相干扰

### 2. **鲁棒性**（Robustness）
- 系统能自动处理四种不同的错误
- 有降级方案（如使用缓存）
- 有防护措施（如重复错误检测、Token 限制）

### 3. **可观测性**（Observability）
- 每条数据都有标记（来源、时间、质量）
- 每个错误都有追踪 ID
- 执行流程可视化（状态图、流程图）

这三点结合起来，就构成了一个"自我修复、自我优化、可被监督"的多智能体系统。
