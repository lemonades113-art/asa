### 4.4 异常检测规则（表格版）

```
【Reviewer 的异常检测规则表】

┌──────────────┬──────────────┬─────────────────┬──────────────────┐
│ 异常类型     │ 触发条件     │ 严重程度        │ 处理方式          │
├──────────────┼──────────────┼─────────────────┼──────────────────┤
│ PE 为负      │ PE < 0       │ 🔴 严重（红）   │ 标记，暂停估值    │
│ PE 过高      │ PE > 100     │ 🟠 中度（橙）   │ 提示高估，需谨慎  │
│ PB 过低      │ PB < 0.3     │ 🟡 轻微（黄）   │ 提示，建议确认    │
│ 股息率异常   │ 股息率 > 10% │ 🟠 中度（橙）   │ 查证分红政策      │
│ 成交量异常   │ 日成交量波动 │ 🟡 轻微（黄）   │ 提示流动性风险    │
│ 停牌复牌     │ 价格跳跃>20% │ 🔴 严重（红）   │ 查证公告，暂停    │
│ 数据缺失     │ 某字段为 NaN │ 🟡 轻微（黄）   │ 使用备选字段      │
│ 数据格式错   │ JSON 解析失败│ 🔴 严重（红）   │ 退回 Coder 修复   │
└──────────────┴──────────────┴─────────────────┴──────────────────┘

【符号说明】
🔴 严重：影响后续分析，需要立即处理
🟠 中度：需要用户注意和确认
🟡 轻微：可以继续，但要在报告中说明
```

### 4.5 Reviewer 的执行流程

```
【Reviewer 完整的执行流程】

开始（接到 Coder 的数据）
  │
  ├─ 【步骤 1】提取数据
  │   ├─ 定位 [DATA] 标记
  │   ├─ 提取 JSON 数据
  │   └─ 验证 JSON 格式
  │
  ├─ 【步骤 2】异常检测
  │   ├─ 逐字段检查（PE、PB、股息率等）
  │   ├─ 与规则库对比
  │   ├─ 生成异常列表
  │   └─ 对异常进行分级
  │
  ├─ 【步骤 3】消息链清洁
  │   ├─ 删除孤立的 ToolMessage
  │   ├─ 删除过期的错误消息
  │   ├─ 去重：只保留最后的 [DATA]
  │   └─ 压缩上下文（防止 Token 爆炸）
  │
  ├─ 【步骤 4】撰写分析报告
  │   ├─ 基本情况：股票代码、名称、价格
  │   ├─ 估值分析：PE、PB、股息率评价
  │   ├─ 财务分析：营收、利润、现金流趋势
  │   ├─ 风险提示：异常值、需要关注的地方
  │   └─ 投资建议：基于数据的结论
  │
  ├─ 【步骤 5】整合输出
  │   ├─ 异常检测结果
  │   ├─ 生成的分析报告（400-800 字）
  │   └─ 消息链（供下一轮对话参考）
  │
  ├─ 【步骤 6】返回给 Supervisor
  │   └─ execution_status: "success" 或 "error"
  │       
  └─ 结束

【决策树】
if JSON 解析失败:
    → 返回 error，派给 Coder 修复
else if 有严重异常:
    → 生成警告报告，继续
else if 数据有效:
    → 生成标准报告
else:
    → 生成有限制的报告
```

### 4.6 Reviewer 生成的报告示例

```
【示例：中国平安估值分析报告】

【基本信息】
股票代码：000001.SZ
股票名称：平安银行
当前价格：9.80 元
数据更新时间：2024-12-09

【估值分析】
PE 比率：9.2 倍（低于行业平均 10.2 倍，说明当前相对便宜）
PB 比率：0.65 倍（低于历史平均 0.75 倍，有一定安全边际）
股息率：4.5%（高于最近 3 年平均 3.2%，收益率提升）

📊 估值结论：
当前估值处于历史低位，具有较好的配置价值。特别是 PE 和 PB
都低于历史和行业平均，说明市场可能对公司前景预期偏悲观。
但这也创造了布局的机会。

【财务分析】
营收趋势：近 3 年营收保持 8-12% 的增长
净利润：营收增速略高于利润增速，说明利润率有压力
现金流：经营现金流稳定，足以支撑分红

📊 财务结论：
公司基本面稳定，无明显恶化迹象。利润率下降可能与市场
竞争加剧有关，但这是行业性问题，不是公司问题。

【风险提示】
1. ⚠️ 金融监管：政策变化可能影响净息差和估值
2. ⚠️ 经济周期：经济下行可能增加不良贷款率
3. ⚠️ 竞争加剧：互联网银行和地方银行分流客户

【投资建议】
★★★★☆（4/5 分）

基于当前估值和基本面，建议：
- 长期投资者：可以逐步建仓，用于长期持有
- 短期交易者：观察技术面，如果跌破 9.5 元可考虑加仓
- 风险厌恶者：先观察，等待进一步确定信号

【数据来源】
数据来自：Tushare Pro API
更新时间：2024-12-09 10:30:00
数据周期：近 3 年日线 + 最近 4 个季度财务数据

【免责声明】
本分析仅供参考，不构成投资建议。任何投资决策
请结合自身风险承受能力和专业咨询意见。
```

### 4.7 Reviewer 与其他模块的交互

```
【Reviewer 作为"枢纽"的角色】

来自 Supervisor 的指令：
  "请检查数据并撰写报告"
  ↓
Reviewer 处理：
  1️⃣ 验证 Coder 的数据
  2️⃣ 检测异常值
  3️⃣ 撰写分析报告
  ↓
输出给 Supervisor：
  "分析报告已生成，包含 X 个异常值"
  ↓
  ├→ 如果有严重异常 → Supervisor 考虑是否回传 Coder 重新获取
  └→ 如果无异常 → Supervisor 派给 ProfileUpdater 更新用户画像

【Reviewer 失败的处理】
如果 Reviewer 报告生成失败（比如 JSON 解析错误）：
  ├─ 第 1 次失败 → Supervisor 将消息重新发给 Reviewer
  ├─ 第 2 次失败 → Supervisor 诊断，可能派给 Coder 修复格式
  └─ 超过 2 次 → 自动结束（reviewer_fail_count >= 2）
```

## 第四部分补充：Reviewer 的关键特性

### Reviewer 和 Coder 的区别

| 特性 | Coder | Reviewer |
|-----|-------|----------|
| **职责** | 执行代码，获取数据 | 质检数据，撰写报告 |
| **输入** | Supervisor 的指令 | Coder 的数据输出 |
| **输出** | [DATA] 标记的原始数据 | 分析报告 + 异常告警 |
| **模型** | smart（编码能力强）| smart（分析能力强）|
| **工具** | 调用 API、执行代码 | 无（只用 LLM） |
| **失败恢复** | ErrorHandler 修改代码 | 无法自救，重新请求 |
| **质量指标** | 数据准确率 | 报告逻辑通顺度 |
