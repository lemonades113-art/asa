【执行流程改动前后对比】

【改动前】（5个节点的串行流程）
User Input
  ↓
Supervisor
  ├→ Coder (获取财务数据 [DATA])
  │   ↓
  ├→ ErrorHandler (如果失败)
  │   ↓
  ├→ Reviewer (撰写报告，只有 [DATA])
  │   ↓
  └→ ProfileUpdater
      ↓
    FINISH

【改动后】（6个节点的可选并行流程）
User Input
  ↓
Supervisor (【新增】识别是否需要新闻)
  │
  ├─ 是 → parallel_task_flag = True
  │
  ├→ Coder (获取财务数据 [DATA]) ──────┐
  │   ↓                                  │
  ├→ ErrorHandler                        │ 【可并行】
  │   ↓                                  │
  └→ Reporter (【新】搜索新闻 [NEWS]) ──┘
      ↓
      【新增同步点】等待两个 Agent 都完成
      ↓
    Supervisor (检查 reporter_ready)
      ↓
    Reviewer (接收 [DATA] + [NEWS]，智能融合)
      ↓
    ProfileUpdater
      ↓
    FINISH

【改动不会出错的关键原因】

理由              验证方法
──────────────────────────────────────
1. State 有默值    检查 get_initial_state() 是否初始化了所有字段
2. 异常处理完善    reporter_node() 的 try-except 是否覆盖所有路径
3. 容错链存在      Reporter 失败时是否能降级到无新闻分析
4. 路由多重        Supervisor 是否有备选路由（降级关键字匹配）
5. Message 累积    新增消息是否自动追加而非覆盖（operator.add）
