## 🏗️ 整体架构

### 1. 高层思路（一句话）

```
用户提问 → Supervisor分解 → Coder执行 → ErrorHandler修复 
→ Reviewer撰写 → ProfileUpdater学习 → 返回报告
```

### 2. 五个智能体及职责

| 智能体 | 职责 | 模型 | 工具 | 备注 |
|:---|:---|:---|:---|:---|
| **Supervisor** | 任务规划者、指挥官 | smart | 无 | 分解任务、路由决策、质检数据 |
| **Coder** | 程序员、执行者 | smart | run_script, search, datetime | 写代码、执行、获取数据 |
| **Reviewer** | 分析师、总结者 | smart | 无 | 撰写投资报告、解释数据 |
| **ErrorHandler** | 修复者、自救者 | fast | 无 | 错误分类、重试逻辑、降级方案 |
| **ProfileUpdater** | 学习者、记忆中枢 | fast | 无 | 更新用户画像、学习偏好 |

**模型分层**：
- **smart模型**（qwen-plus）：Supervisor、Coder、Reviewer - 复杂决策
- **fast模型**（qwen-turbo）：ErrorHandler、ProfileUpdater - 轻量级任务，降低成本

### 3. 状态机（核心流程）

```
MultiAgentState
├── messages              # 完整对话历史
├── next                  # 下一个节点 (Supervisor/Coder/Reviewer/ProfileUpdater/FINISH)
├── execution_status      # 执行状态 (pending/success/error)
├── last_sender          # 上一个发送者（防止虚假路由）
├── task_plan            # 任务分解计划
├── remaining_steps      # 剩余任务步骤
├── error_type           # 错误分类 (code_error/network_error/auth_error)
├── retry_count          # 重试计数
├── user_profile         # 用户画像 {investment_style, risk_preference, ...}
└── last_execution_data  # 上次执行数据（用于去重）
```

## 💡 核心流程详解

### Phase 1: 任务分解（Supervisor.decompose_task）

**输入**：用户的自然语言查询

**过程**：
```
用户问题 (如："查询贵州茅台最近1年的收益率")
   ↓
Supervisor调用 decompose_task()
   ↓
使用smart模型将问题拆分成JSON步骤
   ↓
例如：
{
  "steps": [
    "获取贵州茅台最近1年的日线数据",
    "计算每日对数收益率",
    "计算年化波动率和夏普比率",
    "生成分析报告"
  ]
}
   ↓
保存到 task_plan 和 remaining_steps
```

**关键点**：
- 使用 JSON 格式保证结构化
- 有 fallback 降级方案（无法分解时返回默认步骤）
- 每个步骤必须包含明确的动作（查询/计算/分析）

### Phase 2: 数据执行（Coder节点）

**职责**：
1. **接收指令**：从 remaining_steps 中弹出当前步骤
2. **编写代码**：使用 run_script 工具执行 Python 代码
3. **强制输出**：必须包含 `[DATE]` `[SOURCE]` `[DATA]` 等标签

**代码执行流程**：
```python
# Coder 的代码 = 标准结构

from datetime import datetime
import json
import tushare as ts

# 1. 初始化
pro = ts.pro_api()
current_date = datetime.now().strftime('%Y-%m-%d')

# 2. 获取数据
try:
    df = pro.daily(ts_code='600519.SH', limit=252)
    
    # 3. 数据防护（P0级强制）
    if df.empty:
        print("[WARNING]: 数据为空，尝试自救...")
        # 自救方案 A/B/C
        pass
    
    # 4. 计算指标
    returns = df['close'].pct_change()
    volatility = returns.std() * np.sqrt(252)
    
    # 5. 元数据输出（告诉Reviewer怎么理解这数据）
    print("[META]: 波动率定义 = 年化波动率(日收益率标准差 × √252)")
    
    # 6. 关键输出
    print(f"[DATE]: {current_date}")
    print(f"[SOURCE]: Tushare Pro API")
    print(f"[TIME_RANGE]: 2024-11-28 至 2025-11-28")
    print(f"[DATA]: {json.dumps({'volatility': volatility})}")
    
except Exception as e:
    print(f"[ERROR]: {type(e).__name__}: {e}")
```

**强制输出标签**：
```
[DATE]:        执行日期
[SOURCE]:      数据来源（Tushare Pro API）
[TIME_RANGE]:  数据时间范围
[META]:        元数据（算法定义）
[DATA]:        最终数据（JSON格式，一行）
[IMAGE]:       (可选) 图表路径
```

**成功标志**：
- execution_status = "success"
- 消息中包含 [DATA] 标签
- 没有抛出异常

### Phase 3: 错误自救（ErrorHandler节点）

**触发条件**：Coder执行失败（抛异常）

**工作流程**：
```
Coder抛异常
   ↓
ErrorHandler.classify_error_simple()
   ↓
分类为：code_error / network_error / auth_error / unknown
   ↓
根据错误类型采取不同策略：
   
   ├─ code_error (代码语法错误):
   │   └─ 最多重试3次，每次加重错误信息给Coder
   │
   ├─ network_error (API超时/限流):
   │   └─ 最多重试3次，间隔延长（指数退避）
   │
   ├─ auth_error (权限/认证):
   │   └─ 立即降级到Reviewer，告知用户权限问题
   │
   └─ unknown_error (未知错误):
       └─ 尝试1次重试，失败则触发"动态重规划"
   ↓
重试次数耗尽(>= 3次) → 触发Supervisor的重规划逻辑
```

**关键特性**：
- **分级处理**：不同错误类型采用不同策略
- **渐进降级**：retry_count递增，达到上限自动降级
- **动态重规划**：当某个方案失败时，Supervisor会重新分解任务

### Phase 4: 数据质检（Supervisor.P1质检）

**在派给Reviewer之前，Supervisor检查**：

```
Coder返回结果
   ↓
Supervisor检查：
   ├─ [1] 是否有 [DATA] 标签？
   │   └─ 没有 → 生成诊断建议，退回Coder重新执行
   │
   ├─ [2] [DATA] 内容是否为空？
   │   └─ {} / [] / null / {'error': ...}
   │       → 生成自救建议，退回Coder（扩大日期范围/放宽条件/换API）
   │
   └─ [3] 是否有诊断信息 [DIAGNOSTICS]？
       └─ 有 → 记录诊断，继续推进任务
```

**质检失败时的反馈示例**：
```
❌ 缺少[DATA]标签
修复建议：
  1. 检查代码最后是否包含：
     print(f"[DATA]: {json.dumps(results)}")
  2. 若数据获取失败，也要输出错误JSON：
     print(f"[DATA]: {json.dumps({'error': '原因'})}")

❌ [DATA]为空
修复建议：
  1. 扩大日期范围（5年而非3年）
  2. 放宽筛选条件（包含季报和年报）
  3. 换个API（试试 balancesheet 而非 income）
```

### Phase 5: 报告撰写（Reviewer节点）

**前提**：所有数据收集完毕，execution_status = "success"

**工作流程**：
```
接收Coder的执行结果
   ↓
清洗上下文：
   ├─ 提取第一条 HumanMessage（用户原问题）
   ├─ 提取最后一条成功数据（包含[DATA]标签）
   └─ 过滤掉中间的错误消息和重试日志
   ↓
构建清洁消息列表：
   [SystemMessage] + [用户问题] + [成功数据]
   ↓
调用 reviewer_model 生成报告
   ↓
重试机制：如果回复为空，最多重试3次
   ↓
返回 AIMessage 包含完整报告
```

**报告结构**（标准模板）：
```markdown
## 数据领域简介
- 说明分析的背景和数据来源

## 详细指标分析
- 必须引用[META]中的定义和公式
- 解释每个指标的含义
- 对标历史水平或市场基准

## 投资建议
- 基于数据给出具体建议
- 不能编造数据或违反[META]定义

## 免责声明
- 声明分析的局限性
- 说明历史不代表未来
```

**关键约束**：
- ✅ 必须引用[META]中的权重和定义
- ✅ 必须基于[DATA]中的实际数据
- ❌ 禁止自己编造权重或定义
- ❌ 禁止违反[META]中的计算方法

### Phase 6: 用户画像更新（ProfileUpdater节点）

**目的**：学习用户的偏好，为后续对话优化

**更新内容**：
```python
user_profile = {
    "investment_style": "保守" / "平衡" / "激进",  # 从问题中推断
    "risk_preference": "低" / "中" / "高",         # 从查询内容推断
    "interested_sectors": [...],                  # 访问过的行业

    "update_timestamp": "2025-11-30"               # 最后更新时间
}
```

**更新时机**：在Reviewer之后，作为最后一步

## 🛡️ 核心机制详解

### 1. Supervisor的三大决策逻辑

**逻辑① 状态机路由**：
```python
def supervisor_node(state):
    last_sender = state["last_sender"]
    execution_status = state["execution_status"]
    remaining_steps = state["remaining_steps"]
    
    # 如果是新任务 → 分解
    if last_sender == "User" and remaining_steps == []:
        task_plan = decompose_task(user_query)
    
    # 如果Coder成功 + 还有步骤 → 继续派给Coder
    if last_sender == "Coder" and execution_status == "success" and remaining_steps:
        return {"next": "Coder", "remaining_steps": remaining_steps}
    
    # 如果Coder成功 + 无步骤 → 派给Reviewer
    if last_sender == "Coder" and execution_status == "success" and not remaining_steps:
        return {"next": "Reviewer"}
    
    # 如果ErrorHandler修复成功 → 派给Reviewer
    if last_sender == "ErrorHandler" and execution_status == "success":
        return {"next": "Reviewer"}
```

**逻辑② P1质检**：
```python
# 在派给Reviewer之前检查数据
last_message_content = state["messages"][-1].content

# 检查1：有[DATA]标签吗？
if "[DATA]:" not in last_message_content:
    return {"next": "Coder", "message": "[诊断] 缺少[DATA]标签"}

# 检查2：[DATA]为空吗？
if "[DATA]: {}" in last_message_content or "[DATA]: null" in last_message_content:
    return {"next": "Coder", "message": "[诊断] 数据为空，请自救"}

# 都通过 → 派给Reviewer
return {"next": "Reviewer"}
```

**逻辑③ 动态重规划**：
```python
# 当错误重试耗尽时(retry_count >= 3)
if retry_count >= 3 and error_type == "code_error":
    # 获取失败原因
    error_info = state["messages"][-1].content
    
    # 重新分解任务
    replan_prompt = f"""
    原始请求：{original_query}
    失败原因：{error_info}
    请重新分解任务，避免上一个失败的方案
    """
    
    new_steps = smart_model.invoke(replan_prompt)
    return {
        "next": "Coder",
        "remaining_steps": new_steps,
        "retry_count": 0  # 重置重试计数
    }
```

**防护① 数据检查**：
```python
# API调用后立即检查
if df.empty:

    # 方案B：放宽筛选条件
    # 方案C：换个API
    pass

# 数据质量检查
assert not df.empty, "数据不能为空"

**防护② 类型转换**：

df['revenue'] = pd.to_numeric(df['revenue'], errors='coerce')

**防护③ 异常处理**：

except Exception as e:

**防护④ 元数据输出**：

print("[META]: 数据周期 = 最近252个交易日")
print("[META]: 样本量 = 1支股票")
print(f"[DATA]: {json.dumps(results)}")
```

### 3. ErrorHandler的分级修复

**修复等级**：
```
Level 1: Coder自救（在自己的代码中处理）
  → try-except捕捉异常
  → 数据为空时尝试自救方案A/B/C

Level 2: ErrorHandler一次重试（代码错误）
  → 向Coder提供更详细的错误信息
  → 最多重试3次

Level 3: ErrorHandler网络重试（API超时）
  → 使用指数退避策略
  → sleep(2^retry_count)

Level 4: 降级处理（异常彻底失败）
  → 降级到Reviewer（告知用户数据缺失）
  → 或触发Supervisor的动态重规划
```

## 🎯 向面试官讲解的要点

### 1. **系统整体设计**（2分钟）
```
"这是一个基于LangGraph的多智能体系统。
核心思想是：Supervisor作为指挥官分解复杂任务，
然后派给不同的智能体执行——Coder负责编程和数据，
Reviewer负责分析和撰写报告，ErrorHandler负责错误修复。

这样做的好处是：
1. 职能分工明确，每个智能体只做自己擅长的事
2. 可以按需选择不同能力的模型（省成本）
3. 错误可以自动修复，不用返回给用户
```

### 2. **状态机设计**（2分钟）
```
"系统使用MultiAgentState来管理整个执行过程。
关键字段有：
- execution_status: pending/success/error（执行状态）
- last_sender: 谁发送的这个消息（防止虚假路由）
- remaining_steps: 还剩多少步骤要做（任务进度）
- retry_count: 重试了多少次（错误判断）

这样Supervisor可以基于状态机判断下一步应该干什么，
而不是靠LLM的输出，避免被LLM迷惑。
```

### 3. **数据质检机制**（2分钟）
```
"最关键的是P1质检——在派给Reviewer之前，
Supervisor会检查Coder的输出：
1. 是否有[DATA]标签？
2. [DATA]内容是否为空？

如果检查失败，Supervisor会给Coder诊断反馈，
让它自救（扩大日期范围、放宽筛选条件、换API）。
这确保了Reviewer收到的数据是真实有效的。
```

### 4. **错误自救体系**（2分钟）
```
"系统有四层错误处理：

第1层：Coder自救——在代码中用try-except和数据检查
第2层：ErrorHandler重试——重新执行Coder的代码，最多3次
第3层：网络重试——如果是超时/限流，用指数退避延迟
第4层：动态重规划——如果一个方案彻底失败，Supervisor重新分解任务

最后，如果全部失败，才会降级到Reviewer，告知用户数据缺失。
这样用户体验更好。
```

### 5. **模型分层**（1分钟）
```
"为了降低成本，我们对不同任务使用不同能力的模型：

Smart模型（qwen-plus）：用于Supervisor、Coder、Reviewer
  → 需要强逻辑、决策能力强

Fast模型（qwen-turbo）：用于ErrorHandler、ProfileUpdater
  → 轻量级任务，成本低、足够用

这样成本降低了20%-30%，同时保证了质量。
```

### 6. **代码执行的强制规范**（1分钟）
```
"Coder的每个代码都必须输出五个标签：

[DATE]: 执行日期
[SOURCE]: 数据来源（Tushare Pro API）
[TIME_RANGE]: 数据时间范围
[META]: 元数据（告诉Reviewer怎么理解这数据）
[DATA]: 最终数据（JSON格式）

还有：
[WARNING]: 数据有问题时的警告
[DIAGNOSTICS]: 诊断信息

这样Reviewer就能完全理解数据的来源和定义。
```

8. 返回报告给用户

整个过程中，Coder执行3次、ErrorHandler 0次（没出错）。

### Supervisor.next 的决策树

```python
if last_sender == "User":
    # 新任务 → 先分解再路由
    task_plan = decompose_task(query)
    return {"next": "Coder"}

elif last_sender == "Coder" and execution_status == "success":
    # Coder成功
    # P1质检
    if not has_data_tag or is_data_empty:
        # 质检失败 → 退回Coder重试
        return {"next": "Coder", "message": "诊断建议"}
    
    # 质检通过
    if remaining_steps:
        # 还有步骤 → 继续派给Coder
        return {"next": "Coder", "remaining_steps": remaining_steps}
    else:
        # 无步骤 → 派给Reviewer
        return {"next": "Reviewer"}

elif last_sender == "Coder" and execution_status == "error":
    # Coder出错 → 派给ErrorHandler
    return {"next": "ErrorHandler"}

elif last_sender == "ErrorHandler" and execution_status == "success":
    # 错误已修复 → 派给Reviewer（或如果还有步骤继续派Coder）
    return {"next": "Reviewer"}

elif last_sender == "Reviewer":
    # Reviewer完成 → 派给ProfileUpdater
    return {"next": "ProfileUpdater"}

elif last_sender == "ProfileUpdater":
    # ProfileUpdater完成 → 任务结束
    return {"next": "FINISH"}
```
