# 数独 GRPO 项目的 CoT 设计：为什么这样写 `<think>`，论文怎么支持这个决策

> 论文基础：
> - **PR-CoT**：Enhancing Self-Correction in LLMs through Multi-Perspective Reflection (arXiv:2601.07780, 2026)
> - **PRoSFI**：Learning to Generate Formally Verifiable Step-by-Step Logic Reasoning via Structured Formal Intermediaries (arXiv:2603.29500, 2026)
> - **CRV**：Verifying Chain-of-Thought Reasoning via Its Computational Graph (ICLR 2026 Oral, arXiv:2510.09312)
> - **DeepSeek-R1**：Incentivizing Reasoning Capability in LLMs via RL (arXiv:2501.12948, 2025)
> 代码基础：`sudoku_grpo.py` — `<think>推理</think><answer>81位</answer>` 格式

---

## 一、先把问题说清楚：CoT 在这个项目里是什么角色

在开始讲设计之前，要先把一件事说明白——

**CoT（Chain-of-Thought，思维链）在本项目里不是一个可选项，而是整个 RL 训练的物理基础。**

原因在于 GRPO 的本质是 **在线 RL（Online Reinforcement Learning）**。模型在训练时要自己生成轨迹（trajectory），然后奖励函数对这条轨迹打分，分数的高低决定策略更新方向。这个轨迹的载体就是模型的文本输出——如果没有结构化的 CoT，模型的"推理过程"就是不可观测的黑盒，奖励函数只能盯着最终答案打分。

这正是 DeepSeek-R1 论文的核心发现：*"Format rewards complement the accuracy reward by enforcing specific formatting requirements. In particular, the model is incentivized to put its thinking in `<think>` tag"*。

`<think>` 标签做的事情，本质上是把模型内部的**隐式推理过程（latent reasoning）外化成可观测的文本轨迹**。这个外化一旦发生，就有了以下可能：
- 可以用多维度奖励函数验证推理的中间步骤，而不只是结果
- 可以通过格式奖励的梯度信号，引导 RL 训练收敛方向
- 可以在 SFT 阶段预先注入高质量推理范式，降低 GRPO 的探索成本

这三点构成了本项目 CoT 设计的全部逻辑。接下来逐一展开。

## 二、`<think>/<answer>` 两段式结构：为什么这样划分

### 2.1 结构本身的设计决策

项目的 system prompt 规定输出格式如下：

```python
# sudoku_grpo.py Line 406-421
system_prompt = """你是一个数独专家。...
## 输出格式
请按以下格式回答：
<think>
详细的推理过程，按坐标锁定数字。
</think>
<answer>
81位纯数字答案
</answer>"""
```

这里有两个刻意的设计选择：

**选择 1：强制分离推理过程和最终答案**

`<think>` 和 `<answer>` 是两个独立标签，功能不同：
- `<think>` 块：自由文本，允许任意长度的推理，不做格式约束
- `<answer>` 块：严格约束为 81 位纯数字，不允许任何其他内容

这个分离背后的逻辑，PRoSFI 论文说得最清楚：

> *"outcome rewards often overlook flawed intermediate steps, leading to unreliable reasoning steps even when final answers are correct"*

如果只有一个输出区域（把推理和答案都混在一起），奖励函数无法区分"有效推理后得出的正确答案"和"胡乱猜测后碰巧写对的答案"。两段式强制分离后，`extract_xml_answer()` 精确提取 `<answer>` 内的 81 位数字，而奖励函数的信号只作用于这 81 位——推理过程的质量通过行/列/宫约束的结果来间接衡量。

**选择 2：`按坐标锁定数字` 的 prompt 工程**

系统提示要求模型"按坐标锁定数字"，即推理过程应该包含类似"坐标 (1, 3)：因为第一行已有 1/2/4，第三列已有 3/5，该格只能填 6"这样的表述。

这不是无意义的 prompt 美化。`evaluate_model_fortified()` 函数专门检测坐标：

```python
# sudoku_grpo.py Line 131-137
coords = re.findall(r"坐标 \((\d), (\d)\)", resp)
for r_str, c_str in coords:
    r, c = int(r_str), int(c_str)
    idx_in_str = (r - 1) * 9 + (c - 1)
    if raw_puzzle[idx_in_str] == '0':    # 只对空白格计分
        metrics["coord_acc"] += 1
```

这个坐标准确度检测，是在 SFT 评估阶段用于判断模型是否真正"在解题"而不是在背诵套路。背后的逻辑来自 PRoSFI 的核心思路：**把推理步骤锚定到可验证的结构化形式上**。PRoSFI 用 JSON/YAML 格式的结构化中间步骤，本项目用的是更轻量的"坐标+理由"格式——本质是同一件事：让自然语言推理有可验证的空间定位锚点。

## 三、格式奖励的非对称设计：为什么闭合标签值两倍分

```python
# sudoku_grpo.py Line 349-354
if "<think>" in r: score += 0.1
if "</think>" in r: score += 0.2    # 重点奖励：推理结束
if "<answer>" in r: score += 0.1
if "</answer>" in r: score += 0.2   # 重点奖励：答案完整
```

**[输入]**：模型生成的完整文本
**[核心逻辑]**：开放标签和闭合标签权重不同（1:2），总分 0.6

这个不对称设计，CRV 论文提供了深层的理论支撑。

CRV（Circuit-based Reasoning Verification，ICLR 2026 Oral）的核心发现是：

> *"attribution graphs of correct CoT steps possess distinct structural fingerprints from those of incorrect steps"*（正确的推理步骤和错误的推理步骤，在模型的内部计算图上有不同的结构特征）

什么意思？这说明 **CoT 的完整性（有没有完整闭合）是推理质量的结构性信号**，不只是格式美观问题。一个完整的 `<think>...</think>` 意味着模型启动了推理过程并将其收敛到一个终点；而只有 `<think>` 没有 `</think>` 意味着推理在中途"断掉"了。

工程上，在 RL 训练初期，模型非常容易出现"推理开始但没有闭合"的行为——因为开放标签可以从 prompt 的示例中直接复制，而闭合标签必须由模型自己在推理完成后生成。用 2:1 的权重比强调闭合标签，实质上是用奖励梯度去**惩罚截断行为，奖励完整轨迹**。

进一步，CRV 发现"failures in different reasoning tasks manifest as **distinct computational patterns**"——不同的推理失败模式有不同的计算图特征。这个发现解释了为什么 `soft_format_reward_func` 要分四个维度打分（开思/闭思/开答/闭答），而不是简单地"有没有完整格式"一刀切：四个维度对应四种不同的失败模式，需要分别给出梯度信号。

## 四、PR-CoT 视角：数独的 `<think>` 块里是什么结构的推理

PR-CoT（Poly-Reflective Chain-of-Thought，arXiv:2601.07780）的核心主张：

> *"single-dimensional reflection methods offer insufficient improvements... PR-CoT guides the LLM to self-assess its reasoning across multiple predefined angles: logical consistency, information completeness, biases/ethics, and alternative solutions"*

单维度反思（只检查"答案对不对"）效果不足；多角度反思（逻辑一致性、信息完整性、替代方案）才能显著改善推理质量，尤其在逻辑谜题任务上效果最突出。

把这个框架映射到数独的 `<think>` 块：

| PR-CoT 反思维度 | 数独 `<think>` 对应的思维行为 | 奖励函数对应 |
|---------------|--------------------------|------------|
| **逻辑一致性检查** | "第 3 行已有 5，这格不能填 5" | `row_logic_reward_func` |
| **信息完整性检查** | "这格是空白，线索是 X，附近约束是 Y" | `clue_preservation_reward_func` |
| **替代方案探索** | "如果填 6 不行，试试 7" | `exact_answer_reward_func` 的部分分激励回溯 |
| **全局一致性** | "宫格约束 + 行约束 + 列约束三者同时满足" | `block_logic_reward_func` |

**关键洞察**：本项目的 system prompt 要求"按坐标锁定数字"，这其实是在**用 prompt 工程诱导模型产生 PR-CoT 式的多维反思结构**——每次填入一个数字，必须同时考虑行、列、宫三个约束维度，这就是三个角度的同步反思。

但 PR-CoT 论文同时指出了一个关键局限：**它的多维反思是 prompt-level 的设计，不是训练出来的能力**。PR-CoT 用于 GPT-3.5/4 的 zero-shot 场景，靠 prompt 引导反思结构。

本项目走得更远：通过 GRPO 的奖励信号，试图把这种多维反思能力**训练进模型参数**。这就是为什么奖励函数要覆盖行、列、宫三个独立维度——每一个维度的奖励梯度，都在训练期强化模型某一类反思行为（行约束检查、列约束检查、宫约束检查）。

## 五、PRoSFI 视角：这个项目的 CoT 结构是轻量版结构化中间步骤

PRoSFI（Process Reward over Structured Formal Intermediaries，arXiv:2603.29500）做了一件本项目没做但在同一个方向上的事情：

> *"instead of generating formal proofs directly... the model outputs structured intermediate steps aligned with its natural language reasoning. Each step is then verified by a formal prover. Only fully validated reasoning chains receive high rewards."*

PRoSFI 的管线：
```
自然语言推理步骤
    ↓
结构化中间步骤（JSON/YAML 格式，含 identifier/logic_rule/conclusion/dependencies）
    ↓
形式化证明验证器（formal prover，如 Z3/Prover9）
    ↓
只有全步骤通过验证的推理链才得高分
```

本项目的管线：
```
<think>按坐标锁定数字的自然语言推理</think>
    ↓
<answer>81 位纯数字</answer>
    ↓
7 个奖励函数（clue/row/col/block 验证器 + exact 比对）
    ↓
只有通过所有约束检验的答案才接近满分
```

两者的设计哲学完全一致，区别在于**形式化的深度**：

- PRoSFI 要求模型在 `<think>` 里输出可被 Z3 解析的 JSON 结构，每一个推理步骤都有形式化表示，可以被机器验证逻辑有效性
- 本项目没有形式化到这个程度——`<think>` 内容是自然语言，验证发生在 `<answer>` 的结果层面（81 位数字是否满足数独约束）

**PRoSFI 论文揭示了本项目的一个关键局限**：

> *"RL guided by outcome-only rewards can degrade the faithfulness of reasoning chains... In mathematical tasks, it has been observed that models may arrive at correct answers through invalid arguments"*

同样的问题在数独上存在：一个模型完全可以在 `<think>` 里写一堆不相关的话，然后在 `<answer>` 里填一个恰好正确的答案。row/col/block 奖励函数会因为结果正确而给高分，但无法判断 `<think>` 里的推理是不是真正推导出了这个答案。

这是**本项目 CoT 设计的最大盲点**：推理过程和答案结果是解耦的——格式奖励只验证标签存在，不验证推理内容与答案的因果关联。PRoSFI 的结构化中间步骤正是填补这个盲点的工具。

**[输入]**：模型的 `<think>` 块文本
**[理论上需要做的事]**：PRoSFI 式的步骤级形式化验证
**[实际上做的事]**：只验证最终答案的约束满足程度
**[gap]**：`<think>` 内容可以和 `<answer>` 完全无关，奖励函数无法识别

## 六、Brevity 惩罚的 CoT 含义：防止"推理污染"

```python
# sudoku_grpo.py Line 367-371
if len(r) > 3500:
    penalty = min(0.5, (len(r) - 3500) / 1000)
```

这个设计用 PR-CoT 的视角来解读，意义更加清晰。

PR-CoT 的实验发现：过长的反思链反而会**引入噪声和自相矛盾**——模型在"反思替代方案"时，有时会把一个错误路径越想越真，最终推翻了正确的中间结论。这在逻辑谜题任务上尤其常见。

数独的 CoT 有同样的风险：如果 `<think>` 块过长（超过 4000 token），模型往往陷入两类病态：

1. **重复推导**：对同一格子反复验证行/列/宫约束，不断生成冗余文字但不推进填格进度
2. **虚假探索**：生成大量"假如填 X 则..."的假设分支，实际上在 `<answer>` 里输出的答案并未依循这些假设

3500 字符的阈值设计对应的是：对于高线索密度（70+ 线索）的数独，正常的逐格推理链大约在 1500-3000 字符范围内。超过 3500 意味着模型大概率进入了上述病态。

**[输入]**：模型完整输出的字符数
**[核心逻辑]**：3500 字符以内不惩罚（鼓励充分推理），超出后线性扣分
**[输出]**：0 ~ -0.5 的惩罚
**[实质]**：剪除过度探索的长尾推理，维持 CoT 的信噪比

## 七、SFT 冷启动：为 CoT 提供初始化的"推理范式锚点"

这一节要解释清楚一个问题：纯 GRPO（不做 SFT 冷启动）为什么很难让模型学会有效的 CoT？

DeepSeek-R1-Zero 做了一个历史性的实验——直接用纯 RL（GRPO）训练没有 SFT 的预训练模型，结果出现了一个现象叫 **"aha moment"**：模型在某个训练步骤突然开始自发产生结构化推理行为，包括反思、验证等。

但代价是：过程极不稳定，且生成的文本"可读性差"（*"生成数据可读性差"*，DeepSeek-R1 技术报告）。更重要的是，R1-Zero 用的是 671B 的巨型模型。对于本项目的 3B 小模型，直接 GRPO 的探索空间中有效轨迹的密度远低于大模型——大多数随机生成的文本根本不满足格式要求，奖励全是 0，梯度消失。

PRoSFI 论文对这个问题有明确的论断：

> *"Training models to write formal proofs typically requires much larger model sizes and significant data distillation... models optimized for formal proof generation often struggle with structure and format following"*

小模型没有足够的参数容量来同时学习"如何推理"和"如何格式化"两件事。SFT 的作用是**把"如何格式化"（`<think>/<answer>` 结构）先用监督学习固化下来**，然后 GRPO 只需要专注于优化"如何推理"（推理路径的正确性）。这是任务解耦，不是偷懒。

代码中的 SFT 质量门槛：

```python
# sudoku_grpo.py Line 383-389
success_val = evaluate_model_fortified(model, tokenizer, sft_data, num_samples=5)
if success_val < 3:
    print("⚠️ SFT 格式掌握极差！")
else:
    print(f"✨ SFT 基础尚可，直接进入 GRPO...")
```

Pass@1 < 3/5 时系统报警——这是在检验 SFT 有没有成功把推理范式锚点注入模型。如果锚点没有注入，GRPO 的 RL 训练等于在完全随机的文本空间里探索，没有可利用的初始结构。

**[输入]**：SFT 数据集（`sudoku_reasoning_dataset.json`），高质量 CoT 示例
**[核心逻辑]**：监督学习固化 `<think>/<answer>` 的格式范式和坐标推理模式
**[输出]**：具有格式范式锚点的初始策略，GRPO 探索空间中有效轨迹密度更高
**[本质]**：把"格式学习"和"推理优化"解耦到两个阶段，降低 RL 训练的探索难度

## 八、CRV 的深层启示：为什么这个项目的 CoT 验证是不够的

CRV（Circuit-based Reasoning Verification，ICLR 2026 Oral）发现了一件让人不安的事情：

> *"by scrutinizing a model's computational process, we can move from simple error detection to a deeper, causal understanding of LLM reasoning"*
> *"these signatures are not merely correlational; by using our analysis to guide targeted interventions on individual transcoder features, we successfully correct the model's faulty reasoning"*

CRV 能通过分析模型的**内部激活图（attribution graphs）**直接识别推理步骤哪一步出了问题，并且可以通过针对性干预来纠正——这远超任何基于文本输出的验证方法。

本项目对 CoT 的验证是完全基于文本输出的：`extract_xml_answer()` 提取 81 位数字，然后奖励函数验证这 81 位是否满足数独约束。**这是黑盒输出验证，不是推理过程验证**。

CRV 发现的关键问题：推理错误往往不在输出层面表现出来，而是在模型的内部计算中就已经偏离了正确路径。即，`<think>` 内容看起来是对的，但模型内部的权重更新方向在 RL 训练中实际上强化了一个有偏的计算回路。

具体到数独：假设模型在 `<think>` 里写了"第一行有 1/2/3，这格只能填 4"（看起来正确），但实际上模型的内部表征里并没有真正执行行约束检查，只是模式匹配到了类似的训练样本。这种**伪推理（spurious reasoning）**在 CRV 发现中对应"incorrect steps with correct surface text"，是当前验证体系无法识别的。

**这定义了本项目 CoT 设计的理论天花板**：基于规则的奖励函数能检验结果正确性，但无法验证推理路径的内在有效性。PRoSFI 的结构化中间步骤（配合形式化证明器）和 CRV 的内部激活分析，是两条突破这个天花板的路径。

## 九、四篇论文 × 项目 CoT 设计的完整映射

| CoT 设计决策 | 论文支撑 | 设计本质 |
|------------|---------|---------|
| `<think>/<answer>` 两段分离 | DeepSeek-R1 格式奖励机制；PRoSFI 推理-输出解耦 | 把隐式推理外化为可观测轨迹 |
| `按坐标锁定数字` prompt | PRoSFI 结构化中间步骤思路 | 轻量版推理步骤锚定（空间定位 + 约束理由） |
| 闭合标签权重 2x 开放标签 | CRV 推理完整性是结构性信号 | 奖励完整推理轨迹，惩罚截断行为 |
| 格式奖励分四维打分 | CRV 不同失败模式有不同计算特征 | 四种断点行为对应四个独立梯度信号 |
| SFT 冷启动后才进 GRPO | PRoSFI 小模型无法同时学格式和推理；DeepSeek-R1 cold-start 原理 | 把格式学习和推理优化解耦到两个阶段 |
| `<think>` 内部多维约束检查 | PR-CoT 多角度反思显著改善逻辑谜题表现 | RL 训练期把 prompt-level 多维反思转化为参数级能力 |
| Brevity 3500 字符惩罚 | PR-CoT 过度反思引入矛盾；pruning 无效推理分支 | 维持 CoT 信噪比，防止推理污染 |
| 最大局限：无推理-答案因果验证 | PRoSFI 结构化验证；CRV 内部激活验证 | 当前体系是黑盒输出验证，不是过程验证 |

## 十、[输入] → [核心逻辑] → [输出] 的完整 CoT 因果链

```
[问题输入]
  高线索密度数独（≥70 线索），81 位字符串，'0' 表示空白格
  + system_prompt：要求 <think>按坐标锁定</think><answer>81位</answer> 格式
         ↓

[SFT 锚定层]（已完成，GRPO 之前）
  监督学习固化格式范式 → 初始策略有 >3/5 的 Pass@1
  作用：降低 GRPO 探索空间的随机性，提高有效轨迹密度
         ↓

[GRPO 推理层]（每条轨迹内部）
  Step 1: <think> 标签开启 → 推理过程外化开始
  Step 2: 模型按坐标逐格推理（隐含多维约束检查）
           → 行约束：这行已有 X，不能填 X
           → 列约束：这列已有 Y，不能填 Y
           → 宫约束：这宫已有 Z，不能填 Z
  Step 3: </think> 标签闭合 → 推理过程终止，完整轨迹形成
  Step 4: <answer> 标签开启 → 开始输出 81 位答案
  Step 5: </answer> 标签闭合 → 完整输出轨迹
         ↓

[奖励计算层]（7 个函数并行）
  格式验证：soft_format（0~0.6）— 验证推理轨迹的完整性（闭合权重 2x）
  反作弊验证：clue_preservation（-2~1.0）— 验证答案不是复制或篡改
  过程验证：row/col/block（0~3.0）— 验证答案满足三维空间约束（PRoSFI 思路）

  推理长度：brevity（0~-0.5）— 裁剪过度推理

  问题：推理结构信息在 advantage 计算中全部丢失——
  奖励是标量，不携带"哪一步推理出了问题"的位置信息（CRV 揭示的根本局限）
         ↓

[输出层]
  参数更新后的模型 → 更高概率生成完整格式 + 约束满足的答案
```

## 十一、CoT 改进方向：三篇论文直接给出的路线图

**路线 A（PRoSFI 方向）：引入结构化中间步骤**

把 `<think>` 块的格式从纯自然语言升级为"自然语言 + 结构化标注"：

```
<think>
[Step 1] 坐标(1,3): 行约束: {1,2,4} 已用; 列约束: {3,5} 已用; 宫约束: {1,2,3,4,5} 已用
         → 候选值: {6,7,8,9} - {3,5} = {6,7,8,9}
         → 唯一候选: 无（需结合其他约束）
[Step 2] ...
</think>
```

然后用规则引擎验证每一个 [Step] 块的约束计算是否正确，奖励只给全部步骤验证通过的轨迹。代价：训练数据需要预处理，模型需要额外的 SFT 来学习新格式。

**路线 B（PR-CoT 方向）：多角度反思的显式奖励**

在现有 `<think>` 之后，额外引入一个 `<verify>` 标签，要求模型在给出答案前做一次反向验证：

```
<verify>
行检查：第 3 行 [5,?,1,2,?,?,4,?,?] 中填入的 6 和 8，与现有数字不重复 ✓
列检查：第 2 列 [?,?,?,5,?,?,?,?,?] 中填入的 6，与现有数字不重复 ✓
</verify>
```

对 `<verify>` 块的内容给予额外奖励，奖励"验证结论是否与实际约束一致"。这把 PR-CoT 的多维反思从 prompt 层面推进到了训练层面。

**路线 C（CRV 方向）：推理轨迹的内部一致性监控）**

在 `LogGenerationCallback` 里增加一个分析维度：提取 `<think>` 中提到的坐标和填入数字，验证这些坐标对应的位置在最终 `<answer>` 里是否确实填了模型声称要填的数字。

```python
# 伪代码：验证 think 与 answer 的因果一致性
think_assignments = extract_coordinate_assignments(think_block)
for coord, value in think_assignments.items():
    if answer[coord] != value:
        causal_consistency_reward -= 0.1  # 推理和答案不一致，扣分
```

这不需要 CRV 的神经回路分析，但实现了类似的"推理过程与最终答案因果关联验证"，填补当前系统最大的盲点。
