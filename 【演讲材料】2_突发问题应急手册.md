# 演讲应急手册 - 被问到代码/技术细节时的应对

> **场景**：评委或观众问到关于代码、技术实现的问题，但你（演讲人）不需要完全理解代码。这份文件告诉你如何应对。

---

## 第一部分：如果被问技术问题的通用应答框架

### 框架 1: "这是开发团队的工作"

**情景**：评委问 "你用什么框架做的？"

**应答**：
```
"我负责系统的使用和演示部分。这个项目由【开发同学名字】负责代码实现。
他用了LangGraph框架来构建多智能体系统。如果你对技术细节感兴趣，
我们课后可以约时间让他给你讲解。"
```

**好处**：
- 清晰划分职责
- 显得专业（知道谁负责什么）
- 把技术问题合理地转移给开发人员

---

### 框架 2: "从用户价值的角度解释"

**情景**：评委问 "为什么要用多Agent架构？"

**应答不好的方式**：
❌ "因为...这样可以...降低Token成本...通过...状态管理..."（很技术）

**应答好的方式**：
✅ "我们没有选择传统的单体AI，而是设计了5个专业角色。
   这样做的好处是：
   1. 出错时能快速定位原因（哪个环节出问题）
   2. 质检更严格（Reviewer专门检查数据）
   3. 每个角色专注自己的工作，做得更好
   类比：一个医院，分科室比一个全科医生更靠谱。"
```

**关键**：从"用户能感受到的好处"而不是"技术实现"来解释

---

### 框架 3: "数据支撑你的说法"

**情景**：评委问 "系统真的能工作吗？"

**应答**：
```
"有具体的数据支持：
- 44个真实任务的验证
- 最终成功率 86% 以上
- 一次查询可以处理 100 支股票的数据
- 平均执行时间 10 分钟（对比人工 2-3 小时）

这些都不是我们自己说的，而是通过真实执行记录验证出来的。"
```

**好处**：
- 显得有科学依据
- 证明系统不是纸上谈兵
- 防止被追问"证据在哪"

## 第二部分：常见技术问题及应答

### Q1: 用的什么AI模型？

**问法**：你们用的是GPT、Claude还是...?

**应答**：
```
"我们主要用的是国内的大语言模型（阿里的Qwen系列）。
选择国内模型的原因：
1. API稳定性高（不受跨境网络影响）
2. 成本更优（对标国际模型便宜 30-40%）
3. 中文理解能力更强（我们处理的是中文金融数据）

具体怎么集成的，这个是开发同学的工作。"
```

**备注**：如果演讲人不确定用的是哪个模型，看一下下面的提供信息

---

### Q2: 系统是云端运行还是本地运行？

**问法**：这个系统部署在哪里？需要什么硬件？

**应答**：
```
"目前主要是开发和演示阶段，部署在开发者的本地机器上。
如果要商业化，可以有两种方案：

方案1：云端部署（推荐）
- 优点：用户不需要装软件，直接网页访问
- 缺点：需要服务器成本

方案2：本地部署
- 优点：数据不出去（金融机构喜欢）
- 缺点：需要企业自己维护

我们当前主要做的是功能验证，部署方案由后续的产品团队决定。"
```

---

### Q3: 为什么系统有时候会出错？

**问法**：我看你的演示有失败的情况，是不是系统有问题？

**应答**：
```
"出错很正常，关键是怎么修复。我们的系统有自动故障恢复机制：

一级：代码执行出错
→ 自动改进代码，重试（最多3次）
→ 成功率 90% 以上

二级：API超时
→ 自动等待后重试（1秒、2秒、4秒递增）
→ 解决率 95% 以上

三级：数据格式不对
→ 自动转换格式
→ 基本100%能处理

四级：实在修不了
→ 换个查询策略，换个角度获取数据
→ 大部分情况也能救活

所以虽然有出错，但通过这四层机制，
最终的任务成功率还是达到 86% 的。"
```

**关键**：把"出错能修复"转化为"系统的优势"

---

### Q4: 系统能实时交易吗？能做高频交易吗？

**问法**：这个系统能用来做自动交易吗？

**应答**：
```
"不能。我们的系统是为日级别的研究分析设计的，不是交易系统。

原因有三个：
1. 延迟太高：API调用、AI分析、数据验证这些步骤要 10 分钟
   高频交易需要毫秒级反应
   
2. 设计目标不同：我们关注的是"是否值得持有这只股票 3 个月"
   而不是"今天涨跌多少"
   
3. 风险控制：日级别的分析有时间审核；
   高频交易每秒几百笔，没人能审核

如果要做自动交易，需要完全不同的系统架构。
我们这个系统的定位是'初级分析师的替代品'，
而不是'交易员的替代品'。"
```

---

### Q5: 用户隐私怎么保护？数据安全吗？

**问法**：用户的投资偏好数据会不会泄露？

**应答**：
```
"这是个很重要的问题。当前阶段，系统还在演示和功能验证，
没有真实的用户数据接入。

如果真的要上线，我们会考虑以下方案：

方案1：本地部署
- 用户的数据完全不出去
- 金融机构、基金公司很喜欢这个方案
- 但需要机构自己维护服务器

方案2：云端部署 + 加密
- 用户数据加密存储
- 定期第三方安全审计
- 但企业负责风险

方案3：不存储用户数据
- 每次查询后立即删除
- 隐私最强
- 但用户下次用不了历史记录

现在这阶段，我们重点是验证功能，
数据安全是后续产品化才要重点考虑的事。"
```

---

### Q6: 成本怎么算？需要多少钱？

**问法**：这个系统用一年要花多少钱？

**应答**：
```
"主要成本来自三个方面：

1. API成本（Tushare数据接口）
   - 免费版：无限访问，但有频率限制
   - 付费版：¥5000/年，无限制
   - 一家券商年成本几万块

2. LLM成本（大语言模型调用）
   - 国内模型便宜，大概 ¥0.0001-0.0005 每次查询
   - 一个分析师一年处理 10000 次查询，成本 ¥1-5 块
   - 对比：一个分析师年薪 30-50 万

3. 服务器成本
   - 本地运行：0 块（用自己电脑）
   - 云端运行：¥100-500/月

综合算下来，一家规模 50 人的券商研究部，
用这个系统替代 10 个初级分析师，一年成本约 5-10 万，
vs 人工成本 300 万，节省 97%。"
```

**关键**：用对比法显示经济价值

## 第三部分：如果有人挑战你的项目

### 挑战 1: "44个任务验证，这个数字怎么来的？"

**不好的应答**：
❌ "啊...这个...我得问问开发同学..."（显得不专业）

**好的应答**：
✅ "这个数字来自真实的系统执行日志。
   我们跑了 44 个不同的金融数据查询任务，
   记录了完整的执行过程。
   
   最后统计结果是：
   - 直接成功：30 个（68%）
   - 通过故障修复成功：8 个（18%）
   - 最终失败：6 个（14%）
   - **最终成功率 86%**
   
   这不是预期值，而是真实验证数据。
   详细的执行日志和统计分析，
   可以在【开发文档】中查看。"
```

**好处**：
- 有具体数字支撑
- 承认了失败的部分（显得诚实）
- 指向文档而不是自己编造

---

### 挑战 2: "为什么选择 Tushare？为什么不用 Wind、东方财富？"

**应答**：
```
"很好的问题。我们选择 Tushare 的原因：

1. 数据完整性：A股、港股、美股都有，
   而且包含很多细节数据（如分红记录）
   
2. API 稳定性：Tushare 专业提供数据接口，
   不会因为网页改版而改变数据格式
   （Wind 和东方财富主要是信息平台，不是数据库）
   
3. 成本便宜：Tushare 付费 5000/年，
   Wind 券商版 5-10 万/年
   
4. 开发者友好：有详细 API 文档和开源工具集

当然，如果客户已经有 Wind 的服务，
我们的系统也可以改成接入 Wind 的数据。
这是接口适配问题，技术上没有难度。"
```

---

### 挑战 3: "系统能保证不幻觉吗？"

**应答**：
```
"'幻觉'在 AI 领域是个经典问题。我们的解决方案是：

**不追求 0 幻觉，而是可检测和可修复的幻觉。**

具体做法：

1. 数据驱动的幻觉很少
   → 我们的数据直接来自 API，不是 AI 生成
   → 这部分幻觉率接近 0

2. 分析逻辑的幻觉可能有
   → Reviewer 会检查逻辑是否合理
   → 如果 PE 算出来是负数或 99999，会自动标记为异常
   
3. 重要结论都要人工审核
   → 虽然 AI 生成了报告，但人工投资决策前必须再看一遍
   → 这是金融行业的标准做法

所以我们的定位不是'完全自动化'，
而是'自动化初级分析，专业人士负责最终决策'。"
```

## 第四部分：演讲演砸了的应急预案

### 情景 1: 系统现场演示崩溃了

**不要说**：
❌ "抱歉，有点问题...系统出bug了..."

**要说**：
✅ "现场的网络环境可能有影响。不过我提前录制了演示视频，
   大家可以看看系统的真实运行效果。"
   （播放提前准备好的视频）
   
   "或者我给大家讲一个真实的案例，
   展示系统最后输出的结果。"
   （展示之前执行过的结果截图）

**备心态**：互联网产品演示总是有风险，重要的是有 Plan B

---

### 情景 2: 有人问你超级技术问题

**问题**："消息链完整性验证具体怎么实现的？"

**不要乱说**：
❌ "嗯...应该是通过...LLM...来...验证...?"（瞎编）

**诚实的做法**：
✅ "这是个很深的技术问题。这部分是我开发同学负责的，
   他会在代码审查阶段给我讲解，但细节我现在不掌握。
   
   我知道的是：系统能完整记录每一步的处理过程，
   用户可以追溯'这个数据是从哪来的，怎么处理的'。
   这对金融系统很重要，因为需要可审计。
   
   如果你对技术细节感兴趣，课后我可以约【开发同学】
   给你详细讲解。"
```

**好处**：
- 诚实可信
- 显得专业（知道自己的边界）
- 没有胡乱编造
