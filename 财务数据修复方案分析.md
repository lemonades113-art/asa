## 🔧 建议1：修复现金流字段

### 问题根源

```
错误字段 ❌       正确字段 ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
c_inf_fr_operate_a    n_cashflow_act
(现金流入小计)      (经营现金流净额)
```

**为什么会出错**：
- `c_inf_fr_operate_a` = 现金流**入**小计（如销售收入、税款返还）
- `n_cashflow_act` = 现金流**净额**（流入 - 流出）
- 两者可能相差 **3-5 倍**！

**实际案例**：
```
某公司2023年：
- 经营现金流入: 500亿
- 经营现金流出: 450亿
- 经营现金净额: 50亿  ← 这才是真实的操作现金产生能力

如果误用流入小计(500亿)，会高估10倍！
```

### 修改方向

✅ **完全正确** - 应该在 CODER_SYSTEM_PROMPT 中明确说明：
1. **优先字段**：`n_cashflow_act` (经营活动现金流净额)
2. **禁止使用**：`c_inf_fr_operate_a` (现金流入小计)
3. **备选字段**：如果上述不可用，尝试 `operate_cash_flow` 或 `operating_cash_flow`
4. **验证机制**：输出时必须注明数据来源和字段名

### 风险等级

🔴 **P0 紧急** - 这是致命Bug级别
- 影响：直接导致现金流分析结果偏大 3-5 倍
- 后果：给出完全错误的财务结论
- 修复难度：极低（仅需更新 Prompt）

## 🔧 建议2：修复年份筛选

### 问题根源

```python
# ❌ 原始错误做法
df = df[df['report_type'] == 1]  # 只取年报？

# 问题：Tushare 在某些月份可能：
# - 返回多个 report_type=1 的记录（同年多次更新）
# - 返回错误的 report_type 值（数据滞后）
# - 混入季报或中报数据
```

**真实案例**：
```
某股票2023年数据可能有3条记录：
1. end_date=20231231, report_type=1（正确的年报）
2. end_date=20231231, report_type=1（4月更新版，数据修正）
3. end_date=20231231, report_type=4（误读为中报？）

直接取均值或平均会导致数据混乱！
```

### 修改方向

✅ **完全正确** - 应该改为：
```python
# ✅ 正确做法
# 步骤1：精确定位年报（按 end_date 结尾 1231）
df = df[df['end_date'].str.endswith('1231')]

# 步骤2：去重（同一个 end_date 只保留最新更新）
df = df.drop_duplicates(subset=['end_date'], keep='last')

# 步骤3：排序（确保时间顺序）
df = df.sort_values('end_date')
```

**为什么这样做**：
- 按 `end_date` 末尾筛选比 `report_type` 更可靠（日期不会骗人）
- `drop_duplicates(keep='last')` 保留最新更新版本
- 规避了 API 版本差异和数据延迟问题

### 风险等级

🔴 **P0 紧急** - 数据重复或混淆级别
- 影响：可能在同一年混入多个报告期的数据
- 后果：年度统计数据不准确
- 修复难度：极低（只需几行代码）

## 🔧 建议3：修复资产负债表（Balance Sheet）

### 问题根源

```python
# ❌ 问题
# 1. 直接查询 balancesheet 时 fields 参数可能不全
# 2. 不同表的 end_date 格式不统一（有些是 str，有些是 int）
# 3. 不同表的 end_date 可能错开（income 有数据，balance 没数据）
# 4. 应收账款字段名在不同版本中可能是 ar（应收账款）或 accounts_receivable

df_balance = pro.balance_sheet(ts_code=code, fields='ar,liabilities')
# 这样查可能：
# - ar 字段不存在
# - 应收账款用的是其他字段名（如 note_receivable, bill_receivable）
```

### 修改方向

✅ **完全正确且高级** - 应该采用"分开查询 + 按日期 Merge"的方案：

```python
# 步骤1：查询 income（收入表）
df_income = pro.income(ts_code=code, start_date='20200101')
df_income['end_date'] = pd.to_datetime(df_income['end_date'], format='%Y%m%d')
df_income = df_income[df_income['end_date'].str.endswith('1231')]

# 步骤2：查询 cashflow（现金流表）
df_cashflow = pro.cashflow(ts_code=code, start_date='20200101')
df_cashflow['end_date'] = pd.to_datetime(df_cashflow['end_date'], format='%Y%m%d')
df_cashflow = df_cashflow[df_cashflow['end_date'].str.endswith('1231')]

# 步骤3：查询 balance_sheet（资产负债表）
df_balance = pro.balance_sheet(ts_code=code, start_date='20200101')
df_balance['end_date'] = pd.to_datetime(df_balance['end_date'], format='%Y%m%d')
df_balance = df_balance[df_balance['end_date'].str.endswith('1231')]

# 步骤4：按日期 Merge（确保数据对齐）
df_merged = df_income.merge(df_cashflow, on='end_date', how='inner')
df_merged = df_merged.merge(df_balance, on='end_date', how='inner')

# 结果：同一个 end_date 的数据完整对齐
# 如果某个表缺数据，merge(how='inner') 会自动排除
```

**为什么这样做**：
- 各表独立查询避免字段冲突
- `merge(on='end_date')` 确保数据时间对齐
- `how='inner'` 自动过滤不完整数据
- 避免字段名冲突（多表有相同列名时）

### 风险等级

🟡 **P1 重要** - 数据对齐问题
- 影响：可能查不到应收账款或混淆数据来源
- 后果：财务比率计算不准确
- 修复难度：中等（需要修改数据获取逻辑）

## 🔧 建议4：增速分析 + 净利润 + 毛利率

### 问题4.1：区分"单季"与"累计"增速

**现象**：
```
2024Q3 营收同比增长 5%
  ↑ 这是"累计"增速 vs "单季"增速？

如果是：
- 累计: 2024年1-9月 vs 2023年1-9月的对比 ← 趋势较稳定
- 单季: 2024年7-9月 vs 2023年7-9月的对比 ← 波动更大
```

**为什么会混淆**：
```python
# Tushare income() 返回的数据：
# report_type=1: 年报（全年累计）
# report_type=2: 一季报（1-3月累计）
# report_type=3: 中报（1-6月累计）
# report_type=4: 三季报（1-9月累计）

# 所有这些都是"累计"，不是"单季"！
# 要算单季，需要：单季 = 本期累计 - 上期累计
```

### 修改方向

✅ **完全正确** - 应该在 Prompt 中明确说明：

```markdown
【财务增速计算规范】

1. 累计同比增速（YoY - Year-over-Year）
   - 定义：本期累计 / 去年同期累计 - 1
   - 优先使用 Tushare 的 yoy_growth 字段（如果可用）
   - 如果没有，手动计算：(本期revenue - 去年revenue) / 去年revenue
   - 示例：2024年Q3累计同比 = (2024年1-9月 revenue) / (2023年1-9月 revenue) - 1

2. 单季同比增速（QoQ - Quarter-over-Quarter）
   - 定义：本季 / 去年同季 - 1
   - 计算步骤：
     a) 先算本季数值：本季 = 本期累计 - 上期累计
     b) 再算去年同季：去年同季 = 去年累计 - 去年上期累计
     c) 最后：(本季 - 去年同季) / 去年同季
   - 示例：2024年Q3单季同比 = (Q3累计数 - Q2累计数) / (去年Q3累计数 - 去年Q2累计数) - 1

3. 优先级
   - 优先报告累计同比增速（更稳定，更有参考意义）
   - 附加单季增速（用于观察加速或减速趋势）
```

### 问题4.2：引入净利润 + 毛利率

**现象**："增收不增利"陷阱
```
案例：某公司2023-2024
- 营收增长：+20%（看起来很好）
- 净利润增长：-5%（糟糕！）
- 毛利率：从35% → 28%（成本压力！）

结论：虽然销量增长，但利润反而下降了。
可能原因：原材料成本上升，或竞争加剧导致价格下降。
```

### 修改方向

✅ **高价值建议** - Coder 需要同时输出三个维度：

```python
# 需要 Coder 查询的字段
fields_needed = [
    'revenue',              # 营收（Top Line）
    'n_income_attr_p',     # 归母净利润（Bottom Line，最重要！）
    'gross_profit',        # 毛利（可选，某些API不提供）
    'grossprofit_margin',  # 毛利率（%)
    'net_profit_margin'    # 净利率（%)
]

# 在 Reviewer 报告中应该说
analysis = {
    "revenue_growth": "20%（Top Line增长）",
    "net_profit_growth": "-5%（Bottom Line下降 ⚠️）",
    "gross_margin_change": "35% → 28%（毛利率压缩 ⚠️）",
    "assessment": "增收不增利，反映成本压力和定价权下降"
}
```

### 关于复权问题

**建议中提到**：
> "Prompt 指导：在查询时要注明是否复权（adj='qfq'）"

**分析**：
- `adj='qfq'` 是股价查询时的参数（日线数据 pro.daily()）
- 财务报表（income, balance_sheet 等）**不需要复权**（因为本身是财务数据，不是股价）
- ✅ **建议的意图对**（强调数据规范化），但 ❌ **具体适用场景不对**

**修正建议**：
```python
# ✅ 需要复权的场景：查询历史股价、涨跌幅
df_daily = pro.daily(ts_code=code, adj='qfq')  # 前复权

# ❌ 不需要复权的场景：财务报表、分红数据
df_income = pro.income(ts_code=code)  # 无需 adj 参数
df_dividend = pro.dividend(ts_code=code)  # 无需 adj 参数
```

### 风险等级

🟡 **P1 重要** - 分析深度提升
- 影响：缺少净利润和毛利率，分析不够深入
- 后果：无法识别"增收不增利"的风险
- 修复难度：低（只需扩展字段查询）
