### 3. **补充关键机制说明**

#### 3.1 元数据标签的完整性 ✅
```
v2.0: [DATE][SOURCE][META]
v2.1: [DATE][SOURCE][META][DATA] ✨ + [TIME_RANGE] + [TIME_CALC]

关键：Reviewer在L1130-1140行强制查找[DATA]标记
     没有[DATA] = 执行失败 = 触发重规划
```

#### 3.2 网络错误的独立处理 ✅
```
v2.0: 混在"重试"里
v2.1: network_retry_count 与 retry_count 完全独立
     不会相互干扰，也不会浪费code_error的重试次数

实现位置：L1450-1480
```

#### 3.3 Reviewer的消息清洁 ✅
```
v2.0: 未提
v2.1: L1119-1178的消息清洁机制
     - 删除孤儿ToolMessage（防止API 400错误）
     - 只保留：系统提示 + 首个用户问题 + 最后成功数据
     - 大幅减少token消耗
```

#### 3.4 ProfileUpdater的保守学习 ✅
```
v2.0: 可以学习
v2.1: 有明确的"禁止过度推理"规则（L1273-1287）
     - 不能因为提到"技术指标"就假设用户是高手
     - 不能因为一次查询就更新偏好
     - 必须有"一致的表现"才更新

⚠️ 限制：当前无数据库持久化（系统重启后丢失）
```

#### 3.5 ToolCache的分层策略 ✅
```
v2.0: 完全漏掉
v2.1: L1675-1735的分层缓存
     - realtime: 30秒（实时数据）
     - daily: 300秒（历史数据）
     - default: 60秒（其他）
     
     缓存键：MD5(工具名 + 参数)
     避免了"查中国平安却返回宁德时代"的bug
```

#### 3.6 分层模型成本优化 ✅
```
v2.0: 未提
v2.1: Strong vs Fast模型分层（L1888-1889）
     
     Strong (深思熟虑): deepseek-v3 / gpt-4o
     - Supervisor: 任务分解（需要复杂逻辑）
     - Coder: 代码执行（需要精准）
     - Reviewer: 报告撰写（需要文采）
     
     Fast (轻量级): gpt-4o-mini
     - ErrorHandler: 错误分类（相对简单）
     - ProfileUpdater: 用户画像（相对简单）
     
     效果：相比全用Strong，节省30-40% token成本
```

## ⚠️ 新增的系统限制说明

### v2.1新增的真实限制

| 限制 | 说明 | 解决方案 |
|------|------|---------|
| 无持久化 | ProfileUpdater的学习在系统重启后丢失 | 补充数据库连接 |
| 无Token超限处理 | 长对话可能超过模型限制 | 补充token计算模块 |
| 并发不安全 | ToolCache和state无线程锁 | 高并发时添加threading.Lock |
| 速率限制缺失 | 429错误处理不明确 | 补充等待队列机制 |
