# 智能金融数据分析多Agent系统（ASA v2.0）- 项目简历

## 项目概述
设计并实现了基于LangGraph的**Supervisor多Agent架构**，集成动态技能加载、自适应错误处理、金融逻辑审查的金融数据分析系统。面向股息率、分红、送转等复杂金融指标的语义理解与数据查询。

---

## 核心技术方案

### 1. 多Agent编排框架（LangGraph Supervisor Pattern）
**文件**：`multi_agent.py`

**架构设计**：
- **Supervisor节点**：中心化路由器，根据用户intent分配任务给专用Agent
- **Coder节点**：代码生成与API调用，集成动态技能注入
- **Reviewer节点**：金融逻辑审查，识别数据异常（如"大蓝筹零分红"悖论）
- **ErrorHandler节点**：4级Backtracking策略（优先级降级）
- **ProfileUpdater节点**：用户画像动态更新

**实现细节**：
```
执行流程：
用户输入 → Supervisor(intent解析) 
  → Coder(技能注入 + API调用)
    → Tools(Tushare API)
      → ErrorHandler(失败重试)
        → Supervisor(重新分配)
          → Reviewer(逻辑审查)
            → ProfileUpdater(画像更新)
              → 输出
```

**验证证据**：
- 测试日志（batch_run_20260130_151419.jsonl）第1-26步完整展示了该流程
- Supervisor成功路由Coder执行任务（Step 1-2）

---

### 2. 动态技能加载机制（Skill-based Architecture）
**文件**：`skills.json`、`multi_agent.py`（Coder节点）

**实现方案**：
- **5类技能库**：dividend_expert（分红）、market_expert（港股市场）、valuation_expert（估值）、risk_expert（风险）、accounting_expert（财务）
- **动态注入**：Coder根据用户query的关键词自动load_skill()，获取领域特定的API调用模板与数据处理规则
- **避免硬编码**：技能指导规则动态插入system prompt，每次查询可适配不同分析场景

**关键代码**：
```python
# Coder工具调用
Tool Call: load_skill("dividend_expert")
Response: "[技能注入：分红专家]\n1. 字段冲突提示：Tushare API中的'dv_ttm'是百分比数值...\n2. 单位转换：若原始为小数则×100..."
```

**验证证据**：
- 测试Step 2：`{'name': 'load_skill', 'args': {'skill_name': 'dividend_expert'}}`
- Step 3：Tools返回技能指导内容，Coder据此调用API
- **实际有效**：技能注入后Coder知道如何处理dv_ttm单位转换问题

### 3. 模型分层推理策略
**文件**：`lib.py`（第66-107行：get_chat_model()）

**分层设计**：
| Agent | 模型 | 理由 |
|------|------|------|
| Supervisor | qwen-plus(Smart) | 强逻辑推理，决策关键 |
| Coder | qwen-plus(Smart) | API选择、代码生成精度高 |
| Reviewer | qwen-plus(Smart) | 金融逻辑判断复杂 |
| ErrorHandler | qwen-turbo(Fast) | 轻量级重试逻辑 |
| ProfileUpdater | qwen-turbo(Fast) | 向量更新，计算轻 |

**成本优化**：Fast模型用于非关键路径，预计降低Token消耗15-20%

---

### 4. 金融逻辑审查引擎（Reviewer）
**文件**：`multi_agent.py`（第1345-1496行：REVIEWER_SYSTEM_PROMPT）

**审查规则**：
1. **数据一致性检查**：确保币种、时间周期统一
2. **悖论识别**：检测"大蓝筹零分红"等异常
3. **指标有效性**：验证PE、PB、股息率的合理范围
4. **时间逻辑**：确认分红数据来自最新财报

**实际验证**：
- 测试Step 26：Reviewer输出 `⚠️ **数据异常警告**:系统检测到XXX分红数据存在逻辑悖论...`
- **证明Reviewer确实在工作**，能识别不合理的数据组合

---

### 5. 记忆系统（分层记忆 + 因果图）
**文件**：`memory_system.py`

**三层记忆架构**：
- **短期记忆**：当前对话上下文（容量有限）
- **长期记忆**：ChromaDB向量存储，保存历史查询与结果（测试显示74条记忆已加载）
- **因果图**：追踪"问题→工具选择→结果"的依赖关系，用于RCA（根因分析）

**测试数据**：
- `[LongTermMemory] 加载 74 条记忆`
- `[MemorySystem] 初始化完成 - 短期: 0, 长期: 74, 因果图: 已激活`

---

### 6. 自适应错误处理与恢复
**文件**：`orchestrator.py`

**错误处理流程**：
1. **Coder生成代码** → 调用API
2. **API返回错误** → ErrorHandler捕获
3. **分类错误**：
   - 网络错误 → 重试（max 3次）
   - 认证错误 → 更新Token
   - 数据错误 → 降级策略（更换API、放宽过滤条件）
4. **4级Backtracking**：
   - L1：调整API参数
   - L2：切换API接口
   - L3：使用备选数据源
   - L4：返回部分结果 + 人工提示

**已知限制**：
- ⚠️ **递归循环未完全解决**：当API持续无数据时，Coder→ErrorHandler→Supervisor→Coder循环可能超限（测试中25步后超限）
- 需要状态序列审计，检测"连续3次Coder→ErrorHandler"组合自动熔断

## 已知问题与改进方向

### 问题1：递归循环控制不足 ⚠️
**现象**：当Tushare API返回空数据时，系统进入Coder→ErrorHandler→Supervisor→Coder循环，25步后超限

**根本原因**：
- ErrorHandler缺少"状态序列审计"，无法检测循环模式
- 没有"连续N次无数据→自动降级"的熔断策略

**改进方案**：
```python
# 伪代码
class ErrorHandler:
    def check_loop_pattern(self, last_k_steps):
        # 检查最近5步是否形成"Coder→ErrorHandler"循环
        if all(step.node in ["Coder", "ErrorHandler"] for step in last_k_steps[-5:]):
            return self.fallback_strategy()  # 触发降级
        return self.retry()
```

### 问题2：Tushare API数据真空 ⚠️
**现象**：某些股票的分红数据为空（可能因为上市时间短或无分红）

**当前处理**：重试无效，需人工介入

**建议方案**：
- 调用其他数据源（Wind、同花顺）作为备选
- 返回"部分结果"（如有其他指标可用）
- 主动提示用户"该股票暂无分红数据"

### 问题3：OpenSandbox未实际集成 ⚠️
**现象**：`lib.py`第32-38行try-import OpenSandbox，但OPENSANDBOX_AVAILABLE=False

**影响**：代码执行仍使用本地StatefulPythonKernel，无沙箱隔离

**改进**：若需要沙箱隔离，需完成OpenSandbox SDK集成与权限配置

## 架构优势

### 1. 可扩展的多Agent设计
- 新增Agent只需实现system_prompt + tools
- Supervisor路由规则配置化（可外部配置）

### 2. 动态技能库
- 5类领域知识即插即用
- 避免prompt写死，支持运行时更新

### 3. 分层推理+分层模型
- 关键路径用强模型，非关键路径用快模型
- 预计Token消耗降低15-20%

### 4. 金融特化的逻辑审查
- 不是通用NLP模型，而是金融领域特化的规则引擎
- 能识别"股息率>100%"等明显异常

### 5. 完整的可观测性
- 轨迹记录、因果图追踪、错误分类
- 支持事后分析与模型优化

## 总结

ASA v2.0是一个**功能特化的多Agent金融分析系统**，核心优势是：
1. ✅ 动态技能注入（避免prompt hardcode）
2. ✅ 分层推理（降低成本）
3. ✅ 金融逻辑审查（不只是生成，还要检查）
4. ✅ 完整的可观测性（轨迹+因果图）

当前限制是**递归循环控制**，建议的改进路线是：
1. 实现状态序列审计（检测循环）
2. 完善数据真空的降级策略
3. 集成OpenSandbox沙箱隔离（可选）

该项目可作为**金融领域垂直应用的技术参考**，展示了如何用多Agent架构处理复杂的领域推理问题。
